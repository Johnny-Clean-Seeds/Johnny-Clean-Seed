## Repeating Pattern Families

### 1. Rule Stored But Not Fired

Evidence:

- broad command list says the assistant should not freestyle;
- recursive mirror rule says a rule is alive only when it mirrors through the working body;
- proof-in-pudding rule says dead notes do not count.

Failure shape:

The rule exists, but the action begins without the fired gate.

Fix worn here:

The broad command list was used as the active method before writing this packet.

### 2. Task Success Masking Lifecycle Failure

Evidence:

- root issue diagnosis names lifecycle failure as the major root;
- used-runner rule says a runner is not finished until its mail is closed.

Failure shape:

The rule gets saved or the script gets used, but porch cleanup, runner archive, and receipt are left behind.

Fix worn here:

The loose scripts were moved into a runner custody lane and marked historical.

### 3. Duplicate / Loose Source Copy

Evidence:

- root broad command source and canonical BRAIN_LEARNING copy had identical hash.

Failure shape:

Keeping both loose would create duplicate material and porch clutter.

Fix worn here:

The root copy was moved to processed rule intake instead of rewritten or deleted.

### 5. Shed / Bridge Boundary Pressure

### 6. Notes Piling Up Instead Of Finals

Evidence:

- user asked for note-burning logic;
- broad command list demands ordered output and self-proof.

Failure shape:

Gathered notes stay scattered after the task is done.

Fix worn here:

This packet condenses working notes into ledgers, a final rule card, a boss order, a tabbed list, and a receipt.

## Root Cause Judgment

The house already knows the pattern. The recurring failure is live carry-through:

`source -> fit -> lane -> lifecycle -> proof -> receipt -> Git/return`

The fix is not another giant doctrine file. It is smaller live latches used before and after action.
