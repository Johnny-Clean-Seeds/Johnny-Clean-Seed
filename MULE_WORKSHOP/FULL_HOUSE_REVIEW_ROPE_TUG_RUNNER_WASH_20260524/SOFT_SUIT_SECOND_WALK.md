## What Changed When Wearing The List

The first list got quieter.

The big feeling at first was "there are many issues." With the soft suit on, many became the same
few lifecycle failures:

- notes need disposition;
- fog needs a tug;
- release needs a runner trace;
- repeated review needs a washer habit;
- sorting, mapping, and scheduling need one shared route.

## Second-Walk Findings

| # | Finding | Why it changed |
|---|---|---|
| 1 | The note burn rule should be dedicated. | If the burn rule stays buried in a broad washup, future work can miss it. |
| 2 | Tug should fire before cut. | A cut without a tug can release the wrong thing or lose the root. |
| 3 | Cut should mean active release, not deletion. | The house already says nothing is trash; clean cutting needs custody and trace. |
| 4 | Runner trace can be small. | A giant trace file for every note would become new weight. A packet/receipt/template line is enough when it names root, relation, destination, and return trigger. |
| 5 | "All should link to all" should become controlled relations. | Full graph thinking is too heavy now. Relation words are enough: source, checkpoint, proof, neighbor, parking, blocker, promotion candidate, receipt, custody. |
| 6 | Scheduling should be event-based first. | Automation would be premature. Triggers like root drop, dirty tree, creative pass, final lock, or repeated correction are safer. |
| 7 | Best outside methods agree on small WIP. | Kanban and the house suit both say too many active objects harms flow. |
| 8 | Path/home truth still tugs, but it is not this boss. | It is real fog, but moving path labels during this pass would widen scope. |
| 9 | Repo root tracked artifacts may deserve review later. | They are not parent-drop files and not dirty. Moving tracked root files now would be a different job. |
