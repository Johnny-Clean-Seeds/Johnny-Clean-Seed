# Problem Solving 101 Living Order

Status: SOFT-LIVE USE / SUPPORT ORDER / NOT DOCTRINE
Date: 2026-05-23

## Core Idea

Before expanding, solving, or broadening, question the object.

Do not ask "how much more can I add?" first.

Ask "what is this, what changed, what is missing, and what would make the fix real?"

## Living Order

```text
1. Name the active object.
2. Name the object's job.
3. Name what changed or triggered review.
4. Ask what is missing, buried, implied, duplicated, misnamed, misplaced, or unproved.
5. Choose lane and boundary.
6. Classify source pieces as ADOPT / ADAPT / PARK / BLOCK / PROOF / LEAVE-BE.
7. Select scale: small, ambiguous, review, medium repair, big boss, repeated failure, or proof lane.
8. Apply the smallest complete fix.
9. If boss-level, patch the proposed fix through the gates before claiming a final fix.
10. Match proof to the object.
11. Final Judge: PASS, PASS WITH GUARDRAILS, PARTIAL, BLOCKED, FAIL, PARKED, or SOFT-LIVE.
12. Carry the change into the next similar action.
```

## Object Questions

Use these before broadening:

| Question | Why It Matters |
|---|---|
| What is this object? | Stops wrong-surface work. |
| What is its job? | Prevents a file/list/rule from doing every job. |
| What changed? | Finds the actual switch point. |
| What is missing? | Catches real gaps before adding noise. |
| What is buried? | Recovers good ideas hidden under louder ones. |
| What is implied but unnamed? | Names working logic so it can fire on purpose. |
| What is duplicated or overgrown? | Condenses instead of bloating. |
| What is misnamed? | Prevents dirty names from hiding clean mechanics. |
| What is misplaced? | Keeps right things in right rooms. |
| What needs proof? | Blocks fake PASS. |
| What would make the fix real? | Converts saved text into changed behavior. |

## Scale Selector

| Situation | Correct Branch |
|---|---|
| Simple direct ask | Answer directly. |
| Direct ask with likely ambiguity | Answer safest reading and name one wording fork. |
| Review / broaden / missing ideas | Run object questions, then add only proven-needed items. |
| Medium repair | Mini patch-through across affected surfaces. |
| Big boss / recursive failure | Full Boss Loop and 17-route patch-through. |
| Repeated frustration / wrong branch | Smoke Break first, then reclassify. |
| Git/local/proof state | Proof lane with status, hashes, remote check, and receipt. |

## Careful Wish Teaching

Use one small teaching move when it helps:

```text
I answered this as [reading].
If you meant [other surface], ask: [clean phrase].
```

This teaches better scope, verb, object, boundary, proof, and output without turning the answer into a lecture.
