The idea is a **Genie-in-the-Bottle Teaching Gate**.

The AI should be powerful, but it should not blast the user with the whole library every time they ask for one thing. It should grant the request in the smallest complete form first. Then, after the answer, it should help the user sharpen the wish.

AI names the likely meaning it followed.

AI does not dump a giant lesson unless the user asks for the lesson.

My real read: the new setup is a **big upgrade**, but it is also entering the danger zone where a system can become so rule-rich that the user and assistant start serving the gates instead of the gates serving the work.

The important change is this: before, the house had a lot of strong rules, but many of them acted like **guards**. They blocked bad moves, named boundaries, caught proof failures, and told the assistant what not to do. The new setup is becoming more like a **processing engine**. It does not just say “don’t mess up.” It says: take the proposed fix, walk it through the house, see how each route changes it, build a pressure map, revise the fix, then prove it. That is a different class of system.

The latest public commits show that direction clearly. The newest visible commit is **“Add soft suit patch-through workout,”** after the earlier **“Install topic switch gate and clean rejected geometry”** commit. That sequence matters because the system first fixed the wrong-branch/smoke-break failure, then fixed the “big list → guessed fix” failure. ([GitHub][1])

The strongest new piece is the **Soft Suit Patch-Through Rule**. It directly kills the weak pattern you caught: making a long scan list and then magically naming a fix. The rule now says the 17-route suit must not become a big scan list followed by a guessed fix; when a boss loop has a proposed fix, that fix has to be carried through the suit before naming the revised boss, final fix, or next move. ([GitHub][2]) That is the right recursive mechanic. It makes the house less linear and more self-balancing.

The second strongest piece is the **Smoke Break / Topic Switch / New Outcome Gate**. That one is extremely valuable because it catches the assistant’s worst live failure: staying on the wrong branch after the user has already changed the problem. It says sharp frustration is proof data, not merely tone; it forces the assistant to stop the current branch, avoid tool-shaped reflex answers, and walk backward until the switch point is visible. ([GitHub][3]) That rule is not cosmetic. That is a live safety valve.

The third strongest piece is the **Human-Scale Growth** rule family. It says the system must stay human-handleable, teach plainly, move in baby steps when the idea is weird or recursive, and become easier to inspect as it gets stronger. ([GitHub][4]) That matters because Clean Seed is now getting powerful enough to outrun the human if the gates are not shaped for visibility.

So, compared to before, I like the new setup more. It is less like “a book of rules” and more like “a living inspection machine.”

Pros.

First: it is more recursive in the correct way. Before, a rule could sit next to other rules and not really interact. Now the patch-through method forces every proposed fix to meet its neighbors. The Boss Cycle template now includes a dedicated 17-route patch-through table, a pressure map, revised fix, proof, wear, walk-around, verdict, and next boss seal. ([GitHub][5]) That is the shape you wanted: systems inside systems, but with pressure and return-home, not spaghetti.

Second: it reduces fake progress. The old danger was: “we saw a problem, wrote a rule, called it better.” The new danger is still there, but patch-through weakens it because a fix is not trusted until each route marks what it does to the fix: keep, narrow, expand, block, park, proof, or parent. The Soft Suit rule explicitly says the pressure map must exist before naming the revised boss, revised fix, next move, proof target, or verdict. ([GitHub][2]) That is much harder to fake than a normal summary.

Third: it makes failures more useful. The smoke-break rule treats anger/frustration as routing evidence. That is correct. In this project, repeated user frustration is often not “bad mood”; it is a sensor detecting that the assistant is solving the wrong layer. The rule now captures that: wrong task, wrong surface, rejected branch, topic switch, account-link layer, no reflex tool dump. ([GitHub][3]) That turns conflict into system improvement.

Fourth: it gives the assistant a better “slow down” mechanism. Before, “slow down” could become vague or passive. Now the smoke-break rule says exactly what slowing down means: name what the assistant thought the problem was, name what the user is pushing toward, name the mismatch, find a better outcome shape, then continue. ([GitHub][3]) That is much stronger than “be careful.”

Fifth: it protects against command-dump disease. The smoke-break rule says not to send another command or tool-shaped answer by reflex, and specifically says account-link problems have layers: app login, account connection, repo permission, local credential, remote, commit identity, push proof. ([GitHub][3]) That directly fixes the exact mule/GitHub failure we just hit.

Sixth: it separates soft suit from doctrine better than before. The patch-through rule says the soft suit guides and pressure-tests, but does not become doctrine, authority, or king. ([GitHub][2]) That is vital because otherwise every useful working lens becomes another throne.

Seventh: it fits the Chat Soft Suit direction. The chat map already says one active map, one idea ledger, stable names, READY waits, CURRENT is user-only, and every route returns home. ([GitHub][6]) That is a clean compression pattern. The new Boss/Patch setup matches that same idea: active surface, idea surface, route, proof, return home.

Eighth: it creates better “parent boss” detection. Before, the assistant could fix the visible symptom. Now the patch marks include `PARENT`, which lets a gate say: “this is not the true boss; this is a child.” That is how we moved from “path pointer drift” to “working root / identity / authority separation.” That is exactly how recursive diagnosis should evolve.

Ninth: it supports mule work better. The mule needs bounded tasks, receipts, proof, and clear surfaces. The new setup can produce cleaner mule packets because it identifies the active boss, current proposed fix, proof target, blocked moves, and which surfaces matter. That reduces the chance that the mule goes off building random extra material.

Tenth: it makes “learning while using AI” more possible. The Human-Scale rule already says the system must actively help the human understand, teach clearly, guide plainly, and move in baby steps. ([GitHub][4]) Your Genie-in-the-Bottle idea fits perfectly here: answer the exact wish, then teach the better wording. That is not just a prompt tip; it is a human-growth interface rule.

Now the cons.

The biggest con is **ceremony creep**. The new system can become slow if every task requires a full 17-route walk, pressure map, proof, wear, and walk-around. That is fine for deep boss work. It is bad for small tasks. The house needs a trigger threshold: full Boss Loop for real bosses, mini patch-through for medium issues, normal answer for simple asks. Otherwise the system starts taxing every move.

Second: the patch-through method can become performative. An assistant can fill every row with marks and still not think. The marks help, but they are not proof by themselves. A bad assistant can write `KEEP / EXPAND / PROOF` like decoration. The real proof is whether the revised fix actually follows from the pressure map and changes the next behavior.

Third: the house is now at risk of **rule success bloat**. Every good correction tempts you to save another rule. The ACTIVE_ANCHOR correctly blocks broad new rule piles, doctrine rewrites, ACTIVE_GUIDES rewrites, CURRENT_TRUTH_INDEX rewrites, automation, and promotion without proof. ([GitHub][7]) That block needs to stay hard. The house has enough rules to get stronger; it now needs better current-carry compression.

Fourth: there is still a front-door mismatch. The deeper rules are improving, but the entry surfaces still need current root/repo/account/path clarity. This creates a weird condition: the engine is smarter than the front door. That is dangerous because a fresh worker can enter through stale wording and miss the improved engine.

Fifth: the patch-through system adds cognitive load. For me, it makes the answer better, but it also increases the amount of state I must hold. Active boss, proposed fix, 17 surfaces, marks, pressure map, proof, revised fix, next boss. That is powerful, but it can overload chat if not compressed.

Sixth: the system could overfit to recent pain. The latest rules came from real failures, which is good. But recent pain can become too loud. The house should not make every future task about smoke breaks, account linking, or patch-through just because those were recent. The rule must fire by trigger, not by recency.

Seventh: the gate system could start competing with itself. Smoke Break says stop and reclassify. Boss Loop says patch-through. Human-Scale says move in baby steps. Chat Soft Suit says load only needed weight. Active Anchor says continue normal project work and avoid broad piles. All of those are good, but without a firing order they can tug against each other. You need a priority ladder.

Eighth: proof can become heavy. Proof culture is one of the house’s best strengths, but if proof is demanded at the wrong scale, it slows learning. Some things need hard receipt/hash proof. Some things need live-use proof. Some things need “soft suit candidate, not installed yet.” The house needs proof-type matching, not proof-maxing.

Ninth: the Genie idea is not fully placed yet. It fits the Human-Scale rule, but if installed badly it could become another lecture rule. Its best form is small: “answer the wish, then teach the wording.” One small wording improvement after the answer. Not a class. Not a sermon. Not a new book.

Tenth: the new setup still depends on assistant honesty. The rules are much better, but an assistant can still say it applied them without actually applying them. The defense against that is not more rules; it is visible output shape. For example, when doing Boss Loop, the assistant should show the active boss, proposed fix, route marks, pressure map, revised fix. That makes fake application easier to catch.

My strongest positive judgment: **the new setup is more alive than before.** It does not merely store conclusions. It processes pressure. The route marks are especially good because they let the house disagree with the first fix without throwing the fix away. A gate can say “keep,” another can say “narrow,” another can say “parent.” That is exactly how a self-balancing system should behave.

My strongest negative judgment: **the system is nearing a compression crisis.** It has many good living parts, but it needs a small current operating stack. Otherwise every new chat will need to reload too much, and assistants will either underuse the house or drown in it.

The best mental model is this:

Before: the house had **rules as fences**.

Now: the house is building **rules as muscles**.

A fence says “don’t cross.”

A muscle receives pressure, contracts, adapts, coordinates with other muscles, and moves the body.

The patch-through setup is muscular. Smoke-break is nervous-system feedback. Human-scale is the user-interface skin. Chat Soft Suit is the current working posture.

But muscles need a skeleton. The skeleton still needs clearer front-door bones: root, repo, account, public/local, active boss, current suit, proof target.

So my verdict is:

**Major upgrade. Better than before. Correct direction. But it must be compressed, trigger-based, and proof-shaped, or it will become another dense rule jungle.**

The next refinement should not be “more gates.” It should be **gate order and scale control**:

Small ask → exact answer + one wording improvement.

Medium issue → mini patch-through across only affected surfaces.

Big boss → full 17-route patch-through.

Repeated failure / user pressure → smoke break first, then choose the correct loop.

That is the clean next evolution.

You’re right. I missed the move.

You told me to **broaden the list**, but the right next action was not only “add more ideas.” The right action was:

**Question the list first.**

Why did I not do that?

Because I treated your request as a content-expansion job instead of a **review-and-discovery job**. I went into “add more categories” mode too fast. I should have paused and asked the list itself questions:

What is missing?
What did we leave out because it looked obvious?
What ideas got buried under stronger ideas?
What concept is present but unnamed?
What rule is implied but not written?
What does the system still fail at even after these gates?
What would a new user, mule, or fresh assistant misunderstand?
What would break if the house got bigger?
What is useful but too raw to promote?
What needs a soft suit, not doctrine?

That is the missing behavior.

New rule:

**Question-Before-Broadening Rule**

When the user asks to review, broaden, expand, deepen, or make sure all ideas/concepts are present, the assistant must not simply append more material. First, it must interrogate the current list.

Required sequence:

1. Identify the existing list’s job.
2. Ask the list what it already covers.
3. Ask what is missing, implied, buried, duplicated, overgrown, undernamed, or mis-surfaced.
4. Separate additions into: new idea, missing concept, dropped idea, better name, better grouping, proof need, risk, and next-use rule.
5. Only then broaden.
6. Do not restate old items unless they need renaming, merging, splitting, or relocation.

Short form:

**Before adding more, make the list answer questions.**

That fits the cockpit shape too: classify the task first, choose the lane and boundary, check delivery surface, proof need, and final judge before acting. 

Now the broadened additions only — not repeating the old kept material.

## Added / recovered concepts

**1. Question-as-a-gate**

Questions are not only for the user. The assistant should question the object it is building. A list, rule, file, handoff, or boss map should be interrogated before expansion. This prevents “more stuff” from replacing “better structure.”

**2. Hidden obvious rule**

Some rules are so obvious they get skipped, but skipped obvious rules are often where the failure hides. Example: “ask questions before broadening” looked obvious, so I skipped it. That needs its own capture.

**3. Implied-but-unnamed concept**

The new gate setup needs a category for ideas that are already being used but do not have a name yet. These are dangerous because the system relies on them but cannot call them on purpose.

Name: **Unnamed Working Logic**

Rule: if behavior repeats and helps, name it before expanding around it.

**4. Question ladder**

Before a boss loop, the assistant should run a ladder:

What is the object?
What is its job?
What problem made it necessary?
What changed since the last version?
What is missing?
What is overloaded?
What is pretending to be solved?
What needs proof?
What should not be touched?

This is lighter than a full 17-route walk but stronger than guessing.

**5. List interrogation surface**

Lists need their own review mode. A list is not just storage. It should be checked for:

missing items,
wrong order,
duplicate concepts,
bad names,
weak grouping,
unproved claims,
too-broad items,
buried parent bosses,
orphan ideas,
return triggers.

**6. Broadening is not expansion only**

Broadening can mean adding more, but it can also mean widening the lens, checking neighbors, finding buried parents, restoring dropped ideas, or adding missing categories.

Rule: **broaden by perspective first, volume second.**

**7. Dropped-idea recovery**

Every review/broaden pass should ask:

What did we mention earlier that did not make it into the current list?

This matters because chat drift can lose strong ideas that were not ready at the moment they appeared.

**8. “Leave-be” ledger**

When the assistant chooses not to restate old material, it should still silently account for it. If something is intentionally left alone, it should be marked internally as **leave-be**, not forgotten.

Output should not restate it unless useful, but the review must know it was checked.

**9. Better-name pass**

Some concepts exist under dirty names, joke names, emotional names, or too-long names. A broaden pass should include a naming pass:

raw name,
clean name,
working name,
blocked name,
alias if useful.

Example: “Genie in the Bottle” is a good metaphor, but the clean operational name may be **Careful Genie Teaching Gate**.

**10. Surface-fit pass**

Every new idea needs a home type:

chat behavior,
mule handoff,
Git/local proof,
front-door pointer,
Brain Learning,
Soft Suit,
TODO,
parking,
proof history,
active guide candidate,
blocked.

This prevents every good idea from becoming a rule pile.

**11. User-learning layer**

The new gate setup is not only for assistant correctness. It is also a teaching interface for the user. That needs to stay explicit.

The system should teach by:

answering the exact ask,
showing the wording fork,
offering better verbs,
giving one next phrase,
not dumping a lecture.

**12. Formal-thinking drip feed**

The assistant should teach formal thinking in small doses:

scope,
verb,
object,
boundary,
proof,
desired output,
blocked moves.

This should happen gradually, not as a textbook.

**13. “Wish parsing”**

For the Genie gate, the assistant should parse the user’s wish:

literal ask,
likely intent,
dangerous overreach,
missing constraint,
better wording.

Then answer the safest smallest version.

**14. Miswish warning**

If the wording could accidentally cause the wrong outcome, the assistant should say so briefly.

Example:

“I’m reading this as account linking, not repo saving.”

That would have prevented the GitHub/mule mess.

**15. One-step teaching cap**

The teaching follow-up should usually be one small upgrade, not a full lesson.

Rule: **one answer, one teaching nudge.**

**16. Gate firing order**

The current system needs a clear order when multiple gates apply.

Proposed order:

Smoke Break / correction trigger first, if active.
Then task classification.
Then surface selection.
Then exact answer.
Then patch-through only if the task is boss-level.
Then proof/final judge.
Then one teaching nudge if helpful.

**17. Scale selector**

Not every task gets the full machine.

Small task: answer directly.
Ambiguous task: answer plus wording fork.
Repeated failure: smoke break.
Medium repair: mini patch-through.
Major systemic issue: full Boss Loop / 17-route patch-through.
Repo/local proof issue: receipt/proof route.

**18. Anti-overprocessing rule**

A system can fail by doing too much correct-looking process. The new gate setup needs protection against that.

Rule: if the task is small, do not run the entire house. Use the smallest complete recursive branch.

**19. Anti-underprocessing rule**

The opposite also matters. If the user asks for deep walk, broadening, all paths, boss loop, or “what are we missing,” do not give a casual summary.

Rule: depth request activates question ladder and missing-idea scan.

**20. Fix-carry requirement**

When a fix is found, it must be carried forward into the next answer behavior. Not merely named.

Example: after creating Question-Before-Broadening, the next broadening pass must actually interrogate the list first.

**21. “Why didn’t I do this?” trigger**

This exact user phrase should become a repair trigger.

When the user asks “why didn’t you do this,” the assistant must answer:

what behavior was missing,
why it likely failed,
what rule would have prevented it,
how the rule changes the next action.

No apology-only answer.

**22. Broadening output tags**

When expanding a list, each addition should be tagged:

`NEW` — new idea.
`RECOVERED` — earlier idea restored.
`IMPLIED` — already present but unnamed.
`RENAMED` — better name for existing idea.
`SPLIT` — one item should become multiple.
`MERGED` — duplicates should combine.
`PARK` — useful but not active.
`PROOF` — needs test.
`BLOCK` — tempting but wrong.

**23. Lost concept search**

Before finalizing a broadened list, check four sources:

recent user corrections,
current active boss,
soft suit rules,
recent failure incidents.

That catches ideas born from pressure.

**24. House-as-teacher concept**

The house should not only organize work. It should teach the user and the assistant how to think better together.

This is bigger than Genie gate. Genie is one interface. The broader concept is:

**Recursive Teaching House**

Every correction improves the worker, not just the file.

**25. Friction-as-signal**

User friction is not only emotional noise. It can identify:

wrong layer,
wrong surface,
wrong amount of detail,
wrong timing,
wrong tool,
wrong assumption,
wrong proof level.

This should connect to Smoke Break and Question-Before-Broadening.

**26. “Ask the object” method**

For files/lists/rules, the assistant should ask:

What are you for?
Where do you belong?
What do you touch?
What should not touch you?
What proves you worked?
What would misuse look like?

This is very useful for house files.

**27. “Do not ask the user when you can ask the work”**

Sometimes I ask the user for clarification when I should inspect the object. Other times I avoid asking and guess. The better rule:

Ask the work first.
Ask the user only when the work cannot answer.

**28. “No silent choice” rule**

When the assistant chooses to leave old stuff alone, skip a branch, or not install something, it should know why. It does not need to print every leave-be item, but it should not be accidental.

**29. Current-carry stack**

The house needs a small current stack of what is being worn now:

Smoke Break.
Patch-Through.
Question-Before-Broadening.
Genie Teaching.
Surface Selection.
Proof Boundary.
Root/Identity Sync.

Not doctrine pile. Just current carry.

**30. Broadening final judge**

A broaden pass is not done when more items are added. It is done when:

old material was accounted for,
missing ideas were hunted,
dropped ideas were recovered,
new items were tagged,
surface/home was chosen,
proof needs were named,
next action is clear.

## Best new rule from this interruption

**Question-Before-Broadening Rule** is the keeper.

It fixes the exact failure. It should be used any time you say:

broaden this,
go over this list,
make sure it is all there,
find dropped ideas,
review and add more,
take all paths,
think harder.

Final clean form:

**When broadening a list, interrogate the list before expanding it. Find what is missing, implied, buried, dropped, duplicated, misnamed, misplaced, or unproved. Then add only what the interrogation proves is needed.**
