## 1. One-Line Definition

The **House Nervous System / Alarm-Debugger Mesh** is a routed sensing layer for Mr.Kleen that gives every critical lane local nerve endings, thresholded alarms, named debuggers, safe reflexes, proof-return paths, and memory/decay outputs, so the house can feel weak signals early, stop dirty motion before damage, repair root causes instead of symptoms, and compound clean behavior without letting alarms, rules, tools, memories, or sensors become command authority.

Shorter:

**The House Nervous System lets the house feel, route, debug, reflex, prove, remember, and decay without turning every signal into panic.**

**Sensors everywhere. Alarms only at threshold. No Alarm King.**

## 3. Research Gold From Outside the House

This section converts outside research into house-safe source ore. Outside sources are support, not authority.

### 3.1 Observability: Instrumentation Before Alarms

OpenTelemetry defines observability as understanding internal state from system outputs, usually traces, metrics, and logs. A system must be instrumented before it can be observed. For Mr.Kleen, that means: do not begin with “alarms everywhere”; begin with **nerve endings / sensors** that emit meaningful signals.

House translation:

- Sensor = local instrumentation.
- Signal = observable output.
- Trace = path through rooms/lane decisions.
- Metric = thresholdable measure.
- Log = event evidence.
- Semantic convention = shared naming so signals can correlate.

House rule:

**If the signal cannot be named consistently, it cannot be routed cleanly.**

Source alignment:
- OpenTelemetry: observability through emitted telemetry outputs; standardized semantic conventions improve correlation and consumption.

### 3.2 SRE: Symptom vs Cause

Google SRE monitoring stresses that monitoring should answer two questions:

- what is broken?
- why is it broken?

House translation:

- Alarm names the symptom.
- Debugger hunts the cause.
- Route sends the cause to the right room.
- Proof verifies the repair.

This prevents “fixing the blinking light” instead of fixing the broken wire.

House rule:

**An alarm without a debugger becomes noise.**

### 3.3 SRE Alert Quality: Precision, Recall, Detection Time, Reset Time

SRE alerting on SLOs evaluates alerts by:

- precision: are alerts significant?
- recall: do significant events get caught?
- detection time: how fast does it fire?
- reset time: how long does it keep firing after resolution?

House translation:

- Precision = does this alarm fire only when it matters?
- Recall = does this alarm catch the real failures?
- Detection time = does the house feel the issue early enough?
- Reset time = does the alarm go quiet after repair?

House rule:

**Alarm quality is not “more alarms.” Alarm quality is right signal, right threshold, right reset.**

### 3.4 Autonomic Computing / MAPE-K

Autonomic computing uses control loops often summarized as:

- Monitor;
- Analyze;
- Plan;
- Execute;
- Knowledge.

House translation:

- Monitor = sensors.
- Analyze = debugger.
- Plan = route + action candidate.
- Execute = reflex or approved action.
- Knowledge = proof, memory, residue, decay.

House rule:

**The mesh is not only alarm. It is monitor → debug → route → act/stop → learn.**

### 3.5 NIST Incident Response

NIST incident response guidance emphasizes preparing for incidents, reducing incident impact, improving detection, response, and recovery, and integrating incident response with risk management.

House translation:

- Prepare = nerve map and thresholds.
- Detect = sensors and salience.
- Analyze = debugger.
- Respond = reflex/action/stop.
- Recover = proof and restore clean state.
- Improve = memory residue and rule repair.

House rule:

**Every serious alarm must end in one of: repaired, parked, blocked, decayed, or escalated.**

### 3.6 Resilience Engineering

Resilience engineering identifies four abilities:

- respond;
- monitor;
- learn;
- anticipate.

House translation:

- Monitor = sensors and thresholds.
- Respond = reflex/action/stop.
- Learn = proof/memory/residue.
- Anticipate = pattern watch, neighbor-risk, stage pressure, alarm-storm detection.

House rule:

**A nervous system must not only react; it must learn and anticipate.**

### 3.7 Salience Network / Salience Gate

Salience research supports the idea that intelligent systems need a filter for what matters. If everything enters deeply, the system overloads.

House translation:

- Salience gate decides whether a signal is routine, watch-only, alarm-worthy, council-worthy, proof-worthy, or discard/decay.
- Valence Council should not fire for every task.
- Debuggers should not fire for every tiny change.

House rule:

**No salience trigger, no deep alarm.**

### 3.8 Predictive Processing / Prediction Error

Predictive processing treats cognition, perception, action, and attention around prediction-error minimization. For Mr.Kleen, the useful source pattern is: a signal becomes important when reality differs from expectation, rule, proof, command, or safe operating range.

House translation:

- Difference signal = trigger.
- Prediction error = expected clean state vs observed dirty state.
- Learning = updating the pathway after proof.
- Bad prediction = stale anchor, stale mule packet, wrong authority assumption.

House rule:

**A trigger is not input. A trigger is meaningful difference.**

### 3.9 Ashby’s Law of Requisite Variety

Ashby’s cybernetic law says a regulator needs enough variety to handle disturbances. One alarm cannot govern a complex house. But too much variety without routing becomes chaos.

## 6. Sensors Are Not Alarms

This distinction must be protected.

A sensor says: “I can feel this condition.”  
An alarm says: “This condition crossed threshold.”  
A debugger says: “Here is why.”  
A router says: “Here is where it belongs.”  
A reflex says: “Here is the safe immediate move.”  
Memory says: “Here is what we keep.”  
Decay says: “Here is what fades.”  

If sensors and alarms collapse into one thing, the house becomes noisy.

Clean rule:

**Sensors are everywhere. Alarms are thresholded. Blocks are rare.**

## 7. Alarm Levels

Use this small ladder.

### QUIET

Sensor exists. No action.

### NOTICE

Something changed. No stop. Awareness only.

Use for:
- new file appeared;
- status changed;
- source ore arrived;
- weak signal with no risk.

### WATCH

Pattern may be forming. Keep context. Do not overreact.

Use for:
- repeated friction;
- concept beginning to affect nearby lanes;
- task starting to grow;
- possible stale state;
- low-grade delivery mismatch.

### WARNING

Likely dirty milk, proof weakness, wrong-lane risk, delivery risk, or authority confusion. Pause and inspect.

Use for:
- PASS claim lacks receipt;
- source ore starts steering;
- helper file remains after proof;
- mule output being used without full read;
- concept accelerates without brake.

### BLOCK

Stop forward motion. Authority, proof, safety, file custody, destructive action, or user-command risk.

Use for:
- not in repo root before write;
- dirty repo before save;
- wrong branch;
- destructive wildcard;
- PASS after PowerShell error;
- HEAD mismatch;
- ACTIVE_GUIDES/CTI touched without explicit selection.

### STABILIZE

Too many alarms, repeated failures, overload, runaway recursion, or fog. Stop nonessential motion and restore orientation.

Use for:
- alarm storm;
- active ball overload;
- repeated failed save loops;
- no clear current position;
- multiple bosses colliding.

### SEIZURE / STORM

The mesh itself is unhealthy. Too much firing, no discrimination, no path.

Use rarely. Reflex is:
- freeze optional work;
- name last clean commit;
- name active ball;
- drop all nonessential checks;
- route one root alarm.

## 11. Threshold Design

Thresholds stop sensors from becoming noise.

Threshold classes:

### Frequency threshold

This happened more than once.

Use for:
- repeated wording drift;
- repeated file clutter;
- repeated proof omission;
- repeated mule custody leak.

### Severity threshold

One occurrence is dangerous enough.

Use for:
- destructive command;
- wrong directory write;
- credential leak;
- PASS after error;
- ACTIVE_GUIDES rewrite by inertia.

### Authority threshold

## 15. Pain Model

Pain is information, not command.

Pain classes:

### Sharp pain

Immediate block risk.

Examples:
- destructive command;
- wrong repo;
- credential risk;
- PASS after error.

Response:
- BLOCK.

### Dull pain

Recurring friction.

Examples:
- repeated manual cleanup;
- repeated overlong chat blocks;
- repeated weak handoff shape.

Response:
- WATCH/WARNING, capture if repeated.

### Phantom pain

Old failure fires after repair.

Examples:
- overchecking receipt hygiene after rule already fixed;
- old mule fear blocking current valid output.

Response:
- check current evidence, decay stale alarm.

### Numbness

No alarm when there should be one.

Examples:
- Desktop helper left behind;
- no proof alarm after failed command;
- source ore steering unnoticed.

Response:
- Dead Nerve Detector.

### Over-sensitivity

Alarm fires too often.

Examples:
- every small wording change becomes rule;
- every source becomes high priority.

Response:
- threshold tuning, decay.

### Referred pain

Symptom appears in one lane, root is elsewhere.

Example:
- status bloat symptom; root is source intake not being sorted.

Response:
- debugger route, not symptom patch.

### Chronic pain

Repeated unresolved pattern.

Example:
- repeated false PASS shape.

Response:
- rule/tool/proof repair.

### Growing pain

Discomfort from healthy expansion.

Example:
- new architecture feels big but has clear boundary and stop.

Response:
- source-pattern candidate, tiny test later.

### Contamination pain

Dirty milk entering.

Example:
- web source acting as local truth.

Response:
- authority boundary.

### Coordination pain

Parts are good but not linked.

Example:
- proof lane clean, delivery lane dirty.

Response:
- nervous system link.
