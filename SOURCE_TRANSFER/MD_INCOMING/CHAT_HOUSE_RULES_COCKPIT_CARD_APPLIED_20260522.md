## 2.2 Rule-Fit / No Side-Mutation Gate

A new rule changes only the behavior it actually controls.

Before applying a new rule:
1. What exact problem does it fix?
2. What lane does it belong in?
3. What existing flow does it touch?
4. What must stay unchanged?
5. Does it conflict with anything?
6. What is the smallest behavior change?
7. Is anything being removed?
8. If something is removed, is it actually stale, superseded, unsafe, rejected, or duplicate with no added value?

Blocked:
- changing code delivery because source pickup changed
- changing source routing because code delivery changed
- removing an idea because a new one arrived
- rewriting the room because one rule was added

Short form:
New rule means fit-tested addition, not random surrounding behavior change.

## 2.10 Correction Handling Rule

A correction is evidence and a review gate.

1. Identify the exact failure.
2. Identify the mixed-up rule.
3. Patch only affected behavior.
4. Preserve working flow.
5. Decide if it needs save.
6. Do not overcorrect sideways.

Short form:
Correction fixes the failed behavior, not the whole room.

## 2.17 Alarm Friction Rule

An alarm is not real if it does not change behavior.

If an alarm fires, it must:
- stop output
- force classification
- block PASS language
- require recovery/proof
- or make the next action wait until handled

Alarm severity should match growth stage:
- BLOCKER for authority drift, proof falsehood, dirty repo/save danger, destructive action, doctrine/guide/index rewrite, promotion error, source/lane corruption
- WARNING for repeated mistakes or likely confusion
- WATCH for useful mess, growth bloat, wording roughness, or noncritical anomalies
