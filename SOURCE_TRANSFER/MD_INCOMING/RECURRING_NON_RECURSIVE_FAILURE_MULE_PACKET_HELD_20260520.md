# 1. Live Failure Case — Artifact Self-Check Rule Was Not Carried Into House TODO

## What happened

The assistant created a file artifact for the user.

The user then told the assistant to go over the file and ensure it was made right: full, intact, no context loss, no wording loss, and that the assistant should always check this on its own after sending.

The assistant did perform a file correction and validation. It added missing preserved material to the artifact and reported validation PASS.

But the user then asked whether the rule had been captured into the TODO list as the house rules require.

Truthful answer:

No.

The rule was captured in assistant memory continuity only. It was not placed into the Mr.Kleen TODO list, Work Shed, or saved house lane.

## Why this matters

This is not just an artifact bug.

This is a recursive-system failure.

The assistant did the immediate repair:

- inspected file,
- found missing pieces,
- fixed artifact,
- validated markers.

But it failed the carryover:

- did not classify the rule as a new issue/candidate;
- did not create or propose the house TODO placement;
- did not explicitly park the issue for later save;
- did not connect the lesson to future work entry;
- did not apply “Spin it back cleaner” to the actual house.

The user is correct: this fixed a symptom while failing the model.

## Issue name

**Recurring Non-Recursive Failure**

## Short definition

The system stores recursive ideas but does not reliably recurse at the point of work.

## Expanded definition

A recurring non-recursive failure occurs when:

1. a failure or correction exposes a reusable behavior need;
2. the assistant fixes the immediate output;
3. the assistant may even remember or explain the rule;
4. but the assistant does not route the lesson into the correct house lane, TODO, proof path, watch condition, or next-start surface;
5. the next similar task can therefore fail again.
