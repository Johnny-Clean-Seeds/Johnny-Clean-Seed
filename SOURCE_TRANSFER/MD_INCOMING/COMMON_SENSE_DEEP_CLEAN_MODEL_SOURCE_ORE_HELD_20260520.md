## Top Definition

Common sense is **adaptive relevance control**.

A system has common-sense-like ability when it can:

1. sense enough of its situation;
2. filter what matters from what does not;
3. read what the environment affords or blocks;
4. choose an action that fits the scale and context;
5. detect when reality pushes back;
6. update future behavior from that feedback;
7. carry the lesson into the next start.

Short form:

**Common sense is knowing what matters here.**

Clean Seed form:

**Common sense is the system’s ability to know what matters here, what this touches, what scale of action fits, what proof is enough, and what must carry forward.**

Deep form:

**Common sense is defeasible, situated, relation-aware fit judgment that selects practical action under limited information, then updates from feedback.**

## Common Sense Is Not Just Knowledge

A system can have many facts and still lack common sense.

The missing piece is relevance.

A million facts are not useful if the system cannot tell which fact matters now.

Common sense is therefore not just a knowledge store.

It is a live filter.

It asks:

- What kind of situation is this?
- What matters now?
- What can be ignored safely?
- What does this touch?
- What usually happens next?
- What could break?
- What level of response fits?
- What would prove the move worked?
- What needs to carry forward?

---

## Common Sense Is Defeasible

Common sense is not final authority.

It works as a strong ordinary-world guess until new evidence defeats it.

Clean phrase:

**Common sense proposes. Proof disposes.**

A clean common-sense system must be able to say:

- This seemed right under the old evidence.
- New evidence changes the verdict.
- The earlier conclusion must be adapted, parked, or blocked.
- The next start must change.

This matters because “common sense” is often abused as a way to avoid proof.

Dirty common sense says:

“That is obvious, so I do not need to prove it.”

Clean common sense says:

“This is the best fit judgment now, here is the boundary, and here is what would defeat it.”

## Common Sense Is Situated

Common sense lives inside a situation.

A person, animal, AI, workflow, or group does not judge from nowhere.

It judges from a body, tool surface, memory, environment, lane, rules, affordances, and feedback.

For humans, the body matters.

For animals, sensory and action capacities matter.

For AI, tools, memory, files, permissions, and feedback channels matter.

For Mr.Kleen, the “body” includes:

- files;
- lanes;
- active anchor;
- status;
- TODOs;
- receipts;
- proof history;
- PowerShell;
- Git;
- mule packets;
- user feedback;
- artifacts;
- local folders;
- download links;
- skeletons;
- stop conditions.

A tool-blind assistant has weak local common sense because it cannot touch, check, or verify the local world directly.

## Common Sense Is Affordance Literacy

An affordance is what a thing offers to a particular actor.

A handle affords pulling.

A chair affords sitting.

A red stop sign affords stopping.

A file path affords reading only if the file exists.

A receipt affords proof only if it contains the right claim-scoped evidence.

A mule report affords critique, not command authority.

A status ledger affords history, but not necessarily clean current-state judgment if it is too heavy.

Common sense reads affordances.

It asks:

- What does this object invite?
- What does it block?
- What does it risk?
- Who can use it?
- What wrong action does it tempt?
- What guard should make the right action easier?

Good design embeds common sense in surfaces.

Bad design forces the user to supply all common sense manually.

## Common Sense Exists Beyond Humans

Common sense should not be treated as human-only.

It has different forms at different levels.

### Human common sense

Human common sense includes practical judgment, physical expectations, social judgment, moral expectations, tool use, timing, scale, and ordinary-world assumptions.

### Animal common sense

Animals can show practical world sense without human language.

Examples include:

- route memory;
- object expectations;
- obstacle avoidance;
- tool use in some species;
- reading social signals;
- avoiding traps or danger;
- using smell, sound, motion, and feedback to fit action to situation.

This does not mean animals think in human concepts.

It means practical fit-sense can exist without human-style language.

### Collective common sense

Groups can show common-sense-like behavior that no single member fully holds.

Examples:

- ant colony routing;
- swarm path finding;
- group warning behavior;
- institutional memory;
- team habits;
- workflow defaults.

The intelligence is distributed across signals, roles, memory, feedback, and repeated local actions.

### Minimal life / adaptive fit behavior

Plants and microbes should not be casually described as having human thought.

But adaptive fit behavior exists below human cognition.

Examples:

- roots adjusting growth;
- plants responding to damage, light, gravity, and chemical signals;
- microbes coordinating behavior through chemical communication;
- populations adjusting behavior based on density or environmental signal.

Clean boundary:

This is not human common sense.

It is common-sense-like adaptive fit behavior: sensing, filtering, responding, coordinating, and carrying state.

### Machine / AI common sense

Machines can approximate common-sense ability through:

- knowledge;
- perception;
- simulation;
- memory;
- tools;
- feedback;
- embodied action;
- constraints;
- proof gates;
- learned patterns;
- relation mapping;
- world models.

But a machine does not prove common sense by answering common-sense trivia.

It proves common sense by acting with fit under real constraints and updating after failure.

### Workflow / house common sense

A designed system can embed common sense.

Mr.Kleen can become more common-sense-like when:

- anchors show the next right move;
- TODOs do not command by existence;
- receipts prove only their claim;
- active guides do not get rewritten casually;
- mule is treated as worker, not boss;
- memory is not mistaken for house capture;
- repeated scripts reuse skeletons;
- artifacts are checked before sending;
- failures create carryover;
- broad scans do not become fog;
- relation mapping catches cross-lane issues.

## Core Components Of Common Sense

### 1. Relevance filter

Common sense selects what matters now.

Without relevance control, the system either misses the key issue or overprocesses everything.

### 2. Affordance reading

Common sense reads what the environment offers, blocks, or tempts.

### 3. Scale judgment

Common sense matches response size to leak size.

A typo does not need a doctrine rule.

A repeated proof failure may need a guard.

A transition failure may need a latch.

A doctrine contradiction may need deeper review.

### 4. Timing judgment

A true idea at the wrong time can be dirty milk.

The same idea may be wrong during active mule work and right after mule conflict clears.

### 5. Relation mapping

Common sense sees what a thing touches.

A script touches proof, PowerShell, copy/paste, skeleton reuse, receipts, false-success hazards, and user burden.

A note touches memory, house capture, TODO, anchor, next-start, and proof.

A mule critique touches source custody, authority, disposition, save pressure, and local judgment.

### 6. Defeat switch

Common sense must drop or adapt a conclusion when new evidence defeats it.

### 7. Repair memory

Common sense stores the shape of prior mistakes.

Not as shame.

As reusable prevention.

### 8. Negative judgment

Common sense often knows what not to do before it knows what to do.

Do not trust PASS after a failed guard.

Do not treat memory as house capture.

Do not install all good ideas.

Do not let mule become supervisor.

Do not call future behavior proven because a save exists.

### 9. Feedback correction

Common sense improves when reality pushes back and the lesson carries forward.

### 10. Carryover

A common-sense system closes the loop at the next beginning.

If the next start is not cleaner, the lesson did not fully land.

---

## Common Sense Failure Modes

### 1. Wrong scope

A rule learned in one domain is applied to a domain where it does not fit.

### 2. Stale compression

An old shortcut keeps firing after the environment changed.

### 3. Local success hides system failure

The visible output gets fixed, but the repeated cause remains.

This is a major Mr.Kleen danger.

### 4. Authority fog

Someone or something sounds authoritative, so proof stops.

### 5. Overbuild

The system responds to a small leak with a giant structure.

### 6. Underreaction

The system treats a recurring pattern as a one-off.

### 7. Passive structure

The house has good rules, but they do not fire at the point of work.

### 8. Retrieval fog

The needed truth exists but is buried in a giant ledger or broad search.

### 9. Memory substitution

The assistant remembers a lesson but the house does not receive it.

### 10. Proof overclaim

A receipt proves a file exists, but the assistant claims it proves behavior forever.

### 11. Human-machine custody failure

The code is good in theory but fails through copy/paste, shell state, wrong folder, missing file, or false success output.

### 12. Beautiful language without gate

A motto sounds right but has no trigger, lane, stop condition, action push, or proof.

## Common Sense As A Clean Cycle

1. Sense the situation.
2. Classify the kind of situation.
3. Recall similar repairs or patterns.
4. Map relations.
5. Identify what matters.
6. Choose the smallest fit move.
7. Run proof.
8. Detect mismatch.
9. Adapt, park, block, or save.
10. Carry the lesson into the next start.

This is the recursive loop.

Common sense is not the absence of rules.

It is rule placement, rule timing, rule moderation, and proof correction.

## Common Sense And The Frame Problem

A practical system cannot inspect every possible consequence of every action.

It needs to know what is relevant.

This is close to the frame problem: after an action, what changed, what stayed the same, and what consequences are worth tracking?

Mr.Kleen version:

After a save:

- Did files change?
- Is status clean?
- Did the receipt prove the claim?
- Did active anchor update?
- Did a new TODO need to exist?
- Did the lesson carry forward?
- Did any unrelated authority file change?
- Did the next-start surface get cleaner?

The common-sense answer is not “check everything forever.”

It is “check the consequence set that fits the action.”

## Common Sense And Heuristics

Common sense often uses simple heuristics.

A heuristic is not automatically bad.

Just generate another script.

Good heuristic:

Fixed means done.

Good idea means save now.

## Common Sense And Scale

Common sense is scale control.

Same kind of issue can require different responses depending on repetition, risk, authority, and proof.

Examples:

- One typo: fix.
- Repeated typo in generated artifact: add artifact self-check.
- Repeated false PASS after errors: add proof guard / abort latch.
- Repeated rule not firing: add carryover latch.
- Repeated cousin scripts: add Filing Cabinet method.
- Repeated cross-lane misses: add Relation Method.
- Repeated doctrine contradiction: deeper authority review.

Bad common sense uses the same response size for every issue.

Clean common sense asks:

**What response is large enough to stop recurrence but small enough to avoid bloat?**

## Common Sense And Relation Method

Relation Method is a common-sense engine.

It asks:

- What is the home lane?
- What neighboring lanes does this touch?
- Does it affect proof?
- Does it affect future starts?
- Does it affect handoff or PowerShell execution?
- Does it affect artifacts?
- Does it affect TODO or queue?
- Does it affect memory-vs-house capture?
- Does it repeat under different names?
- Does it collide with an existing rule?
- Does it need a skeleton, Soft Suit card, TODO, receipt, park, or block?

Relation Method catches issues that are “across the board.”

Example:

Repeated deep-scan code was not only a deep-scan issue.

Relation Method shows it touches:

- compute waste;
- copy/paste risk;
- false-success hazards;
- proof consistency;
- Gear Rack;
- artifact self-check;
- file placement;
- local PowerShell execution;
- future start surfaces.

That is why the Filing Cabinet idea was broader than deep scans.

## Common Sense And Filing Cabinet

Filing Cabinet is common sense applied to repeated work.

If a repeated workflow exists, do not rebuild it from scratch.

Retrieve the closest prior skeleton.

Tweak only task-specific parts.

Run proof.

If the tweak is broadly useful, improve the skeleton.

This reduces:

- compute waste;
- token drag;
- copy/paste risk;
- script drift;
- guard mismatch;
- false-success risk;
- inconsistent receipts;
- user burden.

This is not just efficiency.

It is safety.

## Common Sense And Proof

Common sense does not replace proof.

Common sense chooses what to test.

Proof answers whether the move worked.

A clean system does not say:

“This feels right, so it is true.”

It says:

“This feels like the fitting move. Here is the proof needed. Here is what would defeat it.”

Proof must also stay scoped.

A clean git status proves the tree is clean.

It does not prove the rule will fire next week.

A receipt proves the claim it names.

It does not prove the whole model is future-proof.

A mule critique proves a hard outside read exists.

It does not become authority.

---

## Common Sense And Future-Proofing

Nothing here should be called finally future-locked.

The clean term is **future-shaped**.

A future-shaped fix:

- addresses the actual failure edge;
- has a trigger;
- has a lane;
- has proof;
- has a stop condition;
- changes the next start;
- blocks obvious overbuild;
- remains defeasible.

A future-proof claim is usually overclaim.

A clean model says:

“This is future-shaped as far as current evidence shows, and here is the next proof needed.”

**Head-to-Toe Clean Publication Gate**

Purpose:

Before producing a model file as truth, the assistant must prove the file is clean enough to present.

Trigger:

Use before printing or delivering:

- model summaries;
- doctrine-like source ore;
- rule candidates;
- handoff packets;
- mule packets;
- deep scans;
- conceptual maps;
- downloadable files;
- anything the user may treat as durable.

Required checks:

1. Coverage check:
   - Are all major ideas included?
   - Are earlier corrections included?
   - Are missing branches named?

2. Boundary check:
   - Is this truth, source ore, candidate, draft, or installed rule?
   - Is authority clearly marked?

3. Clean fit check:
   - Does each idea sit in the right lane?
   - Are human, nonhuman, machine, system, and house forms separated cleanly?

4. Internal consistency check:
   - Does any section contradict another?
   - Are “common sense,” proof, relation, carryover, and authority kept distinct?

5. Proof check:
   - Does the file claim only what can be supported?
   - Are uncertain areas marked?
   - Are future-proof claims blocked unless proof is named?

6. Bloat check:
   - Is anything decorative without function?
   - Is anything repeated without adding clarity?

7. Carryover check:
   - What rule, method, TODO, or live-use watch did this expose?
   - Is it installed, held, parked, blocked, or memory-only?

8. Artifact check:
   - File exists.
   - File opens.
   - Required sections exist.
   - No placeholders.
   - No truncation.
   - No bad characters.
   - Hash can be provided.

Stop condition:

Do not call the file “truth” unless it passes these checks or the limits are stated.
