# Child Shell Callable Route Install Packet

## Purpose

Install/prove the execution route that lets `nxt` become an action trigger instead of a manual PowerShell request.

This packet is not doctrine. It is an execution-route install/proof packet.

## Target Future Loop

Dropped file -> assistant reads/intakes it -> user says `nxt` -> assistant routes job through Command Center / Child Shell -> local child shell executes approved action -> receipt is produced -> assistant reads proof -> assistant sends the next needed file/artifact.

## Current Boundary

The assistant cannot claim it can execute inside the user’s Windows repo unless an active local bridge/Child Shell route is connected and proven.

Until then, the assistant can:

- read uploaded files
- create next job files
- create scripts/artifacts
- judge uploaded receipts/logs
- route work conceptually

The assistant cannot honestly claim it can:

- directly run scripts in `C:\Users\13527\Desktop\Jxhnny_Kleen_C3dz`
- activate Child Shell from chat without a bridge
- access local files from a hash alone
- use pointer rooms as global PC access

## Install Boss

Build/prove a callable route between ChatGPT-side job files and local Child Shell execution.

## Job File Format

Recommended extension:

`.childjob.json`

Minimum fields:

- job_id
- created_at
- requested_by
- route
- allowed_action
- target_path
- input_files
- expected_receipt
- boundary
- stop_conditions
- hash_requirements

Example shape:

```json
{
  "job_id": "CHILDJOB-20260521-000001-FRONTDOOR-READ",
  "created_at": "2026-05-21T00:00:00-04:00",
  "requested_by": "assistant_via_user_drop",
  "route": "COMMAND_CENTER/CHILD_SHELL",
  "allowed_action": "read_status_only",
  "target_path": "C:\\Users\\13527\\Desktop\\123\\COMMAND_CENTER",
  "input_files": [],
  "expected_receipt": "OUTBOX/CHILDJOB-20260521-000001-FRONTDOOR-READ.receipt.txt",
  "boundary": [
    "read only",
    "no git",
    "no Mr.Kleen write",
    "no delete",
    "no overwrite",
    "no permission expansion",
    "no network service",
    "no junction or symlink"
  ],
  "stop_conditions": [
    "target path missing",
    "job action not allowlisted",
    "receipt path already exists",
    "unexpected dirty state"
  ],
  "hash_requirements": []
}
```

## Allowed Action Ladder

Start with the smallest safe actions.

### Level 0: Probe Only

Allowed:

- confirm folders exist
- list allowed route files
- write probe receipt in Command Center only

Blocked:

- Mr.Kleen write
- Git
- raw shell
- delete
- overwrite
- permission expansion

### Level 1: Read Status

Allowed:

- read selected safe files
- read Command Center cards
- read git status only if target is explicitly Mr.Kleen and action is status-only
- write receipt

Blocked:

- git add/commit/push
- file writes outside receipt/outbox
- delete
- overwrite
- raw shell expansion

### Level 2: Approved Helper Run

Allowed:

- run one approved helper by exact path
- fixed arguments only
- write receipt

Blocked:

- arbitrary command strings
- shell injection
- network service
- broad filesystem crawl
- delete/overwrite unless helper explicitly designed/proven

### Level 3: House Save Route

Allowed only after earlier levels pass.

Allowed:

- run a specific save script generated by the assistant
- must include preflight, proof needles, commit, push, clean final status
- must produce receipt

Blocked:

- unreviewed save script
- doctrine rewrite unless explicitly selected
- active guide/current truth rewrite unless explicitly selected
- permission expansion

## `nxt` Meaning After Install

Before install proof:

`nxt` means: create the next file/job packet or tell the user what proof is missing.

After install proof:

`nxt` may mean: route the next approved job file to Child Shell and wait for receipt.

The assistant must not claim the after-install meaning until the route proves itself.

## Pass Conditions

### Probe Pass

- Command Center exists
- Child Shell folders exist or are created by an approved setup script
- Local Hard Bridge target exists if used
- a probe receipt is produced
- no Mr.Kleen write
- no git
- no permission expansion

### Read Status Pass

- a `.childjob.json` is accepted
- the runner performs only the allowlisted read
- receipt appears in OUTBOX
- receipt includes job ID, action, paths, hashes when relevant, boundary, and verdict
- no blocked action occurs

### Callable Route Pass

- assistant creates or provides one job packet
- user does not manually run the internal task itself
- local child-shell runner consumes the job
- receipt is produced
- assistant reads/judges receipt
- next file/artifact can be sent

## Stop Conditions

Stop and ask for user/system intervention if:

- no local runner/watcher exists
- Child Shell cannot consume job files
- receipt does not appear
- job action is not allowlisted
- bridge path is missing
- local state is dirty before a write job
- permissions would need expansion
- raw shell would be required
- delete/overwrite would be required
- assistant cannot see receipt/readback

## Hidden Trap Watch

Do not confuse these:

- job file exists != job executed
- hash exists != path opened
- pointer exists != access granted
- receipt exists != action passed unless receipt content proves it
- runner exists != assistant can call it from chat
- local script exists != bridge is live
