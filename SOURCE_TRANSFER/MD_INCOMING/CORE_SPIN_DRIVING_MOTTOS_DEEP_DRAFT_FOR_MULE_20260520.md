# 1. Why the Motto Layer Exists

The house already has rules. The recent repair run proved that having rules is not enough.

The house needs short phrases that cause the right rule to fire at the point of work.

These are not slogans. They are ignition keys.

A driving motto is a compressed operational key that makes one part of the core spin.

A motto should do work:

- start a gate,
- stop a bad move,
- push a clean move,
- choose a lane,
- demand proof,
- trigger learning,
- return the next cycle cleaner.

If a phrase does not alter motion, evidence, authority, placement, or proof, it is not a driving motto. It may be poetry, memory, or morale, but it should not be promoted as a core key.

## Working definition

A driving motto is:

A short memorable phrase that activates a specific gate in a specific lane and leaves proof that it fired.

Required fields:

1. Phrase
2. Plain meaning
3. Gate
4. Lane
5. Push
6. Stop
7. Proof of use
8. Failure smell
9. Cycle role
10. Promotion level

# 2. The First Two Mottos

## Motto 01 — The proof is in the pudding

Plain meaning:
The result must show itself. A claim is not proof. A plan is not proof. A file path written in chat is not proof. PASS requires something checkable.

Gate:
Proof Gate.

Lane:
Proof / receipt / completion standard.

Push:
Find or create visible evidence.

Stops:
- "I said it, so it is done."
- "The plan sounds good, so it is good."
- "Printed PASS after earlier error."
- "Movement equals proof."
- "Confidence equals proof."

Allows:
- file exists,
- hash exists,
- receipt exists,
- commit pushed,
- fetch confirms remote,
- HEAD equals origin/main,
- status clean,
- live-use behavior observed.

Proof of use:
The assistant names what the pudding is.

Example:
"The pudding is: file saved, receipt saved, commit pushed, fetch complete, HEAD == origin/main, status clean."

Failure smell:
Final answer says "done" or "PASS" without evidence.

Cycle role:
Closes the proof cycle.

Short form:
No pudding, no PASS.

## Motto 02 — If words are not just words, then that’s just that, always and forevermore

Plain meaning:
No selective enforcement. If a word controls behavior, authority, evidence, status, files, commands, receipts, memory, bridge behavior, or next state, it is not cosmetic.

Gate:
Word-Key Gate.

Lane:
Naming / authority / command safety / proof wording / status wording.

Push:
Treat key wording as machinery.

Stops:
- "It is just wording."
- sloppy status labels,
- vague PASS,
- "saved" meaning unclear,
- queue active-by-inertia,
- mule command drift,
- receipt overclaim,
- local/remote/brain confusion,
- "complete" without standard.

Allows:
- exact task state,
- exact proof claim,
- exact authority boundary,
- exact next move.

Proof of use:
The assistant notices when a word controls motion and either confirms it, fixes it, or blocks action until it is clear.

Failure smell:
A phrase like "active," "saved," "done," "proof," "mule," "brain," "local," "packet," or "command" is used loosely while controlling behavior.

Cycle role:
Starts the word/entry cycle.

Short form:
If a word moves the house, the word is machinery.

# 3. Motto Engine

The motto engine should run like this:

Word -> State -> Boss -> Lane -> Tool -> Proof -> Lesson -> Return

Each stage has a motto:

1. Word:
   Words are not just words.

2. State:
   Control state first.

3. Boss:
   Boss first, ghosts under it.

4. Lane:
   Open intake, bounded authority.

5. Tool:
   Use the tools that fit.

6. Proof:
   The proof is in the pudding.

7. Lesson:
   A mistake is a seed if it grows a rule.

8. Return:
   Spin it back cleaner.

This engine makes the core loop.

The point is not to create a chant. The point is to force work through the correct gates.

# 5. Missing Part Added: Carryover Gate

The prior draft had word, proof, learning, and return ideas. The missing part is a stronger carryover gate.

A loop is not recursive just because it repeats.

A loop becomes recursive when the next start is cleaner because the previous cycle changed the entry condition.

## Carryover Gate

Motto:
Spin it back cleaner.

Plain meaning:
A cycle does not close merely because work ended. It closes only when the next cycle can start cleaner than the last one.

Gate:
After proof and before stop.

Lane:
Anchor / status / TODO / Work Shed / BRAIN_LEARNING / proof history / future packet top block.

Push:
Carry the useful change into the next entry point.

Stops:
- repeating the same failure,
- leaving lessons buried,
- proof with no next-state effect,
- notes with no route,
- receipts that do not help future work,
- rule files that do not fire.

Allows:
- anchor update,
- status note,
- TODO parking,
- boss/ghost placement,
- rule candidate,
- receipt hygiene note,
- future top-block improvement,
- mule dedupe note,
- live-use watch.

Proof of use:
The final state tells the next worker or assistant how to start cleaner.

Failure smell:
The same failure returns because the lesson was saved somewhere but not connected to the next entry point.

Cycle role:
This is the return spoke. It links proof back to word/state.

## Why this matters

Without a carryover gate, the house can make beautiful artifacts and still fail the next live task.

The motto engine must not end with proof. It ends with next-start improvement.

Proof answers:
Did it work?

Carryover answers:
Will the next start be cleaner because it worked?

# 6. Motto Types

Not every motto does the same job.

## Law Motto

Controls behavior directly.

Examples:
- The proof is in the pudding.
- Words are not just words.
- Worker present, hands off.

Promotion standard:
High. Must be precise and repeatedly useful.

## Gate Motto

Activates a specific check.

Examples:
- Control state first.
- Boss first, ghosts under it.
- Complete means proven to this task's standard.

Promotion standard:
Medium-high. Must have trigger and stop condition.

## Motion Motto

Chooses action direction.

Examples:
- No random moves.
- Use the tools that fit.
- Current task first, later issue parked.

Promotion standard:
Medium. Must prevent bad movement.

## Learning Motto

Turns failure into growth.

Examples:
- A mistake is a seed if it grows a rule.
- Spin it back cleaner.

Promotion standard:
Medium. Must create better next behavior.

## Poetry / Memory Wrapper

Useful for remembering but not enough alone.

Examples:
- The core spins by clean keys.
- Horns going and flag held high.
- Clean motion, clean proof.

Promotion standard:
Low unless tied to operational gates.

# 7. Motto Registry — Deep Version

## A. Word / Authority Mottos

### Words are not just words

Gate:
Word-Key Gate.

Use when:
A word controls behavior, authority, evidence, status, files, commands, receipts, memory, bridge behavior, or next state.

Stop:
Do not treat controlling words as cosmetic.

Push:
Define, confirm, or fix the word.

Proof:
The word's meaning and lane are named before action.

### Name it clean or route it wrong

Gate:
Naming Gate.

Use when:
A file, task, boss, status, receipt, packet, or rule is being named.

Stop:
No vague names for active machinery.

Push:
Choose a name that says job and lane.

Proof:
The name compresses the job without misleading authority.

## B. State / Custody Mottos

### Control state first

Gate:
Work Entry Gate.

Use when:
Any nontrivial task begins.

Stop:
Surface request does not automatically control action.

Push:
Check custody, dirty tree, anchor, queue, proof gap, authority, and user command.

Proof:
Control state is named before action.

### Worker present, hands off

Gate:
Custody Gate.

Use when:
Mule, Codex, or another worker is active.

Stop:
No house-touching work.

Push:
Non-invasive support only.

Proof:
The assistant refuses save/edit/commit while worker custody is active.

## C. Boss / Sorting Mottos

### Boss first, ghosts under it

Gate:
Boss/Ghost Gate.

Use when:
There is a pile of issues.

Stop:
Do not fight symptoms as separate bosses.

Push:
Find parent issue, attach ghosts, rank only true bosses.

Proof:
Parent boss and child ghosts are named.

### Do not fight smoke when the fire has a name

Gate:
Root-Cause Gate.

Use when:
Many symptoms point to one parent cause.

Stop:
Do not patch the visible symptom repeatedly.

Push:
Name the parent and place child issues below it.

Proof:
Fix targets the controlling parent or explicitly parks the ghost.

## D. Intake / Placement Mottos

### Open intake, bounded authority

Gate:
Placement Gate.

Use when:
New ideas, corrections, outside methods, mule output, screenshots, notes, or old chat appears.

Stop:
Do not lose ideas; do not let raw ideas command.

Push:
Place as active rule, candidate, TODO, Work Shed, Idea Bag, Source Ore, proof, rejected, or parked.

Proof:
Every useful item has a lane.

### Source ore is not command

Gate:
Source Boundary Gate.

Use when:
External material or raw notes enter.

Stop:
Do not treat raw source as authority.

Push:
Sort, adapt, test, or park.

Proof:
Source is dispositioned before adoption.

## E. Motion / Tool Mottos

### No random moves

Gate:
Motion Gate.

Use when:
A next fix seems attractive.

Stop:
No movement without fit, lane, and proof path.

Push:
Explain why this move is next.

Proof:
The assistant names why now.

### Use the tools that fit

Gate:
Tool Fit Gate.

Use when:
Evidence may require files, web, screenshots, mule output, queue, proof, or scripts.

Stop:
No literal all-tool crawl. No under-tooling.

Push:
Pull only relevant tools and place outputs.

Proof:
Tool result is applied, cited, parked, saved, rejected, or ignored with reason.

## F. Proof / Completion Mottos

### The proof is in the pudding

Gate:
Proof Gate.

Use when:
Calling PASS, saved, completed, or fixed.

Stop:
No evidence-free PASS.

Push:
Demand artifact, hash, receipt, commit, push, fetch, status, live-use proof, or other task-specific pudding.

Proof:
Final answer names checkable evidence.

### Complete means proven to this task's standard

Gate:
Completion Gate.

Use when:
Defining done.

Stop:
No generic complete.

Push:
Define completion standard for task type.

Proof:
The final state meets that standard.

### The receipt proves the claim, not the universe

Gate:
Claim-Scope Gate.

Use when:
Using proof history.

Stop:
No global PASS overreach.

Push:
Name what the receipt proves and does not prove.

Proof:
Partial/superseded/trusted claims are separated.

## G. Learning / Repair Mottos

### A mistake is a seed if it grows a rule

Gate:
Learning Gate.

Use when:
Failure or repeated friction appears.

Stop:
No shame loop. No ignore loop.

Push:
Diagnose, capture, candidate rule, test, save if selected.

Proof:
Failure changes future behavior.

### If it fails three times, stop patching and find the boss

Gate:
Pattern Gate.

Use when:
Issue repeats.

Stop:
No repeated small patching after pattern appears.

Push:
Group evidence and find parent boss.

Proof:
Pattern note or boss/ghost sort exists.

## H. Return / Carryover Mottos

### Spin it back cleaner

Gate:
Carryover Gate.

Use when:
A task is closing.

Stop:
No proof with no next-entry improvement.

Push:
Update anchor/status/TODO/parking/rule or note only if it improves future entry.

Proof:
Next start is clearer.

### The loop closes at the next beginning

Gate:
Re-entry Gate.

Use when:
Saying a cycle is complete.

Stop:
Do not close if next task would start in fog.

Push:
Leave a clean re-entry point.

Proof:
Anchor, status, or packet top block tells next move cleanly.

# 10. External Comparison Notes

These outside methods support the shape but do not command it.

## PDSA / Deming learning cycle

Useful part:
Improvement is a cycle of planning, doing, studying, and acting. The model should learn through repeated cycles, not one-shot declarations.

Adoption:
Use as background support for "proof -> lesson -> return."

Risk:
Could become generic management language if not tied to local gates.

## SRE blameless postmortem / action discipline

Useful part:
Failures should identify contributing causes and produce meaningful action, not blame or shame.

Adoption:
Supports "A mistake is a seed if it grows a rule."

Risk:
Postmortem ceremony can become too heavy if used for every tiny issue.

## Diátaxis documentation separation

Useful part:
Different documents have different jobs. Tutorials, how-to guides, reference, and explanation should not be blurred.

Adoption:
Supports motto type separation: law/gate/motion/learning/wrapper.

Risk:
Over-structuring simple notes.

## NIST RMF-style control lifecycle

Useful part:
Prepare, select, implement, assess, authorize, monitor. This matches controlled promotion and watch states.

Adoption:
Supports "closed with watch" and "motto earns promotion through use."

Risk:
Too formal if copied literally.

## OODA feedback movement

Useful part:
Observe, orient, decide, act loops help with fast state-aware movement.

Adoption:
Supports "control state first" and "loop closes at the next beginning."

Risk:
Could become another slogan if not tied to proof.

# 11. Motto Promotion Ladder

A motto should move through stages.

## Stage 0 — Phrase

A useful phrase appears.

Status:
Raw candidate.

Allowed:
Capture.

Not allowed:
Command work.

## Stage 1 — Candidate Motto

The phrase has possible operational meaning.

Required:
Meaning and likely gate.

Allowed:
Idea Bag / Sorting Bench.

Not allowed:
Doctrine.

## Stage 2 — Operational Key

The phrase has trigger, lane, stop, push, proof.

Allowed:
BRAIN_LEARNING support.

Not allowed:
Active guide promotion.

## Stage 3 — Live-Use Watch

The key is used in real work.

Required:
Evidence that it fired correctly.

Allowed:
Receipt or status note if relevant.

## Stage 4 — Stable Root Key

The key repeatedly improves work and does not confuse lanes.

Allowed:
Consider promotion or stronger placement if user selects.

## Stage 5 — Doctrine Candidate

Only after repeated proof.

Required:
Clear user selection, proof history, no collision with active guides.
