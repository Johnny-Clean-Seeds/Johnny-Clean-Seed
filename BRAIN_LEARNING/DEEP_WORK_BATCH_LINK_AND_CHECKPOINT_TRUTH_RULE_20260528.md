## Core routine

Use this only for large, deep, boss, source-heavy, save-route, or repeated-failure work.

Route:

front door -> rope -> named batch -> source/object links -> findings -> park before rule -> Soft Suit if safe -> washer -> A progress checkpoint -> B end-state checkpoint -> next return point

## Batch discipline

Large work must be split into named batches with links/pointers, source/object references, checkpoint notes, parked findings, candidate rules/tools, proof needs, and next return point.

## Finding cards before rules

Useful ideas do not become doctrine raw. First make a finding card with:

- finding;
- role;
- lane;
- why it matters;
- proof needed;
- Soft Suit yes/no;
- save route yes/no;
- return trigger.

## Park then Soft Suit

Safe assistant-facing behavior may be worn while waiting for lock/save only if clearly labeled UNSAVED / NOT DOCTRINE.

## Dead-weight drop on continuation

When the user says `nxt`, continue only the approved sequence. Drop repeated source hallway, old proof chatter, stale helper details, broad-scan impulse, and furniture that can be carried as a key.

Do not drop receipts, source custody, proof history, active object, current blocker, or next return point.

## A/B checkpoint truth

A checkpoint A is progress while work is alive.

A checkpoint B is the end-state when motion stops.

Allowed B labels:

- CLEAN_CLOSE;
- BLOCKED_WITH_RETURN_POINT;
- CHOKE_STOP;
- CHOKE_CAUSED_END;
- USER_STOP.

CHOKE_CAUSED_END is never done, never PASS, and never clean.

## Size guardrail

Do not drag the full ritual into tiny/direct tasks. Use it when depth, risk, proof, source volume, save routes, or repeated failure justify the structure.
