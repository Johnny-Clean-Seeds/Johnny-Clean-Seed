# Soft Suit Trigger Event Proof-Before-Mark Rule

Date: 2026-05-20

## Status

ACTIVE BEHAVIOR RULE.

## Problem

A Soft Suit trigger-event script was generated too early after raw Growth Cycle mule review intake.

The code tried to mark a trigger event before the mule return had been read and dispositioned enough to prove what the suit actually blocked, allowed, clarified, improved, or failed to handle.

Raw intake is custody proof.

Raw intake is not event proof.

## Rule

A Soft Suit triggered event may be marked only after proof shows the suit fired in a specific scenario.

## Required Proof Before Marking

Before marking a trigger event, identify:

1. Scenario.
2. Suit involved.
3. What triggered the suit.
4. What the suit blocked.
5. What the suit allowed.
6. What the suit clarified.
7. What proof improved.
8. What bloat or wrong move was prevented.
9. Whether the suit failed or was insufficient.
10. Why this is a new event/new scenario and not a duplicate.

## Not Enough

The following are not enough to mark a trigger event:

- raw intake;
- file presence;
- hash verification;
- user excitement;
- a strong mule report existing;
- a future possible use;
- an assistant assumption that the suit probably helped.

## New Event Rule

New event plus new scenario is captured.

Same suit or same general issue does not make it duplicate.

But it must still be proven.

## Park Rule

If trigger-event code is generated before proof, park it.

Mark it as:

PARKED / NOT APPROVED / NOT RUN.

Then return to the missing proof step.

## Boundary

This rule does not split the Soft Suit.

It does not install gates.

It does not promote doctrine.

It protects event capture quality.
