# Rule Bloom and Stack Capture Rule

## Motto

Bloom every fix. Collect every stack. Collapse only after pattern proof.

## Purpose

When an issue, idea, fix, rule, correction, or missed behavior is addressed, it must not remain only chat talk.

It becomes an active provisional rule or held provisional idea unless it is deliberately rejected.

This creates a controlled rule-bloom phase so repeated odd/narrow rules can reveal their shared family and later collapse into a stronger Broad-Bite / bulldog rule.

## Core Machine

addressed issue -> provisional capture -> stack capture if present -> live firing evidence -> grouping -> collapse/compress -> promote, park, or retire

## Trigger

This rule fires when any of these appear:

- a failure is named
- a correction is named
- a "what I should have done" behavior is named
- a new method is proposed
- a rule gap is noticed
- a stack/pile of related symptoms appears
- a repeated pattern begins to show itself

## Required Action

1. Capture the addressed item as an active provisional rule or provisional idea.
2. If several related items appear, capture the whole stack, not only the loudest item.
3. Label each item as active provisional, held idea, needs proof, parked, rejected, or candidate for collapse.
4. Use active provisional rules in the next relevant action.
5. Watch firing evidence.
6. Later collapse proven families into small, dense Broad-Bite rules.

## Stack Capture Requirement

If a pile appears, the pile itself is evidence.

Do not compress too early.

Do not grab only the top item and lose the pattern underneath.

A stack may include:

- rules
- fixes
- symptoms
- mistakes
- user corrections
- assistant missed behaviors
- repeated wording issues
- route failures
- source/proof failures
- copy/paste delivery failures
- room/lane placement failures

## Bloat Boundary

Rule bloom is not final doctrine.

Temporary bloat is allowed only as controlled growth pressure.

The purpose of rule bloom is later collapse, not permanent rule clutter.

## Collapse Condition

Collapse begins only when there is enough pattern evidence to identify:

- shared trigger
- shared choice-family
- neighboring surfaces
- allowed outputs
- blocked outputs
- proof of activation

## Ghost/Boss Watch

A stack may hide the real boss.

A provisional rule may create a ghost if it:

- names the wrong family
- mimics an existing house result
- feels familiar but misses the actual trigger
- creates the same visible output without changing the behavior
- collapses too early

Guard: keep provisional items scoped, labeled, and separate from final authority.

## Boundary

This rule does not rewrite doctrine.
This rule does not rewrite ACTIVE_GUIDES.
This rule does not rewrite CURRENT_TRUTH_INDEX.
This rule does not make Soft Suit into Hard Suit.
This rule does not promote source ore into authority.
This rule does not remove proof requirements.
