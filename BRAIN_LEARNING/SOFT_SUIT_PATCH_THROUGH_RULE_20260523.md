# Soft Suit Patch-Through Rule

Status: SUPPORT RULE / BOSS LOOP REPAIR / NOT DOCTRINE
Date: 2026-05-23

## Purpose

The 17-route suit must not become a big scan list followed by a guessed fix.

When a boss loop has a proposed fix, the assistant must carry that fix through the active suit and record what each route does to it before naming a revised boss, final fix, or next move.

## Trigger

Use this rule for:

- Boss Loop runs;
- 17-route house walks;
- boss-gate workouts;
- "apply this fix through the house" tasks;
- repeated failures where a first fix may be too narrow;
- any run where the user asks whether the fix is being applied at each gate.

## Required Header

Start every patch-through run with:

- Active boss:
- Current proposed fix:
- Soft suit being used:
- Boundary:
- Proof target:

Boundary must say that Soft Suit guides and pressure-tests. It does not become doctrine, authority, or king.

## Route Questions

For every route or gate, ask:

1. Does this gate support the fix?
2. Does this gate change the fix?
3. Does this gate show a neighbor problem?
4. Does this gate block part of the fix?
5. Does this gate need proof before PASS?

## Patch Marks

Each route must mark how it affects the fix:

- KEEP: fix works here.
- NARROW: fix is too broad here.
- EXPAND: fix must include this surface.
- BLOCK: fix would violate boundary here.
- PARK: related, but not current boss.
- PROOF: cannot claim until verified.
- PARENT: this gate shows a deeper boss.

## Required Ledger Row

Use this row shape:

| Route | Gate / Surface | Mark | What It Does To The Fix | Revised Fix Note | Proof Needed |
|---|---|---|---|---|---|

## Pressure Map Before Fix Claim

After all routes are marked, build a pressure map:

- gates that agree;
- gates that narrow;
- gates that expand;
- gates that block;
- gates that park;
- gates that need proof;
- gates that reveal a parent boss.

Only after the pressure map exists may the assistant name:

- revised boss;
- revised fix;
- next move;
- proof target;
- final verdict.

## Boss Loop Seal

Use:

`BOSS -> PROPOSED FIX -> 17-ROUTE PATCH-THROUGH -> PRESSURE MAP -> REVISED FIX -> PROOF -> WORN -> WALK-AROUND -> NEXT BOSS`

## Moving Forward

From now on, when I use the 17-route suit inside a Boss Loop, I will not merely scan the routes and summarize. I will carry the active fix through each route, mark what each route does to the fix, revise from the pressure map, and only then judge the fix.

This is better because it prevents list-to-guess failure, forces the fix to survive neighbor gates, and shows when the real answer is a parent boss rather than the first visible patch.

## Proof

The proof is a completed patch-through ledger where every route has a mark and the revised fix follows from the pressure map.
