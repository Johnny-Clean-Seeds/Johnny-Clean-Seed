## Purpose

Swift Scan is the longer-flow reset-review system for evidence saturation, stale-file bloat, repeated proof loops, and batch momentum.

It stops the pile, compresses live carry, reviews proof, checks freshness, blocks false promotion, pulls in only useful method keys, uses Creative Room synthesis to make the method house-native, records one compact decision, and chooses the next smallest clean action.

Swift Scan does not mean fast scanning. It means clean movement without dragging the whole house into live chat.

## Entry triggers

Use Swift Scan when any of these appear:

- evidence saturation;
- stale drop-copy buildup;
- repeated proof loops;
- batch momentum replacing judgment;
- expensive chat/file load;
- proof chain needs review before the next run;
- repeated failure suggests the rule or assumption needs inspection.

## Exit criteria

Do not leave Swift Scan until these are named:

- active object;
- current pointer;
- proven items;
- not-proven items;
- stale/proof-only material;
- blocked lanes;
- lost leftovers recovered or dropped;
- decision record;
- next smallest clean action or stop state.

## WIP / one-pointer rule

Carry one active object, one active source pointer, and one next condition.

Do not carry every ledger, drop-copy, proof receipt, failed script, or old suit card in live chat. Older material stays in house/proof history unless exact inspection is needed.

## No-more-proof rule

Do not run another probe, canary, or scale bump unless it tests a new risk or answers a live uncertainty.

More rows are not proof by themselves.

## Room roles

### 1. Stop / Set Object

Reason: no room walk matters until the active question is named.

Ask:
- What are we deciding right now?
- What must not change?
- What closes this pass?

Output:
- one locked object.

### 2. Wash Live Load

Reason: heavy live carry distorts judgment.

Ask:
- What can drop from live chat carry?
- What stays only as proof history?
- What is the one current key?

Output:
- active carry list and stale/proof-only list.

### 3. Situation Awareness / Freshness

Reason: see the current state before reasoning.

Ask:
- What is the newest valid pointer?
- What did it replace?
- What should not be reloaded?

Output:
- current pointer and stale-source list.

### 4. Proof / AAR Room

Reason: compare intended proof against actual proof before promotion pressure starts.

Ask:
- What was supposed to happen?
- What happened?
- What changed?
- What was repaired?
- What is only identity/hash?

Output:
- proven / not-proven split.

### 5. Boundary / Promotion Pressure

Reason: repeated success creates false permission.

Ask:
- What does this not authorize?
- Which lane is trying to sneak open?

Output:
- hard blocked list.

### 6. Premortem / Failure Forecast

Reason: test the next action before doing it.

Ask:
- If the next action failed or wasted motion, why?
- What risk would it actually test?
- Is this just bigger, or is it smarter?

Output:
- next-action risk check.

### 7. Think Tank / Method Match

Reason: borrow useful outside method keys without making them authority.

Adopted method keys:
- OODA: orient before action.
- AAR: review after evidence.
- safe-fail probes: probe only while uncertainty remains.
- ADR: record context, decision, consequence.
- premortem: assume next action failed before doing it.
- A3/current-condition thinking: current condition, target, cause, countermeasure, evidence.
- Kanban/WIP: one active pointer and limited live work.
- double-loop learning: repeated failure can require changing the rule, not only the action.
- lessons-learned application: captured lessons must affect the next work.
- cognitive-load reduction: reduce extraneous live information.

Output:
- adopted keys only; no method becomes doctrine by itself.

### 8. Creative Room / Synthesis

Reason: turn parts into one house-native shape.

Creative synthesis:
Swift Scan is a broom plus a flashlight plus a judge. The broom clears chat load. The flashlight finds current proof and source. The judge decides whether more motion has a real job.

Output:
- one clean system shape.

### 9. Lost Leftovers Sweep

Reason: compression can accidentally drop useful guards.

Ask:
- Did we lose a guard, receipt, blocker, or current pointer?
- Which subtools remain active?

Recovered subtools:
- 333 as inspection subtool.
- User-Said-First as entry rule.
- Crime Scene / Powerplay as failure rule.
- Desktop Alert as attention rule.
- Copy Field rule as copy-surface rule.
- PowerShell operator-binding guard as script-generation rule.

Output:
- recover/drop list.

### 10. Decision Record / Map

Reason: one compact record replaces repeated file drops.

Shape:
- Context.
- Evidence.
- Decision.
- Consequences.
- Next condition.

Output:
- compact decision record.

### 11. Final Judge / Omega

Reason: close without fake motion.

Ask:
- Is another action needed now?
- Is the next action smaller and cleaner than more proof?
- Should this stop, move, save, park, or review?

Output:
- final state and next smallest clean action.
