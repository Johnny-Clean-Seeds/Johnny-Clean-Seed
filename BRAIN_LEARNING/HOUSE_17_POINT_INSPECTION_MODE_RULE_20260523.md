# House 17-Point Inspection Mode Rule

Status: SUPPORT INSPECTION MODE / HOUSE COVERAGE / NOT DOCTRINE
Date: 2026-05-23

## Purpose

Run a bounded house inspection that checks whether the main gate family, object types, proof habits, porch discipline, and closure routines are working together.

This is not a growth gate by itself. It is an inspection mode.

## Trigger

Use this mode when:

- the house feels smart but crowded;
- a helper/assistant keeps producing the same issues;
- gate runs repeat without improvement;
- porch/guest artifacts are leaking;
- object types blur;
- proof language gets too loose;
- the user asks for a 16 or 17 point house run.

## 17 Inspection Points

1. Front Door / Porch Discipline
2. Router Hall
3. Layer Room
4. Object Room
5. Gate Firing Bench
6. Root-Fix / Powerplay Bench
7. Boundaries / Permission Chamber
8. Guest Boundary Workshop
9. Return-Home Sink
10. Enforcement Ladder Garage
11. Proof Locker
12. Hindsight Room
13. Sequencer Desk
14. Gate Inventory / Coverage Workshop
15. CONSOLIDATOR Table
16. Alpha Omega / Final Judge Closure
17. Resident Parking / Deep Scan Bay

## Required Questions

For each point, ask:

- What is this point supposed to protect?
- Is the current issue touching it?
- Does it change the active fix?
- Does it reveal a missing gate, duplicate gate, or non-gate support object?
- Does it require proof, parking, return-home, or a TODO?
- Does the previous point's change still hold here?

## Gate Available Is Not Gate Active

A gate being available does not mean it fires every time.

For each gate, name the trigger:

- Always-on behavior support.
- Fires only on new object.
- Fires only on guest/porch intake.
- Fires only on proof claim.
- Fires only on repeated failure.
- Fires only on permission/boundary crossing.
- Fires only after Creative/Growth creates a candidate.
- Fires only at closure.

No gate should run just because it exists.

## Object-Type Hardening

Classify useful material before saving or promoting it:

- gate;
- tool;
- scaffold;
- card;
- ledger;
- parking item;
- receipt;
- script;
- doctrine candidate.

If the type is not clear, park it with a return trigger.

## One Clean Root Fix

When a root issue is visible, prefer one clean swipe at the root.

Rule:

`one fix -> same problem -> proof -> freeze -> next item`

Do not chase every shadow until the root fix has been tested.

## Proof Locker Rule

No PASS language without evidence.

Minimum proof:

- what changed;
- source basis;
- hash or receipt when files change;
- same-problem rerun when behavior changes;
- final status;
- blocked or unknown items named.

## Return-Home Rule

Every artifact must leave with a destination:

- keep in lane;
- archive;
- resident parking;
- pickup;
- TODO;
- delete-candidate only if explicitly allowed later.

Nothing should finish in the air.

## Resident Parking Boundary

Symbolic, final-ideas, Day Dreams, Cube, and similar material stays local-only and clearly pointed to.

It can be visible, useful, and warm without becoming gate authority, Final Order, public proof, or doctrine.

## Boundaries

- Not doctrine.
- No ACTIVE_GUIDES rewrite.
- No CURRENT_TRUTH_INDEX rewrite.
- No public authority change.
- No Git commit/push by itself.
- No delete.
- No broad crawl.
- No gate promotion by inspection alone.

## Proof

Proof is a completed inspection ledger showing the point, mark, issue, change to the fix, proof needed, and final disposition.
