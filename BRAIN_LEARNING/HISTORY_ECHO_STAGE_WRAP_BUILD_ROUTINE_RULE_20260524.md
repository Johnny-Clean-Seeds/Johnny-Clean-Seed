# History Echo Stage Wrap Build Routine Rule - 2026-05-24

Status: support rule, not active doctrine.
Route: history -> evidence -> stage card -> washer -> next move.
Authority: lower than ACTIVE_ANCHOR and CURRENT_TRUTH_INDEX. Do not use this rule to rewrite active guides by momentum.

## Purpose

Use old work, shed work, LocalBot, proof packets, chat transcripts, and current build history as source ore for a clean stage wrap. The history is allowed to teach pattern, sequence, and failure shape. It is not allowed to become a live burden just because it exists.

The goal is less live load with more usable truth.

## Trigger

Use this routine when a task asks for any of these:

- compare old and current build history
- find the sequence of tightening, rigidity, or model behavior
- use the shed, LocalBot, root drops, or proof packets
- tighten a rough wrap into a real build feel
- find similarities across work systems
- run the whole house through a washer after a history pass

## Core Rule

One source object becomes one stage object before it becomes a rule.

Do not carry the whole history forward. Select one bounded object, give it a source pointer, extract its useful pattern, condense it into a stage card, replay it once, wash it, then either promote, park, or burn active weight.

## Clean Stage Routine

1. Source map: name the exact files, folders, commits, hashes, or receipts inspected.
2. Sequence pass: order the evidence from oldest to newest where possible.
3. Similarity pass: identify repeated shape across systems, not repeated wording.
4. Clean bloat pass: allow enough detail to prove the stage, but mark what is excluded.
5. Condense pass: reduce the result to one stage card and one next move.
6. Replay pass: walk the same house again while wearing the condensed result as a soft suit.
7. Washer pass: compare first read, replay read, outside challenge, and final route.
8. Close state: promote, park, or release active weight with a runner trace.

## Agent Frame / Freedom Ratio

Use the weakest useful frame that still protects the stage.

| Stage | Job | Suggested frame | Suggested freedom |
| --- | --- | ---: | ---: |
| Stage 0 | seed and boundary | 40 | 60 |
| Stage 1 | tiny working loop | 55 | 45 |
| Stage 2 | failure handling | 65 | 35 |
| Stage 3 | recovery and replay | 70 | 30 |
| Stage 4 | practical use | 80 | 20 |
| Final wrap | closeout and handoff | 85 | 15 |

If the agent becomes rigid, soften the next pass by asking for contrast, disproof, and what would be lost if the rule is wrong. If the agent drifts, tighten the next pass with source paths, one object, and a close condition.

## Rope Handling

- Tug rule: tug the rope when path fog hides the next evidence object.
- Note scale rule: notes get louder by proof, fog, repetition, time risk, and neighbor fit, not by drama or age alone.
- Burn rule: a note that no longer changes route is released from active weight.
- Cut rule: cutting the rope means removing active weight only after leaving a runner trace to source, rule, room, and next retrieval path.

## Outside-Room Imports

These outside ideas fit the house without taking over:

- Kanban WIP limit: keep live work small enough to finish and expose blockage.
- Event sourcing: preserve the sequence of state-changing events and derive current truth from the log instead of rewriting the past.
- Requirements traceability: every promoted rule should trace backward to source and forward to use/proof.
- Phase/stage gate: a phase closes only when the gate has evidence, a verdict, and a next allowed move.
- Cognitive load control: compress live context into a stable schema so working memory is not forced to carry every raw note.
- OODA/Cynefin style sensemaking: observe and orient before deciding whether the object is clear, complicated, complex, or chaotic.

## Dirty Signals

Stop and wash if any of these appear:

- the note is getting louder without new proof or route change
- a support file starts acting like active authority
- old path signs disagree with the live house path
- a TODO says "next use" but has no owner, trigger, or close condition
- duplicated evidence surfaces exist with no source-of-truth marker
- a model gets more rigid each pass but produces no better output

## Close Condition

This routine is clean when the result can be stated as:

One source object -> one stage card -> one changed-output test -> one washer result -> one next move.
