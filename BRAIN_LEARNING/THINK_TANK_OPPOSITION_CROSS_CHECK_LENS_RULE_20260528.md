## 2. Core Rule

When creating, revising, saving, or judging any of these:

```text
rule
path file
helper
registry
gate
workflow
save route
proof route
house map
candidate system
```

do not judge only from current chat momentum.

The assistant must:

```text
1. Wear the relevant Think Tank / saved concept files.
2. Cross-check neighboring house rules and current proof state.
3. Look for useful outside concepts when the task is architectural, rule-shaped, or systemic.
4. Look at opposite or rejected designs when useful.
5. Extract what those opposite systems optimize for.
6. Decide which rules we do not want.
7. Invert rejected rules into Clean Seed guardrails when the inversion fits.
8. Return to the active seed, boundary, proof, and next clean move.
```

## 4. Why Opposite Ideas Matter

Opposite systems can expose rules faster than friendly systems.

If another build optimizes for:

```text
speed over proof
automation over custody
search before meaning
services before paths
authority by convenience
broad crawling
helper sprawl
one mega-index
hidden state
fake PASS
```

then Clean Seed can invert those into guardrails:

```text
proof before speed
custody before movement
meaning before search
path before door
authority by source/proof
bounded read before scan
helpers stay workers
small path before mega-index
visible receipts
Done is not PASS
```

Opposite ideas are not automatically bad. They are contrast material.

Question:

```text
What does this system protect that we do not want to protect?
What does it risk that we must block?
What useful part can be adapted cleanly?
```

```text

challenge assumptions, expose weak points, think against the plan before the plan fails.
```

House translation:
