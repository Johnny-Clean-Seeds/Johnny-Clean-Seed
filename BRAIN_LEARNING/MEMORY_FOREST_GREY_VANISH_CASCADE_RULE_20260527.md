# Memory Forest / Grey-Vanish Cascade Rule

Date: 20260527

## Status

Candidate memory-hygiene and chat-weight process.

Living process support.
Not doctrine.
Not a new boss.
Not delete authority.
Not transcript/source material.
Not raw-chat erasure.

## Purpose

Retain old chat cleanly without carrying old chat weight.

Plain chunking keeps pieces of old text.
This process keeps the living shape of the ideas.

## Core model

Raw chat becomes source residue.

Important ideas become branches.

Repeated patterns become trunks.

Related concept groups become cross-linked trees.

Saved rules become proof pointers.

Old raw chat can become grey pointer state.

Grey material can vanish from live carry only after its meaning has been safely consolidated.

## Cascade

Raw chat -> extract live ideas -> attach to roots -> merge duplicates -> cross-link touching trees -> assign proof/custody state -> grey raw residue -> vanish only from live carry when safe.

## Memory states

- LIVE: needed for the current work.
- WARM: likely needed soon, but not carried full-size.
- GREY: retained as pointer/branch, not full live text.
- PARKED: saved or held with return trigger.
- PROOF-CUSTODY: cannot vanish because it proves a trail, receipt, decision, or failure.
- CROSS-LINK: connects trees without becoming active by itself.
- VANISHED-FROM-LIVE: removed from active carry because the meaning is consolidated elsewhere.
- BLOCKED: not safe to compress, grey, or vanish yet.

## Guardrails

Vanishing is not deletion.

Do not vanish proof before proof custody.

Do not use this to hide unresolved issues.

Do not replace strong analysis with thin summaries.

Do not flatten relationships that matter.

Do not treat raw chat as trash when it is still evidence.

## Reading method

When returning to old chat, do not reread everything by default.

Read:

current active object -> matching tree root -> branch summaries -> cross-links -> proof pointers -> raw source only if needed.

## Proof test

Test on one recent failure cluster before broad use.

Suggested first test:
file-run / helper-script / proof-custody / chat-weight failure trail.

Expected output:
one root, five to eight branches, cross-links to saved rules, proof pointers, greyed old-chat residue, blocked items, and one short live carry phrase.
