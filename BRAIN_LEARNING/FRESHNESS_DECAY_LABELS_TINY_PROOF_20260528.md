Purpose: test whether a small freshness/decay label set can reduce stale/live confusion across one saved chain without becoming a dashboard, registry, helper, or authority layer.

---

## 1. Clean Thesis

Freshness labels do not create authority.

Freshness labels describe live-use state.

They answer:

```text
Is this object current?
Is this object ready for narrow use?
Is this object watch-only?
Is this object proof-parked?
Is this object stale?
Is this object superseded?
Is this object blocked?
```

Short form:

```text
Freshness labels tell how to carry an object.
They do not tell the house what is true.
```

## 4. Candidate Label Set

Use this minimal set:

| Label | Meaning | Use Case | Risk If Missing |
|---|---|---|---|
| CURRENT | active object or active decision point | work is currently centered here | active state becomes vague |
| READY_FOR_NARROW_USE | proved enough for bounded reuse | small repeated use is allowed | good proof gets buried |
| WATCH | useful but not settled | needs caution or later review | unstable object overpromotes |
| PROOF_PARK | preserved as evidence, not live instruction | old proof remains findable | old logs stay carried live |
| SUPERSEDED | replaced by a later object | old version should not guide new work | outdated object keeps firing |
| STALE | aged or context-distant | read before reuse | old claim acts current |
| NEEDS_REVIEW | cannot be reused without inspection | uncertain or incomplete state | bad reuse |
| BLOCKED | explicitly not allowed forward | unsafe, dirty, or wrong-route object | repeated failed route |
| LOCAL_ONLY | useful locally but not URL/authority | local helper, drop-copy, temporary artifact | local object becomes house authority |

## 6. What This Proves

This tiny proof shows:

1. Freshness labels clarify what to carry live.
2. Old proof can stay findable without staying active.
3. `READY_FOR_NARROW_USE` is different from doctrine.
4. `CURRENT` is temporary, not permanent authority.
5. `WATCH` protects useful but unsettled objects.
6. `PROOF_PARK` reduces log drag without deleting evidence.
7. `BLOCKED` and `SUPERSEDED` are needed to stop repeat failure.

The mechanism works because it separates:

```text
truth authority
proof existence
live carry state
reuse permission
```

Those are not the same thing.

---

## 7. What This Does Not Prove

This does not prove:

- a global freshness system should be built;
- every file should be labeled;
- a dashboard should be created;
- a helper should relabel files;
- freshness labels can replace receipts;
- freshness labels can replace CURRENT_TRUTH_INDEX;
- freshness labels can replace ACTIVE_GUIDES;
- labels can promote a candidate to doctrine;
- labels should be applied retroactively without receipt.

## 8. Pass / Watch / Fail

### PASS

PASS if the labels make the saved chain easier to carry:

```text
current work is clear
parked proof is not carried live
narrow-use objects are reusable but not promoted
watch objects stay cautious
blocked/superseded states are possible
```

### WATCH

WATCH if:

```text
labels multiply
labels become decorative
labels start acting like commands
labels hide uncertainty
labels are applied without receipts
labels duplicate status files
```

### FAIL

FAIL if:

```text
labels become authority
labels replace proof
labels rewrite history
labels trigger broad relabeling
labels start dashboard pressure
labels cause more bloat than they remove
```
