# Clean Seed Rule — Sort Before Judgment

Lock status: RULE CANDIDATE REVIEWED / READY FOR BRAIN LOCK
Lane: brain/learning + house-work method
Active doctrine effect: method rule for large issue lists and boss triage

## Rule

Issue sorting must always filter before final judgment, no matter how large the list is.

Never judge a large issue list as one flat whole.

First:

1. Capture the whole list.
2. Split items into groups.
3. Identify parent bosses versus ghosts, symptoms, tools, checks, causes, helper pieces, and future upgrades.
4. Collapse children under larger seams.
5. Rank only the true parent bosses.
6. Fight only the current highest-leverage boss.

If the list is too large, create more groups or use a tournament/bracket system:

- group items by lane or failure type
- choose each group winner
- compare group winners
- rank the final parent bosses
- park the rest under their parent boss with status

## Purpose

This prevents the house from fighting little ghosts while a larger boss seam remains alive.

The system must not rank raw piles. A raw pile is source material, not a decision surface.

## Mechanical Equivalent

Pile -> Groups -> Parent Bosses -> Ranked Parent Bosses -> One Active Boss -> Parked Children

## Pass Condition

A boss list passes triage only when:

- every item has a parent group or is intentionally marked as a parent boss
- symptoms are not ranked beside root seams
- tools/checks are not mistaken for bosses
- only parent bosses are ranked at the top level
- only one boss is active
- parked children retain their parent link and next-action status

## Failure Condition

This rule fails if the assistant:

- ranks a large raw list without grouping first
- treats every valid issue as a separate main boss
- fights symptoms before identifying the parent seam
- activates too many bosses at once
- loses parked children after grouping
- uses tournament language as ceremony without reducing the list

## Fit Review

Verdict: PASS.

Reason: This rule fits the existing Clean Seed flow. It protects one-boss-at-a-time repair, reduces bloat, keeps source-ore useful, improves boss selection, and strengthens recursive house sorting without changing the core idea.

This rule is a refinement, not a doctrine replacement.
