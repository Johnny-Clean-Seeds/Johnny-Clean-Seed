# Word-Key Clarity Checker Rule

Status: ACTIVE LEARNING RULE
Authority: assistant behavior rule; not command authority

## Purpose

The Bell Logic Gate rings when word-key risk appears.

This rule inspects the words after the bell rings.

Words are keys.

A loose word can become a wrong route.

A vague label can become false authority.

A similar term can split the model into duplicate paths.

## Core Rule

Before saving or using a new important model word, label, status, room name, method name, gate name, proof label, boss name, or command-like phrase, check whether the word is clean.

A clean word must have:

- clear meaning
- job
- lane
- authority boundary
- neighbor relationship
- lifecycle state
- proof/use condition

## Trigger

Use this checker when:

- a new model term appears
- a term sounds similar to an existing term
- a word could become command-like
- a word could affect authority, proof, lifecycle, source, support, active state, or promotion
- a phrase feels vague, weird, broad, overloaded, or too cute
- two terms may mean the same thing
- one term may be doing two jobs
- a status word appears, such as active, current, live, final, installed, promoted, parked, source, proof, pass, done, anchor, boss, gate, map, index, receipt, rule, guide, tool, or command

Do not use this checker for ordinary prose that does not become a route, label, status, or project key.

## Clarity Checks

For each important word/key, ask:

1. What does this word mean here?
2. What job does it do?
3. What lane owns it?
4. Can it command anything?
5. What neighboring words/files can it be confused with?
6. What lifecycle state does it imply?
7. Is it too similar to an existing word?
8. Is it vague or overloaded?
9. Does it preserve source/support/authority separation?
10. What proof or condition allows it to be used?

## Bad Word-Key Signals

Flag the word when:

- it has no defined job
- it sounds active but is only support
- it sounds proven but is only saved
- it sounds installed but is only parked
- it sounds command-like but has no authority
- it duplicates another word
- it hides a lifecycle state
- it mixes source/support/authority
- it makes a receipt sound like doctrine
- it makes a TODO sound like anchor
- it makes a tool sound like command
- it uses final/current/live/pass/proof loosely
- it creates a new route without a sign

## Fix Options

When a word fails clarity:

- define it
- rename it
- narrow it
- move it to the right lane
- mark it support-only/source-only/history-only
- pair it with its neighbor term
- add a non-inference rule
- park it until it is ready
- reject it if it fights the core

## Pairing With Bell Logic Gate

Bell Logic Gate asks:

Is this word clean enough to become a key?

The alarm and checker work together.

Bell rings first.

## Closing Rule

No weird keys.

No vague routes.

No duplicate labels without a reason.

No status words without lifecycle truth.

A word that guides the model must be clear enough to carry authority boundaries without drift.
