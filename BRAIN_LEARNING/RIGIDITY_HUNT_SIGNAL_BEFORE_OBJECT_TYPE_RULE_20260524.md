# Rigidity Hunt / Signal Before Object Type Rule

Status: SUPPORT ROUTINE / ANTI-RIGIDITY / NOT DOCTRINE
Date: 2026-05-24

## Purpose

Prevent the house from becoming so efficient that it classifies strange or weak signals too early.

The problem is not proof discipline, boundaries, or routing. The main rigidity risk is early certainty pressure:

`observe thing -> classify thing -> assign thing -> move thing`

The safer shape is:

`observe thing -> hold the signal briefly -> classify thing -> assign thing -> move thing`

## Placement

Use this after the router names the task family and before Gate Inventory assigns a final object type.

`Router / task family -> Signal Observation Station -> Object Type -> normal gate flow`

This is a station, not a full gate.

## Signal Observation Station

Ask:

1. What is strange, surprising, or repeated?
2. What does not fit the current rooms?
3. What seems absent?
4. What is overloaded or doing another room's work?
5. What feels like a weak signal rather than proof?

Then continue to object typing.

## Signal vs Proof

Signal means an observed pressure, clue, absence, repeated shape, or friction point.

Proof means the claim has evidence, receipt, hash, rerun, or other scoped verification.

Signal is:

- not truth;
- not doctrine;
- not trash;
- allowed to park, watch again, become source ore, or become a future test.

Proof is required before adoption, promotion, PASS language, or authority change.

## Proto-Object Holding

If the object is still unclear, place it in proto-object holding.

Allowed contents:

- fragments;
- strange links;
- partial ideas;
- repeated symbols;
- absence signals;
- unknown pressure;
- possible room overload.

Proto-object holding has no authority. It must leave as one of:

- checkpoint;
- tool;
- card;
- ledger;
- parking item;
- source ore;
- TODO;
- rejected stale;
- proof-first candidate.

## Empty Seat Test

Use this when a run flatlines or keeps returning the same result:

1. What should be here but is not?
2. Which room keeps needing help?
3. Which rule family is overloaded?
4. Which neighboring gate is doing borrowed work?
5. Is the missing thing a gate, checkpoint, tool, card, ledger, parking item, or proof surface?

## Controlled Variant Check

When repeated loops return the same answer, test one small route variation:

- move one checkpoint earlier or later;
- ask one missingness question before proof;
- run Evidence / History before Adoption;
- run Coverage before Evidence;
- compare the result to the normal route.

This is controlled variation, not random drift. If it does not improve the active issue, park or reject it.

## Soft Suit Fit Check

The current route can be worn as a temporary test suit for the active issue.

Ask:

- Does this route help the issue move cleaner?
- Does it make the assistant more capable without hiding uncertainty?
- Does it preserve proof discipline?
- Does it keep local source material local?
- Does it reduce rigidity without making fog?

If yes, apply once to the same issue and rerun the same issue through the changed route.

## Boundary

- Not doctrine.
- Not public authority.
- No ACTIVE_GUIDES rewrite.
- No CURRENT_TRUTH_INDEX rewrite.
- No proof bypass.
- No broad transcript import.
- No raw source text in Git-facing support files.
- No full gate promotion by itself.

## Proof

Proof is a live run where this station prevents premature classification, captures a weak signal, parks unknown material cleanly, or finds a missing room/checkpoint that a normal route missed.
