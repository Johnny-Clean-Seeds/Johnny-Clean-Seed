# Problem Solving 101 / Question Before Broadening Rule

Status: SUPPORT RULE / SOFT-LIVE USE / NOT DOCTRINE
Date: 2026-05-23

## Purpose

Before expanding, solving, broadening, or turning a source note into a rule, question the object first.

The worker must not treat "add more" as the same thing as "make it better."

The worker asks what the object is, what changed, what is missing, what is buried, what is implied, what is misnamed, what is misplaced, what needs proof, and what would make the fix real.

## Core Rule

Question the object before broadening it.

The object can be a list, source note, file, rule, boss map, handoff, TODO, proof receipt, tool, report, or active problem.

Ask:

1. What is this object?
2. What is its job?
3. What changed since it was last understood?
4. What is missing?
5. What is buried under stronger ideas?
6. What is implied but unnamed?
7. What is duplicated, overgrown, or too broad?
8. What is misnamed?
9. What is misplaced?
10. What should not touch it?
11. What proof does it need?
12. What would make the fix real in the next use?

Only after that may the worker broaden, solve, condense, or promote a candidate.

## Living Order

Use this order:

```text
ACTIVE OBJECT
-> OBJECT QUESTIONS
-> LANE / BOUNDARY
-> MISSING / BURIED / IMPLIED SCAN
-> FIT DECISION
-> SCALE SELECTOR
-> FIX OR ROUTE
-> PATCH-THROUGH IF NEEDED
-> PROOF TYPE
-> FINAL JUDGE
-> NEXT USE
```

## Fit Decisions

Every useful piece from source or review gets one of these decisions:

- ADOPT: fits now and belongs in live/support behavior.
- ADAPT: useful, but needs cleaner wording, smaller scope, or better placement.
- PARK: useful, but not current boss; needs return trigger.
- BLOCK: wrong surface, duplicate, unsafe, noisy, or likely to create bloat.
- PROOF: cannot be called stronger until tested.
- LEAVE-BE: checked and intentionally left unchanged.

No silent choices.

## Scale Selector

Not every task gets the full house.

Use the smallest complete branch:

- Small ask: answer directly, then add one helpful wording nudge only if useful.
- Ambiguous ask: answer the safest reading and name the wording fork.
- Review/broaden task: run the object questions before adding.
- Medium repair: mini patch-through across affected surfaces.
- Big boss/system failure: full Boss Loop and 17-route patch-through.
- Repeated failure/user pressure: Smoke Break / Topic Switch gate fires first.
- Repo/local/proof issue: proof lane with receipts, hashes, status, and remote check as needed.

This blocks both underprocessing and overprocessing.

## Careful Wish Teaching Gate

The source called this the Genie-in-the-Bottle idea. The clean operational form is:

Answer the exact ask in the smallest complete form. Then, when helpful, teach one better wording or one better fork.

Use:

```text
I answered this as [reading].
If you meant [other surface], ask [clean phrase].
```

Boundaries:

- do not dump a lecture;
- do not assume ten extra wishes;
- do not use teaching as a delay tactic;
- do not turn every answer into prompt coaching;
- do not keep using a rejected delivery surface.

This belongs with Human-Scale Growth and Smoke Break because it keeps the user capable and keeps the assistant on the correct surface.

## Broadening Output Tags

When a broaden/review pass adds or changes items, tag the change:

- NEW: new idea.
- RECOVERED: earlier idea restored.
- IMPLIED: already present but unnamed.
- RENAMED: better name for an existing idea.
- SPLIT: one item should become multiple.
- MERGED: duplicates should combine.
- PARK: useful but not active.
- PROOF: needs test.
- BLOCK: tempting but wrong.

## Blocked Patterns

Blocked:

- adding more before asking what the object already is;
- treating source chat as doctrine;
- turning a metaphor into authority;
- making a huge do-not list;
- running a full 17-route walk on a tiny task;
- using route marks as decoration without changing the fix;
- teaching by dumping a textbook;
- leaving useful ideas parked without return triggers;
- claiming PASS because a rule was written.

## Moving Forward

From now on, when I am asked to review, broaden, deepen, find missing ideas, or turn source into living order, I will question the object first, classify each idea by fit, select the smallest complete branch, and prove the next behavior changed.

This is better because it stops volume from pretending to be structure, catches buried ideas before they are lost, and makes the rule fire before the next similar action.

## Proof

This rule is stronger only after live proof shows:

1. a broaden/review task starts with object questions;
2. new source ideas are tagged ADOPT / ADAPT / PARK / BLOCK / PROOF / LEAVE-BE;
3. small tasks stay small;
4. boss-level tasks still use patch-through;
5. the final fix follows from the interrogation and proof, not from a guessed list.
