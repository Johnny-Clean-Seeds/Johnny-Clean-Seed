# If The Shoe Fits / Broad Command Washup Rule

Status: SUPPORT RULE / LIVE-USE REPAIR
Date: 2026-05-23
Source: root porch broad-command list plus recurring history/pudding evidence

## Purpose

When the user gives a broad command, a cleanup command, or a "use all the house logic" command, the assistant must not turn the request into a loose pile of notes.

Use the smallest complete recursive branch that can carry the job:

`anchor -> gather -> branch -> fit-test -> clean / rename / drop -> order -> output -> self-proof`

If the shoe fits, wear it.

If it only partly fits, rename or adapt it.

If it does not fit, drop it or park it with a return trigger.

## Live Gate Card

Before acting on a broad house-touching command, name:

- active boss;
- lane and boundary;
- source custody state;
- evidence already in the pudding;
- root versus symptom;
- neighbor rules already present;
- object lifecycle;
- moving-forward rule;
- proof target;
- Final Judge verdict options.

This card is not ceremony. It must change what happens next.

## Sliding List Method

Use a sliding list when material is scattered across notes, TODOs, proof, tools, source ore, and idea bags.

Tabs:

1. Active Boss
2. Evidence
3. Fit Decisions
4. Clean Finals
5. Parked / Return
6. Proof / Judge

Each row needs:

- item;
- source;
- fit decision;
- action;
- proof;
- next trigger.

## Note-Burning Rule

Working notes are allowed while gathering.

Before close, burn dead weight by converting notes into:

- final rule;
- final checklist;
- evidence ledger;
- parked item with return trigger;
- rejected item with reason;
- receipt.

Do not leave orphan scratch when a final object now carries the job.

## Moving Forward

From now on, when a broad command asks for ideas, rules, resources, toolboxes, TODOs, ladders, and house history, I will:

1. anchor the real task;
2. gather source and history;
3. branch the material;
4. fit-test every object;
5. clean, rename, link, park, drop, or block;
6. order by dependency;
7. output a usable packet;
8. prove target files, porch state, and Git state before PASS.

This is better because it fixes loose notes, duplicate rule cards, stale porch files, false PASS, and rule-storage-without-rule-firing.

## Guardrails

- Do not rewrite active guides or CURRENT_TRUTH_INDEX from this card alone.
- Do not restore shed or bridge lanes unless the user explicitly opens that lane.
- Do not run historical save runners just because they were preserved.
- Do not treat source custody copies as new doctrine.
- Do not claim Git completeness until status, commit, push, and origin proof are checked.

## Proof

Proof is changed behavior:

- root porch loose files moved into custody;
- source copy matched canonical rule hash;
- stale runners were preserved but marked historical;
- a tabbed list was created for human use;
- final receipt hashes written;
- Git save checked after commit/push when authorized.
