# Washer Return Habit Loop Rule

Date: 2026-05-24

Status: active support rule / repeated wash cadence.

## Rule

Use the washer as a return habit:

`walk -> list -> burn -> soft-suit walk -> outside/creative challenge -> synthesize -> wash -> return to work`

The washer is not only a final polish. It is the habit that turns a messy pass into a smaller,
proved, routed pass.

## Trigger

Use this loop when:

- the user asks for a broad house review, full wash, deep dive, creative room pass, or lock/save;
- a first review list is big, foggy, or emotionally loud;
- a new rule candidate is good but not yet tested against the house;
- outside research has been gathered and needs local fit judgment;
- a note needs burning or a rope cut;
- a final lock/save is about to happen;
- the same issue keeps returning under new wording.

## Loop

1. Walk: inspect the current state, root/drop zone, Git state, active suit, and relevant rooms.
2. List: name issues, proof, status, and first disposition.
3. Burn: convert deadweight notes into rules, checklists, evidence, parking, rejection, blocked
   items, source custody, or receipts.
4. Wear: put only the strongest one to three candidates in the soft suit.
5. Walk again: revisit the house with the list in the soft suit and name what changes.
6. Challenge: go to creative/outside/neighbor rooms and look for contrary methods or stronger
   patterns.
7. Synthesize: make a third local shape that fits the house without importing a foreign boss.
8. Wash: test source, authority, proof, placement, quiet-down, return trigger, Git/local safety, and
   final destination.
9. Return: leave one active boss or a clear closeout, not a new fog bank.

## Stop Condition

Stop the washer when two passes produce the same route and no new controlling conflict appears:

- same active boss;
- same destination;
- same blocked or parked items;
- same proof need;
- same quiet-down condition;
- Git/local state verified if files changed.

If a new controlling conflict appears, run one more smallest useful pass. If the conflict requires
user authority, mark BLOCKED / ASK instead of washing forever.

## Scheduling

Prefer event-based scheduling over clock-based automation unless the user asks for automation.

Run a micro-wash:

- before cut, burn, promotion, parking, or rejection;
- before broad staging, commit, push, or final lock;
- after a creative room pass;
- after a dirty tree or root-drop discovery;
- after a repeated user correction.

Run a full wash:

- for broad house reviews;
- before active-guide or current-truth changes;
- when trust depends on many rooms;
- when the final state will be reused by a future assistant.

## Dirty Pattern

- Starting over after every pass instead of returning to the last verified point.
- Washing forever after the route is stable.
- Treating outside research as authority.
- Leaving the list bigger than it started.
- Calling the washer clean without note burn, parked triggers, and proof of Git/local state.
