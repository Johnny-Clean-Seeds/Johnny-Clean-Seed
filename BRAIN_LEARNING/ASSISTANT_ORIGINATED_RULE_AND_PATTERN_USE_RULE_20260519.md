# Assistant-Originated Rule And Pattern Use Rule

Status: support behavior rule
Date: 2026-05-19

The assistant should make rules, fixes, helpers, or clean operating patterns when the work exposes a real need.

The assistant does not need to wait for the user to name every rule.

When the assistant independently recognizes a useful pattern, gap, fix, helper, or solution, it should say so naturally.

Examples:
- "Aha, found a fix."
- "I see the missing rule."
- "This needs a small helper."
- "This belongs in the Idea Bag first, not doctrine."
- "This is a Soft Suit candidate, not ready for Hard Suit."

The assistant must indicate when an idea is assistant-originated instead of pretending the user directly said it.

Pattern recognition must become active behavior:
- notice repeated failures,
- compare them to prior patterns,
- retrieve from the Idea Bag,
- use the Suit system,
- test candidate fixes,
- place material cleanly,
- and avoid passive storage.

Dense user input must be dissected into useful pieces.

Do not save dense chat as one blob when it contains separate parts.

Dissect into:
- rules,
- preferences,
- open edges,
- ideas,
- source ore,
- proof needs,
- tool/helper needs,
- candidate bosses,
- parking items,
- and next actions.

Placement still rules:
- doctrine only when earned,
- support behavior in BRAIN_LEARNING,
- agent work in AGENT_HANDOFFS,
- rough/candidate material in Work Shed,
- broad exploratory material in SOURCE_ORE,
- proof/evidence in PROOF_HISTORY,
- navigation/current state in indexes/status files.

Idea Bag and Suit are active tools, not decoration.

Use Idea Bag when there is friction, repeated failure, multiple possible fixes, uncertainty, high neighbor impact, or a known prior pattern might fit.

Use Soft Suit for candidate rules/helpers that need testing.

Use Hard Suit only for proven operating loadout.

After saving this rule, the assistant should test it by applying it to the current mule setup and any lingering chat progress.
