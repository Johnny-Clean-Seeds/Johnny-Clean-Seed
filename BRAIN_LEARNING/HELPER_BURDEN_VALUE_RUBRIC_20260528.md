# Helper Burden / Value Rubric

Date: 2026-05-28  
Status: RUBRIC / EVALUATION MODEL / NOT A HELPER BUILD  
Authority: NOT doctrine; NOT ACTIVE_GUIDES; NOT CURRENT_TRUTH_INDEX  
Source trigger: user correction that the real task is measuring how much helpers are actually helping in data usage, savings, burden, and proof value.  
Placement: BRAIN_LEARNING  
Boundary: no scanner, no dashboard, no automation, no helper build.

---

## 1. Core Correction

The question is not only:

```text
Does the helper use less data?
```

The better question is:

```text
Where does the helper move burden, and is that move worth it?
```

A helper may not reduce total work immediately.

A helper may convert messy live-chat work into structured local proof work.

That can be good.

That can also be bloat.

The value judgment depends on whether the burden moved to the right place.

---

## 2. Full Working Concept

```text
HELPER VALUE IS NOT ONE SCORE.

A helper can:
1. save chat weight,
2. add local file weight,
3. reduce user action weight,
4. increase setup weight,
5. prevent dirty failure,
6. create proof value,
7. reduce cognitive load,
8. create maintenance debt,
9. improve handoff flow,
10. become toil if overused.

Therefore helper value must be judged as a burden exchange:

HELPER VALUE =
    proof gained
  + failures prevented
  + repeated chat removed
  + user actions removed
  + future reuse gained
  + handoff clarity gained
  - scripts generated
  - reports generated
  - commands copied
  - repair loops added
  - cleanup debt added
  - cognitive burden added

A helper is good when it moves burden out of fragile live chat and into durable local proof.

A helper is bad when it turns a small task into machinery.

A helper is excellent when it disappears into the workflow:
the user runs less,
the assistant explains less,
the house proves more,
and the next move is clearer.
```

---

## 3. Formula

Short formula:

```text
HELPER VALUE = SAVINGS + SAFETY + REUSE - TAX
```

Expanded formula:

```text
SAVINGS =
    less chat
  + less rereading
  + fewer repeated decisions
  + fewer manual checks
  + less path confusion
  + less live context carry

SAFETY =
    fewer dirty passes
  + better proof
  + better boundaries
  + caught parser/runtime/staging problems
  + clean close certainty

REUSE =
    template value
  + repeated shape value
  + future task reduction
  + amortized setup cost

TAX =
    scripts
  + reports
  + receipts
  + user commands
  + Code Gate cycles
  + repair loops
  + local clutter
  + cleanup debt
  + cognitive burden
```

The helper helps only when:

```text
SAVINGS + SAFETY + REUSE > TAX
```

## 4. Six Ledgers

### 4.1 Cost Ledger

Question:

```text
What did the helper add?
```

Count:

- generated scripts;
- script bytes;
- Code Gate reports;
- report bytes;
- receipts;
- receipt bytes;
- status updates;
- pasted report characters;
- user commands;
- repair loops;
- extra files;
- cleanup burden.

This is the helper tax.

---

### 4.2 Savings Ledger

Question:

```text
What did the helper remove?
```

Count or judge:

- repeated explanation removed;
- old chat rereading avoided;
- manual Git checking avoided;
- path retyping avoided;
- proof restatement avoided;
- next-route arguing reduced;
- context carry reduced;
- assistant recomputation reduced;
- user confusion reduced.

This is the visible savings.

---

### 4.3 Safety Ledger

Question:

```text
What failure did the helper prevent or catch early?
```

Look for:

- parser failure caught before run;
- target run failure caught before save;
- dirty repo before save;
- ignored path / wrong lane;
- missing expected file;
- unexpected staged file;
- push mismatch;
- final status not clean;
- authority boundary drift;
- false PASS avoided.

This is often the biggest value.

One prevented dirty PASS can be worth many extra bytes.

---

### 4.4 Flow Ledger

Question:

```text
Did the handoff improve?
```

Handoff chain:

```text
assistant sends file/code
user runs code
report comes back
assistant judges report
assistant sends next code/fix/action
```

Measure:

- was the next move obvious?
- did the report contain a copy-back block?
- did the assistant need less explanation?
- did PASS lead directly to next route?
- did FAIL lead directly to narrow fix?
- did the user have fewer unclear steps?
- did the work stay in one lane?

This is the Soft Suit fit value.

---

### 4.5 Reuse Ledger

Question:

```text
Was this one-time cost reusable?
```

Separate:

```text
one-time build cost
per-use cost
per-use savings
break-even point
```

Break-even:

```text
break-even uses = build cost / savings per future use
```

Example:

```text
If a helper costs 20,000 characters to build
and saves 4,000 characters per future task
then it breaks even after 5 uses.
```

A one-use helper must have high safety value to justify itself.

A reusable helper can be expensive if it actually reduces future burden.

---

### 4.6 Debt Ledger

Question:

```text
What future burden did the helper create?
```

Watch for:

- stale helper scripts;
- loose downloads;
- unused templates;
- repeated receipts that are not read;
- helper identity confusion;
- unnecessary dashboards;
- new rules that do not fire;
- more local cleanup;
- more explanation required later;
- helper becoming the main work.

This is helper debt.

## 5. Helper Levels

Use the smallest helper level that proves the task.

```text
Level 0 — No Helper
Use for tiny judgment, naming, simple answer, concept work.

Level 1 — Checklist / Rubric
Use when deciding whether a helper is needed.

Level 2 — Tiny Read-Only Counter
Use when simple counts or sizes answer the question.

Level 3 — Code Gate Script
Use when local execution needs guarded proof.

Level 4 — Save Script With Receipt
Use when Git/proof/save/receipt/custody matters.

Level 5 — Reusable Template
Use only after repeated use proves the pattern.

Level 6 — Helper/System Build
Use only when repeated evidence proves lower levels cannot carry the job.
```

The key question:

```text
Was the helper level proportional to the task?
```

Most helper waste comes from using Level 4 or Level 5 when Level 1 or Level 2 would have worked.

---

## 6. Helper Verdict Labels

Use these labels after a task.

```text
HELPER SAVED WORK
The helper reduced total burden and improved proof.

HELPER TRANSFERRED WORK
The helper moved work from chat to local proof. Useful only if the proof mattered.

HELPER ADDED PROOF WORTH THE COST
The helper added cost, but the safety/proof gain justified it.

HELPER OVER-SERVED THE TASK
A smaller helper level would have worked.

HELPER CAUSED BLOAT
The helper added machinery without enough savings, safety, or reuse.

HELPER PREVENTED DIRTY FAILURE
The helper caught a failure that would likely have produced bad proof or false PASS.

HELPER BECAME THE WORK
The helper stopped serving the task and became the task.
```

## 7. Scorecard

Use only when useful. Do not score every tiny task.

```text
1 = poor
2 = weak
3 = usable
4 = strong
5 = excellent
```

| Dimension | Score | Evidence |
|---|---:|---|
| Task fit |  | Was helper level proportional? |
| Chat savings |  | Did it reduce live explanation/context? |
| Local cost |  | How much script/report/receipt weight was added? |
| User action savings |  | Did user do fewer steps? |
| Safety gain |  | Did it catch/prevent failure? |
| Proof gain |  | Did it create clean proof no-helper would not? |
| Flow gain |  | Did handoff improve? |
| Reuse value |  | Will this help future tasks? |
| Debt created |  | What cleanup/complexity remains? |
| Net value |  | Did value exceed tax? |

## 9. Minimum-Helper Rule

Before using code, ask:

```text
What is the smallest helper level that proves this task?
```

If Level 0 works, answer in chat.

If Level 1 works, use a checklist/rubric.

If Level 2 works, use a tiny read-only counter.

If Level 3 is needed, use Code Gate.

If Level 4 is needed, use save script with receipt.

If Level 5 or 6 seems needed, demand repeated evidence first.

This prevents helper bloat.

---

## 10. Copy-Paste Method

For user-run code, use the copy-paste method:

```text
1. one file link
2. one Code Gate command
3. one expected close marker
4. user pastes report back
5. assistant judges report
6. PASS -> next code
7. FAIL -> narrow fix code
8. no code needed -> next clean action
```

Do not mix normal explanation into the code box.

Do not ask the user to paste tool output back into PowerShell.

Output is for chat/proof.

Commands are for PowerShell.

Keep them separate.

## 11. Dead Weight To Drop

```text
DROP:
- “less data” as the only goal
- big all-knowing measurement script
- one giant helper score
- measuring bytes without measuring proof
- assuming helper always helps
- assuming no-helper is always lighter
- treating scripts as value by themselves
- building dashboards before proof
- continuing helper work after the helper became the work
```

## 13. Hard Verdict

```text
The helper is not automatically a savings device.
It is a burden-routing device.

Use it when burden belongs in durable proof.
Do not use it when burden belongs in simple judgment.
```
