# Adapt / Adopt Critique Learning Gate

Date: 2026-05-19.
State: active learning behavior rule.
Authority: operating behavior rule, not doctrine install.
Starting base: main @ 14c762b.

## Core Rule

Critiques, proof history, notes, ideas, handy bags, bitstrings, receipts, failed runs, successful runs, outside-review reports, old check scripts, and parked candidates are growth material.

They are not passive archives.

They must be used as learning material when judging new critiques, adapting patterns, selecting fixes, classifying lanes, and proving behavior.

## Main Law

Do not blindly copy the newest critique.

Do not let latest report win by freshness alone.

Learn the useful pattern behind the input, reshape it to Mr.Kleen's lanes and proof flow, test it, then lock/save only after fit and proof.

## Sources To Pull From

- outside critiques and reviewer reports,
- proof history,
- receipts,
- failed saves,
- successful saves,
- notes and ideas,
- source ore,
- handy bags,
- bitstrings and sorting reviews,
- parking lanes,
- old check scripts,
- current status and anchors.

## Learning Loop

1. Intake boundary: classify the input as user instruction, source ore, proof, critique, tool candidate, or command authority.
2. Retrieve related house material: proof/history/notes/ideas/handy bags/bitstrings/status before judging the new input alone.
3. Extract atomic pieces: claims, triggers, patterns, warnings, tools, rejects, proof gaps, wording fixes.
4. Find the useful pattern: identify what the input is really teaching.
5. Sort each piece: adopt, adapt, park, reject, or test.
6. Map lanes: decide which room/lane the piece belongs to and what neighbors it touches.
7. Fit check: compare against core, authority, boundary, proof need, current load, and existing patterns.
8. Install small: choose the smallest behavior-changing card, rule, gate, template, or checklist if needed.
9. Prove: test against a recent failure or next live task.
10. Lock/save only after proof: update Mr.Kleen files, receipt, status, commit, push, and verify clean state.

## Labels

Adopt: fits the house with minimal reshaping.
Adapt: useful pattern, but must be reshaped for house lanes and proof flow.
Park: useful later, not now.
Reject: wrong, too heavy, unsafe, redundant, or mismatched.
Test: promising but unproved.

## Mimic Rule

Mimic the pattern, not the costume.

Copying the surface of an outside method can add bloat.

Adapting the working mechanic can improve the house.

## Behavior Test

A critique is not learned until it changes behavior at the point of work.

Good signs:

- it catches a failure earlier,
- it prevents wrong-lane movement,
- it improves classification,
- it strengthens proof,
- it reduces fog,
- it makes a smaller cleaner next move possible.

## Boundary

No doctrine install.
No active guide rewrite.
No runtime automation.
No automatic adoption from any critique.
No blind push of parked material.
