## Purpose

The Intake Gate System is a bounded selected-object intake washer.

It turns messy input into keyed chunks, relation piles, ledger decisions, map routes, proof receipts, operator/helper handoffs, and one next action.

It does not broad crawl, execute, promote, delete, or fake PASS.

## Core rule

Intake does not decide final destiny alone.

Intake opens, keys, chunks, groups, and prepares. Word Gate clarifies. Decision tables choose candidate routes. Ledger records. Map routes. Specialist gates test. Proof checks. Final Judge closes or blocks.

## Default boundary

Selected object only. Read/report first. No Git. No move/delete. No network. No broad crawl. No doctrine promotion. No ACTIVE_GUIDES. No CURRENT_TRUTH_INDEX. Output local-only until explicitly routed.

## Gate stack V1.1

0. Front Door / Intake Authority Shell — locks object, source, mode, lane, boundary, proof level, allowed actions, blocked actions, output surface, and close condition.
1. Source Custody Gate — captures origin, source hash, actor, raw/source-heavy status, local-only/Git-safe status, and source-ore status.
2. Object Shape Gate — classifies text, chat paste, transcript, log, receipt, code, command, error output, rule idea, gate idea, map/ledger item, handoff, package/folder, mixed object, or unknown.
3. Intake Data Contract Gate — checks expected fields, metadata, quality expectations, transformation limits, and missing context.
4. Context Carry Gate — carries WorkKey, TraceId, SourceKey, ObjectKey, ParentKey, SourceHash, actor, boundary, gate, risk, proof pointer, and next surface.
5. Helper Capability Gate — checks what the next helper can read, write, must not touch, proof needed, stop condition, and approval need.
6. Orientation Surface Gate — places small support orientation in the right files so future helpers can see the route without treating it as doctrine.
7. Chunk Gate — chunks only when routing, proof, retrieval, or decision quality improves.
8. Chunk Quality Gate — checks chunk health.
9. Word Gate — checks authority words, action words, scope words, raw/source words, ambiguous house words, room collisions, and route-sensitive phrases.
10. Entity / Object Extraction Gate — extracts paths, filenames, hashes, commits, scripts, gate names, proof receipts, dates, actors, commands, URLs, branches, errors, and tools.
11. Relative Pile Gate — groups chunks by relation.
12. Decision Table Gate — turns facts into repeatable route decisions.
13. Code Branch Gate — routes code/commands/scripts/config/terminal output to Code Gate or blocks until approval.
14. Risk / Blast Radius Gate — labels LOW, MEDIUM, HIGH, or BLOCKED.
15. Backpressure / Choke Regulator Gate — slows, batches, parks overflow, carries key not furniture, or stops with return point.
16. Duplicate / Stale / Data Waste Gate — marks live carry, proof trail only, superseded, duplicate, stale watch, burn from live carry, keep as receipt, or unknown compare. It never deletes.
17. Ledger Fit Gate — decides whether a chunk becomes no ledger, intake event, pile row, decision row, proof pointer, incident row, self-reflection row, route row, status row, blocked row, or return-trigger row.
18. Map Route Gate — routes piles to the correct next surface.
19. Dead-Letter / Unknown Gate — creates recoverable custody for unclear, risky, source-missing, proof-missing, conflict-route, or choke-stopped chunks.
20. Handoff Gate — builds a compact handoff for the next helper or operator.
21. Readiness Review Gate — checks whether the next helper can proceed.
22. Operator Approval Gate — fires when the next move crosses a boundary.
23. Feedback Click Gate — records when the method starts working in practice without promoting it.
24. Balance / Wobble Gate — checks whether the system is upright or overreaching.
25. Escalation Gate — routes to operator, Code Gate, Word Gate, Proof Gate, Final Judge, parking, or blocked lane when the current helper lacks authority/context/capability/proof/safety.
26. Runbook / Playbook Gate — writes the minimum repeatable procedure after proof.
27. Proof Gate — checks every route claim.
28. Output Gate — writes predictable outputs only.
29. Final Judge Gate — checks role fidelity, chunk fate, route proof, boundary, unknown return triggers, code routing, raw source protection, and one next action.

## Orientation Surface rule

Support files should carry small orientation blocks early when useful, but those blocks must be labeled as support/candidate/local orientation when appropriate.

The file does not need to worry about "learning." Its job is to grow and do its current job. Repeated clean exposure to context, boundaries, proof, and handoff lets the click happen naturally.

## Manual test protocol

Before any scripted intake helper, run one manual intake test on one selected mixed object containing a rule idea, one command or code block, one error output, one proof/hash line, one next-action request, and one loaded word such as all, save, final, proof, crawl, move, run, or PASS.

Manual outputs:

- INTAKE_CONTRACT
- CHUNK_INDEX
- PILE_LEDGER
- WORD_KEY_REPORT
- CODE_BRANCH_REPORT
- MAP_ROUTE_TABLE
- DEAD_LETTER_REVIEW if needed
- HELPER_READINESS_REVIEW
- INTAKE_OPERATOR_HANDOFF
- PROOF_RECEIPT
- NEXT_ACTION_CARD
- FINAL_JUDGE

Pass condition:

Every chunk has a fate. Every route has a reason. Every risk has a boundary. Every unknown has a return trigger. Every helper-facing next action has context, capability, proof, stop, and handoff. Code is not executed. No broad crawl. No save. One next action.
