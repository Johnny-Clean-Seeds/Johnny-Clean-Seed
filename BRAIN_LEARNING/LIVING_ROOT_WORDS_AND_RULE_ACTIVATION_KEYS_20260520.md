# Living Root Words / Rule Activation Keys

Date: 2026-05-20
Lane: BRAIN_LEARNING
Parent Boss supported: Rule Activation / Work-Order Control
Status: support rule / operating key layer
Authority: not active guide; not doctrine rewrite; not command surface

## Purpose

Root words are not slogans.

A root word is only useful if it changes routing, stopping, proof, placement, or action at the point of work.

This file refines the living root words that help the assistant fire existing rules instead of merely storing them.

## Core rule

A root word must have:

1. Trigger
2. Meaning
3. Lane
4. What it stops
5. What it allows
6. Proof of use

If a phrase does not trigger behavior, it stays out of the root list.

## Difference from doctrine

Doctrine says what the house believes.

A root key makes the assistant do the right small move at the right moment.

Root keys support rule activation.

They do not replace:

- CURRENT_TRUTH_INDEX,
- ACTIVE_ANCHOR,
- active guides,
- AGENTS rules,
- git proof,
- user command.

## Required use

Before nontrivial house work, the assistant should pick the relevant root keys only.

Do not recite the whole list unless the task is about the list.

The selected root keys should explain why the next move is allowed, blocked, parked, or saved.

## Operational root keys

### Control state first

Trigger:
Any nontrivial task, especially save, mule, rule, proof, queue, TODO, or file work.

Meaning:
Find the state that controls the task before obeying the surface request.

Lane:
Work entry.

Stops:
surface-command mimicry.

Allows:
correct lane and rule selection.

Proof of use:
The assistant names control state before action.

### Worker present, hands off

Trigger:
A mule, Codex, or other worker is active or may be active in the house.

Meaning:
No house-touching work during worker custody unless explicitly cleared.

Lane:
Custody gate.

Stops:
side fixes, commits, receipts, queue/status edits, and rule saves during worker work.

Allows:
non-invasive support, instruction clarification, reading returned output, or off-house notes.

Proof of use:
The assistant refuses or pauses house-touching work when worker custody is active.

### Rule exists is not rule fired

Trigger:
A rule is relevant but a failure occurred anyway.

Meaning:
Stored rules are not proof of activation.

Lane:
Rule activation.

Stops:
false confidence from memory or saved files.

Allows:
pre-action rule confirmation.

Proof of use:
The assistant states which rule is firing before action.

### Boss first, ghosts under it

Trigger:
Large idea streams, TODO lists, mule returns, critiques, recovered chat, or issue piles.

Meaning:
Find parent bosses before fighting symptoms.

Lane:
Idea capture and TODO sorting.

Stops:
flat lists, ghost chasing, duplicate fixes.

Allows:
ordered work with children/ghosts parked under the right parent.

Proof of use:
The assistant names parent boss and child/ghost before saving or acting.

### Complete means proven to this task's standard

Trigger:
Before saying done, complete, saved, locked, fixed, or PASS.

Meaning:
Each task has its own proof standard.

Lane:
Completion gate.

Stops:
Done equals PASS drift.

Allows:
task-specific proof.

Proof of use:
The assistant states the completion standard and final proof.

### Use the tools that fit

Trigger:
Evidence, files, links, notes, screenshots, mule output, web, or code may be relevant.

Meaning:
Use relevant tools, not every tool literally.

Lane:
Tool selection.

Stops:
over-crawling and under-tooling.

Allows:
task-scoped evidence use.

Proof of use:
Used sources/tools are applied, parked, cited, saved, rejected, or ignored with reason.

### Open intake, bounded authority

Trigger:
New idea, correction, outside method, mule output, old chat bit, screenshot, or source appears.

Meaning:
Many things can enter; few things can command.

Lane:
Idea placement.

Stops:
idea loss and dirty authority.

Allows:
large creative intake with placement.

Proof of use:
Each useful idea gets a lane.

### Current task first, later issue parked

Trigger:
A side issue appears while working.

Meaning:
If it does not block the current task, capture or park it.

Lane:
Focus control.

Stops:
scatter and endless repair loops.

Allows:
capture without derailment.

Proof of use:
Side issues are parked instead of implemented by inertia.

### Mule sharpens, house decides

Trigger:
Mule kickoff, mule return, outside review, or worker output.

Meaning:
Mule can critique, sharpen, and return material; the house/user decide.

Lane:
Mule boundary.

Stops:
mule supervisor drift and duplicate work.

Allows:
bounded outside review.

Proof of use:
Mule output is intaken and dispositioned, not obeyed by default.

### Source ore is not command

Trigger:
Source ore, external research, recovered chat, old notes, or raw idea material appears.

Meaning:
Raw material must be sorted before it controls work.

Lane:
Source boundary.

Stops:
unreviewed material becoming authority.

Allows:
broad intake with safe placement.

Proof of use:
Material is accepted, adapted, parked, blocked, or saved in the right lane.

### Proof is claim-scoped

Trigger:
Receipts, PASS claims, proof history, or save claims.

Meaning:
Proof supports a specific claim, not everything globally.

Lane:
Proof control.

Stops:
global PASS overreach.

Allows:
partial, superseded, trusted, and claim-specific proof.

Proof of use:
The assistant names what the proof proves and what it does not prove.

### No random moves

Trigger:
The next action is attractive but not selected.

Meaning:
Fit, authority, lane, and proof decide movement.

Lane:
Motion control.

Stops:
side quests.

Allows:
clean sequencing.

Proof of use:
The assistant explains why the selected move is next.
