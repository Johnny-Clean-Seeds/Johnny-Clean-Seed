# Hold Signal Before Shape Soft Suit Candidate

Status: SOFT SUIT CANDIDATE / TINY UNCERTAINTY BEHAVIOR / NOT DOCTRINE
Date: 2026-05-24

## Purpose

Prevent the house from rushing identity.

The current rigidity source is not mainly bad rules. It is early identity pressure:

`signal -> rule -> gate -> test -> candidate -> save -> proof -> next action`

The safer shape is:

`signal -> hold -> compare -> name only after pressure repeats`

## Core Question

Before naming a new object, ask:

`Am I trying to make this clean before I have let it be strange?`

## Trigger

Use this candidate when:

- a strange signal appears;
- the house wants to immediately classify a signal as a rule, gate, test, candidate, save, proof, or next action;
- a model, helper, or worker sounds plausible but unproven;
- a symbolic structure appears, such as cube, atom, ring, map, city, or grammar;
- a repeated flatline may be caused by the route becoming too familiar;
- Gate Inventory has not yet named the right object type.

## Behavior

1. Hold the signal as signal.
2. Do not call it proof.
3. Do not call it doctrine.
4. Compare it against neighbor rules, prior watch items, and current route pressure.
5. Ask whether the signal repeats across at least two surfaces.
6. If it repeats, give it a provisional shape.
7. If the shape is still unclear, park it in proto-object holding.
8. Let Gate Inventory decide whether it is a gate, checkpoint, tool, card, ledger, parking item, proof surface, or reject.

## Relationship To Existing House

This candidate narrows and renames the useful part of:

- Rigidity Hunt / Signal Before Object Type;
- Gate Family Checkpoint Grammar;
- Gate Inventory / Coverage;
- Worker Symptom Diagnostic;
- Simplicity Build Compression;
- Soft Suit Patch-Through.

It does not replace them.

## Clean Output

Allowed outputs:

- WATCH AGAIN;
- proto-object holding;
- provisional candidate;
- Gate Inventory label;
- parking item with return trigger;
- same-issue rerun request;
- support note.

Blocked outputs:

- direct doctrine;
- active-guide rewrite;
- current-truth rewrite;
- promotion;
- final PASS;
- proof claim from signal alone;
- new full gate from one interesting phrase.

## Proof

Proof is a live run where this candidate prevents premature naming and produces a cleaner result than the old route.

Minimum proof:

1. A signal was held before shape.
2. A comparison surface was named.
3. Gate Inventory or neighbor rules decided the object type.
4. The same issue reran cleaner or parked cleaner.
5. The result stayed support-only unless later proof opened promotion.

WATCH AGAIN.
