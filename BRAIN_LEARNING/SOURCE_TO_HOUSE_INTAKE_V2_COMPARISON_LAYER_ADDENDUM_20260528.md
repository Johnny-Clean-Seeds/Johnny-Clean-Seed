## 4. First Comparison Layer — Review and Decision Methods

### 4.1 ADR / Decision Record

Use for:

Capturing the chosen decision, context, rejected alternatives, consequences, and possible supersession.

Add to report:

Decision Record section.

Required questions:

- What decision is being made?
- Why one mechanism extraction report?
- Why not all 24 files?
- Why not direct rules?
- Why not helper build?
- What are the consequences?
- What would supersede this later?

### 4.2 Premortem / Failure-First Pass

Use for:

Assuming the report failed later and asking why.

Add to report:

Failure-first pass.

Prompt:

It is two weeks later and this report caused bloat, confusion, or wrong authority. What caused the failure?

Likely causes:

- ADOPT became active install;
- ADAPT became rule pile;
- proof support became authority;
- raw chunks entered Git;
- report ended with too many next routes;
- helper build happened too early;
- save target went into ignored path;
- candidate language became doctrine language.

### 4.3 FMEA / Failure Mode Register

Use for:

Structured failure control.

Add to report:

Failure Mode Register.

Columns:

- failure mode;
- cause;
- effect;
- detection sign;
- prevention;
- verdict.

Example:

Failure mode: ADAPT becomes rule pile.  
Cause: useful ideas feel urgent.  
Effect: bloat and premature authority.  
Detection sign: separate ADAPT rule names appear.  
Prevention: compress ADAPT into pressure themes.  
Verdict: BLOCK / COMPRESS.

### 4.4 TRIZ / Contradiction Scan

Use for:

Handling opposing needs without accepting a crude tradeoff.

Current contradictions:

- narrow vs complete;
- source proof vs no raw chunks;
- future usefulness vs no bloat;
- pressure test vs no stall;
- save durability vs no premature authority;
- orientation map vs no second authority spine;
- candidate extraction vs no rule installation.

Add to report:

Contradiction Scan.

Output:

Name the contradiction, name the clean resolution, name what is blocked.

### 4.5 Alternatives Considered

Use for:

Preventing the chosen method from becoming automatic.

Compare at least:

- one narrow mechanism extraction report;
- all-24-file archive;
- direct rule pack;
- helper-built ledger;
- do-nothing / park;
- build Keyring first.

Output:

Chosen route plus rejected alternatives and why.

### 4.6 Kill Criteria

Use for:

Stopping role drift early.

Kill criteria:

- raw chunks appear;
- ADAPT becomes separate rules;
- helper/code appears without evidence need;
- save language outruns proof;
- one next route becomes many;
- proof support becomes authority;
- report starts rewriting authority files;
- report becomes all-24-file summary.

### 4.7 Supersession Rule

Use for:

Allowing better future methods without rewriting history.

Rule:

If a better method is later found, do not erase this addendum. Create a superseding decision record and link back.

## 5. Second Comparison Layer — Systems and Learning Methods

### 5.1 Context Classifier

Use for:

Choosing the right work shape before deciding the method.

Context types:

- clear;
- complicated;
- complex;
- chaotic;
- disorder.

Current object classification:

Complicated plus complex.

Reason:

There are known parts and rules, but interactions between mechanisms, proof, authority, route, and future house structure are not fully predictable.

Implication:

Do not claim final best method. Use best current candidate method with proof route.

### 5.2 A3 Problem Spine

Use for:

Keeping the report as a clean problem story, not a pile of ideas.

Add micro-section:

- current condition;
- target condition;
- root cause;
- countermeasure;
- proof check;
- follow-up.

Current condition:

Focused packet is clean but contains many useful objects that could become bloat if saved directly.

Target condition:

One narrow mechanism extraction report with candidate mechanisms, compressed ADAPT pressure, proof support, and one next route.

Root cause:

Too much project value can arrive at once without enough role separation.

Countermeasure:

Use Chuck Norris pressure plus comparison layer before report build.

Proof check:

Report stays candidate-level and avoids raw chunks, authority rewrite, helper build, and ADAPT rule pile.

### 5.3 Pugh-Style Decision Matrix

Use for:

Comparing options against criteria instead of choosing by vibe.

Alternatives:

- narrow mechanism report;
- all-24 archive;
- direct rule pack;
- helper ledger;
- do-nothing / park;
- build Keyring first.

Criteria:

- authority safety;
- source custody;
- proof value;
- bloat risk;
- retrievability;
- speed;
- future reuse;
- weak-PC friendliness;
- save cleanliness;
- next-route clarity.

Output:

A compact matrix or scored comparison.

Expected likely winner:

Narrow mechanism report with comparison and pressure layers.

### 5.4 Dependency / Neighbor Map

Use for:

Preventing isolated saves that disturb neighbor concepts.

Map relationships among:

- House Keyring Graph;
- Center Room / Common Room;
- Dead Chat Shed;
- Wayfinding save-selection;
- Wayfinding event-chain;
- Proof-memory system;
- Freshness / decay labels;
- Helper confidence ladder;
- Helper output shape contract;
- Lane-Locked Manifest Staging;
- Source-to-House Intake V2.

Ask:

- Which mechanisms depend on which?
- Which should not be promoted first?
- Which are support surfaces?
- Which are proof surfaces?
- Which are orientation surfaces?
- Which can stay candidate-only?

### 5.5 Governing-Assumption Audit

Use for:

Finding hidden assumptions that might be wrong.

Assumptions to test:

- more pressure equals cleaner;
- saving means progress;
- one report is enough;
- ADOPT means close to active;
- ADAPT means not important yet;
- the house already has the right rooms;
- BRAIN_LEARNING is always the right candidate lane;
- no helper is always better;
- comparison prevents bloat by itself;
- every useful idea needs a named section.

Output:

Keep, revise, or watch each assumption.

### 5.6 Maturity / Evolution Labels

Use for:

Preventing immature ideas from wearing mature clothes.

Labels:

- source ore;
- candidate;
- pressure-tested candidate;
- proof-ready;
- active pattern;
- saved rule;
- authority.

Apply to each mechanism.

Default for this report:

Candidate or pressure-tested candidate only.

Blocked:

No mechanism jumps to saved rule or authority because it is well-worded.

### 5.7 Readiness Review

Use for:

Checking whether the report is ready to save/use.

Readiness questions:

- Is the object locked?
- Is source custody clear?
- Are boundaries explicit?
- Are alternatives compared?
- Are dependencies named?
- Are assumptions checked?
- Are failure modes named?
- Are proof needs clear?
- Is one next route named?
- Is authority language bounded?
- Is save location clean?
- Is final status honest?

### 5.8 OODA-Style Re-entry Loop

Use for:

Ensuring the report does not become a dead artifact.

Re-entry loop:

Observe:
What did the report reveal?

Orient:
Which mechanism is strongest and where does it fit?

Decide:
Which one mechanism deserves proof first?

Act:
Run only that proof route.

Then:
Shed dead weight and update status.

Blocked:

Do not open ten next routes.
