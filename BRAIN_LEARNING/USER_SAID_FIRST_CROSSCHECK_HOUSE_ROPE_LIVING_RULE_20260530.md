## Trigger

Use this rule when the user posts a proof/status/result message, asks what is next, gives a correction about reasoning order, or asks to lock a new thinking/routing behavior.

## Required order

1. Go over what the user actually said first.
2. Use the Git URL/repo when available to cross-check the commit, current head, status claims, and hashes before trusting the state.
3. Cross-check the freshest loaded project sources already present in chat/project context, and prefer the newest living source over stale drop-copies.
4. Enter through the house door with the rope: name the object, source, lane, boundary, proof need, and blocked lanes.
5. Pick the best-fit room for the current task/message, not the first obvious room from momentum.
6. Then say what is best moving forward, including disagreement or a cleaner route when the posted next move is not best.

## Response shape

Use this shape when useful:

- You said: <plain read of the user proof/status/correction>.
- Cross-check: <Git/current-source result, including stale guard if needed>.
- Loaded sources: <fresh project cards/rules that matter>.
- House-rope read: <front door/object/lane/room/boundary>.
- Best move: <assistant judgment, not automatic agreement>.

## Stale-source guard

Do not bring in stale ideas just because they are visible in chat. Check Git/main, file recency, source role, and whether a newer living card supersedes an older drop-copy. If origin/main changed after the claimed base, stop and report STALE_BASE rather than saving.

## Hash/key rule

Major files created or touched by this rule must be named in a legend/ledger/keymap with WorkKey, SourceKey, FileKey, role, path, authority, and SHA256 where computable. Self-hashes belong in the receipt when writing the hash inside the file would change the file.

## Exposure / powerplay companion

When a failure exposes a missing companion rule or route gap, treat the exposure as POWERPLAY / CRIME SCENE. Freeze the scene, preserve the failure evidence, identify the violated or missing mechanic, repair the route, and resume only after proof. The concept can be right while the save body is not yet house-native.

## Boundary

This rule changes assistant ordering and save-route behavior. It does not prove content quality by itself. It does not promote doctrine. It does not rewrite ACTIVE_GUIDES or CURRENT_TRUTH_INDEX. It does not authorize automation, watcher, implementation, full batch, or Whirlpool.
