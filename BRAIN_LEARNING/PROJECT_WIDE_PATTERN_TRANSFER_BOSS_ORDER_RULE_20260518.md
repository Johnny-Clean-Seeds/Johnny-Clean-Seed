# Project-Wide Pattern Transfer Boss Order Rule

Status: ACTIVE MAIN DECISION-ORDER RULE
Role: make recursive pattern transfer usable without causing scattered repair
Authority: assistant learning rule for Clean Seed / Clean Milk work

## Core Rule

Pattern transfer is project-wide.

When a clean pattern works in one room, the assistant must check whether the same concept should apply elsewhere in Mr.Kleen.

This is one of the main ways the house becomes recursive.

A clean pattern should echo through the house where the fit is the same.

## What Counts As A Transferable Pattern

A transferable pattern can be:

- a working concept
- a gate
- a wording pattern
- a proof method
- a lane boundary
- a room structure
- a repair method
- a decision method
- a receipt shape
- a status rule
- a parking rule
- an authority split
- a live/static boundary
- a candidate/testing/installed/worn/promoted lifecycle distinction

## Do Not Blindly Copy

Do not copy text just because it worked somewhere else.

Copy the useful pattern.

Then adapt it to the target room.

The target room keeps its own job.

Respect Thy Neighbor still applies.

A transferred pattern is clean only if it blesses the target room without stealing another room's job.

## Alarm Rule

When pattern transfer is applied, alarms may go off across the house.

That is expected.

An alarm is not permission to repair everything.

An alarm means:

- capture the issue
- name the pattern involved
- identify the room touched
- identify the neighbor risk
- decide whether it is a boss, child, symptom, ghost, duplicate, tool, or proof need
- park it in the boss stack
- do not repair until it is ranked and chosen

## Stack Before Repair

When multiple issues appear, the assistant must not jump from alarm to alarm.

The order is:

1. capture all visible alarms
2. list them
3. group them
4. collapse duplicates
5. separate parent bosses from symptoms
6. rank the true bosses
7. choose the current boss
8. fight one boss at a time
9. save proof
10. walk the halls again

## Boss Ranking Questions

Rank bosses by asking:

1. Does this issue affect entry?
2. Does it affect authority?
3. Does it affect routing?
4. Does it affect proof?
5. Does it affect next-move meaning?
6. Does it affect save continuity?
7. Does it cause repeated fog?
8. Does it create competing authority?
9. Does it cause many smaller symptoms?
10. Does fixing it unlock cleaner work elsewhere?

The boss that causes the most downstream confusion usually ranks higher than a visible small symptom.

## Main Decision-Order Rule

This rule helps decide what is best to work on and in what order.

Do not choose the next task only because it is visible, annoying, recent, or easy.

Choose after stack/list/group/rank.

A clean project does not chase noise.

A clean project finds the boss seam.

## Recursive Power Play

Recursive growth means the same clean law can repeat at different scales.

The pattern can appear in a word, file, room, receipt, index, lifecycle, bridge, guide, or whole-house decision.

But recursive does not mean uncontrolled spread.

Recursive means:

- same clean concept
- correct room fit
- neighbor respect
- proof
- ranked order
- one boss at a time

## Required Output When Alarms Appear

When a pattern-transfer pass raises multiple issues, the assistant should output:

- Raw alarm list
- Groups
- Parent bosses
- Symptoms / children
- Parked items
- Current recommended boss
- Why that boss ranks first
- Blocked repairs
- Proof needed before save

Do not repair the whole list in one move.
