## 1. The Chuck Norris Test

This test asks:

Can the object take a hard hit and still stay in its lane?

If pressure makes the report become a different object, it fails.

The report passes only if it keeps its job under pressure:

- source pressure;
- rule pressure;
- proof pressure;
- helper/code pressure;
- ADOPT pressure;
- ADAPT pressure;
- dead-chat pressure;
- next-action pressure;
- authority-language pressure;
- save-location pressure;
- consolidation pressure;
- no-parking pressure.

The main question is not:

"How long is it?"

The main question is:

"Did it stay what it is?"

## 2. Three-Pass Review Rule

Before finalizing the report shape, run three passes.

### Pass 1 — Extract and Map

Map the object.

Ask:

- What is the active object?
- What source packet is being judged?
- Which items are ADOPT?
- Which items are ADAPT?
- Which items are PROOF_PARK?
- What is the report allowed to do?
- What is it blocked from doing?
- What proof would make the next move real?
- What should be burned from active carry?

Output of Pass 1:

A clean map of source group, role, lane, boundary, and next proof need.

### Pass 2 — Cross-Examine and Pressure-Test

Do not trust the first map.

Hit it with pressure.

Ask:

- Is any source bulk sneaking in?
- Is any ADAPT item becoming a separate rule too early?
- Is any mechanism acting like doctrine?
- Is any proof support acting like authority?
- Is any helper/code route being opened without need?
- Is any old chat weight being carried because it feels familiar?
- Is the report creating too many next moves?
- Is the report trying to solve the whole house?
- Is the report skipping fit decisions?
- Is it using PASS language without proof?
- Is it making a save claim before the save route exists?
- Is it confusing local-only material with Local + URL durable material?
- Is it trying to consolidate after one signal instead of repeated evidence?

Output of Pass 2:

A pressure map:

- KEEP;
- NARROW;
- EXPAND;
- BLOCK;
- PARK;
- PROOF;
- PARENT.

No list-to-guess. Patch the report shape through the rules, then judge from the pressure map.

### Pass 3 — Rewrite Clean Final

Burn the dead weight.

Rewrite only the clean final object.

Final output must be:

- one report;
- candidate/mechanism language only;
- no raw chunks;
- no helper build;
- no doctrine promotion;
- no ACTIVE_GUIDES rewrite;
- no CURRENT_TRUTH_INDEX rewrite;
- no broad rescan;
- one next route.

Output of Pass 3:

A clean Source-to-House Intake V2 Mechanism Extraction Report shape ready to build or save.

## 4. How We Know What Is Too Much

Too much is not measured only by size.

Too much means the object starts doing a job it was not allowed to do.

A section is too much if it causes role drift.

A mechanism extraction report becomes too much when it starts behaving like:

- an authority file;
- a source dump;
- a rule install;
- a helper build;
- a doctrine rewrite;
- a proof replacement;
- a broad scan;
- a cleanup script;
- a whole-house map;
- a final answer to all future movement.

The clean limit:

The report may describe candidate mechanisms.
It may not grant them authority.

## 5. Pressure Zones

### 5.1 Source Pressure

Allowed:

- source name;
- source role;
- source group;
- source pointer;
- short mechanism summary;
- proof relevance.

Blocked:

- raw chunks;
- long copied source bodies;
- transcript slabs;
- pasted evidence bulk;
- all-file archive behavior.

PASS:

The report can prove what it used without carrying the whole source body.

FAIL:

The report needs raw chunks to feel complete.

---

### 5.2 Mechanism Pressure

A mechanism is not just an interesting idea.

A mechanism must answer:

- what it does;
- when it fires;
- what input it takes;
- what output it gives;
- what lane it belongs in;
- what it must not become;
- what proof would test it.

PASS:

The report extracts repeatable motions.

FAIL:

Every cool phrase becomes a mechanism.

---

### 5.3 Authority Pressure

Danger words:

- installed;
- doctrine;
- active rule;
- final truth;
- must always;
- rewrite guide;
- update CURRENT_TRUTH_INDEX;
- now governs.

Clean words:

- candidate;
- mechanism;
- pressure;
- pattern;
- route;
- fit;
- proof-needed;
- not promoted;
- support only.

PASS:

The report stays candidate-level.

FAIL:

The report starts sounding like house law.

---

This is a judgment/report-shape task.

Blocked:

### 5.8 Dead-Weight Pressure

Burn from active carry:

- old failure chatter;
- broad locator attempts;
- duplicate queue loops;
- old PowerShell logs;
- direct REUSE labels on code files;
- old helper identity;
- Codex/mule dependency while Codex is unavailable;

- impulse to scan everything again.

Keep as proof trail only:

- failed Code Gate reports;
- old helper versions;
- blocked runs;
- read/report receipts;
- raw evidence pointers.

PASS:

Carry the key, not the furniture.

FAIL:

The report drags the whole old chat trail forward.

---

### 5.9 Next-Action Pressure

A clean report does not open ten new doors.

Allowed next route:

Review mechanism extraction -> choose one mechanism for proof -> then decide save/promote/build separately.

Blocked:

- ten next moves;
- broad house cleanup;
- helper build as automatic next step;
- doctrine promotion as automatic next step;
- all ADOPT items moving to proof at once.

PASS:

One next route.

FAIL:

The ending becomes a new maze.

### 5.10 Save-Location Pressure

Durable project save normally goes Local + URL unless marked special.

But raw/source-heavy material stays local unless routed.

For this report:

Allowed durable save later:

- mechanism extraction report;
- proof receipt;
- status/index update if the save route is explicitly run.

Blocked:

- raw chunk save;
- raw source pack Git staging;
- local-only artifact falsely described as URL-pushed;
- URL-push claim without proof;
- save claim before commit/push/clean-status proof.

PASS:

Save report splits Local + URL from Local-only.

FAIL:

Save language outruns proof.

### 5.12 Consolidator Pressure

One signal means WATCH AGAIN.

PASS:

The report notes overlap but does not merge rules early.

FAIL:

The report packs gates together because they feel related once.

### 5.13 Gate-Hallway Pressure

Normal output should not narrate every gate.

Allowed:

PASS:

The report uses gates to judge, not to perform a hallway show.

### 5.14 Misc Drawer / Porch Pressure

Mixed material must be read and classified before action.

The Misc Drawer is not trash.
The porch is not storage.
Drop means event.
Folder means package.
Move, receipt, classify.

For this report:

PASS:

Cleaned report material goes to the correct lane.

FAIL:

Generated files, helper reports, source packets, or handoff drops sit loose in root/Desktop/Downloads with no receipt.

---

### 5.15 Code Gate Pressure

If code is eventually needed, Code Gate comes first.

Code Gate PASS is not job PASS.

For this step:

No code.

If later code appears:

- parse first;
- risk classify;
- block unsafe writes unless allowed;
- run bounded;
- capture output;
- prove target job separately.

PASS:

No generated helper unless evidence requires it.

FAIL:

A convenience script appears before the road is proven.

## 7. ADOPT Mechanism Candidates

### 7.1 Helper Confidence Ladder

What it does:

Ranks helper trust by proven behavior instead of vibes.

Possible ladder:

- unknown helper;
- read-pattern only;
- dry-tested helper;
- Code Gate passed;
- bounded run passed;
- output verified;
- repeated clean use;
- candidate active pattern;
- saved helper with proof.

Must not become:

Automatic trust or authority.

Proof needed:

A helper must produce expected output under bounded conditions and survive independent check.

---

### 7.2 Helper Output Shape Contract

What it does:

Defines what a helper must output so the assistant can judge it cleanly.

Expected output shape:

- start marker;
- object;
- root/path;
- mode;
- counts;
- skipped items;
- warnings;
- blockers;
- output files;
- hash/receipt if relevant;
- final verdict;
- next action.

Must not become:

Fancy output theater.

Proof needed:

Assistant can read output and make a correct next decision without guessing.

---

### 7.3 House Keyring Graph

What it does:

Turns house orientation into keys, addresses, rooms, links, proof pointers, and return paths.

It helps answer:

- where am I?
- what is this?
- what does it connect to?
- what proof exists?
- what should not be touched?
- what is next?

Must not become:

A second authority spine or replacement for CURRENT_TRUTH_INDEX.

Proof needed:

It reduces repeated orientation burden without overriding authority order.

---

### 7.4 House Walk Routine

What it does:

Gives a bounded entry/reset method before focused work.

Clean walk:

- release old room;
- identify active object;
- check source/authority;
- check lane;
- check boundary;
- check proof need;
- return to focused work.

Must not become:

Broad audit, wandering, cleanup trigger, or process performance.

Proof needed:

It improves orientation and then narrows work.

---

### 7.5 Lane-Locked Manifest Staging

What it does:

Stages files or report outputs by lane before save or movement.

Clean fields:

- object;
- source;
- lane;
- intended destination;
- authority status;
- local-only or Local + URL;
- proof need;
- blocked moves.

Must not become:

Bulk staging of raw source.

Proof needed:

Manifest prevents wrong-room saves and accidental promotion.

---

### 7.6 Solution-to-Concept Loop

What it does:

Turns a working fix into a reusable concept only after cross-exam and proof.

Loop:

fix -> concept -> cross-exam -> fit -> route -> prove -> shed

Must not become:

Every fix instantly becomes a rule.

Proof needed:

The concept survives pressure and improves future movement.

---

### 7.7 Dead Chat Shed

What it does:

Burns old live chat burden while preserving useful proof pointers.

Keeps:

- active object;
- trigger;
- current blocker;
- proof pointer;
- next condition.

Burns:

- repeated logs;
- failure chatter;
- duplicate links;
- stale explanation;
- old helper identity;
- already-processed queue weight.

Must not become:

Evidence deletion.

Proof needed:

Future work can resume from key pointers without dragging the furniture.

---

### 7.8 Source-to-House Intake V2 Design Spec

What it does:

Defines the mechanism for turning source material into house-fit candidates without raw source pollution.

Working chain:

source -> custody -> read -> map -> claim ledger -> mechanism extraction -> fit decision -> proof route -> save/park/block.

Must not become:

Raw source Git staging or direct source authority.

Proof needed:

Cleanly separates source signal, proof support, mechanism candidate, and authority boundary.

---

### 7.9 Think Tank Opposition Lens

What it does:

Forces a candidate to face opposition before it is trusted.

Questions:

- what argues against this?
- what would break it?
- where does it overreach?
- what neighbor rule does it conflict with?
- what proof would make it real?
- what should stay parked?

Must not become:

Endless debate or refusal to act.

Proof needed:

It improves the candidate without bloating it.

---

### 7.10 Wayfinding Save-Selection

What it does:

Chooses what deserves to be saved based on role, lane, proof, and future retrieval value.

Save only if it has:

- stable object;
- useful role;
- clear lane;
- proof or proof need;
- boundary;
- retrieval value;
- next route.

Must not become:

Save everything because it feels important.

Proof needed:

Saved objects are findable and useful later.

---

### 7.11 Wayfinding Event-Chain Branch

What it does:

Links events in a chain so future workers can see cause, movement, proof, and next state.

Chain fields:

- event;
- prior state;
- action;
- proof;
- result;
- remaining blocker;
- next route.

Must not become:

A giant timeline dump.

Proof needed:

It helps resume without rereading everything.

## 10. Chuck Norris Pass / Watch / Fail Criteria

### PASS

The report:

- stays one report;
- extracts mechanisms only;
- uses candidate language;
- carries source roles, not raw chunks;
- compresses ADAPT into themes;
- parks PROOF_PARK as support;
- burns dead chat weight;
- names one next route;
- keeps authority boundaries;
- shows proof needed;
- avoids helper/code build.

Verdict:

READY FOR NARROW REPORT BUILD.

### PASS WITH WATCH

The report is mostly clean but has one or more watch risks:

- ADAPT section is getting heavy;
- mechanism list is too broad;
- proof language is fuzzy;
- next route is too open;
- source pointers are weak;
- save boundary needs clearer wording.

Verdict:

NARROW BEFORE SAVE.

### FAIL

The report:

- promotes doctrine;
- rewrites ACTIVE_GUIDES;
- edits CURRENT_TRUTH_INDEX;
- saves raw chunks;
- stages raw source to Git;
- builds helper code;
- performs broad rescan;
- makes all 24 files the save target;
- turns ADAPT into rule pack;
- lets proof support become authority;
- ends with many next actions;
- says PASS without proof.

Verdict:

BLOCK / REWRITE SHAPE.
