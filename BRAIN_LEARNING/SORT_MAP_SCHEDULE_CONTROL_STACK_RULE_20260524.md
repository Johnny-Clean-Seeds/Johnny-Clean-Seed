# Sort Map Schedule Control Stack Rule

Date: 2026-05-24

Status: active support rule / method stack.

## Rule

Use one small control stack for sorting, mapping, and scheduling. Do not make each method a new boss.

The stack is:

`capture -> classify -> limit -> map -> prioritize -> schedule -> prove -> close`

## Stack

| Layer | House behavior | Outside support lens |
|---|---|---|
| Capture | Keep raw source, root drops, corrections, and notes from disappearing. | GTD capture, source custody. |
| Classify | Sort before judgment: object, intent, mode, place, authority, staleness, proof, disposition. | GTD clarify/organize, PARA actionability. |
| Limit | Keep active work small and visible. | Kanban WIP limit, soft suit max 3 candidates. |
| Map | Show level of zoom and source route. | C4 context/container/component/code, W3C PROV provenance. |
| Prioritize | Choose the next boss by proof, risk, delay, impact, effort, and confidence. | WSJF and RICE as rough lenses only. |
| Schedule | Use event triggers and cadences to revisit the right notes at the right time. | Kanban feedback loops, GTD reflect, PARA review horizons, Scrum inspect/adapt, PDCA. |
| Prove | Check evidence, counterproof, affected scope, Git/local state, and authority. | PROV, PDCA check/study, proof gate. |
| Close | Burn, cut with runner trace, park, reject, block, commit, receipt, or return. | Zettelkasten structure notes, PARA archives, house receipts. |

## Fog Domain Check

Use Cynefin as a small uncertainty check:

- Clear: use the known rule and do not over-research.
- Complicated: analyze with sources, compare methods, and choose.
- Complex: run small safe tests and washer passes.
- Chaotic: stabilize first; stop destructive or authority-risk movement.
- Confused: tug the rope and classify the domain before acting.

## Mapping Zoom

Use this map shape before creating more structure:

| Zoom | House question |
|---|---|
| Context | What house, user command, outside source, or boundary is this inside? |
| Container | Which room owns it? |
| Component | Which object, note, rule, TODO, packet, or receipt is moving? |
| Line | Which exact file, line, hash, command, or proof decides it? |

Do not jump to line detail when context is foggy. Do not stay in context when an exact file decides
the move.

## Priority Lens

When multiple candidates are truly ready, rank by:

1. Safety or false-PASS risk.
2. User's active command.
3. Source custody or lost-trace risk.
4. Cost of delay.
5. Impact.
6. Confidence.
7. Effort.
8. WIP load.

RICE and WSJF may help compare candidates, but scores are not proof. If scores disagree with source
truth, source truth wins.

## Schedule Lens

Every parked or quieted object needs one of:

- event trigger;
- review cadence;
- blocked-until proof;
- no-return rejection reason;
- archive/source custody only.

No parked object should tug forever without a named return condition.
