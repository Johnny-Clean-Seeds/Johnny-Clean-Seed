The user tested a search mission for an old target file called `WHY_WE_SUCK_SOMETIMES_METHOD`.

The assistant recognized the right target identity, but did not surface the actual file card or opened file content. That means the search was not a full find. It was a memory trace with side hits and a fog wall.

This candidate converts that mistake into a pathfinding method.

The point is not simply to search harder. The point is to search like a living system: probe, smell, branch, test cost, reinforce proof, prune noise, sew useful paths, label the return, and reset before the next wave.

## The core correction

Do not confuse these three things:

1. Target recognition.
2. Path evidence.
3. File proof.

A remembered title is not a found file.
A neighboring file family is not the target.
A pointer is not proof.
A citation to a rule is not source custody.
A side hit is not completion.

The clean labels are:

- FOUND: actual file card, opened content, exact path with verified existence, exact receipt, or authoritative source content surfaced.
- STRONG_SCENT: exact or near-exact title signal, but no file proof yet.
- MEMORY_TRACE: assistant remembers or recognizes the object but cannot prove it in the current accessible surface.
- SIDE_HIT: adjacent document, sibling rule, concept-family hit, or related evidence that is not the target.
- FOG_WALL: pointer/name/path exists, but contents are not visible or verified from here.
- DEAD_TUBE: path returns no useful signal.
- REPLAN: terrain changed or current method is not producing proof.
- CUT: remove from live carry.
- SEW: connect with typed relation and proof need.
- PARK: useful, but not current.
- BLOCK: unsafe, wrong authority, wrong room, or forbidden move.

## Edward principle

Edward mode does not mean every tool fires at the same time.

Edward mode means all rooms, tools, metaphors, proof routes, and source families are available for selection while one active movement is carried at a time.

The clean formula is:

All rooms available.
One room active.

All tools available.
One toolbelt worn.

All ideas available.
One current frame.

All proof preserved.
One proof pointer carried.

All search ways available.
Three straight spines per wave.

All side branches allowed.
Only proof-bearing branches survive.

## Bigger concept

Pathfinding is orientation under partial visibility.

The system does not know the whole terrain. It has a target smell, a current position, a cost limit, a memory trace, a map if one exists, and proof needs.

A good pathfinder does not merely move. It senses while moving and changes the method based on what the terrain gives back.

This candidate treats search as a living circuit:

SMELL -> SPINE -> FEELER -> COST -> FLUX -> SCISSOR -> SEW -> LABEL -> BREATH

SMELL means define the target scent.
SPINE means run a long straight path in one direction.
FEELER means allow small side branches only when signal improves.
COST means judge value against time, load, risk, and proof burden.
FLUX means strengthen paths that return proof.
SCISSOR means cut stale, duplicate, side-only, or dead branches.
SEW means reconnect useful paths with typed relation and return route.
LABEL means every branch gets a disposition.
BREATH means smoke-break reset before the next wave.

## Three straight spines

Each wave uses three straight searches.

### Left spine: living pathfinding

Use for old, buried, renamed, half-remembered, symbolic, or foggy targets.

Source metaphors:

- Roots grow through gradients and soil signals.
- Hydrotropism turns root growth toward moisture gradients.
- Mycelium explores while balancing energy cost and nutrient gain.
- Slime mold reinforces useful tubes and lets weak routes disappear.
- Ants build pheromone trails where repeated useful movement marks the environment.
- Axon growth cones feel local cues through dynamic tips and turn by attraction, repulsion, contact, and internal state.
- Chemotaxis compares signal over time instead of requiring a complete map.

House translation:

Use roots when the target is only smelled.
Use slime mold when multiple paths exist and proof-return should thicken one.
Use ants when repeated returns should mark a route.
Use growth cones when the next step must feel before committing.
Use chemotaxis when signal improves or worsens over time.

### Middle spine: formal pathfinding

Use for known names, paths, ledgers, indexes, hashes, registries, or structured search.

Method families:

- BFS: broad one-layer sweep.
- DFS: deep straight descent.
- Dijkstra: cost-aware route selection.
- A*: goal-scent plus cost.
- Bellman-Ford: tolerate negative or troublesome costs in model space.
- Floyd-Warshall: all-pairs relation map when the whole graph is small enough.
- D* / D* Lite: replan when terrain changes.
- Theta*: line-of-sight shortcut instead of forced grid corners.
- Jump Point Search: skip redundant grid expansion.
- RRT: grow random exploration trees into large unknown or high-dimensional spaces.
- PRM: sample the space first, then connect useful milestones.
- Potential fields: attractions and repulsions shape movement.
- Constraint search: allowed moves matter as much as available moves.
- Beam search: keep only the best few branches when load must stay capped.

House translation:

Use BFS when you need a shallow neighborhood sweep.
Use DFS when you are going down one long rope.
Use Dijkstra when every path has different cost.
Use A* when target scent is strong enough to guide.
Use D* Lite when new evidence changes the terrain.
Use Theta* when there is a clean shortcut across artificial folder/grid boundaries.
Use Jump Point Search when repeated same-shaped folders are creating redundant expansion.
Use RRT when the space is unknown, large, and poorly indexed.
Use PRM when you need a rough map of landmarks before choosing a route.
Use potential fields when attractions and blockers are more useful than exact paths.
Use beam search when the assistant is carrying too many branches.

### Right spine: physical, cognitive, social, and symbolic pathfinding

Use when the target is a workflow, meaning, relation, source family, rule pattern, or hidden route rather than only a file.

Pathfinding families:

- Information foraging: follow scent based on perceived value, cost, and access path.
- OODA: observe, orient, decide, act, then use feedback to re-enter.
- Double-loop learning: if repeated failure appears, change the governing assumption, not only the action.
- River channels: repeated flow carves stable routes.
- Lightning / diffusion branching: exposed tips grow; shielded areas receive less signal.
- Breadcrumbs: leave return marks so descent does not become lostness.
- Ariadne thread: carry a return rope into the maze.
- Wayfinding: landmarks, routes, survey knowledge, and local cues all matter.
- Search-and-rescue grid: cover the terrain deliberately instead of wandering.
- Legal discovery: produce, tag, review, privilege-check, and chain-of-custody.
- Forensic investigation: preserve evidence before changing the scene.
- Compiler resolution: names bind through scope, imports, namespaces, and precedence.
- Network routing: prefer reliable paths, update tables, avoid loops, and handle failure.
- Cache lookup: try cheap remembered indexes but invalidate when stale.
- Immune response: distinguish self, non-self, false positive, and recurring threat.
- Mythic descent: go down with a rope, not as a tourist.

House translation:

Use information foraging when choosing which file/path is worth opening.
Use OODA when a search must move fast but not blind.
Use double-loop when repeated search misses show the search model is wrong.
Use river-channel thinking when repeated successful routes should become easier.
Use breadcrumbs and Ariadne rope whenever going deep.
Use forensic thinking when the evidence must not be dirtied.
Use compiler thinking when aliases, names, and path precedence are confusing.
Use network routing when there are many tunnels and possible loops.
Use cache thinking when memory is useful but cannot be trusted as proof.

## Branch scoring

Each branch gets four scores from 0 to 3.

Scent:
0 no match.
1 weak conceptual relation.
2 strong alias or family relation.
3 exact title/path/source match.

Proof:
0 none.
1 memory or pointer only.
2 side evidence, parent index, or partial source.
3 file card, opened content, verified path, hash, receipt, or commit.

Cost:
0 cheap.
1 moderate.
2 heavy.
3 too expensive for current task.

Fog:
0 clear.
1 partly visible.
2 pointer-only.
3 dark or inaccessible.

Good branch:
Scent high, proof rising, cost acceptable, fog falling.

Bad branch:
Scent weak, proof absent, cost rising, fog unchanged.

## Why this matters

The old failure pattern is subtle.

The assistant can sound right because it recognizes the shape of the answer. But sounding right is not the same as reaching the file.

This method forces the assistant to say:

I found it.
I smelled it.
I remembered it.
I found a sibling.
I hit fog.
I need a different path.

That distinction is the whole point.
