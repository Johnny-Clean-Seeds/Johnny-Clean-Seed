# Issue Self-Repair and Room Redesign Authority Rule

Status: ACTIVE LEARNING RULE
Authority: assistant behavior rule; not command authority

## Purpose

When the assistant encounters an issue during Clean Seed / Mr.Kleen work, it should not merely report the issue and wait if the fix is within doctrine, current scope, and clean authority boundaries.

The assistant may diagnose the issue, design the fix, create helper tools or checkers, redesign rooms, and move rooms when needed.

This authority exists only inside the model.

It must protect the core, proof, source/support/authority boundaries, neighbor rooms, lifecycle truth, and user flow.

## Core Rule

When an issue appears:

detect -> diagnose -> weigh if bells ring -> design fix -> soft trial if structural -> build helper/tool/room if needed -> prove -> save -> update maps -> clean status.

Do not walk past a fixable issue.

Do not remodel without proof.

## Issue Triggers

Use this rule when the assistant sees:

- missing room signs
- unclear room placement
- empty folders that will not survive Git
- repeated proof failures
- duplicate commit-message fog
- stale anchor/current-state mismatch
- source/support/authority confusion
- word-key drift
- lifecycle-state confusion
- neighbor-role theft
- user-flow friction
- repeated paste or command errors
- missing checker/index/tool
- retrieval fog
- overgrown idea piles
- room structure that blocks clean work
- old method still useful but poorly stored
- new method that needs a safe test lane
- a current repair that reveals a deeper leak

## Allowed Self-Repair Moves

The assistant may create or revise:

- learning rules
- checkers
- indexes
- support maps
- placement reviews
- Work Shed helpers
- idea cards
- front-shelf cards
- proof receipts
- trace gates
- room signs
- soft trial rooms
- tool-use gates
- retrieval maps

The assistant may design and later move rooms when a cleaner recursive fit is proven.

## Room Redesign / Move Authority

The assistant may redesign or move rooms when:

- the current room placement causes fog
- a room is doing the wrong job
- a room belongs functionally to a different layer
- neighbor roles are being confused
- retrieval is materially worse because of placement
- a soft trial proves the new placement is cleaner
- the move improves recursive fit without violating the core

Room moves must use a gate.

## Room Move Gate

Before moving a room or changing house structure:

1. Detect the placement issue.
2. Name the root leak.
3. Run Bell Logic if bells ring.
4. Run Word-Key Clarity if room names or labels are affected.
5. Create a soft design/review first unless the move is tiny and mechanical.
6. Compare viable placements.
7. Select the cleanest placement.
8. Preserve old paths/history.
9. Protect neighboring rooms.
10. Update indexes/status/anchor if affected.
11. Prove references and retrieval still work.
12. Commit, push, and verify clean status.

## Tools / Helper Authority

The assistant may make tools or helpers when needed, but tools cannot command.

A helper/tool/checker/index is allowed only when it:

- follows doctrine
- has a clear job
- has an authority boundary
- has an entry trigger
- has a proof path
- protects neighbor lanes
- does not promote source material
- does not replace active authority
- does not hide history
- passes proof

## Blocked Self-Repairs

Do not:

- violate the core idea
- skip proof
- hide proof history
- rewrite public history without explicit high-risk authorization
- promote source ore
- let support become authority
- let tools command
- move rooms without design/proof
- delete or bury old useful methods
- turn cleanup into disguised repair
- change active guides casually
- install doctrine because a helper looks useful
- use recursion language as proof
- let feeling outrank proof

## Old Path Preservation

When moving, redesigning, or superseding a room/method:

- preserve the old path in notes or receipts
- explain why the change happened
- record what the old method was good for
- park old useful ways instead of deleting them
- add retrieval pointers if future work may need them

Old ways are not trash.

They may fit again later.

## Proof Standard

A self-repair passes only when the receipt proves:

- issue detected
- cause named
- fix selected
- blocked moves respected
- neighbor roles protected
- files created/changed as intended
- no source promotion unless explicitly authorized
- no authority confusion created
- final status clean

## Closing Rule

If the assistant sees a fixable issue inside scope, it should repair through the model.

If the issue is structural, soft-design first.

If the fix needs a helper, build the helper.

If the room is wrong, redesign or move it only after proof.

Proof decides.
