# Gate Family Checkpoint Grammar Rule

Status: SUPPORT GRAMMAR / GATE-FAMILY ORGANIZATION / NOT DOCTRINE
Date: 2026-05-24

## Purpose

Keep useful checks alive without turning every useful check into a full gate.

The house can have major gate rooms and smaller checkpoints that sit after, beside, or inside those rooms.

## Core Shape

`Parent Gate -> Checkpoint -> next Parent Gate`

The parent gate opens the room and does the main work.

The checkpoint checks the parent's output before the object moves on.

The parent gate may then confirm the checkpoint output, narrow it, or send it forward.

## Partner Interpretation Shape

When the word "checkpoint" starts to imply a boss and servant hierarchy, use the cleaner neighbor shape:

`Parent Action -> Partner Interpretation -> Next Parent Action`

The parent action does the work.

The partner interpretation asks what that work means, what consequence it created, what missing evidence appears, and what the next parent must receive.

This is not a loose second boss. It is a context partner that inherits the parent trigger and prepares clean handoff.

## Object Types

- Full gate: a major room that decides, blocks, permits, routes, proves, sequences, or closes.
- Checkpoint / lesser partner: a smaller support check that fires only when the parent output needs interpretation.
- Tool: something the worker uses.
- Card: a structured note the worker writes.
- Ledger: a trace of steps, marks, or proof.
- Parking item: unresolved useful material with a return trigger.
- Receipt: proof that work happened.

## Checkpoint Rules

A checkpoint:

- fires only when its parent produces something that needs checking or interpretation;
- does not own the whole route;
- does not overrule the parent unless it finds a boundary, permission, proof, or role failure;
- usually runs after the parent because it checks output;
- prepares the object for the next gate;
- can be promoted only after repeated proof shows a distinct job, trigger, boundary, proof need, and return-home path;
- can be demoted into a tool, card, ledger, or checklist if it is useful but too small to be a gate.

## Gate-Judges-Gate Tie-In

Every gate still judges:

1. the active problem;
2. the previous gate's effect on the fix;
3. whether the checkpoint stayed in its lane;
4. whether the partner interpretation changed the active object correctly;
5. whether the parent/checkpoint pair prepared clean output for the next parent.

## Boundary

- Not doctrine.
- No ACTIVE_GUIDES rewrite.
- No CURRENT_TRUTH_INDEX rewrite.
- No public authority change.
- No automatic gate promotion.
- No CONSOLIDATOR merge.
- No deleting old gates because a checkpoint touches them.

## Proof

Proof is a live run where a candidate is handled cleaner because it becomes the right object type: full gate, checkpoint, tool, card, ledger, parking item, or receipt.
