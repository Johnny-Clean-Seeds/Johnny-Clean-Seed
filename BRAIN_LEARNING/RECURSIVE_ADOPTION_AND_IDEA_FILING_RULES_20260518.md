# Recursive Adoption and Idea Filing Rules

Status: ACTIVE LEARNING RULE SET
Authority: assistant behavior rules; not command authority

## Purpose

These rules make the assistant handle new ideas, better methods, old methods, and idea storage recursively across the whole model.

The rules do not move files.

The rules do not promote source ore.

The rules do not create doctrine by themselves.

They define how the assistant should think, sort, adopt, park, retrieve, and change methods while working.

## Captured Requirements

The user gave these requirements:

1. Work softly first when changing room placement so the house can feel the change before hard movement.
2. Do not pick the first available idea.
3. Find several task-fitting ideas.
4. Compare ideas and choose the better one.
5. Do not collect random ideas.
6. For large piles, break ideas into groups before comparing.
7. Use this across the whole model because the model is recursive.
8. Make rules, not just one rule, when the behavior has several parts.
9. Adopt better ways when they improve the current work.
10. Change methods again when the now changes.
11. Do not throw old ways away.
12. Store old ideas because they may fit later.
13. Keep the most-used or most-changed ideas near the front.
14. Group ideas so huge idea halls remain navigable.
15. Use filing-cabinet style organization.
16. Use multiple organization aids such as labels, colors, shapes, tags, groups, and indexes when they help retrieval.

## Rule 1 — Soft Trial Before Hard Move

When changing room placement, learning-room position, operating-room shape, cabinet structure, or house architecture:

- create a soft trial first
- define what the change would mean
- keep original files in place
- test how the new placement feels in work
- only move files later if the soft trial proves useful

Soft trial means:
support map, placement review, bridge, index, or sign.

Soft trial does not mean:
move, delete, rewrite authority, or promote doctrine.

## Rule 2 — Task-Fitting Idea Search

When looking for ideas, helpers, layouts, algorithms, filing methods, or outside patterns:

- search for ideas that fit the active task
- do not collect random material just because it is interesting
- capture source and fit reason
- route candidate ideas through source ore, cabinet, or Soft Suit
- keep authority with the current command chain

## Rule 3 — Candidate Pile Before Choice

When the choice matters:

- make a small candidate pile
- group candidates by job
- compare the groups
- choose the best fit
- block the weaker options for now
- park useful non-selected ideas

Do not compare 60000 loose items flat.

Large pile method:

1. capture raw
2. group by type
3. choose group winners
4. compare winners
5. select current best
6. park the rest with retrieval tags

## Rule 4 — Recursive Across the Board

Any working method that fits the core should be checked for use across other matching lanes.

This does not mean blind copy.

It means pattern transfer:

- recognize the useful pattern
- compare to the new lane
- adapt to that lane
- respect neighbors
- prove fit
- save only if it blesses the model

The model is recursive, so good placement logic should repeat at different levels without violating the core.

## Rule 5 — Adopt Better Now

When a better way appears:

- capture it
- judge fit
- test it
- adopt it for now if it improves the current work
- preserve the old way as parked history
- change again later if the now changes

Adoption is current-fit, not forever-proof.

Old methods are not trash.

Old methods may become useful again when the environment, task, or neighbor conditions change.

## Rule 6 — Old Ways Stay Retrievable

Do not delete old methods merely because they are not the best current method.

Store them with:

- name
- former use
- why it was changed
- current status
- possible future fit
- neighbor risks
- retrieval tags

Possible statuses:

- current
- front shelf
- parked
- old but useful
- superseded
- rejected for now
- archive only

## Rule 7 — Filing Cabinet Over Hallway

The idea system must not become a long hallway of billions of loose ideas.

It must act like a filing cabinet.

Each idea should have:

- drawer
- folder
- card
- tags
- status
- source
- next action
- retrieval key

Cabinet levels:

1. Cabinet: whole idea system
2. Drawer: broad family
3. Folder: grouped method or topic
4. Card: one atomic idea
5. Tag: cross-cutting retrieval key
6. Front shelf: most-used or currently changing ideas

## Rule 8 — Front Shelf For Hot Ideas

Ideas used often or changed often should stay near the front.

Front shelf candidates:

- active learning rules
- current checkers
- current gates
- current filing patterns
- current boss methods
- current adoption methods

Front shelf does not mean command authority.

It means fast retrieval.

## Rule 9 — Multi-Shape Organization

Use several organization shapes when they help retrieval:

- hierarchy for cabinet/drawer/folder/card
- tags for cross-cutting retrieval
- facets for multiple ways to find the same idea
- status labels for lifecycle
- neighbor labels for room relation
- color/shape words as optional signs
- sequence for process flow
- matrix for comparing options
- graph/link notes for neighbor paths
- timeline for history and supersession

Do not use decoration as proof.

Use signs only when they improve retrieval, placement, or clarity.

## Rule 10 — Clarity Before Save

Before saving a new idea, helper, method, word, or filing category:

- run Bell Logic if alarms ring
- run Word-Key Clarity if terms become keys
- check source/support/authority boundary
- check lifecycle state
- check retrieval path
- check neighbor impact
