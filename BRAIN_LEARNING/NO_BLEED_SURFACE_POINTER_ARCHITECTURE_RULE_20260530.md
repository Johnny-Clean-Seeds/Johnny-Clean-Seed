Date: 2026-05-30
Status: SUPPORT RULE / NO-BLEED POINTER ARCHITECTURE
Authority: not doctrine, not ACTIVE_GUIDES, not CURRENT_TRUTH_INDEX

## Correction

Git needs its durable anchor files.
AssistantLocal needs its chat-load pointer files.
Chat needs minimal live keys.
User-local 123 files need their own system and must not be mixed with assistant-local storage.

The concepts can be the same.
The placement, authority, density, codes, receipts, and stale checks must differ.

## No-bleed hard blocks

No assistant-local pointer files in user Command Center unless explicitly requested for a user-facing bridge.
No user-local root drop files stored in AssistantLocal.
No chat raw transcript stored as Git truth.
No Git proof file treated as local execution authority.
No user-local operation treated as assistant memory unless saved/pointed by receipt.
No pickup copies unless the target surface's purpose requires it and the bridge reason is named.

## Bridge exception

A user-local file may exist for assistant's sake only when it serves a named purpose:
pickup, command packet, receipt, context cart, or operator handoff.

That bridge file must declare:
why it exists, who reads it, which Git anchor it points to, whether it is local-only, and when it is stale.

## Supersession note

This rule supersedes any draft updater that writes assistant pointer copies into user Command Center/PICKUP by default.
