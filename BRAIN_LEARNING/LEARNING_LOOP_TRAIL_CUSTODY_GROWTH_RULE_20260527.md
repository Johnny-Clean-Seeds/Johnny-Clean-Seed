## Founding question

How do I learn to grow and learn if I cannot get this right?

## Clean answer

Growth is not perfect first execution.

Growth is making sure failure cannot pass through unchanged.

A final successful repair does not close a compounding failure trail. It only proves the last repair. The whole trail must still be taken into custody, washed, mapped, and converted into changed behavior.

## Core loop

Experience -> Freeze -> Trail Custody -> Cause Map -> Existing Rule Check -> New Lesson Extract -> Think Tank Expansion -> Gate Wash -> Soft Suit Apply -> Proof Against Original Problem -> Save One Object -> Shed Weight -> Resume.

## Rule

After a compounding failure trail, do not close from the last successful command, last successful cleanup, or last clean git status.

Close only when:

1. the active issue is named;
2. the whole trail is preserved as evidence;
3. the cause map names what made the wrong move possible;
4. existing rules that failed to fire are identified;
5. new living lessons are extracted;
6. clean lessons are applied to the Soft Suit before next action;
7. the corrected route is proofed against the original failure, not merely against the final patch;
8. one object at a time is saved or parked;
9. saved material is shed from active suit/chat weight into pointer form;
10. remaining residue is classified;
11. execution is reopened only after the route passes.

## The trail is evidence

The trail includes:

- failed commands;
- bad assumptions;
- parser errors;
- wrong delivery surface;
- unverified file paths;
- missing custody checks;
- loose helper scripts;
- root/porch residue;
- ignored lanes;
- untracked receipts;
- git wrapper mistakes;
- user corrections;
- assistant over-motion;
- final repair proof.

The trail must not be erased by the final success.

## Existing rule check

If a rule already existed but failed to fire, the problem is not lack of words.

The question becomes:

- Was the trigger missing?
- Was the Soft Suit not wearing it?
- Was the active object misclassified?
- Was code allowed before issue resolution?
- Did the assistant confuse knowing the rule with applying the rule?
- Did the script prove the wrong thing?
- Did proof happen too late?

## Soft Suit adoption requirement

A clean lesson is not learned because it was written.

A lesson is learned when it changes the next move.

Before the next action, the assistant must show the behavior change directly.

## Proof against original problem

Do not prove only the patch.

If the original failure was "rushed into code before route was resolved," then a successful script does not prove the fix.

The proof is:

- no code while route is unresolved;
- active issue named first;
- ammo pouch checked;
- residue classified;
- route accepted;
- then execution.

## Too-many-tries drift rule

If the same class of failure repeats, stop the current lane.

Verdict becomes:

DRIFT DETECTED / STOP EXECUTION / RECLASSIFY ROUTE.

Do not keep patching in the same surface.

Change the route by reducing scope, changing delivery method, requesting inventory, parking the object, or pausing for trail map.

## Redundancy / Waste relationship

After save/lock, drop dead chat/suit weight.

Carry only:

- pointer;
- trigger;
- file/home location;
- return condition.

Saved proof is not meant to be carried as full live load.

## Porch / root relationship

Do not treat root mess as unrelated.

Loose helper scripts, receipts, source files, and generated artifacts are residue from the work trail.

They must be classified before movement.

## Short form

Failure cannot pass through unchanged.
