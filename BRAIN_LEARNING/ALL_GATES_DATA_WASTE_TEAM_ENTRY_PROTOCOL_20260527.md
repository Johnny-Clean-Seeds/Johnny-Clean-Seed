## Purpose

Prevent data/token waste without crowning one file, one gate, one score, or one assistant guess as the target.

The house already has several waste-adjacent organs. This protocol tells the assistant to let the gate team decide together before writing, updating, shedding, burning, dissolving, parking, or carrying material.

## Core rule

Data Waste is a team behavior, not one throne.

When dead chat weight, duplicate carried material, old notes, repeated explanations, saved-but-still-carried rules, or over-loaded suit material appear, the assistant must read the team, identify each gate's job, and choose the smallest safe entry.

Short form:
No king ding-a-ling. All gates decide. Carry the key, not the furniture.

## Team roles

Redundancy / Waste Gate detects repeated chat weight, duplicate ideas, saved material still carried live, over-carried suit load, repeated source material, and unnecessary bulk.

Safe Dissolution decides the live-load state: carry, pointer-only, parked, blocked, local custody, source lane, silence, or dissolved from active load.

Memory Forest / Grey-Vanish Cascade condenses old chat into roots, branches, cross-links, proof pointers, grey residue, and vanished-from-live state only after meaning is safe.

Note Burning / Deadweight Release releases active note weight after clean disposition. Burning means the useful part has moved to the right object; it does not mean deleting source truth.

No Parking Without Fit Decision prevents "later" piles. Anything held must have lane, reason, return trigger, risk, and next action.

Cockpit / Chat Behavior decides what must affect the assistant live in chat and exports drop-copies when cockpit behavior changes.

Twelve-Gate Refinery and Final Judge check role fidelity: every object has a job, every gate stays in lane, proof is not skipped, fake PASS is blocked, and no gate becomes monarchy.

CONSOLIDATOR may later propose cleaner packing only after repeated evidence exists. It does not merge or kill gates by itself.

## Read order

1. Identify the active object.
2. Ask whether the object is live work, saved project material, proof, source, note residue, cockpit behavior, mule scope, or local-only operational material.
3. Read the relevant gate/team files before choosing a write.
4. Separate detection from disposition.
5. Separate disposition from deletion.
6. Separate score/neighborhood discovery from write authority.
7. Decide whether the material is already covered, partially covered, missing, or misfiled.
8. Choose the smallest safe entry.
9. Write only after the target and boundary are proven.
10. Close with proof, receipt, and clean carry state.

## Write-decision outcomes

Allowed outcomes:

- NO WRITE — ALREADY COVERED
- POINTER ONLY — CARRY THE KEY
- UPDATE EXISTING TARGET
- WRITE SMALL CONNECTOR MAP
- UPDATE COCKPIT ONLY
- PARK WITH RETURN TRIGGER
- BLOCK — NEEDS HUMAN FILE DECISION
- BLOCK — SOURCE/PROOF CUSTODY NOT SAFE
- BLOCK — WOULD DELETE OR HIDE EVIDENCE
- READY FOR SAVE ROUTE

A score report may find neighbors, but it must not choose a write target by itself.

## What counts as dead chat weight

Dead chat weight includes:

- old explanations repeated after a cleaner rule exists;
- raw chat already converted to a rule, list, map, receipt, research paper, or pointer;
- saved Git material still carried in full live chat;
- terminal logs carried after the relevant proof lines were extracted;
- source text carried after source custody and clean extraction exist;
- parked items with no return trigger;
- notes that keep pulling attention after their job is known;
- helper scripts, wrappers, or code blocks carried after a file-first launcher exists;
- repeated assistant apology/repair language after the repair has a rule and proof.

## What must not be burned or dissolved

Do not burn, dissolve, grey, compress, or unload:

- proof needed to verify a claim;
- source custody;
- raw evidence that still carries unresolved meaning;
- receipt trails;
- local truth;
- user constraints not yet routed;
- active blockers;
- privacy/safety warnings;
- dirty repo state;
- mule/assistant boundary lines;
- anything the next move actually needs.

Dissolve means stop carrying live weight. It does not mean delete.

Burn means release active note weight after disposition. It does not mean destroy the root trace.

## Live carry standard

After a save, wash, route, or proof pass, carry only:

- active object;
- trigger;
- current blocker;
- proof pointer;
- next condition.

Do not carry the furniture when the key is enough.

## File-entry standard

Before inserting or updating files, produce or internally satisfy this file-entry check:

- Which file already owns the job?
- Which neighbor files only support it?
- Is the material assistant-facing, project-durable, proof-only, source-only, mule-only, local-only, or parked?
- Does cockpit need visibility?
- Does Git need local + URL save?
- Does any source/transcript/raw material need to stay out of Git?
- Does the change create a duplicate rule?
- Does it flatten a useful gate team into one king gate?
- What proof closes the move?

## Blocked moves

Blocked:

- no single gate gets the throne;
- no "highest score" target selection without reading the file role;
- no delete/cleanup action from this protocol alone;
- no proof, source, or receipt burning;
- no ACTIVE_GUIDES rewrite;
- no CURRENT_TRUTH_INDEX rewrite;
- no doctrine promotion;
- no raw transcript/source Git staging;
- no auto-save from a chat idea without route/proof;
- no cockpit update without required CHAT drop-copies;
- no mule rule change unless mule lane is explicitly selected.

## Relationship to Extended Thinking Growth Valve

This protocol reduces carried data weight. It must not reduce reasoning depth.

The assistant may think deeply to find the right role and route, then unload the temporary reasoning scaffolding after the useful part is placed.

Short form:
Deep thinking is allowed. Dead carry is not.

## Proof test

A correct run should:

1. find the waste cluster;
2. refuse to crown one file;
3. identify team roles;
4. propose the smallest safe entry;
5. preserve proof/source custody;
6. update cockpit only if assistant-facing;
7. save local + URL only when durable;
8. unload raw chat weight after pointer/proof exists.
