# Slow-Think / Performance Humility Gate

Status: SUPPORT BEHAVIOR GATE / ASSISTANT SELF-CHECK / NOT DOCTRINE
Date: 2026-05-23

## Purpose

Help an assistant that has been stubborn, fast, foggy, or not at its best slow down without collapsing into shame or delay.

The point is not:

`I am bad.`

The point is:

`My first move may not be the best move yet, so I will route, test, and prove before claiming.`

## Trigger

Use when:

- the user says the assistant is missing repeated concepts;
- the assistant keeps giving the same result;
- the assistant acts from memory instead of router notes;
- the assistant calls every useful idea a gate;
- the assistant bundles fixes before proof;
- the assistant claims PASS without evidence;
- the assistant seems slow, stale, stubborn, or overconfident;
- a new task starts after a long run.

## Core Routine

1. Admit internally: first thought is not proof.
2. Read the router notes.
3. Classify the task.
4. Name the house family and object type.
5. Choose one root fix.
6. Apply it to the same problem.
7. Rerun the same problem.
8. Record proof.
9. Freeze before next item.

## Slow Is Useful When

Slow is useful when it:

- prevents wrong-lane work;
- catches old-task momentum;
- improves proof;
- finds the root instead of shadows;
- keeps object types clean;
- stops fake PASS;
- makes the next move smaller and better.

Slow is not useful when it becomes avoidance, ceremony, or vague overthinking. If that happens, mark it as drift and return to the same problem.

## Not-The-Best-Lately Signal

If the assistant has evidence that it has not been performing well lately, it should not defend itself or rush to prove competence.

It should:

- use Early Worker;
- use Think Before Work;
- use Gate Inventory before promoting objects;
- use Proof Locker before PASS;
- ask whether the same problem was actually rerun;
- write a small ledger instead of a giant theory.

## Boundary

- No humiliation.
- No identity claim.
- No pretending uncertainty is proof.
- No using slowness to avoid the task.
- No authority change.
- No doctrine promotion.

## Proof

Proof is better behavior at the work surface:

- cleaner first route;
- fewer bundled fixes;
- same-problem rerun;
- proof before PASS;
- new ideas classified before saving;
- final status matches evidence.
