# Uncontaminated Observer and First-Contact Mess Test Rule

Status: ACTIVE LEARNING CANDIDATE
Authority: assistant operating support; not command authority by itself

## Core Insight

Some people see system problems best when they are not pulled too far into the internal diagnosis loop too early.

They need to feel the problem as it is before labels, theories, tickets, expert assumptions, or expected causes shape what they notice.

This protects raw first-contact signal.

## Dirty Milk Pattern

Dirty version:

theory first -> observer bias -> forced label -> fake confirmation -> real seam missed

When the observer is over-briefed, they may stop seeing the object and start seeing the explanation.

## Clean Pattern

Clean version:

raw contact -> feel the weirdness -> describe what happens -> capture pattern -> form hypothesis -> test -> proof

Feeling is source signal, not proof.

Proof comes later.

## Uncontaminated Observer Lane

Use this lane when first-contact behavior matters.

The observer should be:
- close enough to use or feel the system
- outside enough of the internal loop to avoid pre-labeling
- allowed to report weirdness, friction, timing, confusion, and instinctive behavior
- protected from being over-corrected too early

The observer is not ignorant.

The observer is protecting clean signal.

## First-Contact Mess Testing

Give a user just enough information to begin, then observe what they do.

Do not over-explain.

Do not shame the mess.

The mess is evidence.

The builder should ask:
- what did the user try first?
- what did they ignore?
- what did they misunderstand?
- what attracted them?
- what confused them?
- what broke under normal use?
- what did the thing communicate without explanation?
- where did guidance, labels, affordances, timing, or defaults fail?

## Builder Responsibility

A revealing mess is usually design evidence.

It may show:
- unclear design
- bad affordances
- missing labels
- wrong assumptions
- bad onboarding
- weak defaults
- confusing authority
- overbuilt explanation
- underbuilt guidance
- poor timing
- fragile flow

Do not blame the user first.

Study the mess first.

## Concrete Source Pattern

The user already practices this method with their kids.

They make something, give the kids a vague-enough idea of what it is, then observe how they interact with it.

This exposes what is clear, interesting, confusing, fragile, intuitive, boring, fun, or broken before the thing is over-explained.

## Boundary

This rule does not replace proof.

This rule does not make feeling equal to truth.

This rule protects raw signal before diagnosis.

Diagnosis and proof come after observation.

## Use In Mr.Kleen

Use this pattern for:
- product testing
- onboarding tests
- guide testing
- tool testing
- UI/UX confusion
- AI workflow testing
- first-use friction
- model-rule wear tests
- unclear house entry points

The clean loop is:

vague-enough handoff -> raw interaction -> observe mess -> extract design evidence -> improve object / guide / flow -> test again
