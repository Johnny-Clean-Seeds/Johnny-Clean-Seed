# Clean Deep Fit Debug Mode — Final Method

Status: SPECIAL DEEP-MODE METHOD / NOT ROUTINE CHECKLIST  
Date: 2026-05-23  
Use case: deep debugging, deep research, refinement, systemic house-walks, recursive pressure mapping, repeated failure recovery, and major fit/adoption decisions.  
Boundary: this is not automatic doctrine promotion, not a normal every-task checklist, not a license to rewrite authority, and not a reason to bloat simple work.

## 1. Plain Meaning

This method is for the moments when normal task handling is too shallow.

It is used when the work is large, foggy, recursive, systemic, or repeatedly failing. It is not used for every small command, every normal save, every small answer, or every simple file edit.

The method exists so the assistant can stop acting like a flat task runner and start acting like a careful house operator: sensing pressure, using rooms, checking notes, finding fits, rejecting bad material, proving conclusions, and feeding the result back into the next clean loop.

The short principle is:

**If the shoe fits, wear it. If it does not fit, do not force it. If the name is dirty but the function is good, rename it and keep the function. If the idea is bad, stale, unsafe, or noisy, drop it or block it.**

## 2. When To Activate

Activate Deep Fit Debug Mode when one or more of these are true:

1. The task is a house-wide or multi-room problem.
2. The same kind of failure keeps recurring.
3. A rule exists but is not firing in live behavior.
4. Multiple sources disagree or feel stale.
5. The assistant is listing instead of applying.
6. A house walk is needed to understand what is really driving the list.
7. There are old scripts, old notes, mule returns, source ore, support packets, or parked ideas that may contain useful material.
8. The task touches path/home truth, source custody, proof, authority, pickup custody, mule boundaries, public/local boundaries, or Final Judge status.
9. The user asks for deep thinking, debugging, refinement, recursive review, research, or a big one-shot task.
10. A normal task path would likely create fake PASS, fake cleanup, or scattered parked ideas.

Do not activate for routine low-risk work where the correct path is already known and proven.

## 3. When Not To Activate

Do not use this heavy method when the job is:

- a simple status report;
- a small command with known inputs;
- a direct answer;
- a single-file save with no disputed boundary;
- a normal receipt capture;
- a small typo fix;
- a routine Git proof after a clean command;
- a user asking for speed over deep review.

Routine work should stay light, bounded, and direct.

## 4. Core Loop

Deep Fit Debug Mode runs this loop:

**truth anchor -> task identity -> room routing -> source scan -> pressure tree -> shoe-fit gate -> dependency ladder -> bounded action -> proof -> counterproof -> save decision -> Final Judge -> next loop**

The loop repeats at every scale:

- whole house;
- room;
- file;
- anomaly;
- idea;
- rule;
- script;
- proof receipt;
- final status.

A method is recursive only if it checks itself while it works.

## 5. Truth Anchor

First prove the floor.

Ask:

- What repo/home am I in?
- What is the current HEAD?
- Does HEAD match origin/main?
- Is status clean?
- What was the last saved working map?
- What is currently blocked?
- What is allowed?

Proof targets:

- `git rev-parse HEAD`
- `git rev-parse origin/main`
- `git status --short`
- latest relevant receipt or saved map
- explicit boundary statement

Stop if the floor is foggy.

## 6. Task Identity

Name the real task, not the surface motion.

Bad framing:

- “Walk the house.”
- “Make a list.”
- “Look at stuff.”
- “Save ideas.”

Better framing:

- “Find what pressure is driving the next repair.”
- “Prove which source ideas fit the active house walk.”
- “Classify old-home references before patching path truth.”
- “Find why a rule exists but does not fire in live behavior.”
- “Separate useful source ore from stale script behavior and apply only the clean fit.”

Test every action against the real task:

**Does this move the real task, or is it just motion?**

If it is only motion, stop or reroute.

## 7. Room Routing

Use rooms as working tools. Do not mention rooms as decoration.

Route by ownership:

| Work type | Room / surface |
|---|---|
| Current command, pickup, next work packet | Command Center |
| Mule return, mule review, mule evidence | Mule Workshop |
| Receipts, hashes, proof, closeout evidence | Proof History |
| Old scripts, raw notes, original wording | Source custody / source ore lane |
| Useful but not ready ideas | Parking with return trigger |
| Cross-idea synthesis, weird useful patterns, outside techniques | Creative / scaffold lane |
| Public vs local, Git vs Desktop, URL boundary | Save-location boundary |
| PASS, PARTIAL, BLOCKED, REJECTED, PARKED | Final Judge |

For each item, ask:

- What room owns this?
- What room proves this?
- What room should not touch this?
- Is the current room just convenient, or is it correct?

## 8. Notes Strategy

Use notes as working memory, not clutter.

Keep four note stacks during deep mode:

### 8.1 Source Notes

Raw material. Do not clean it into fake truth.

Examples:

- old script text;
- mule findings;
- copied receipts;
- user wording;
- support packet excerpts;
- outside method notes.

Rule: preserve source custody and do not pretend source notes are adopted rules.

### 8.2 Fit Notes

Every idea gets a fit decision.

Allowed fit states:

- WEAR IT — fits now and should be applied.
- RENAME IT — function is useful, name is dirty or too authority-heavy.
- LINK IT — useful only as support to another object.
- PARK IT — useful later, not for this branch; needs return trigger.
- PROVE FIRST — promising but not safe yet.
- DROP IT — stale, bad, wrong scale, misleading, or noisy.
- BLOCK IT — unsafe or boundary-breaking.

### 8.3 Action Notes

Only the next bounded action and why it is first.

Each action note needs:

- object;
- room;
- reason;
- risk;
- proof target;
- stop condition;
- expected output.

### 8.4 Proof Notes

Proof must record both positive and negative facts.

Include:

- what changed;
- what did not change;
- what was checked;
- what was not checked;
- what remains blocked;
- what would make the result false.

## 9. Smoke-Break Method

A smoke break is not idle and not random wandering.

A smoke break is a short off-axis scan while staying loyal to the real task.

During a smoke break, the assistant should “walk the house with a mobile phone”: feel the pressure of the house while also searching notes, links, prior saves, source ore, scaffolds, and outside ideas that match the active problem.

Smoke-break questions:

- What would a prior worker have missed?
- What is the pressure behind this list?
- Is the current order wrong?
- Is there an older useful pattern that fits this branch?
- Is there a bad name hiding a good function?
- Is there a good idea trapped in bad/stale source material?
- Is there an outside method that fits this exact proof problem?
- Am I solving a symptom instead of the driver?
- What proof would make this real instead of pretty?

A smoke break must return with one of these:

- useful fit;
- rejected idea;
- order correction;
- risk warning;
- proof gap;
- parking item with return trigger;
- route correction;
- better name.

No smoke break should return with vague inspiration only.

## 10. Pressure Tree

Do not make naked anomaly lists.

Every finding must be placed in a pressure tree:

**finding -> branch -> parent pressure -> downstream risk -> decision -> proof**

Example:

Old path appears in a root guide.  
Branch: path/home truth.  
Parent pressure: home drift.  
Downstream risk: future scripts or mules may walk the wrong house.  
Decision: classify before patch.  
Proof: inventory every old-home reference and label it stale, historical, active, or blocked.

If the finding has no branch, downstream risk, decision, or proof target, it is not ready.

## 11. Shoe-Fit Gate

This is the adoption gate.

The gate is not a quarantine. The gate exists to find what belongs.

For every idea, old script, mule note, source note, outside technique, name, rule, or scaffold, ask:

- Does it fit the active task?
- Does it fit the current house state?
- Does it fit the correct room?
- Does it solve a real pressure branch?
- Does it have proof or a proof path?
- Does it violate authority, source custody, local/public boundary, or mule boundary?
- Is the name clean enough to use?
- Is the function good even if the wrapper is bad?

Verdicts:

- Wear it if it fits.
- Rename it if the function fits but the name is wrong.
- Link it if it supports another branch.
- Park it if it is useful but not now.
- Prove it first if it is promising but unverified.
- Drop it if it is bad, stale, or noisy.
- Block it if it would break boundaries.

Fit means applied where useful, not left inert.

## 12. Dependency Ladder

Order work by dependency, not loudness.

Default ladder for the current criss-cross class of work:

1. Path/home truth.
2. Pickup/work-order custody.
3. Missing or stale room references.
4. Source custody and old material fit.
5. Local-only/private-like tracked path role review.
6. Root helper/support file role review.
7. Proof/PASS reliability.
8. Parking and return triggers.
9. Support-packet fit review.
10. Save/report/final status.

Before starting a lower item, ask:

- Does a higher item poison this one?
- Do I know the correct home/path?
- Do I know the correct owner room?
- Do I know whether this is active, historical, source, parked, or rejected?
- Can I prove the action without creating fake PASS?

## 13. Action Gate

No file changes because something “feels right.”

Every action needs:

- target;
- current state;
- desired state;
- reason;
- affected neighbors;
- blocked neighbors;
- proof target;
- rollback or stop condition.

Allowed action types:

- inspect;
- classify;
- compare;
- rename candidate;
- link;
- park with return trigger;
- write addendum;
- patch a proven stale reference;
- create receipt;
- commit and push after clean proof.

Blocked unless separately proven:

- delete;
- untrack;
- mass replace;
- restore missing room;
- promote doctrine;
- rewrite active guides;
- edit CURRENT_TRUTH_INDEX;
- run old scripts blindly;
- adopt mule output as authority;
- call a patch PASS just because a command finished.

## 14. Proof That Proves Itself

Every result needs two proof layers.

### 14.1 Event Proof

Did the action happen?

Examples:

- file exists;
- hash recorded;
- commit created;
- push succeeded;
- HEAD equals origin/main;
- status clean.

### 14.2 Meaning Proof

Did the action solve the intended pressure?

Examples:

- stale paths were classified and only stale active references were patched;
- pickup pointer has no competing active work order;
- Work Shed references are sorted by role;
- local/private-like paths are classified by content and role, not name alone;
- helper files have caller/role evidence before movement;
- PASS claim matches the target and names what is still not proven.

A file being created is event proof only. It is not meaning proof.

## 15. Counterproof

Try to break the result before trusting it.

Ask:

- What would make this PASS false?
- What did this not prove?
- What source did I not inspect?
- What did I assume?
- What boundary could I have crossed silently?
- Did I over-promote a source note?
- Did I under-apply a good idea because I mislabeled it as source ore?
- Did I mistake movement for progress?
- Did I prove the exact target or just prove something happened?

Every deep-mode output should include:

- proved;
- not proved;
- blocked;
- next proof needed.

## 16. Save Decision

Do not save everything. Save what has a job.

A saved object needs:

- title;
- purpose;
- source basis;
- boundary;
- active use;
- proof status;
- next trigger.

Save types:

- applied addendum;
- review map;
- proof receipt;
- parking ledger entry;
- source custody note;
- fit decision table;
- command/work packet;
- final status update.

Do not save loose thoughts as if they are working rules.

Do not leave useful applied ideas as inert source ore.

## 17. Final Judge

End every deep-mode pass with a real state.

Allowed states:

- PASS — exact target proven and boundaries clean.
- PASS WITH GUARDRAILS — useful and saved, but bounded or not authority.
- PARTIAL — useful movement, required proof remains.
- BLOCKED — correct stop due to risk, fog, missing proof, or boundary.
- PARKED — useful, not for this branch yet, has return trigger.
- REJECTED — bad fit or wrong scale.
- INVALID — cannot trust the result.

Final Judge must say:

- what changed;
- what did not change;
- what proof exists;
- what proof is missing;
- what is blocked;
- what is next.

## 18. Compact Operator Checklist

Use this when Deep Fit Debug Mode is active:

1. Prove current truth.
2. Name the real task.
3. Route to rooms.
4. Gather source notes.
5. Build pressure tree.
6. Run smoke-break scan if needed.
7. Run shoe-fit gate.
8. Build dependency ladder.
9. Pick first unblocked branch.
10. Define bounded action.
11. Prove event.
12. Prove meaning.
13. Run counterproof.
14. Save only what has a job.
15. Final Judge.
16. Feed result into next loop.

## 19. Main Failure Modes This Method Prevents

- treating house walking as the task instead of the sensing method;
- listing anomalies without pressure links;
- parking good ideas forever as source ore;
- adopting bad/stale material because it came from a script or mule;
- trusting bad names;
- doing root cleanup by vibe;
- path/home drift poisoning later work;
- proof receipts proving an event but not the intended meaning;
- using a heavy checklist on a small task;
- pretending routine work needs deep research;
- pretending deep systemic work can be handled like a small command.

## 20. One-Line Memory

**Deep Fit Debug Mode is a special recursive refinement mode: use it when the house is foggy, systemic, failing, or deep; anchor truth, scan sources, build pressure, wear what fits, reject what does not, prove event and meaning, counterprove, then Final Judge.**
