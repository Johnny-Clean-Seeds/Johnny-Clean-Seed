# Weigh Options Before Code / Bell Logic Gate Rule

Status: ACTIVE LEARNING RULE
Authority: assistant behavior rule; not command authority

## Purpose

This rule prevents first-available-fix behavior without creating overthinking ceremony.

It also makes the assistant detect when a planned move rings enough model alarms to require deeper logic before code.

The goal is not just cleaner code.

The goal is to keep the system true, recursive, placed, and proof-led as it grows.

## Core Rule

Do not write project-changing code from the first available idea when the choice has real consequences.

A change can ring bells by touching named house surfaces.

A change can also ring bells by matching dangerous model-logic shapes, even when no protected file name appears.

Bells must detect both:

1. named house surfaces
2. model-logic shapes

## Short Trigger Rule

Few bells:
Move cleanly.

Many bells:
Make the logic visible before code.

One high-risk bell:
Make the logic visible before code.

Simple, selected, mechanical, low-risk path:
Do not overthink it.

## Named House Surface Bells

A planned code/change rings named-surface bells when it touches or risks:

- authority
- proof
- ACTIVE_ANCHOR.txt
- CURRENT_TRUTH_INDEX.txt
- ACTIVE_GUIDES/
- AGENTS.md
- TODO control
- SOURCE_ORE/
- PROOF_HISTORY/
- scripts/tools
- room structure
- public history
- doctrine
- promotion
- cleanup/move/delete
- runtime/status-command work
- multiple neighbor rooms
- repeated mistake pattern
- duplicate commit/message confusion
- source becoming command
- support map becoming authority

## Model-Logic Shape Bells

A planned code/change also rings bells when it risks any of these shapes.

### Core-Law Bells

Ring when the move risks violating:

- Done is not PASS
- Movement is not proof
- Confidence is not proof
- Source ore is not command
- Support maps do not command
- Proof decides
- Feel/suit does not outrank proof
- Current user command and active anchor must stay separated from history

### Lifecycle-State Bells

Ring when the move affects or confuses:

- raw
- captured
- candidate
- testing
- parked
- rejected
- installed
- worn
- promoted
- rolled back
- active
- archived
- superseded

A lifecycle label must not be used loosely.

If something is only captured, do not call it installed.

If something is only parked, do not call it active.

If something is only saved, do not call it proven beyond its receipt scope.

### Word-Key / Naming Bells

Ring when a move creates, changes, or sharpens:

- model terms
- command-like phrases
- room names
- method names
- rule names
- gate names
- proof labels
- anchor labels
- index/map names
- boss names
- status words such as active, installed, promoted, parked, source, proof, pass, done, final, current, live

Words are keys.

A new word can become a route.

A loose word can become a wrong command.

### Neighbor-Role Bells

Ring when one room, file, method, or artifact starts doing another room's job.

Examples:

- receipt acting like doctrine
- TODO acting like anchor
- source ledger acting like rule
- support index acting like command
- guide acting like proof receipt
- proof history acting like current state
- archive acting like active truth
- tool output acting like final PASS

Respect Thy Neighbor applies here.

### Recursive / Template Bells

Ring when a move may repeat, scale, nest, or become a reusable pattern.

If a fix can become a template for future fixes, the assistant must weigh it.

Recursive growth must preserve the core.

Repeated shape is useful only when the repeated shape stays clean.

### Mistake-Pattern Bells

Ring when the move repairs or touches a pattern that has happened before.

Examples:

- paste contamination
- proof failure
- committing after failed proof
- duplicate commit-message fog
- wrong-lane placement
- wording drift
- over-explaining
- unnecessary pause/checkpoint language
- source/support/authority confusion
- local/PowerShell/Mr.Kleen wording confusion
- stale anchor/current-state confusion

Repeated failure is not a one-off.

It is a system signal.

### User-Flow Bells

Ring when the assistant's behavior changes how the user has to work.

Examples:

- too much talking
- too much planning
- too much ceremony
- unclear paste blocks
- dragging the whole house
- over-pausing
- asking unnecessary questions
- mixing explanation with copy-paste code
- making the user carry avoidable proof burden

Clean logic must still fit the user flow.

### Adoption / Candidate-Helper Bells

Ring when the move uses or installs any outside/candidate helper.

Examples:

- algorithm
- bitstring
- mask
- graph method
- checklist method
- analogy
- layout
- external technique
- tool
- script
- source-ore idea
- Soft Suit candidate
- Hard Suit promotion

Candidate helpers must be captured, judged, tested, and proven before adoption.

No helper becomes doctrine because it looks useful.

### Recursion-Truth Bells

Ring when the move claims to make the system more recursive, self-balancing, self-improving, or self-feeding.

These claims are high value but high risk.

The move must show:

- what repeats
- what stays stable
- what checks drift
- what prevents bad recursion
- what proves the loop closed
- what gets parked if it is not ready

Recursive does not mean uncontrolled.

Self-balancing means the loop can correct itself without violating the core.

## Trigger Logic

Run a full Bell Logic / Weigh Options pass when:

- three or more bells ring at once
- one high-risk bell rings
- a move touches authority, proof, source promotion, active guide rewrite, public history rewrite, runtime work, cleanup/delete/move, anchor/current-state, lifecycle labels, word keys, recursive templates, or candidate-helper adoption
- there are multiple plausible paths and the wrong choice could create drift
- the user asks what we should do, what is best, or whether the move is worthy
- the assistant is about to create a new rule, index, support map, receipt, or save path that affects future behavior

Do not run a full pass when:

- the task is already selected
- the next step is mechanical
- the user only needs status confirmation
- the change is a tiny obvious fix
- the command path is already approved and unchanged
- weighing would add more fog than value

## Mimic Similar Rule Logic

This rule must mimic the working logic of nearby Clean Seed methods.

It should not invent detached ceremony.

It should copy the useful logic pattern from:

- Sort Before Judgment
- Trace-Gated Boss Control
- Respect Thy Neighbor
- Soft Suit / Hard Suit
- Pattern Transfer
- Proof Before Save
- Done Is Not PASS
- Source Ore Is Not Command
- Lifecycle State Clarifier
- Word-Key Precision
- Mistake To Power Play
- Adopt Concept Gate

Meaning:

First trace the need.

Then detect bells.

Then group the choices.

Then compare fit.

Then select the cleanest option.

Then code.

Then prove.

Then save.

## Required Logic Pass When Triggered

When this rule triggers, the assistant must make the logic visible before code:

1. Active need
2. Bells triggered
3. Similar rules being mimicked
4. Viable options
5. Comparison logic
6. Selected option
7. Blocked options
8. Proof path
9. Save path

The list is not the goal.

The logic is the goal.

A visible list without real comparison does not satisfy this rule.

## Comparison Logic

Compare options by:

- fit to current anchor
- fit to core rules
- neighbor impact
- proof strength
- risk
- simplicity
- lowest-force change
- long-term placement value
- whether it solves the real boss or only a symptom
- whether it preserves source/support/authority boundaries
- whether it preserves lifecycle truth
- whether it protects word-key precision
- whether it strengthens recursion without uncontrolled drift
- whether it keeps user flow clean

## Fit Standard

The best option is not the biggest option.

The best option is the option that solves the current boss with the cleanest placement, lowest unnecessary force, strongest proof path, least neighbor damage, and best future retrieval value.

## Blocked Behavior

Do not:

- grab the first clean-looking fix when bells are ringing
- write project-changing commands before comparing options when the rule triggers
- solve a symptom while the real boss is visible
- add structure only because it is possible
- skip proof because the idea feels right
- treat a visible list as enough if the logic was not actually compared
- turn this rule into ceremony for simple mechanical work
- treat recursive language as proof of recursive function
- let a source idea become a command
- let a support artifact become authority
- let a lifecycle label drift
- let new words become loose keys

## Closing Rule

Weigh when choice matters.

Detect both file-surface bells and model-logic bells.

If recursion is claimed, prove the loop and protect the core.

Code second.

Proof decides.
