# Router-First Same-Problem Fix Ledger Rule

Status: SUPPORT RULE / ROUTER AND PATCH-THROUGH SCAFFOLD / NOT DOCTRINE
Date: 2026-05-23

## Purpose

Make the assistant start from router notes, classify the work, apply one clean fix to the same exposed problem, rerun that same problem, and record the result before moving on.

This rule prevents the common failure where a boss loop finds a good idea, judges it in isolation, saves a loose note, then starts the next gate as if the previous gate taught nothing.

## Trigger

Use this scaffold when the user says or implies:

- start from the router notes, not memory;
- classify first;
- route before acting;
- apply the fix to the same problem;
- rerun after the fix;
- keep a live fix ledger;
- do not bundle fixes without proof;
- freeze and review before moving to the next item;
- make sure gate effects carry forward.

## Router-First Flow

1. Start from the router notes, not memory.
2. Classify the task first.
3. Route to the right house-rule family.
4. Walk the source as terrain.
5. Identify the layer and object type before acting.
6. Apply one fix to the same problem.
7. Re-run the same problem after the fix.
8. Record proof, receipts, and any new idea.
9. Keep a small live fix ledger.
10. If a new idea appears, classify it as gate, tool, scaffold, parking item, or proof surface before saving it.
11. Do not bundle fixes unless the proof shows the bundle is cleaner.
12. Freeze, review, then move to the next item.

## Object-Type Classifier

Before saving a new idea, classify it as exactly one primary object type:

- GATE: decides, blocks, permits, or changes route.
- TOOL: performs a repeatable action.
- SCAFFOLD: supports another gate or process without authority.
- PARKING ITEM: useful but not active yet.
- PROOF SURFACE: records evidence, receipt, hash, trace, or validation.

If the type is unclear, park it with a return trigger instead of promoting it.

## Same-Problem Rule

The fix must be applied to the problem that exposed the gap.

Do not prove a fix by using it on a new easier example.

Minimum proof loop:

`issue -> router note -> classification -> one fix -> same issue rerun -> proof -> freeze`

## Gate Effect Carry-Forward Rule

Every gate or route row must record what it did to the current fix before the next gate starts.

Use the existing patch marks:

- KEEP: the fix works here.
- NARROW: the fix is too broad here.
- EXPAND: the fix must include this surface.
- BLOCK: the fix would violate a boundary here.
- PARK: related, but not the current boss.
- PROOF: cannot claim until verified.
- PARENT: this gate reveals a deeper boss.

The next gate starts from the revised fix, not the original fix.

## Gate-Judges-Gate Rule

Each gate must judge both:

1. the active problem;
2. the previous gate's effect on the active fix.

Ask:

- Did the last gate improve the fix?
- Did it make the fix too broad?
- Did it hide a boundary?
- Did it expose a neighbor gate that must fire now?
- Did it create a new idea that needs object-type classification?

This keeps the loop from spinning around the same unpolished checklist.

## Live Fix Ledger

Use a small ledger, not a giant report.

Required columns:

| Step | Router Source | Task Class | House Family | Layer / Object Type | Current Fix Variant | Gate Mark | Same-Problem Rerun | Proof | New Idea Disposition | Freeze Status |
|---|---|---|---|---|---|---|---|---|---|---|

The ledger is complete only when the same problem was rerun and the freeze status is named.

## Bundle Brake

Default to one fix per problem.

Bundle fixes only when proof shows the bundle is cleaner than separate moves.

Bundle proof must name:

- shared root problem;
- why separate fixes would create more fog;
- how the bundle will be rerun against the same problem;
- what rollback or park lane exists if the bundle fails.

## Repeated Result Plateau Test

If repeated boss loops return the exact same result, do not treat that alone as proof.

Classify it as one of two cases:

- STABLE FIXED POINT: different routes, starts, or source slices still converge.
- STALE LOOP PLATEAU: the same template, same inputs, or same memory path is replaying.

Test by changing the route order, start lens, or source slice while keeping the same problem.

If the result survives that variation, confidence rises.

If it changes, capture what the old loop was missing.

## Freeze Rule

Before moving to the next item:

- name the current fix variant;
- name what changed from the starting variant;
- name the proof;
- classify any new idea;
- park or TODO unresolved pieces;
- state whether the item is PASS, PASS WITH GUARDRAILS, PARTIAL, BLOCKED, PARKED, or REJECTED.

## Boundaries

- This is a support scaffold, not doctrine.
- Do not rewrite ACTIVE_GUIDES from this rule alone.
- Do not rewrite CURRENT_TRUTH_INDEX from this rule alone.
- Do not use it as public authority by itself.
- Do not use it to make small tasks ceremonial.
- Do not let the ledger replace real proof.

## Neighbor Rules

- Think Before Work / Intake Alarm Gate starts the route.
- Stop / Issue Notes / Options / Weigh / Verify Gate holds the same-problem repair line.
- Soft Suit Patch-Through Rule carries the fix through each gate.
- 17-Route Sequential Learning Walk Process Rule gives the broader route family walk.
- Broad Command Recursive Trigger List gives the small recursive branch.
- Rule-Touch Spiral can vary route order when repeated results look suspicious.
- CONSOLIDATOR Trial may later watch repeated overlap, but does not merge this scaffold by itself.

## Proof

This rule is proved only by future work where:

- the router source is read first;
- task classification happens before action;
- the same exposed problem is rerun after the fix;
- gate effects carry forward;
- the live ledger stays small and usable;
- the final status follows the proof.
