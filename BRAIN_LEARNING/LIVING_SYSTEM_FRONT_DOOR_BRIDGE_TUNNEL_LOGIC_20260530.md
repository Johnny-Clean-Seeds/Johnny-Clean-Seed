# Living System Front Door / Bridge-Tunnel Logic

Date: 2026-05-30
Status: BRAIN LEARNING / DESIGN-PILOT LOGIC / NOT DOCTRINE
WorkKey: LIVING-SYSTEM-FRONT-DOOR-BRIDGE-TUNNEL-20260530-V1-4

## Core problem

The house has ledgers, maps, keys, proof receipts, Chat Drop copies, source reports, tool cards, and cockpit rules. The burden problem is not lack of material. The burden problem is navigation, stale carry, and weak entry routing.

## Core answer

Build a front-door-led bridge/tunnel layer that connects important living objects through typed, proof-aware, tool-bearing tunnels.

The front door points to the current living ledger.
The living ledger points to the right hub.
Each hub has a toolbelt.
Each tunnel has a relation type, proof need, carry-out line, leave-behind rule, and return path.
Chat loads the manifest, not the whole furniture pile.

## Rule

No front door without a current living ledger pointer.

No tunnel without a toolbelt.

No all-ways linking without legal relation types.

No checkpoint without smoke-break reset before the next wave.

## Borrowed source methods

- Information scent: labels and surrounding cues must tell the worker where a route leads.
- RDF-style triple: source object, relation type, target object.
- C4-style zoom: house, hub, file.
- Diataxis-style separation: operation card, template/reference, explanation/source fit, walkthrough/checkpoint.
- Progressive disclosure: load the manifest first, then follow a tunnel only when needed.

## House-native translation

The system is not a graph database.
The system is not a universal crawler.
The system is not every file linking to every file.

It is a selected current-orientation overlay.

## Boundary

Design/map/pilot only. No broad refactor, delete, move, watcher, automation, Whirlpool, doctrine promotion, ACTIVE_GUIDES rewrite, CURRENT_TRUTH_INDEX rewrite, universal mapper, graph database, or whole-house crawl.
