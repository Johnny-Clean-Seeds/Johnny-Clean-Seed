Date: 2026-05-30

## Core carry

Organic pathing is not folder neatness. It is path ecology.

A normal file path answers:

``text
Where is it?
``

A living path answers:

``text
How does the house remember movement without carrying the whole house?
``

A path becomes alive when it is walked, marked, judged, and reused.
A path becomes dangerous when it grants access without role, proof, cost, boundary, or return.

## Soft suit spine

``text
scent + footsteps + toolbelt + trace + return
``

- Scent: can the route be smelled before opening?
- Footsteps: has this route been walked enough to matter?
- Toolbelt: what method must be worn to travel?
- Trace: what small environmental memory does the movement leave?
- Return: where does the result go back?

## Path ecology

The house does not need a perfect map first. It needs a path ecology:

- roads for common movement;
- trails for emerging movement;
- gates for dangerous movement;
- signs for scent;
- breadcrumbs for return;
- fences for boundaries;
- landmarks for orientation;
- footprints for repeated use;
- pruning for stale paths;
- compost for source ore;
- bones for proof;
- breath between waves.

## Center-strength pathing

A path is clean only when it strengthens the surrounding centers.

Examples:

- README strengthens START_HERE.
- START_HERE strengthens FRONT_DOOR_LEDGER.
- FRONT_DOOR_LEDGER strengthens CHAT_LOAD_MANIFEST.
- CHAT_LOAD_MANIFEST reduces live burden.
- BRIDGE_REGISTRY gives typed routes.
- TOOLBELT_REGISTRY gives method.
- PROOF_HISTORY gives bone.
- CURRENT_STATUS supports but does not outrank authority.

A path that works while weakening proof custody, room smell, or authority boundaries is dirty.

## Stigmergic breadcrumbs

Path traces are environmental memory.

Trace ladder:

- SCUFF: weak sign that a file or route was touched.
- FOOTPRINT: repeated use noticed.
- BREADCRUMB: route hint left behind.
- PATH_CARD: reusable route card.
- TUNNEL_ENTRY: official typed route.
- LEDGER_LINE: state or decision captured.
- RECEIPT: proof/custody captured.
- RULE: behavior adopted only after proof.

Not every movement deserves the same artifact.

## Role-signaling paths

A file path should signal legal use before entry.

- PROOF_HISTORY signals evidence, not instruction.
- BRAIN_LEARNING candidate files signal possible behavior, not doctrine.
- CHAT_COCKPIT suit cards signal chat behavior support, not source authority.
- SOURCE_ORE signals raw material, not rule.

## Cognitive offload paths

Good pathing moves burden out of chat and into the house.

Bad pathing forces the assistant to reconstruct context, authority, proof, boundary, status, and next move live.
Good pathing lets the assistant smell the route, wear the toolbelt, touch the object, leave the right trace, and return cleanly.

## Breadcrumb spine

Prefer a controlled spine over a spiderweb.

Use restrained directions:

- UP: return toward parent/master orientation.
- DOWN: enter child/detail.
- SIDE: neighboring support route.
- BACK: source/proof/history route.
- FORWARD: next allowed action/proof route.

## Desire path court

Repeated shortcuts are evidence, not automatic authority.

Verdicts:

- PAVE: turn into official typed tunnel.
- WATCH: useful but not proven.
- REPAIR: official route is too expensive or foggy.
- PARK: good idea, wrong time.
- BLOCK: bypasses proof, authority, or boundary.

## Choke-to-path repair

A repeated confusion is path evidence.

Ask:

- What path signal was missing?
- What room smell was wrong?
- What proof pointer was absent?
- What return route was unclear?
- What shortcut did the worker take instead?
- Should that shortcut be paved, blocked, watched, parked, or repaired?

## Path health score

Use only when a route matters enough to judge.

- Scent
- Authority
- Toolbelt
- Cost
- Proof
- Return
- Burden
- Boundary
- Reuse
- Prune

A major tunnel should be strong on most of these. A minor movement may only need a breadcrumb.

## Path metabolism

The file system should behave like a living system:

- Intake: source ore, chat drop, user paste, helper output.
- Digest: Creative Room / Work Shed / fit report.
- Circulate: typed tunnels and load shelves.
- Use: active task/toolbelt.
- Store: rule, receipt, ledger, registry, candidate.
- Excrete: drop from live carry, stale copy, superseded proof-only.
- Heal: choke repair, broken return path, dirty shortcut.
- Grow: paved tunnel after repeated clean use.

## Helper rule

Helpers are helping only when they reduce fog, attach keys to ledgers/maps, preserve return paths, and return work in a form the house can ingest.

A helper is not helping when it only adds concepts, reports, files, or links without:

- source/target;
- relation type;
- toolbelt;
- proof/custody;
- load shelf;
- boundary;
- return path;
- next condition.

## Intake Gate connection

The Intake Gate should not only receive material.

It should classify incoming material into path metabolism:

- intake;
- digest;
- circulate;
- use;
- store;
- excrete;
- heal;
- grow.

It should join keys, ledgers, maps, helper output, and return triggers before any concept becomes living house material.
