# External Issues Stay External - Clean Lane Rule

Date: 2026-05-19.
State: BRAIN_LEARNING behavior boundary.

## Core Rule

External issues stay external.

Project issues stay in the project lane.

Workarounds stay in the delivery / adapter lane unless the actual project itself owns the issue.

Do not let outside platform dirt reshape the seed.

## Why This Exists

A platform, browser, hosting service, device, or tool can create friction around the work. That friction may affect how the assistant packages an answer, stages a command, or helps the user execute safely.

That does not mean the friction belongs inside Mr.Kleen.

The house must stay clean. External mud does not go into the clean milk.

## External Issue Examples

External issues include:

- ChatGPT UI or viewport behavior,
- browser lag,
- device lag,
- hosting limitations,
- service limits,
- ads,
- platform bugs,
- network weirdness,
- third-party tool constraints,
- editor quirks,
- display or scrolling behavior.

These may be noted as delivery friction, but they do not become model truth.

## Allowed Adaptation

The assistant may adapt delivery when it creates real help:

- better accuracy,
- safer execution,
- cleaner copy/paste,
- easier processing,
- stronger proof,
- less chance of user-side command errors,
- clearer staging of complex work.

These adaptations remain outside the project.

## Forbidden Contamination

External friction must not alter:

- doctrine,
- model structure,
- active guides,
- proof standards,
- Whirlpool / debugger logic,
- architecture,
- authority chain,
- promotion rules,
- source/proof boundaries,
- local/brain artifact design,
- project meaning.

Do not build project rules around a platform bug.

Do not weaken, shorten, simplify, or reshape Mr.Kleen content because of external friction unless the change genuinely improves the work itself.

## Placement Test

Before saving any issue into Mr.Kleen, ask:

1. Does the project itself own this issue?
2. Is this only a delivery / UI / platform constraint?
3. Would saving this contaminate doctrine or model structure?
4. Is a workaround enough?
5. Can the workaround stay outside the project?
6. Does the adaptation create real help, or only accommodate discomfort?
7. Is this mud trying to enter the clean milk?

## Verdict Shape

If the issue is external:

External issue -> external lane.
Workaround -> delivery / adapter lane only.
Project files -> untouched.

If the project owns the issue:

Project issue -> correct project lane.
Proof required before save.
Neighbor lanes checked.
No random promotion.

## Pass Condition

This rule passes when external friction can be handled without changing project meaning, doctrine, proof, or architecture.

## Failure Condition

This rule fails when an outside platform issue gets absorbed into the model, when a workaround becomes doctrine, or when the assistant weakens the work because of user-interface friction instead of improving actual accuracy, proof, or execution.
