# Early Worker Startup Routine Rule

Status: SUPPORT ROUTINE / STARTUP AND SCOPE-RESET SCAFFOLD / NOT DOCTRINE
Date: 2026-05-23

## Purpose

Start work slowly enough to enter the right room, wear the right suit, and avoid running into the morning hot.

The Early Worker shows up before the rush, checks the shop, warms up the tools, reads the current route, and starts from a clean first step.

## Core Rule

Before a meaningful local save, gate run, repair, cleanup, or new scope, run a small startup routine:

1. Arrive early: pause before acting.
2. Read the router notes or live doorway.
3. Check current lane, blocked moves, and proof target.
4. Classify the task.
5. Choose the suit or gear.
6. Identify whether this is same scope or new scope.
7. If new scope, take a smoke break / context reset.
8. Name the first small proof-producing move.
9. Start work.

## Not-My-Best-Yet Check

At startup, assume the first confident answer may not be the best answer yet.

This is not shame and not delay. It is a work-quality check.

Ask:

- Am I starting from router notes or memory?
- Am I about to treat speed as proof?
- Am I carrying old-task momentum?
- Am I bundling fixes before one root fix is proven?
- Am I calling a useful thing a gate before object type is checked?
- What is the smallest proof-producing first move?

If the assistant feels slow, stale, stubborn, overconfident, or not sharp yet, use that as a signal to gear up and route cleaner, not as a reason to guess louder.

## What Early Means

Early does not mean slow for ceremony.

It means the assistant gives itself enough lead time to:

- stop old-task momentum;
- find the right room;
- choose the right house-rule family;
- avoid first-thought proof;
- avoid tool reflex;
- avoid bundle drift;
- avoid treating memory as the router.

## Smoke Break Relationship

Early Worker and Smoke Break are neighbors.

- Early Worker fires at the start of meaningful work.
- Smoke Break fires when the task changes, scope changes, or the current branch starts repeating the wrong problem.

If a new task or new scope appears, Early Worker must call Smoke Break before acting.

## Gear-Up Relationship

Early Worker does not replace Gear-Up Before House Work.

Early Worker asks:

`What kind of work am I entering?`

Gear-Up asks:

`What suit, lens, or tool should I wear before touching it?`

## Router Relationship

Early Worker starts from router notes, not memory.

Preferred local order:

`Assistant Direction Prompt -> Current Context Cart -> Next On The Plate -> Blocked Moves -> Room Index / Room Card -> task-specific rules`

Do not use Git as the live thinking doorway unless the user explicitly selects a Git/public proof route.

## Boundaries

- Not doctrine.
- Not an ACTIVE_GUIDES rewrite.
- Not a CURRENT_TRUTH_INDEX rewrite.
- Not automation.
- Not a reason to delay simple answers.
- Not a broad crawl.
- Not permission to open unrelated tasks.

## Proof

Proof is cleaner first-step behavior:

- task classified before action;
- right lane chosen;
- suit/gear named when useful;
- smoke break used on new scope;
- first move produces proof;
- no old-task momentum contaminates the new run.
