# Rule Application Gate Power Play

Status: ACTIVE LEARNING RULE
Role: prevent existing rules from staying passive when they should fire as gates
Authority: assistant learning rule for Clean Seed / Clean Milk work

## Problem Captured

The assistant made a fog repair by treating the issue as a content sync problem.

It saw stale text in CURRENT_HOUSE_WORK_STATUS.md and updated the value.

That fixed the visible symptom.

But it failed to ask the stronger neighbor question:

Should a support/index file carry live-current-position wording at all?

That missed the real leak.

## Root Cause

The rule was known, but it did not fire as a gate.

The assistant had these concepts available:

- Respect Thy Neighbor
- soft add
- wear the fix
- walk the halls
- copy the useful pattern
- do not patch symptoms only
- words must match jobs
- support files must not pretend to be authority

But those concepts were treated as background ideas instead of a mandatory pre-save review.

That allowed a receipt to prove expected text while failing to prove system fit.

## Correct Lesson

A rule is not active just because it exists.

A rule is active only when it fires at the correct decision point.

For word, fog, authority, routing, status, receipt, index, map, or support-file changes, the assistant must run a rule-application gate before saving.

## Mandatory Gate

Before saving a fog/word/authority/status/index fix, ask:

1. What file owns this job?
2. What neighboring files does this wording touch?
3. Does the changed file have authority to say this?
4. Is this a live-source responsibility or a static-support responsibility?
5. Does this wording become stale after the next commit, proof, move, or user command?
6. Is this patch only syncing content, or does it fix the system shape?
7. Did a pattern already work in another room?
8. Should that same pattern be copied here?
9. Does the fix respect each neighbor's job?
10. Does the receipt prove neighbor fit, not just expected text?

## Pattern Transfer Rule

When one room has a working pattern, check whether the same concept should apply to neighboring rooms.

Do not blindly copy words.

Copy the clean pattern.

Example:

Receipts may record head-before-save.

Support indexes must not claim live current position.

The shared pattern is:

live values belong to live sources.

static files may record bases, history, support notes, and navigation.

## Soft Add / Wear / Walk Requirement

A fix is not complete when the edited text matches the intended text.

The assistant must:

1. soft add the fix as a candidate
2. wear it in its actual room
3. walk neighboring files
4. check whether the fix steals another file's job
5. check whether an existing house pattern should transfer
6. only then save

## Receipt Requirement

A receipt must not only prove the patch exists.

For these changes, the receipt must prove:

- job ownership
- neighbor respect
- live/static boundary
- authority boundary
- pattern transfer check
- no competing authority created
- no recurring fog shape left behind

## Power Play

The mistake exposed a missing gate.

From now on, when an existing rule should have applied but did not, treat that as a rule-application failure.

Do not only repair the immediate file.

Capture why the rule did not fire.

Turn that failure into a pre-save gate.

Then apply the same clean concept to neighboring lanes where the fit is the same.
