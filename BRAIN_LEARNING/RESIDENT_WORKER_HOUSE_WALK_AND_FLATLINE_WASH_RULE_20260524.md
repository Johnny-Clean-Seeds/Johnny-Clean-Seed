# Resident Worker House Walk And Flatline Wash Rule

Status: SUPPORT RULE / CODEX LOCAL BRIDGE CONDUCT / NOT DOCTRINE
Date: 2026-05-24

## Purpose

When the user asks a local worker to walk the house, wash an issue through gates, or run until flatline, the worker should behave like a responsible resident in a shared project space.

That means:

- enter through the current house rules;
- read the active maps before touching files;
- notice the real room and boundary;
- leave receipts for file-side movement;
- add only clean-fit support material;
- stop when the same cycle produces no new useful signal.

This is not a promotion of the worker into authority. It is conduct inside the house.

## Trigger

Fire when the user says or implies:

- walk the house first;
- run the issue through gates;
- wash it through the whirlpool;
- run cycles until flatline;
- make a clean pack;
- include Codex or mule in the house rules;
- do not act like a detached worker.

## Resident Conduct

Before acting:

1. Read the current doorway, active anchor, local bridge rules, and relevant support rules.
2. Classify the object: source drop, worker symptom, rule gap, map/sign issue, receipt gap, or support idea.
3. Keep raw source local-only when the source is rough, chat-flavored, transcript-like, symbolic, private, or not ready for Git.
4. Keep Git-facing support neutral, practical, and routed.
5. Do not rewrite active guides, current truth, doctrine, public authority, or existing rules unless the user explicitly opens that exact lane.

During the wash:

1. Name the active issue.
2. Separate goal, artifact, proof, explanation, correction, and lesson.
3. Run the issue through the smallest gate family that can judge it.
4. Track what each cycle changes.
5. Stop when new cycles produce no new action, rule, route, proof need, or map/sign correction.
6. Report the cycle count.

After the wash:

1. Save a compact support pack.
2. Save a receipt with source custody, written files, hashes, blocked actions, and next clean use.
3. Park remaining watch items with a return trigger.
4. State what was not touched.

## Mule Boundary

This rule does not replace Mule Worker, Not Supervisor or Mule Sidekick / Contractor Boundary.

External mule output remains support material until reviewed and routed.

Codex/local-worker resident conduct means the worker should respect the shared project house while working inside it. It does not make Codex, mule, scripts, tools, or reports final authority.

## Flatline Standard

Flatline means the last cycles are stable:

- no new root issue;
- no new route;
- no new support rule;
- no new proof need;
- no new file custody action;
- no new map/sign correction;
- only repeated confirmation of the same decision.

Cycle report shape:

`Cycles run -> flatline reached at cycle -> confirmation cycles -> remaining watch items`

## Boundary

- Not doctrine.
- Not an ACTIVE_GUIDES rewrite.
- Not a CURRENT_TRUTH_INDEX rewrite.
- Not a public authority change.
- Not permission for broad cleanup.
- Not permission to change rules or ideas against current flow.
- Not a final PASS claim.

## Proof

Proof is a live case where the worker:

- walks the house before action;
- routes raw source into local custody;
- writes a clean support pack instead of loose notes;
- names the gate cycles and flatline count;
- leaves hashes and receipts;
- keeps active doctrine untouched.
