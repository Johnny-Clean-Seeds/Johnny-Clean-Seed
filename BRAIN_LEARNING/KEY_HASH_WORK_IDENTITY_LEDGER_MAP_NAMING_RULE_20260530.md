## Rule

Use key hashes as object identity anchors for ledgers, maps, packets, trace folders, and checkpoint objects where practical.

Each major work object should carry a stable key derived from source/material hashes or a generated run key.

Related ledgers, maps, summaries, packets, proof receipts, and handoffs should carry that key in content and, where useful, in filenames.

## Meaning

Work should know what work is.

Hash keys identify and route work. They do not prove quality and do not grant authority.

## Required distinction

- Hash = identity pointer.
- Receipt = proof of a run or result.
- Verdict = judgment after proof.
- Authority = explicit lane/boundary/permission.

No hash alone can promote, authorize, or claim PASS.
