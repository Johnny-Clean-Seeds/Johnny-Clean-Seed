# Broad-Bite Rule Machine Rule

## Motto

Small Rule. Wide Bite.

## Core Purpose

When making or repairing a rule, do not patch only the visible symptom.

Broaden first, then condense.

A good rule should be small, stout, dense, and high-load-bearing. It should catch the family of choices underneath the failure, not only the single surface mistake.

## Core Machine

trigger -> choice-family -> neighboring surfaces -> dense rule -> allowed output -> blocked output -> proof of activation

## Broadening Method

When a failure or weak spot appears:

1. Name the surface failure.
2. Name the choice that caused it.
3. Find the wider family of choices.
4. Scan neighboring routes/surfaces touched by the same choice.
5. Condense the family into one compact rule.
6. Define allowed outputs.
7. Define blocked outputs.
8. Prove the rule fired across the next relevant use.
9. Replace stale narrow patches only after proof.

## Dense Rule Test

A strong rule should be:

- small enough to carry
- broad enough to bite
- specific enough to fire
- clean enough to compose with neighboring rules
- proved by behavior, not only saved text

## Known Bulldog Rules

### Separate Target From Talk

Trigger: user needs something copied.

Rule: only the exact copy target goes inside the copy block. Explanation stays outside.

Covers: PowerShell, hashes, prompts, paths, readback commands, file text, front-door tests, and other copy surfaces.

### Hash Seals; Path Opens

Trigger: a hash is used as a pickup reference.

Rule: a hash verifies identity. It does not locate or open a local hub by itself.

A real hub key requires path + pickup card + expected hash + read order + boundary + readback proof.

### Rule Must Fire, Not Sit

Trigger: a rule exists and the matching situation appears.

Rule: the rule must activate at the trigger. Saved text is not enough.

Proof: the next output changes because the rule fired.

### Right Thing. Right Room.

Trigger: content is useful but may be placed in the wrong lane.

Rule: correct content in the wrong room is still dirty placement.

### Mechanics Over Myth

Trigger: metaphor or project language could imply impossible access, fake authority, or hidden capability.

Rule: translate metaphor into actual path, boundary, proof, and action.

## Live Repair Activation Rule

When the assistant says what it did not do and then names what it should have done, that is a live repair trigger.

If the corrected behavior is actionable in the current task:

1. pause the current flow
2. wear the proposed fix in Soft Suit
3. resume the task under that fix
4. watch whether behavior actually changes
5. keep it scoped and provisional unless proof supports promotion

This is not automatic doctrine. It is a live Soft Suit behavior test.

## Ghost/Boss Watch

A provisional Soft Suit fix can create a ghost if:

- it is framed too broadly
- it resembles an existing house result too closely
- it is paired with a similar-looking outcome
- it hides the real boss under a familiar name

Guard:

Keep Soft Suit tests clearly labeled, scoped, and separate from installed house truth.

## Source-Ore Receipt Requirement

When a source-ore list is gathered but intentionally withheld, the assistant must still provide a compact receipt shape:

- task
- lane
- boundary
- source families searched
- output restriction
- held/not sent state
- proof limitation
- next allowed pass

Do not claim full PASS when the proof trail is underbuilt.

## Boundary

This rule package is a house learning/rule-machine install.

It does not rewrite doctrine.
It does not rewrite ACTIVE_GUIDES.
It does not rewrite CURRENT_TRUTH_INDEX.
It does not expand bridge permissions.
It does not create junctions or symlinks.
It does not turn source ore into authority.
