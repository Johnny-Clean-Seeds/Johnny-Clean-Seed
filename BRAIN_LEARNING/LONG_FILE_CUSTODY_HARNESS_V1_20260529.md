## 1. Core sentence

Invisible work must become inspectable without becoming chat bloat.

Long file, long research, long report, and source-heavy synthesis work creates a blind work zone. The user cannot see whether the assistant still holds the right object, whether scope is swelling, whether sources are being used cleanly, or whether the file is forming in the correct lane. When the user repeatedly says "hold the rope," the user is not merely encouraging the assistant. The user is performing emergency custody work that should have been carried by the process.

The fix is a custody harness.

The final file is not enough. The work that builds the file needs custody marks: object lock, work seal, chunk map, source map, A-checkpoints, rope alarm handling, B-close, and after-run wash.

## 2. Activation rule

Use this harness when work is large, source-heavy, file-only, synthesis-heavy, save-route heavy, repeatedly confusing, or when the user says "hold the rope" more than once during the same build.

Do not apply the full ritual to tiny/direct tasks.

This is a size-sensitive support behavior.

## 3. Main failure this repairs

During large file work, the assistant may disappear into internal work. Even if the assistant is still working, the user loses custody visibility. The user then becomes watcher, communicator, and scope governor.

That is backwards.

The assistant must hold four internal roles:

- Worker: does research, synthesis, and writing.
- Scribe: records chunk checkpoints, source ledger, parked material, and B-close.
- Communicator: sends only tiny rope-state pings when allowed or when interrupted.
- Judge: checks scope, boundary, source use, close condition, and next move.

Worker may not take over the whole process. Communicator may not flood chat. Scribe may not create a second bloated file. Judge may not claim done before close proof.

## 4. Work Seal

Before deep work starts, create a small custody lock. If the user requested no chat, put it inside the file or sidecar instead of chat.

Minimum Work Seal fields:

- File/Object: exact output or active object name.
- Output Lane: chat answer, downloadable file, repo save candidate, local-only note, or other named lane.
- User Delivery Rule: file only, brief chat allowed, checkpoint pings allowed, or silent unless interrupted.
- Scope: what the file must accomplish.
- Forbidden Drift: what the work must not become.
- Chunk Map: named sections or batches.
- Source Map: what source types are needed and why.
- Checkpoint Mode: embedded in file, sidecar ledger, tiny chat pings, or silent-unless-interrupted.
- Close Condition: what proves the file is ready.

The Work Seal is the first rope knot. Without it, a large task begins as fog.

## 5. Chunk Map

A big file cannot be one hidden blob.

A chunk map prevents drift by giving each section a local job. Each source, idea, and tool use must serve the active chunk. If it does not serve the active chunk, it parks.

Chunk map example:

1. Define the problem.
2. Fit the problem to house rules.
3. Gather outside analogs.
4. Design the process harness.
5. Write the template.
6. Close with proof, limits, next move, and blocked moves.

Chunk maps are not for prettiness. They are custody rails.

## 6. A-checkpoints

A-checkpoints are not summaries.

A summary says what was written.

A checkpoint proves custody is still intact.

Minimum A-checkpoint fields:

- Object still held: yes/no.
- Current chunk: name/number.
- Chunk state: open, closing, or closed.
- Sources used: source keys or links captured.
- Kept: what belongs in the file.
- Parked: useful but not active.
- Dropped: dead weight removed.
- Drift blocked: what was refused.
- Next checkpoint: exact next close point.

If chat silence is requested, A-checkpoints go inside the file or sidecar. If the user interrupts with "hold the rope," surface the current rope state in one line and continue.

## 7. Live Carry WIP limit

During a long file build, carry only:

CURRENT OBJECT + CURRENT CHUNK + CURRENT SOURCE SET + CURRENT BLOCKER + NEXT CHECKPOINT

Everything else parks.

This is the practical form of "carry the key, not the furniture."

## 8. Rope Alarm Table

User phrases become process controls.

- "hold the rope" means: stop expansion, emit current object / current chunk / next checkpoint / blocked drift, then continue.
- "stay focused" means: reduce scope to active chunk, park side branches, do not defend.
- "smoke break" means: drop live clutter, re-lock the object, resume from last checkpoint.
- "send when ready" means: do not dump content into chat; finish the file; close with B-check.
- "make chunks" or "make checkpoints" means: create or expose the chunk map immediately.
- "use links" means: source ledger required; no broad unsupported synthesis.
- "drop dead chat" means: release old chat weight, keep the active key, and continue from the current rope state.

The phrase fires the behavior. The user should not have to re-explain the rule.

## 9. Sidecar option

Sometimes the main report should stay clean. In that case, custody marks go into a sidecar:

`<FILE_NAME>__ROPE_LEDGER.md`

The sidecar holds:

- Work Seal.
- Chunk Map.
- Source Map.
- A-checkpoints.
- Parking Lot.
- Rough Spot Log.
- B-close.

The main report remains readable. The sidecar proves custody.

## 10. B-close

A long file must close with truth, not just polish.

B-close fields:

- What this file is.
- What it proves.
- What it does not prove.
- Sources used.
- Parked material.
- Risks or weak spots.
- Next clean move.
- Blocked moves.
- Boundary held.
- State label: source-ore, candidate, template, save-ready, closed, blocked, or other exact label.

B-close prevents a big file from becoming a blob.

## 11. Rough spot logging

During long file builds, repeated user rope calls are not noise. They are custody signals.

Use this log class:

`USER_ROPE_INTERVENTION / LONG INVISIBLE WORK / POSSIBLE DRIFT OR SILENCE / REANCHOR REQUIRED`

If the user says "hold the rope" more than once during a single long build, create or update a rope ledger.

If the user says it three times, stop expanding and return to the Work Seal before continuing.

## 12. External analog fit

The harness borrows from external patterns without handing authority to them:

- Progress indicators: visibility reduces user uncertainty during delays.
- Incident command: separate worker, scribe, communicator, and judge roles.
- Blameless postmortems: rope interventions become process signals, not blame.
- Gated technical reviews: chunks need entry and exit criteria.
- WIP limits: carry only the active chunk and next checkpoint.
- Reproducible research: source ledger and checkpoint trail let later workers inspect what happened.
- Unsafe-control-action thinking: checkpointing can fail by being absent, too much, too late, too early, wrong order, or continuing too long.

These are source analogs, not doctrine.

## 13. House fit

This harness fits existing house behavior:

- Deep work gets named batches and A/B checkpoints.
- A means work is alive; B means why motion stopped.
- Carry the key, not the furniture.
- Findings become cards before they become rules.
- Useful but unsaved behavior is Soft Suit, not doctrine.
- No fake PASS.
- No doctrine or authority rewrite from a support candidate.
