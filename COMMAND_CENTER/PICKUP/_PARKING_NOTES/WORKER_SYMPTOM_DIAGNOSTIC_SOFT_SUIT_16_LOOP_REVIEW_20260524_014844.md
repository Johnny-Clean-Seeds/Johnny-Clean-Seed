## Active Issue

The observed symptom is not simply "the other assistant is bad."

The active issue is:

`a lower-depth worker rushes, communicates less clearly, names rules without fully applying them, and closes work before the same issue is rerun through the changed route.`

This is a worker-surface and rule-application problem.

## Soft Suit First

Soft suit under test:

`Worker Symptom Diagnostic -> Capability Fit -> Communication Surface -> Rule-Application Checksum -> Same-Issue Rerun`

This suit is not doctrine. It is a support route for repeated worker symptoms.

## Revision Result

The parent/checkpoint grammar needed a correction.

Better shape:

`Parent Action -> Partner Interpretation -> Next Parent Action`

Why:

- Parent/subordinate language can create a false hierarchy.
- The house often works through neighbors that interpret consequences.
- The partner does not rule the parent; it explains what the parent action produced and prepares the next parent.

## What Works

- Partner interpretation is more accurate than hierarchy.
- Diagnostic wrapper stops the house from adding random gates for every assistant miss.
- Broad worker-surface wording catches model/helper/script variants.
- Rule-application checksum directly targets the "knows rules but does not use them" failure.
- Focus reset addresses rushing without adding a big ceremony.

## What Does Not Work

- Calling the worker the problem is too shallow.
- Adding another creative room without a diagnosis lane would create bloat.
- Telling a lower-depth worker to "think harder" is not enough.
- Treating communication as a final-summary issue misses the route/proof surface.
