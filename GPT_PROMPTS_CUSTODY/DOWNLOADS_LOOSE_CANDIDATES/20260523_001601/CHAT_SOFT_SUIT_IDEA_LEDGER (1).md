# CHAT_SOFT_SUIT_IDEA_LEDGER

STATUS: LOCAL-ONLY IDEA LEDGER / SUPPORT SURFACE / NOT ACTIVE RULE AUTHORITY / NOT GIT / NOT PUBLIC / NOT DOCTRINE

PURPOSE:
Hold raw and shaped ideas without turning every idea into a new rule file.

RELATION TO `CHAT_SOFT_SUIT_MAP.md`:
- `CHAT_SOFT_SUIT_MAP.md` = one active rule map.
- `CHAT_SOFT_SUIT_IDEA_LEDGER.md` = one idea inbox / sorting ledger.
- Ideas live here until they are distilled, tested, blocked, parked, archived, or promoted into the map.
- An idea is not an active Chat Soft Suit rule until it is intentionally folded into `CHAT_SOFT_SUIT_MAP.md`.

CORE LAW:
Do not make a new file for every idea.
Do not make a new rule for every idea.
Do not let ideas disappear in chat.
Hold ideas in this ledger, then promote only the clean distilled rule into the map.

---

## 1. Idea States

Use one of these states for each idea.

RAW:
Captured but not judged.

TRIAGED:
Read and separated from noise.

ADOPT:
Fits current active problem and should be applied now.

ADAPT:
Useful, but needs local Clean Seed / Chat Soft Suit shape.

PARK:
Useful later. Must have source and return trigger.

BLOCK:
Duplicate, dirty, risky, unclear, or boundary-breaking.

PROMOTED_TO_MAP:
Clean distilled rule was added to `CHAT_SOFT_SUIT_MAP.md`.

ARCHIVED:
Preserved for history/source, not active.

---

## 2. Links Ladder for Ideas

Each idea should carry only the links it needs.

Source Link:
Where did it come from?

Home Link:
Where does it belong now?

Neighbor Link:
Which Suit/node handles it next?

Proof Link:
What proves its state?

Return-Home Link:
What closes the loop?

Bell/User-Action Link:
Does the user need to do something?

Short form:

`source -> home -> neighbor -> proof -> return-home -> bell if needed`

---

## 3. Promotion Gate

Before any idea becomes a rule in `CHAT_SOFT_SUIT_MAP.md`, answer:

1. What problem does this solve?
2. Is it already covered?
3. Is it a rule, a note, a metaphor, a source, or a parked idea?
4. What section of the map would it improve?
5. Can it be stated as one clean rule?
6. What gets removed, merged, or simplified so the map does not bloat?
7. What proof or behavior change shows it is useful?
8. What is the return-home state?

If the answer is weak, keep the idea in the ledger. Do not promote.

## 8. Bell Rules

Ring the bell when:
- the ledger has many RAW ideas and no triage;
- an idea should be promoted but has not been folded into the map;
- a rule update is sitting in `READY_TO_DROP_IN_CHAT`;
- an idea repeats across entries and should be merged;
- the map starts growing into spaghetti again.

Bell wording:

`BELL: idea backlog needs triage before more rule work.`

---

## 9. Cleanup Rule

Clean idea handling means:

- capture the idea;
- give it an ID;
- decide state;
- link source/home/neighbor/proof/return;
- promote only distilled rules;
- archive or block the rest;
- do not create extra active rule files.

Final judge:
Ideas are not lost, and rules do not become a spaghetti monster.
