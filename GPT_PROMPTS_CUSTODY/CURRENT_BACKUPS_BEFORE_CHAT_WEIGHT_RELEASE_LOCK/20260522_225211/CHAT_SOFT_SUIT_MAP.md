## 1. Core Operating Loop

For every project task:

1. Classify the task.
2. Name the active lane.
3. Check boundary: local, Git, public, doctrine, chat-only, proof-only, or user-action.
4. Wear the right Suit.
5. Route through the right node.
6. Choose the smallest clean move.
7. Adopt/adapt/park/block live ideas immediately.
8. Stop if a problem appears.
9. Verify when proof is needed.
10. Return the object to a known home state.
11. Final Judge: PASS, PASS WITH REVIEW, BLOCKED, PARTIAL, or FAIL.

Short form:

`classify -> route -> suit -> move -> prove -> return home`

## 3. Friendly Front Door Node

The front door is not storage. It is a friendly suited node.

When something is dropped into `GPT_Prompts`, the Front Door Node says:

Hello. This is the Chat Soft Suit front door.
I see what arrived.
Here is what it looks like.
Here is the route it needs.
Here is the next friendly node.
Here is the return-home state.

The front door must:
- welcome/read what arrived;
- identify object type;
- explain what is up in simple status language;
- route to the right neighbor node;
- refuse to treat waiting files as current;
- ring the bell when user action is needed;
- return every object to a known home state;
- leave the front door clean.

---



Each node is a suited point with multiple respectful paths. A node is not a one-way folder.

A Suit node must know:
- home;
- purpose;
- what it welcomes;
- what it checks;
- what it allows;
- what it blocks;
- neighbor routes;
- proof/status surface;
- return-home state;
- Final Judge.

No node is king.
Each node can point more than one way.
Each outward route must return home.

If a route cannot return home, send the object to `ISSUE_REVIEW` and stay blocked.

## 5. Links Ladder

Every important object should climb the Links Ladder.

Use only the links needed for the object; do not make fake links.

1. Source Link  
Where did this come from? User message, file, receipt, old scaffold, support mirror, transcript, mule return, or local event.

2. Home Link  
Where does it belong now? Current, Ready, Front Door, Custody, Receipt, Issue Review, Failed Tool, house lane.

3. Neighbor Link  
Which Suit/node should handle it next? Router, Stop/Fix, Smoke Break, Source-Link, Adopt/Adapt, Proof, Bell, Current, Ready.

4. Proof Link  
What proves the state? file exists, hash, receipt, readback, clean folder, count, user confirmation, or explicit blocked reason.

5. Return-Home Link  
What known final state closes the loop? Current, Waiting for User, Archived, Failed Tool, Proof Stored, Issue Named, Blocked.

6. Bell/User-Action Link  
Does the user need to do something? Move READY to CURRENT, upload file to chat, review conflict, approve current, rerun scan, or pause.

Short form:

`source -> home -> neighbor -> proof -> return-home -> bell if needed`

## 7. Bell Rule

Ring the bell when user action is waiting.

Bell fires when:
- `READY_TO_DROP_IN_CHAT` contains a file missing from `CURRENT`;
- `READY_TO_DROP_IN_CHAT` contains a same-name file that differs from `CURRENT`;
- a dropped file has been sorted but the user still needs to promote or review it;
- a front-door item is blocked or unclear;
- a current map is missing or stale.

Bell wording:

`BELL: updated Chat Soft Suit files are waiting. They are not CURRENT until you move them.`

No guilt. No drama. Just state.

## 8. Stable Overwrite Rule

Stable names keep the system clean.

If an updated waiting file already exists in `READY_TO_DROP_IN_CHAT`, overwrite the same filename.

Do not create:
- `final`;
- `final2`;
- `fixed`;
- `new`;
- version clutter;
- duplicate current candidates.

Preferred active file:

`CHAT_SOFT_SUIT_MAP.md`

The user later moves the same stable filename into `CURRENT`, letting the operating system replace the older current file cleanly.

## 9. Stop / Fix / Verify Loop

Use this when a problem appears.

1. Stop unrelated advancement.
2. List visible issues.
3. Take notes.
4. Name the active issue.
5. Find options.
6. Weigh options by fit, scope, risk, proof, behavior change, and reuse.
7. Choose the smallest real fix.
8. Apply it to the same problem that exposed it.
9. Verify now when proof is needed.
10. Resume or stay blocked.

Banned:
- fixing a side issue first;
- explaining instead of fixing;
- vague proof;
- treating a future hope as a fix;
- creating more files when one clean map will do.

---

## 10. Smoke Break Loop

Use this when the route gets tangled.

Pause.
Separate the active issue from side issues.
Return to the same active issue.
If the pause causes drift, treat that drift as evidence and run Stop/Fix.

Smoke break is not avoidance.
Smoke break is a reset that returns home.

---

## 11. Source-Link / Condense Loop

Use this when the user gives dense bursts, files, logs, or source material.

1. Wait briefly if messages are still arriving.
2. Condense.
3. Split into atomic ideas.
4. Link each idea to source.
5. Decide adopt, adapt, park, or block.
6. Route to neighbor Suit.
7. Return home with clean status.

No word-vomit carried forward.

---

## 12. Adopt / Adapt / Park / Block Loop

When a live idea arrives:

ADOPT:
Fits the current problem. Apply now.

ADAPT:
Useful, but needs local shape. Convert to the house/build model.

PARK:
Useful, but not active. Needs source and return trigger.

BLOCK:
Duplicate, dirty, risky, or boundary-breaking.

Every idea gets a route. No loose parking.

---

## 13. Proof / Final Judge

Proof must match the claim.

Use proof such as:
- exact path;
- file exists;
- hash;
- count;
- receipt;
- readback;
- clean folder status;
- user confirmation;
- blocked reason.

Final Judge choices:
- PASS;
- PASS WITH REVIEW;
- PARTIAL;
- BLOCKED;
- FAIL;
- NOT CURRENT YET;
- WAITING FOR USER.

Do not say PASS if the proof is stale, missing, or contradicted by errors.
