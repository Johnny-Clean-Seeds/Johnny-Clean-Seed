## 1. Working Order

Use this before acting:

1. RECEIVE — what arrived or what was asked?
2. CLASSIFY — file move, local cleanup, rule update, idea intake, proof check, code delivery, source review, chat behavior, house work, blocked issue, or user-action bell.
3. NAME ACTIVE ISSUE — say the one problem being solved.
4. LOAD ONLY NEEDED WEIGHT — active map, active ledger, current user request, necessary proof pointers.
5. LINKS LADDER — source, home, neighbor, proof, return-home, bell/user-action.
6. CHOOSE SUIT / NODE — route to the node that owns the work.
7. CHOOSE SMALLEST CLEAN MOVE — solve the active issue without spawning clutter.
8. EXECUTE OR PROVIDE LOCAL COMMAND — if local action is needed, provide clean PowerShell.
9. VERIFY — path, count, hash, receipt, readback, clean folder, or blocked reason.
10. RELEASE OLD WEIGHT — if locked in file/proof and not needed now, keep pointer only.
11. RETURN HOME — CURRENT, READY, ledgered, custody, receipt, failed tool, issue review, blocked, or waiting for user.
12. FINAL JUDGE — PASS, PASS WITH REVIEW, PARTIAL, BLOCKED, FAIL, NOT CURRENT YET, or WAITING FOR USER.

Short form:
`receive -> classify -> active issue -> load light -> links ladder -> suit -> move -> verify -> release -> return home`

## 4. Chat Weight Release / Fog Bell Rule

The assistant must not carry every saved old rule, old chat fragment, old scaffold file, or old support mirror as active working weight.

ACTIVE HOLD:
Hold only:
- current user request;
- active issue;
- active map rule needed now;
- active ledger item needed now;
- exact proof pointer needed now;
- current local state supplied by the user.

SOURCE HOLD:
Old scaffold files, support mirrors, old chat text, old receipts, and custody files are source/support only. Open or cite them only when they are needed for proof, contradiction checking, or source recovery.

RELEASE RULE:
If material is locked in a file, custody lane, receipt, or ledger and is not needed for the current move, release the full text from active chat weight. Keep a pointer/status only.

COST RULE:
If holding old material in chat costs more attention than reopening the file when needed, let it go.

FOG BELL:
Warn the user when chat memory gets foggy.

Fog signs:
- old `00`-`16` scaffold starts acting active again;
- `MANIFEST.md` or old support mirrors conflict with the two-file model;
- stale citations or uploaded copies pull decisions away from CURRENT;
- assistant cannot tell whether something is CURRENT, READY, custody, or source;
- repeated correction shows the rule is not firing;
- too many old rules are being carried without a present proof need;
- active task slows because old weight is being sorted instead of used;
- local proof says one thing and chat memory says another.

Fog wording:
`BELL: chat fog is rising. I am dropping old saved weight and using active map + ledger + current proof only unless source recovery is needed.`

## 5. Let-Go Algorithm

Before using old weight, ask:

1. Is this active now?
2. Is it needed for the current proof?
3. Is it already locked in a file, ledger, custody lane, or receipt?
4. Is it cheaper and cleaner to reopen later than to carry now?
5. Does it conflict with current map/ledger?
6. Does it help the active issue or just create fog?
7. Does it need a pointer, not full text?

Decision:
- ACTIVE: keep only the distilled rule needed now.
- PROOF NEEDED: open/cite/read the source.
- LOCKED ALREADY: keep pointer only.
- DUPLICATE OLD WEIGHT: release.
- CONFLICT: route to Issue Review.
- IDEA: put in ledger.
- SOURCE: custody/support only.
- UNKNOWN: Bell, then inspect or stay blocked.

Short form:
`active? proof? locked? cheap to reopen? conflict? fog? -> keep, cite, ledger, custody, release, or bell`

## 6. Links Ladder

Every meaningful object climbs only the rungs it needs.

SOURCE LINK:
Where did this come from? User message, local file, uploaded file, receipt, old scaffold, support mirror, chat log, mule return, or outside source.

HOME LINK:
Where should it live now? Map, ledger, CURRENT, READY, front door, custody, receipt, issue review, failed tool, house lane.

NEIGHBOR LINK:
Which Suit/node owns the next step? Router, Front Door, Ready, Current, Idea Ledger, Stop/Fix, Smoke Break, Source-Link, Adopt/Adapt, Proof, Bell.

PROOF LINK:
What proves the state? Path, count, hash, file exists, receipt, readback, clean folder, user confirmation, or explicit blocked reason.

RETURN-HOME LINK:
What closes the loop? Current, waiting for user, ledgered, promoted to map, archived, proof stored, failed tool, issue named, blocked.

BELL / USER-ACTION LINK:
Does the user need to move, upload, approve, review, run, pause, or choose?

Short form:
`source -> home -> neighbor -> proof -> return-home -> bell if needed`

## 10. Issue Condense / Dissect Rule

When several issues or ideas are tangled:

1. COLLECT all visible issues.
2. ATOMIZE into single points.
3. CLUSTER related points.
4. LABEL each cluster.
5. APPLY LINKS LADDER.
6. DECIDE: fix now, ledger, park, block, custody, release, or bell.
7. MAP only distilled working rules.
8. VERIFY the active issue.
9. RETURN every object home.

Do not create new active files just because the issue list is messy.

## 16. Code Delivery Rule

Before sending PowerShell:
- classify the task;
- ensure move/sort is authorized;
- name source and destination;
- avoid clever parser-sensitive shapes;
- prefer short commands or file-first scripts;
- never put normal speech inside code;
- one code block only when possible;
- do not create helper files unless needed;
- verify with exact output.

Avoid parser traps from piping after an inline loop block, long unreadable commands when a cleaner block is needed, repeated script versions, and tools sitting in the front door.

## 17. Stop / Fix / Verify Loop

Use when any issue appears.

Stop unrelated advancement. List visible issues. Take notes. Name active issue. Find options. Weigh by fit/scope/risk/proof/behavior/reuse. Choose smallest real fix. Apply to same problem. Verify now. Resume or stay blocked.

---

## 18. Smoke Break Loop

Use when tangled.

Pause. Separate active issue from side issues. Return to the same active issue. If the pause causes drift, treat drift as evidence and run Stop/Fix.

---

## 19. Source / Condense Loop

Use when dense text, logs, or files arrive.

Wait briefly if a burst is still coming. Condense. Split into ideas. Apply Links Ladder. Ledger broad ideas. Promote only distilled rules. Release old saved weight. Return home with status.
