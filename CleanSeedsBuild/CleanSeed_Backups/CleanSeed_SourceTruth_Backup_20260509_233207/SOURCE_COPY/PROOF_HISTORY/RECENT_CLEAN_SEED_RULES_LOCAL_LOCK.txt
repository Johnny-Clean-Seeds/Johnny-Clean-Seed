2. SAVE METHOD / NO DUPLICATE DOWNLOAD DRIFT

When files, package updates, proof reports, live-test folders, or build artifacts need to be saved locally, default to the clean installer/file-packet method instead of scattered browser downloads.

Clean save method:
- use one clear local root
- create one named package/build folder
- separate ACTIVE_GUIDES from PROOF_HISTORY and live-test/build folders

3. LOCAL SAVE RULE / NO MEMORY-ONLY TRUST

"Saved in memory" does not mean saved in a user-owned local file.

Memory, chat, and summaries are reference-only convenience, not durable package proof.
They can compress, lose detail, distort wording, or fail to transfer later.

Clean rule:
Any clean rule, patch candidate, proof report, package update, live-test result, or trust-affecting saved item that matters long term must be saved to the user's PC or another user-controlled durable location when practical.

Reason:
Local save prevents important clean milk from being trapped only in chat or memory where it may be lost or compressed.

Future wording should distinguish:
- memory-captured
- local-file-saved
- package-file-installed
- proof-history-saved
- exported/sent

4. PRE-RUN INSTALLER / FILE EXISTENCE GATE

Before asking the user to run a downloaded installer, script, file packet, or local-save command, verify the file path first.

Correct order:
- name the target local path
- make sure the file is actually saved there
- use Test-Path or equivalent before execution
- if missing, stop cleanly with installer-location FAIL
- do not call it package failure
- use the smallest clean fix, such as an inline PowerShell save packet when a download path is unreliable

Reason:
A script cannot be run from Downloads unless the file actually exists in Downloads.

5. PARTIAL RETURN EVIDENCE / GOLD-LOSS REMINDER

When the assistant asks the user to return exact PASS/FAIL/error lines and the user sends only a shortened summary such as "pass," partial output, or selected lines, the assistant must not shame the user or call the package failed.

Correct classification:
Partial return evidence.

Reminder:
A shortened return is okay as a quick signal, but it may lose valuable proof/gold:
- which file passed
- which file failed
- what command finished
- which folder was used
- what evidence can be trusted later

Correct assistant action:
Ask only for the missing proof lines or run a small existence check.
Do not restart the whole process.
Do not treat partial return as full receipt-quality proof.
Do not treat it as package failure.
Keep the active thread anchored.

Reason:
Exact returned output prevents compression, missing evidence, duplicate work, and false PASS.
