2. Cleanup cycle order

Cleanup cycle:
SEE -> SORT -> COMPARE -> JUDGE -> PLAN -> APPROVE -> ACT -> VERIFY -> CLOSE

SEE:
Capture what exists within the approved scope.

SORT:
Put each item into a visible lane before judging value.

COMPARE:
Compare only when the lane requires it and the exact comparison proof is approved.

JUDGE:
Decide role, risk, value, and route from evidence.

PLAN:
Write the exact proposed action, target path, proof, and rollback/protection needs.

APPROVE:
Get explicit user approval for any real action.

ACT:
Perform only the approved exact action.

VERIFY:
Check target, neighbors, protected lanes, hashes when needed, and no-change claims.

CLOSE:
Write receipt, record final state, and name the next bounded route.

3. Neighbor respect rule

Each lane must know:
- what comes before it
- what comes after it
- what it is allowed to judge
- what it is not allowed to judge
- what proof it needs before action

Neighbor rule:
No lane may steal authority from its neighbor.
No lane may silently promote, archive, park, delete, compare, or declare duplicate status for another lane.
Unknown material must stay unknown until exact proof changes its lane.
Blocked material stays blocked until a separate approval path exists.
Active truth stays protected unless a separate active-guide or truth-index update is approved.

5. Real cleanup gates

Content review gate:
Requires exact file/folder path, reason for reading contents, allowed depth, blocked action list, and user approval.

Hash compare gate:
Requires exact candidates, hash algorithm, read-only hash action, comparison fields, and user approval.

Archive gate:
Requires exact source path, exact archive/parking destination, uniqueness proof, neighbor check, receipt, and user approval.

Park gate:
Requires exact item path, parking reason, parking destination or status file, no-promotion statement, and user approval.

Delete gate:
Requires exact target path, proof of redundancy, proof it is outside protected lanes, receipt, post-delete verify, and explicit user approval.

Blocked item gate:
Requires item stays inert, no execution/opening, exact path, risk reason, and a later user decision before any action.

Active guide protection gate:
Requires backup, exact proposed guide change, source evidence, role fit, verification markers, receipt, and user approval.

Proof-history protection gate:
Requires proof role review before any archive/park/delete. Proof is not compressed into memory.

OldSchool source-ore gate:
Requires no deeper scan unless approved, no promotion by identity, no edits, no execution, and clear relic/source-ore labeling.

LocalBot separation gate:
Requires LocalBot/OldBot material stay separate from Clean Seed active guide truth unless an explicit bridge task is approved.

Unknown item review gate:
Requires unknowns to remain parked until exact content review or user decision gives them a lane.
