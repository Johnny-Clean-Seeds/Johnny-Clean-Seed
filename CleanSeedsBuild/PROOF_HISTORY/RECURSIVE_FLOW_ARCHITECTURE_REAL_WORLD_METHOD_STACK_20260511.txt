SECTION 3 - STATION-BY-STATION ARCHITECTURE

1. AUTHORITY

Receives:
Project root, source-of-truth files, user scope, current task.

Allowed:
Identify what is active truth, what is support proof, what is local bridge guidance, and what is blocked.

Must not:
Treat proof-history, memory, chat, or good-sounding ideas as active truth.

Proof needed:
Path, file role, hash when judging active files, patch/state IDs when present.

Output:
Authority boundary for the active cycle.

Stop when:
Authority source is missing, conflicted, stale, or outside scope.

Real-world methods:
Zero trust, requirements traceability, audit control, legal authority check.

2. ACTIVE BALL

Receives:
The current target or batch.

Allowed:
Name what is in hand and what state it is in.

Must not:
Silently switch tasks or keep multiple hidden active balls.

Proof needed:
Target path or statement, scope, owner, status, next gate.

Output:
One named active target.

Stop when:
The active ball is unclear.

Real-world methods:
Kanban WIP limit, cockpit task discipline, incident command role clarity, working memory.

3. FREEZE

Receives:
Active ball plus risk or review need.

Allowed:
Stop movement and prevent state change.

Must not:
Delete, move, rename, overwrite, cleanup, promote, run scripts, extract zips, edit active guides, or change current truth.

Proof needed:
What is frozen, why it is frozen, what has not changed.

Output:
Stable scene.

Stop when:
State cannot be preserved or the user has not approved required action.

Real-world methods:
Crime-scene preservation, forensic imaging, legal hold, jidoka stop-the-line, inhibition.

4. CAPTURE / COLLECT

Receives:
Frozen scene or bounded target.

Allowed:
Preserve visible evidence: paths, names, timestamps, sizes, hashes when needed, file lists, command output, receipts, logs, current state.

Must not:
Interpret, judge, fix, clean, move, delete, promote, or restructure.

Proof needed:
Captured facts and source path.

Output:
Raw evidence set.

Stop when:
Evidence is not preserved enough for the next station.

Real-world methods:
Forensics collection, chain of custody, lab specimen receipt, field sampling.

5. INVENTORY

Receives:
Captured evidence.

Allowed:
Turn captured evidence into a structured list.

Must not:
Judge value, delete, promote, or fix.

Proof needed:
Name, path, size, type, modified time, source, role guess, evidence present, evidence missing.

Output:
Structured item list.

Stop when:
Identity, path, or source is unclear.

Real-world methods:
Lab accessioning, library cataloging, asset inventory, metadata inventory.

6. SORT

Receives:
Structured inventory.

Allowed:
Group like with like by role, source, risk, authority, status, and next gate.

Must not:
Decide final value, fix, delete, promote, or merge.

Proof needed:
Pile criteria and why each item belongs there.

Output:
Clean piles.

Stop when:
Items have mixed authority/risk and cannot be safely grouped.

Real-world methods:
Faceted classification, taxonomy, affinity mapping, triage prep, library subjects.

7. JUDGE

Receives:
Sorted piles or one item from a pile.

Allowed:
Decide value, risk, authority, evidence strength, next gate, and whether optional triage is needed.

Must not:
Treat judgment as permission to change files.

Proof needed:
Criteria, comparison target, evidence, risk, unknowns, affected neighbors.

Output:
Provisional status: save, adjust, deny, park, block, review, promote candidate, or yield.

Stop when:
Evidence, authority, user approval, or scope is missing.

Real-world methods:
Scientific method, differential diagnosis, FMEA, evidence admissibility, red-team review, decision tables.

8. PLACE / FILE

Receives:
Judged item with provisional status.

Allowed:
Route to correct lane: proof-history, parked, blocked, active-guide candidate, current-truth candidate, review candidate, receipt, source-ore, rest stop.

Must not:
Treat placement as promotion.

Proof needed:
Lane reason, authority boundary, next gate.

Output:
Findable item with status.

Stop when:
No correct lane is proven.

Real-world methods:
Records management, archival appraisal, library indexing, warehouse slotting.

9. SAVE

Receives:
Approved save target.

Allowed:
Create proof-history file, receipt, backup, hash record, or decision record when approved.

Must not:
Overwrite, move, delete, or promote unless separately approved.

Proof needed:
Exact path, purpose, boundary, save scope.

Output:
Saved durable evidence.

Stop when:
Path exists unexpectedly, scope changed, or approval is unclear.

Real-world methods:
Lab notebook, audit trail, version control, records retention.

10. SEAL

Receives:
Saved or reviewed item.

Allowed:
Record path, size, hash, what changed, what did not change, status, blocked actions, and next clean action.

Must not:
Turn file-save success into final PASS.

Proof needed:
Size, hash, boundary confirmation.

Output:
Closed proof state.

Stop when:
Verification cannot be completed.

Real-world methods:
Audit seal, release verification, chain of custody, reproducibility check.

11. REVIEW / HANDOFF

Receives:
Sealed item or bounded target for another reviewer.

Allowed:
Inspect, challenge, compare, recommend, and return provisional judgment.

Must not:
Final PASS, promote, edit active truth, or exceed role.

Proof needed:
Review target, evidence used, findings, weak points, next gate.

Output:
Review result.

Stop when:
Reviewer authority is unclear or evidence is insufficient.

Real-world methods:
Peer review, red-team review, incident command handoff, legal review, lab second review.

12. PROMOTE / PARK / BLOCK

Receives:
Reviewed item with next-state decision.

Allowed:
Promote only after separate authority gate; park useful not-now items; block unsafe or unready items.

Must not:
Promote by identity, delete by disappointment, or block without reason.

Proof needed:
Promotion approval, park reason, block reason, reopen condition.

Output:
Clear next state.

Stop when:
Decision needs user approval or proof.

Real-world methods:
Release management, quarantine, archival appraisal, records retention, change control.

13. CONTINUE

Receives:
Closed state from prior station.

Allowed:
Return to authority/active-ball check and choose the next clean action.

Must not:
Drift into a new task without closure.

Proof needed:
Current state and next clean action.

Output:
Recursive next cycle.

Stop when:
Open item remains in hand.

Real-world methods:
OODA loop, lessons learned, process control, feedback learning.


SECTION 4 - HUMAN-BRAIN-LIKE MAPPING

This is not magic. It is distributed coordination with gates.

Attention:
Maps to ACTIVE BALL.
It chooses the current target.

Working memory:
Maps to current plate / active surface.
It holds what is in hand without pretending to be long-term truth.

Long-term memory:
Maps to PROOF_HISTORY and source ore.
It stores useful material without deciding everything.

Prefrontal judgment:
Maps to JUDGE.
It filters value, risk, authority, and next gate.
