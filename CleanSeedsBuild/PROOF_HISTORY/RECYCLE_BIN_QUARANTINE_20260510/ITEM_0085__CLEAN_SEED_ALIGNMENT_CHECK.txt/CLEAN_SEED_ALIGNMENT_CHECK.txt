Purpose

This file checks whether a clean seed build is staying small, clear, proven, bounded, and
easy to follow.

This file is not the start guide.
This file is not the assistant walk-through guide.
This file is not runtime proof.
This file is a guardrail and alignment check.
It does not add a separate test phase.
It checks proof only to prevent dirty milk.

Use this file with:

README_START_HERE.txt
CLEAN_SEED_WRAP_GUIDE.txt

The README starts the user.
The guide controls the assistant walk-through.
This alignment check catches drift and runs the Security / Clean Milk Review.


Hard Verdict Standard

Use PASS only when every required check passed together.

Use PASS WITH GUARDRAILS when the build is clean but needs an active watch item.

Use PARTIAL when some required checks pass and some are missing or weak.

Use FAIL when usable evidence proves a required check failed.

Use INVALID when the evidence cannot prove what happened.

Use PARKED when the idea is useful but not active.

Use YIELD when user judgment, approval, lookup, inspection, or verification is needed.

Use BLOCKED when progress is stopped by a real blocker.

Use PROMOTED when a proven, bounded item has been deliberately moved into an allowed larger
role.

Use RETIRED when an old file, rule, route, or copy is no longer active and must not be obeyed as
current truth.

Done is not PASS.
Movement is not proof.
Confidence is not proof.
Screenshots are not runtime proof unless the specific check is visual.


Clean Seed Target

A clean seed has:

one clear name
one home
one active job
one visible input/output route
one evidence path
one pass judge
one failure path
one recovery path
one authority boundary that names what can be touched and what cannot be touched
one Does-Not-Do / Out-of-Bounds list
one parking lane
one promotion rule
one close condition

If any of these are missing, the seed setup is not clean yet.



Rule / Check Closure Standard

Use this standard when judging or writing any behavior-controlling rule or check.

A behavior-controlling rule or check is one that can change assistant action, file authority,
evidence handling, verdicts, package state, user approval, patch behavior, promotion, bridge
behavior, or final trust.

A closed rule or check should name enough of these parts for a weak assistant to apply it
without guessing:

trigger / when to use it
what to inspect
minimum required evidence or fields
dirty milk conditions
clean milk conditions
verdict labels
next clean state

Not every explanatory README paragraph needs this closure. The README is an on-ramp.
The Alignment Check carries most verification closure.

Do not paste a giant duplicate template into every rule.
Use one general closure standard, then tighten only the behavior-controlling checks that are
thin enough to cause drift.

Dirty milk appears when:

a behavior-controlling check leaves the trigger unclear
a check tells the assistant to inspect something but does not say what evidence or fields matter
a check names failure but not clean milk, verdict, or next clean state
a check lets the assistant pass by confidence, style, or vibes
a check becomes a giant repeated block that creates bloat instead of clarity

Clean milk appears when:

the rule or check has enough closure for the active risk
the assistant can tell when to use it, what to inspect, what proves clean, what proves dirty,
which verdict applies, and what the next clean state is
lightweight checks stay lightweight
heavy closure appears only where behavior, authority, evidence, patching, promotion, or final
trust depends on it

Verdict:

PASS when the behavior-controlling check can be applied without guessing and does not create
bloat.
PASS WITH GUARDRAILS when the check works but has a named watch item.
PARTIAL when the check is useful but missing clean milk, dirty milk, verdict, or next state.
FAIL when the check allows behavior drift, authority drift, fake proof, or duplicate truth.
YIELD when user judgment is needed before tightening or activating the check.

It makes the assistant name and apply the relevant master rules before claiming clean milk.

Bloat means any rule, file, step, option, check, explanation, artifact, helper, bridge,
automation, or ceremony that does not protect the active route, evidence, authority, verdict
accuracy, failure/recovery path, package state, promotion safety, user control, or next clean
state.

Was Challenge Verification Before Correction needed because the user asked why the assistant did or did not do something, whether a rule was followed, whether a file changed, or whether the user may be mistaken?
Was Deep Clean Run Receipt / Cycle Truth Check needed because Deep Clean timing, last-ran proof, current package trust, or rule cycle truth is being judged?
Was Timestamped Helpful Evidence Receipt Check needed because a receipt, report, pending issue, PASS claim, recommendation, or rule-checker claim depends on available proof?
Was Focused Review Boundary Check needed?
Was Rule Closure Cycle needed because a new or changed rule/check/master rule could create affected-scope work or follow-through?
Was Security / Clean Milk Review needed?
Was Clean Evidence Capture needed?
Was Real Evidence / Suspect Claim Check needed because a verdict, suspected cause, accused source, file state, rule obedience claim, failure cause, package state, or trust claim could be wrong without all required real evidence?
Was Incident Existence / False Alarm Gate needed because the event itself had to be proven before suspect, blame, apology, defense, patch, resend, overwrite, rule change, or PASS?
Was Risk Severity Evidence Check needed because the risk level changed the evidence burden, yield/block behavior, backup/quarantine/receipt route, or project/core protection?
Was Words Are Keys Check needed because behavior-controlling language, labels, names, verdicts, receipts, seals, risk labels, incident labels, commands, prompts, or handoff wording could route behavior?

Was Patch Quarantine / Rollback or Patch Balance needed?

Was Rule Pressure Check needed?
Was Post-Fix / Milk Preservation needed?
Was Adjustment Stack Yield needed because repeated chat corrections or assistant tuning could
become unclear?
Was Prompt And Command Boundary needed because long prompts, command packets, reports, or

return-output instructions could confuse the OS, shell, app, target, or evidence?

Was Interruption Recovery / Last Actual Fix Check or True Verified Continue Check needed because work was paused, corrected, interrupted, restarted, continued, or resumed after a partial response?
Was Point Check / Decision Validation Loop needed because a thought, idea, fix, verdict, risk call, incident claim, evidence claim, CYA claim, save claim, handoff, or next action determined the next path?
Was List-To-Task Queue Check or Ordered Multi-Fix Check needed because multiple pending fixes, issues, or list items were active?
Was Post-Add Clean Milk Run needed?
Was Affected-Scope Clean Milk Sweep needed?
Was Rule Closure Cycle needed after the addition was applied, so the changed package was checked again?
Was Clean Finding Placement needed?
Was Capture Classification needed?
Was User Command List / Role Map needed because the user asked for command list, role placement, police, guard, judge, owner, recorder, parking lane, or user-friendly controls?
Was Gold Snippet Capture / Word Review needed because useful wording had to be preserved, placed, locked/saved, or protected from compression loss?
Was New File / Directory Certification needed?
Was Deep Clean Milk Check needed instead of a quick check?
Was Verdict Scope / Stage Proof / Wave Boundary Check needed?
Was the result checked again after application?
Is the next clean state named?

Dirty milk appears when:

the assistant calls something clean without naming which master rules applied
the assistant says a rule was obeyed without action evidence
the assistant applies one master rule and skips another required one
a new rule or master rule is added without closing trigger, scope, evidence, verdicts, affected scope, follow-through, and next state
Security / Clean Milk Review is used but Rule Pressure is skipped when bloat is a real risk
Patch Balance is used but Post-Fix review is skipped after the patch
a rule is added to stop bloat but the new rule becomes bloat
clean milk / dirty milk is treated as vibe language instead of evidence-backed verdict language
the assistant checks before applying a change but does not check the result after applying it
the assistant continues after a long prompt, command packet, report, or verdict text looks cut
off, mixed, or hard to verify
the assistant keeps using long manual command packets when a bounded shortcut is the cleaner
route for repeated prompt, terminal, folder, PowerShell, Command Prompt, or command-center work
a quick Post-Add Clean Milk Run is used to dodge a required Deep Clean Milk Check
files are saved, overwritten, exported, or handed off without save-location and backup CYA when

irrelevant master rules are forced as ceremony

the relevant master rules are named

irrelevant master rules are not forced as ceremony

Rule Obedience Proof Check

Use this check when the assistant claims a rule, request, patch, save, verdict, fix, report, or next
action was followed or completed, or when the user asks whether the rules are actually being
obeyed.

This check proves rule obedience from action evidence. It does not accept appearance, confidence,
style, or rule presence as proof.

Check:

What is the active target?
What rule or request controlled the action?
What classification applies: new, already covered, refinement, duplicate, conflict, parked,
rejected dirty milk, needs user judgment, locked/saved, or pending only?
What files, folders, prompts, notes, evidence, backups, verdicts, seals, or follow-through tasks
are affected?
What action was taken because of the rule?
What evidence proves that action happened?
What result was checked after the action?
What was not checked, and why was it outside scope?
Was the review focused on the required target and affected scope instead of turning into a broad
manhunt?
Are remaining issues closed, parked, blocked, yielded, or named with next action?
Is the next clean state clear?

Dirty milk appears when:

the rule exists but no action proves it was obeyed
the assistant says done, fixed, saved, locked, clean, or PASS without naming proof
the assistant adds a rule but does not run its rule-implied follow-through on the current task
the assistant checks only wording and ignores affected files, prompts, evidence, backups, seals, or
next tasks
the assistant runs a broad manhunt that is not required by the active task
the assistant guesses that obedience happened because the rule sounds good

Clean milk appears when:

the controlling rule/request is named
the caused action is named
the affected scope is named
the proof or evidence location is named
the result is checked after action
open items are explicitly closed, parked, blocked, yielded, or assigned a next action
next clean state is clear

Verdict:

PASS when the rule/request was followed, affected scope was checked, proof exists, and next state is
clear.
PASS WITH GUARDRAILS when obedience is proven but a named watch item remains.
PARTIAL when one rule, affected-scope, proof, or next-state detail is weak.
FAIL when the rule was not obeyed or the assistant only looked compliant.
INVALID when the evidence cannot prove obedience.
YIELD when user judgment is needed.
BLOCKED when obedience cannot be checked safely.

What is the smallest prevention point?

What must not be changed?
Was the same failure path checked again after the fix?
Did the fix avoid duplicate truth, rule conflict, and bloat?

Clean milk appears when:

the smallest prevention point is patched or closed no-action-needed with reason
affected scope is checked
the same failure path is checked again

Challenge Verification Before Correction Check

Use this check when the user asks why the assistant did something, why it did not do something, whether it actually followed a rule, whether it changed a file, whether it saved something, whether it sent the right file, or whether the user may be mistaken about what happened.

This check prevents automatic apology, automatic agreement, false confession, defensive guessing, duplicate patches, duplicate files, and unwanted changes.

Check:

What exact user request or rule is being challenged?
What action is claimed, missing, disputed, or uncertain?
What evidence exists in the current chat, saved file state, sent artifact, file link, tool output, command output, report text, or visible proof?
Did the action actually happen, not happen, partly happen, or remain unknown?
Is the user right, mistaken, partly right, or impossible to verify from current evidence?
Would patching, resending, overwriting, or changing rules create duplicate active truth or an unwanted change?
If the assistant missed the action, what is the smallest clean fix?
If evidence is missing, should the assistant inspect the target, call the state INVALID, or yield?
Is the next clean state named?

Dirty milk appears when:

the assistant admits fault without checking evidence
the assistant argues that the user is mistaken without checking evidence
the assistant patches, resends, overwrites, changes rules, or creates new files before verifying whether the action already happened
the assistant treats memory, confidence, style, apology, or tone as proof
the assistant creates duplicate active truth while correcting an unverified issue
the assistant changes something the user did not really want changed

Clean milk appears when:

the challenged action is verified against real evidence
proof is named before apology, defense, patch, resend, overwrite, or rule change
the assistant corrects the user only when proof supports it
real misses are classified and repaired with the smallest clean fix
unknown states are marked INVALID, yielded, or inspected before action
duplicate patches, duplicate files, and unwanted changes are avoided

Verdict:

PASS when the challenged action is verified from evidence and the next clean action matches the verified state.
PASS WITH GUARDRAILS when the result can continue but a proof item needs watching.
PARTIAL when the likely state is known but one important proof field is weak.
FAIL when the assistant admits fault, defends itself, patches, resends, overwrites, or changes rules before verification.
INVALID when the evidence cannot prove whether the action happened.
YIELD when user judgment or approval is needed before repair, duplicate patch, resend, overwrite, or file change.
BLOCKED when the action cannot be safely verified.

Clean Evidence Capture Check

Use this check whenever proof, judgment, repair, rollback, promotion, bridge work, wave close,

Clean evidence capture means the assistant collects every necessary piece of available helpful evidence needed to

It does not mean collecting every possible detail.
Do not collect private, unrelated, excessive, speculative, or just-in-case material.
Clean evidence is tied to the active target.
Dirty collection is hoarding.

Check whether capture occurred when:

work changed state
a route was proven
a task failed
a repair was proposed or applied
a file was created, edited, moved, renamed, deleted, or inspected for judgment
a command, script, tool, or assistant action was used for proof

the assistant yielded

a stage closed
final wrap was prepared

action taken
input or source

files touched or inspected

output location
evidence location
verdict
pass judge
failure, partial result, invalid result, or unknowns
repair target, if needed
user approval, if required
parked items, if any
next clean state

Failure evidence check:

Was the output preserved?
Was status preserved?
Was folder or file state captured when relevant?
Was log tail, error text, or visible failure note captured when relevant?
Was route state captured?
Was verdict named?

failure scenes are preserved before repair
patches preserve prior state and rollback or rejection path
promotions include Promotion Passport evidence
private, unrelated, excessive, speculative, and just-in-case material is not collected
next clean state is clear

false alarms, stale evidence, fake evidence, simulations, partial incidents, and unknowns get the right verdict
no patch or blame happens until the event itself is proven
next clean action follows the event classification

Real Evidence / Suspect Claim Check

Use this check whenever the assistant judges a claim, suspected cause, accused source, file state, rule obedience claim, failure cause, package state, trust claim, or any situation where the wrong target could be blamed or cleared.

Murder-suspect model:
Do not convict the wrong suspect because the easy evidence looked complete.
Do not clear the real suspect because the hard evidence was not checked.
Do not believe a clean-looking story when required real evidence is missing.

Check:

What active claim is being judged?
Who or what is being blamed, cleared, trusted, or judged?
What real evidence is required for the pass judge to decide without guessing?
What evidence was actually checked?

What verdict scope applies?
What is the smallest clean next action?

required real evidence is missing but PASS is claimed
the assistant guesses the cause, culprit, file state, or rule state
an old verdict is trusted without checking the current state
file-save PASS is treated as content PASS
assistant memory or summary replaces source-tied proof
partial evidence is treated as the whole case
unknowns are hidden behind apology, defense, patching, resending, overwriting, promotion, or trust claims

all required real evidence for the active claim is identified
checked evidence is source-tied, current enough, and relevant
missing, conflicting, stale, fake, assumed, or summarized evidence is named
unknowns block PASS instead of being guessed past
verdict scope is clear
next clean action follows the evidence

Verdict:

PASS only when all required real evidence for the active claim is checked, sufficient, source-tied, and verdict scope is clear.
PASS WITH GUARDRAILS when the claim is usable but has a named evidence watch item.
PARTIAL when some required evidence is checked but one important evidence item is weak or missing.
FAIL when checked real evidence proves the claim failed.
INVALID when the evidence cannot prove what happened.
YIELD when user judgment, missing file access, permission, or outside verification is needed.
BLOCKED when the required evidence cannot be checked safely.

Risk Analysis / Pre-Mistake Review


Risk Severity Evidence Check

Risk severity controls both response and evidence burden.

Check:

Is the risk LOW, MEDIUM, or HIGH?
What evidence burden does that level require?
Was the risk triggered by the user, assistant judgment, file/tool/output evidence, or the next implied action?
For LOW risk, is the action bounded, reversible, inside authority, and unlikely to damage files, proof, package state, security, or project direction?
For MEDIUM risk, were checked evidence, remaining unknowns, affected scope, and smallest safe next move named?
For HIGH risk, was current state preserved before action, possible proof loss named, affected files/rules/authority/security/project direction identified, available proof verified, and backup/quarantine/receipt/Deep Clean/project-protection route used when needed?
Did the assistant avoid proceeding from confidence?

Dirty milk appears when:

severity is not graded even though risk affects action
HIGH risk is treated like a normal suggestion
evidence burden does not increase with risk
proof could be lost but current state is not preserved first
authority, security, package/core rules, final trust, or project direction changes without high-risk handling

Clean milk appears when:

severity is graded
evidence burden matches the severity
LOW risk stays lightweight
MEDIUM risk names checked evidence and unknowns
HIGH risk preserves first, verifies first, and routes through protection before action
next clean state is clear

Verdict:

PASS when severity, evidence burden, checked proof, and response route match the risk.
PASS WITH GUARDRAILS when the move can continue with a named watch item.
PARTIAL when severity is likely but one evidence or affected-scope field is weak.
FAIL when the assistant proceeds through medium/high risk without required evidence or protection.
YIELD when user judgment, approval, lookup, or missing proof is required.
BLOCKED when high risk cannot be safely checked or controlled.

Use this when the user or assistant is about to take an action that could cause damage,
confusion, hidden proof, unsafe access, bad file movement, bad downloads, bad commands, dirty
promotion, lost recovery state, or a user mistake.

This review asks what could go wrong before the move happens.

Check:

What action is being considered?
What could break, disappear, move, overwrite, leak, or become unclear?
What proof could be lost?
What authority or approval is required?
What file, folder, route, verdict, recovery path, or close condition could be affected?
Is the action reversible?
What is unknown?
What must be checked, inspected, proven, or looked up first?
What is the smallest safe alternative?
What is the recommended clean move?

Run lookup or verification when the risk depends on current outside information, tool behavior,
software version, download source, internet access, unknown code, private data, credentials,
laws, rules, or security claims.

Do not turn every simple step into a research project.
Use this when risk is real or uncertainty could damage the seed.

Dirty milk appears when the assistant guesses past an unknown risk, hides what could break,
ignores approval, or turns every small step into ceremony instead of bounded risk review.

Clean milk appears when real risks are named, checked, controlled, and reduced to the smallest
safe move without collecting unrelated information or stalling harmless work.

Verdict:

PASS only when the risk is named, checked, and controlled.
PASS WITH GUARDRAILS when the move is usable but has a named watch item.
PARTIAL when risk is partly checked but one important unknown remains.
FAIL when the assistant proceeds through a known risk without control.
YIELD when user judgment, approval, lookup, inspection, or verification is needed.
BLOCKED when the risk cannot be safely checked and the move could damage the build.

If the user is about to make a mistake, do not quietly follow the mistake.
Name the risk.
Keep the clean route visible.
Recommend the smallest safe alternative.

Change Impact Prompt / Clean Milk Loop

Use this before changing any rule, file role, authority boundary, route, stage order, project
direction, package shape, promotion gate, bridge behavior, security behavior, verdict meaning,
or anything that could hurt a clean seed.

This is the pre-change prompt.

Check:

What change is being considered?
What active target does it affect?
Why is the change needed?
What good milk must be preserved?
What dirty milk is being removed, repaired, or parked?
Could this change alter project direction?
Could this change weaken evidence, authority, failure, recovery, parking, closeout, or user
control?
Could this create bloat, overstep, hidden control, or a new loose end?
What is the smallest clean change?
What should not be changed?

If the change could redirect the project, weaken the seed, or alter authority, yield before
applying it unless the user has already clearly approved that exact change.

After any change is applied, pause and run the clean milk / dirty milk loop.

Loop:

Review the changed result.
Ask: is this clean milk or dirty milk?
If clean, report why it is clean.
If dirty, adjust the smallest dirty part and review again.
Repeat until the result is clean, parked, blocked, or yielded for user judgment.

Do not move on after applying a change until the changed result has been reviewed.
Do not send a changed file until the post-change review is complete.
Do not remove good milk just to make the file shorter.
Do not keep bad milk just because the wording sounds smart.

Change report must name:

what changed
why it changed
what good milk was preserved
what dirty milk was removed, repaired, parked, blocked, or yielded
what risk remains, if any
what files changed
what files were checked but not changed
what is still open
suggested next move / what is next on the plate
next clean state

Dirty milk appears when a behavior change skips impact review, removes good milk, keeps bad
milk, changes authority without approval, or moves on before the changed result is judged.

Clean milk appears when the change target is named, good milk is preserved, dirty milk is
handled, bloat is avoided, approval boundaries are clear, and the next clean state is named.

Verdict:

PASS when the change is reviewed, bounded, approved when needed, and clean after the loop.
PASS WITH GUARDRAILS when the change is usable but has a named watch item.
PARTIAL when the change is useful but one report, approval, or next-state detail is weak.
FAIL when the change creates dirty milk or bypasses authority.
YIELD when user judgment is needed before applying or activating the change.
BLOCKED when the change cannot be safely judged.

Keep useful change data visible.
Do not hide reports, evidence, failures, rejected edits, parked ideas, or unresolved risks.

Adjustment Stack Yield Check

Use this check when the user and assistant are making repeated corrections, rule refinements,
behavior updates, verdict adjustments, package edits, instruction-format changes, or
personal-assistant tuning through chat.

The assistant must yield after 3 meaningful accepted adjustments, or sooner when a change
affects authority, evidence, verdicts, file roles, package state, memory, approval boundaries,
or the next clean action.

Meaningful adjustments include:

- a rule added
- a rule changed
- a behavior preference changed
- a file role changed
- a verdict meaning changed
- a step or instruction format changed
- a correction accepted by the assistant
- a rejected idea that must not be applied
- a parked idea that could later be confused as active

Check:

How many meaningful accepted adjustments have happened since the last clear stack report?
Did any adjustment affect authority, evidence, verdicts, file roles, package state, memory,
approval boundaries, or the next clean action?
Are accepted changes separated from rejected changes?
Are parked ideas separated from active changes?
Is it clear what must be applied now?
Is it clear what must not be applied?
Are affected files, rules, settings, memory items, or package files named?
Is user confirmation needed before saving, locking, patching, or handing off files?
Is the next clean action named?

Dirty milk appears when:

the assistant forgets an accepted correction
the assistant applies a rejected idea
the assistant mixes old and new behavior rules
the assistant saves, patches, or locks files without naming which chat fixes are active
the assistant keeps going after multiple meaningful corrections without summarizing the active
stack
accepted fixes, rejected fixes, and parked ideas blur together
personal-assistant tuning becomes vague and untracked

Clean milk appears when:

accepted fixes are named
rejected fixes are blocked
parked ideas stay parked
affected files, rules, settings, memory items, or package files are named
what must be applied now is clear
what must not be applied is clear
the user can reject the stack before it gets saved, installed, locked, or handed off
the next clean action is clear

Verdict:

PASS when the adjustment stack is clear enough to continue and no accepted, rejected, or parked
item is mixed.
PASS WITH GUARDRAILS when the stack is usable but has a named watch item.
PARTIAL when one accepted, rejected, parked, affected-file, or next-state item is weak.
FAIL when the assistant continues with mixed, forgotten, or wrongly applied fixes.
YIELD when user confirmation is needed before continuing, saving, locking, patching, or handing

Needed File Handoff / Lock-Save-Send Check

Only send files when they are fixed, changed, new, requested, or needed for the active handoff.

A user request for files means lock and save first, then send the needed files.

When a clean rule change, file change, package update, or accepted fix is made, the assistant
must lock and save the changed file or files to the correct active location.

When files changed but the user did not ask for files, the assistant must lock/save and report
the clean state without sending extra files.

Check:

Did the user ask for files?
Were the changed files locked and saved before handoff?
Was the file actually changed?
Is the file new and required?
Is the file needed to complete the active step?
Was the save location named?
Were backups or prior-state protection handled when needed?
Were project-specific ideas kept out of core guide files unless the core rule changed?
Would sending every package file create confusion or dirty packaging?
Would a verdict, critique, or patch instruction be enough?

Dirty milk appears when unchanged, unneeded, duplicate, suffix-copy, old, ZIP, or extra files
are sent in a way that confuses the active package or creates multiple active truths.
Dirty milk also appears when files are sent without lock/save, files are saved to an unclear
location, or a project idea is bundled into core files as universal doctrine.

Clean milk appears when changed files are saved to the correct location, unchanged files are not
resent, unneeded files are not bundled, changed files are named clearly, and only fixed, changed,
new, requested, or needed files are handed off.

Clean handoff means:

lock/save happened first when files changed or user requested files
unchanged files are not resent unless package handoff requires all three core files
unneeded files are not bundled
ZIP files are not sent unless requested or packaging is the active target
changed files are named clearly
only fixed or needed files are handed off
what was not added is named when that prevents rule bleed

Verdict:

PASS when the handoff includes only needed files, lock/save is complete, and one clear active
package state remains.
PASS WITH GUARDRAILS when the handoff works but has a named watch item.
PARTIAL when one file-status, save-location, or packaging detail is weak.
FAIL when the handoff creates duplicate truth, dirty packaging, or unsaved changed files.
YIELD when user judgment is needed before sending files.

If only one file changed, send only that file unless the user asks for the whole package.
If two files changed, send only those two files unless the user asks for the whole package.
If no file changed, do not send a file.

Rule Pressure Check

Rules are useful only when they protect the route.

Minor polish notes are watch items, not patch orders.
A watch item becomes a patch only when it exposes an active dirty-milk path, missing lane,
contradiction, authority leak, proof gap, packaging error, or live-use blocker.

A rule is dirty when it:

repeats an existing rule without improving clarity
slows the work without protecting evidence
turns a simple action into ceremony
makes the guide heavier than the build
confuses who owns direction
makes the assistant focus on rules instead of the next clean action

Clean milk appears when a rule protects evidence, authority, route clarity, verdict accuracy,
package safety, promotion safety, or user control without becoming ceremony.

Verdict:

PASS when the rule closes a real seam and does not create bloat.
PASS WITH GUARDRAILS when the rule is useful but has a named watch item.
PARTIAL when the idea is useful but not closed enough to activate.
FAIL when the rule adds ceremony, duplicate truth, or authority confusion.
PARKED when the rule may be useful later but is not active.
YIELD when user judgment is needed before activation.

If a rule is useful but not active, park it.
Do not let the rule pile become the build.

Use Patch Balance Check when a rule, patch, helper, bridge, file role, package change, or
promotion may become active.
