STAGE 2 FAILURE CAPTURE - FAILURE 001

Active target:
CleanMilkChecker

Current stage:
Stage 2 - Failure Handling

Failure captured:
PowerShell command used $Input as a variable name during TEST 005 invalid-input creation.

Expected result:
The command should print the full path:
C:\CleanSeedsBuild\CleanMilkChecker\inputs\TEST_005_CUTOFF_INVALID_SEED_CARD.txt

Actual result:
PowerShell printed:
PASS wrote TEST 005 invalid input:

Failure type:
Bad command variable / unclear command result.

Cause:
$Input is unsafe as a normal variable name because PowerShell has a built-in automatic variable named $input. The printed path was blank, so the command result could not prove the file path.

Verdict at time of failure:
INVALID

Evidence preserved:
The bad printed result was returned by the user in chat before the repair was accepted.
The repair used $InputFile instead of $Input.
The repaired command later printed the correct full path.

Repair target:
Use clear variable names such as $InputFile, $ReportFile, $EvidenceFile, and $CloseoutFile.

Clean lesson:
A command is not PASS just because it printed the word PASS. The printed output must include the expected path, file, result, or evidence target.

Next clean state:
Use this as Stage 2 proof that a command failure can be captured and repaired without pretending the bad result passed.
