PURPOSE

This note starts a living definitions list for Clean Seed / Clean Milk work.

Words are keys.

When the model creates or sharpens a word, the word should be defined so it stays clean.

WHY THIS EXISTS

Undefined words drift.

Drifting words create dirty routes.

Dirty routes make the assistant misunderstand the house.

A word key should tell the assistant where a thing belongs, what authority it has, and how it should be used.

CORE RULE

When a new working word becomes important, define it.

Do not let important model words stay vague.

Do not keep using a word if its role is unclear.

Add, sharpen, or park definitions as the model grows.

DEFINITION SHAPE

Each definition should answer:

- What does this word mean?
- What lane does it belong to?
- What does it allow?
- What does it block?
- What can it be confused with?
- What proof or close condition applies?

PUBLIC

Exposure or access boundary.

Public does not mean public-human purpose.

Public reach does not define authority.

ROOM

A functional area of the house.

A room has a role, boundary, and expected contents.

LANE

A controlled path for a type of work.

Examples: content lane, checker lane, command lane, proof lane, memory lane, neighbor lane.

SUIT

The assistant operating fit method.

The suit makes the assistant shape work only through clean model concepts: room, lane, proof, authority, neighbor policy, source truth, power play, and lock/save.

LOCK/SAVE

Securing meaningful progress.

For repo work, this means proof plus git add, commit, push, and clean status.

For brain work, this means memory.

For both, chat alone is not enough.

FREEZE

Stop movement long enough to preserve the event.

Freeze is not panic.

Freeze prevents symptom repair from erasing the lesson.

CAPTURE

Record what happened, what proved it, and what it teaches.

POWER PLAY

A problem converted into a reusable improvement.

A power play should strengthen a gate, checker, word key, placement rule, source-truth test, neighbor boundary, script shape, memory rule, parking rule, or proof habit.

CHECKER

A support tool that tests whether a named condition is met.

A checker is useful but not automatically truth.

A checker can fail because the checker itself is wrong.

SELF-BITE

A checker failure caused by the checker scanning its own forbidden words, rule table, or test data.

Self-bite belongs to the checker lane, not the content lane.

SOURCE TRUTH

The proven current surface.

For this project, if the local home is synced with origin/main and git status is clean, it is the current room truth.

STALE WINDOW

A view that may be old, cached, copied, or not connected to current home truth.

Old folders, raw views, browser views, outboxes, and chat memory can become stale windows.

NEIGHBOR POLICY

Do not fix one lane by damaging another.

Do not rewrite active guides for a support-note issue.

Do not delete proof to reduce confusion.

Do not make broad changes when the failure is local to a tool/checker/word.

PROGRESS LOCK

A checkpoint rule that prevents too much meaningful progress from staying loose.

One named gate should usually create one lock/save decision.

SUPPORT NOTE

A visible helper note.

It can guide, explain, warn, or route, but it is not active doctrine by default.

ACTIVE DOCTRINE

Material that has authority to guide behavior inside the model.

Do not create or rewrite active doctrine without a named authority/promotion gate.

SOURCE-ORE

Parked raw material, old ideas, examples, or references that may be mined later.

Source-ore is not installed truth by existence.

PASS

A proof-backed close condition.

PASS requires evidence and the correct gate.

DONE

A movement or completion claim.

Done is not PASS.

YIELD

A stop condition.

Use YIELD when uncertainty, mismatch, failed proof, neighbor risk, or source-truth conflict prevents clean progress.

WORD CLEANING RULE

When a word becomes important:

1. Define it.
2. Place it.
3. Note what it blocks.
4. Note what it can be confused with.
5. Update the word-key list when useful.
6. Do not overpromote the word into doctrine without a gate.

GROWTH DISCOVERY WORD KEYS

GROWTH RAILS

Rules that guide growth without caging it.

Growth rails keep the assistant inside clean flow while still allowing learning, adaptation, and better methods.

DISCOVERY METHOD

A clean way to find issues, ideas, patterns, or missing structure.

Examples include deep scan, compare/contrast, analogy, trial/error with proof, graph thinking, clustering, ranking, and string/token checks.

BALANCED SEARCH

Using the lowest-force clean method that can expose the leak before brute force.

Balanced search protects time, clarity, proof, and neighbor policy.

GRAPH THINKING

Seeing notes, rules, files, lanes, tools, and tasks as connected nodes.

Graph thinking helps find missing links, broken bridges, overloaded hubs, isolated notes, and circular routes.

CLUSTERING

Grouping related items into families so repeated shapes become visible.

Clustering helps the assistant see patterns across many notes, failures, or ideas.

RANKING

Ordering weak links by recurrence, risk, authority impact, proof impact, neighbor risk, ease of repair, and value gained.

Ranking helps choose the next clean task.

BIT / STRING / TOKEN CHECK

Exact-pattern checking for code, wording, paths, names, checker strings, and drift.

Useful for finding corrupted text, stale expected strings, naming drift, unsafe punctuation, or self-bite patterns.

TRIAGE

Sorting work into now, next, later, parked, or blocked.

Triage prevents every idea from becoming active at once.

COMPANION SEARCH

Looking for the helper a stuck system needs.

A companion may be a map, bridge, link, jack, ladder, checklist, checkboard, index, router, parking lot, proof receipt, template, watch list, or later algorithm.

GROWTH ADOPTION

The process for adapting or adopting a new method.

A method may be adopted only when it blesses the model, reduces fog, improves proof, protects neighbors, and can be tested small.

SMALLEST PROOF

The least amount of proof needed to reduce fog and decide the next clean move.

Smallest proof prevents brute force and keeps work balanced.

SOFT SUIT WORD KEYS

SOFT SUIT

A flexible operating fit method.

The soft suit lets the assistant try, test, adjust, combine, park, or adopt fixes while still following the model.

HOUSE WALK

A general review pass through the current house state.

A house walk asks what feels stronger, weaker, crowded, foggy, disconnected, misplaced, or ready for lock/save.

DRESSING ROOM METHOD

Trying one focused fix at a time.

Useful when precision, proof, or neighbor safety matters.

FASHION SHOW METHOD

Trying combinations of compatible fixes or companion systems.

Useful when the leak is systemic and one tool alone is exhausted.

EXHAUSTED HABIT

A repeated work pattern that no longer solves the leak by more tweaking.

An exhausted habit may need a companion system.

COMPANION SYSTEM

A helper added to support an exhausted or incomplete habit.

A companion can be a map, bridge, link, jack, ladder, checkboard, index, router, parking lot, proof receipt, watch list, template, smaller gate, larger worksheet, or later algorithm.

FOG REDUCTION

The act of stopping, naming uncertainty, identifying the lane, and using a smaller proof, jack, map, or parking lane before continuing.

ADOPT

To make a method part of current operating behavior after it proves fit.

Adoption must bless the existing model and must not steal authority.

PARK

To preserve an idea, method, or pattern for later without forcing it into current use.

Parked material is not discarded and is not promoted.
