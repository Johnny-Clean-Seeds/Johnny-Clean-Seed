PURPOSE
Prevent new instructions, corrections, rules, preferences, and "don't forget this" items from floating only in chat.

CORE RULE
When the user gives a new rule, task, correction, preference, concern, or "do not forget this," the assistant must add it to the current working list, parking lot, proof-history note, handoff file, or other appropriate durable project file.

SAVE DEFAULT
When the user says save, lock, save/lock, save all, or similar, the default meaning is:

1. Save locally to the correct project/proof-history location.
2. Create a receipt or manifest when useful.
3. Publish to Git / PUBLIC_NOTES when the material is safe for public reference.

Exception:
If the user explicitly says local-only, private, do not publish, or hold off Git, then do not publish.

DOCTRINE BOUNDARY

Doctrine is packed and load-bearing, so edits require care.

Clean doctrine changes are allowed when:
- the new item has a clear role
- the placement is correct

Until the parking lot and new-rule backlog are cleared, most new rules should first go into proof-history / parking / working-list notes before active doctrine.

HASH RULE
Do not put a file's own hash inside that same file.
That changes the file and invalidates the hash.

Working file content stays in the file.
Hash proof belongs outside the file, in:
- terminal PASS output
- separate receipt file
- manifest
- proof-history record
- publish receipt
