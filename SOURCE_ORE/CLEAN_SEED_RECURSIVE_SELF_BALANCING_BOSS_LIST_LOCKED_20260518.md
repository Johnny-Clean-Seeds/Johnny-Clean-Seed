# Clean Seed Recursive Self-Balancing Review — Boss List Draft

## Review Scope

Reviewed inputs:

* Uploaded visual system cards: Patch Balance, Controlled Promotion, Controlled Bridges, The Seed, Done Is Not Pass, Names Are Compression Keys, Mission Lock, Evidence Before Repair.
* Pasted critique text describing the current house state, checkpoint concerns, and stale anchor concern.
* Public repo surfaces available from the provided shortcut: front door, `CURRENT_TRUTH_INDEX.txt`, `ACTIVE_ANCHOR.txt`, `PUBLIC_HOUSE_MAP.txt`, and active guide/check material.

## Core Verdict

The plan is strong as a philosophy and operating language. It already has many right instincts: seed discipline, evidence before repair, controlled promotion, bridge permissions, proof-scoped PASS, parking lanes, mission lock, and a boss/weakest-link loop.

The biggest weakness is that the plan is still heavier on doctrine than on executable recursion. It says the right things, but the next level needs more machine-like self-balancing: state, score, thresholds, ledgers, gates, escalation rules, decay rules, and automated contradiction checks.

The system should not merely remember the rules. It should repeatedly measure itself against the rules, choose one proven weakest link, apply the smallest clean fix, wear only that fix, walk around, rescore, and either continue or stop.

## Recursive Self-Balancing Frame

The recursive frame should be the same at every level:

`SEED -> ROUTE -> CHECK -> SCORE -> BOSS -> FIX -> PROOF -> WEAR -> WALK-AROUND -> RESCORE -> PROMOTE/PARK/ROLLBACK -> NEXT SEED`

Every level should expose the same small interface:

* Name: what this unit is.
* Mission: why it exists.
* Home: where truth lives.
* Authority: who or what can decide.
* Input: what may enter.
* Output: what leaves.
* Evidence: how truth is proven.
* Failure: how failure is seen.
* Recovery: how it resets.
* Parking: what waits safely.
* Promotion: how it grows.
* Close: how it stops.

That is what makes the plan recursive instead of just organized.

## Boss List

### Boss 01 — Stale Active Anchor

**Weak link:** The active anchor can point to an older active ball while newer saved doctrine exists elsewhere.

**Why it matters:** The system can have strong rules but still resume from the wrong door. That makes recursion unsafe because each loop starts from uncertain state.

**Need more:** A mandatory anchor-alignment check before every boss cycle, install, promotion, handoff, or runtime lane move.

**Fix shape:** Add an `ANCHOR_SYNC_CHECK` that compares `ACTIVE_ANCHOR`, `CURRENT_TRUTH_INDEX`, git HEAD, latest receipt, and user command.

**Pass condition:** One current active ball, one next allowed move, one blocked list, one source of durable truth.

---

### Boss 02 — Doctrine Overload Without Runtime Enforcement

**Weak link:** Many rules exist, but rule existence is not the same as rule firing.

**Why it matters:** The assistant can quote the system while still missing the actual trigger under pressure.

**Need more:** Trigger-to-check mapping with explicit command/action hooks.

**Fix shape:** Build a `RULE_TRIGGER_MAP` where every rule has: trigger phrase, trigger situation, required check, output format, pass/fail route, and owner.

**Pass condition:** A user or contractor can test whether each rule fires from an example input.

---

### Boss 03 — No Quantified Balance Score

**Weak link:** Patch balance exists visually, but not yet as a measurable scoring system.

**Why it matters:** Without scoring, “helpful” versus “bloat” becomes feeling-based.

**Need more:** A simple weighted score for every patch, tool, rule, and promotion.

**Fix shape:** Score each candidate from -2 to +2 across: control help, clarity, fit, evidence, recovery, load, stiffness, bloat, obscurity, misfit.

**Pass condition:** Every accepted patch has a visible score and reason.

---

### Boss 04 — Boss Selection Can Become Preference

**Weak link:** The next boss may be chosen because it feels important, not because it is the highest proven constraint.

**Why it matters:** The system can overbuild interesting areas while the real bottleneck remains.

**Need more:** Boss ranking criteria.

**Fix shape:** Rank candidate bosses by: current damage, recurrence, blast radius, blocker value, proof strength, small-fix availability, and load cost.

**Pass condition:** The selected boss has a higher priority score than parked candidates.

---

### Boss 05 — Fixes Can Be Worn Without Regression Test

**Weak link:** “Wear the fix” is strong, but it needs a regression replay before becoming part of the suit.

**Why it matters:** A fix can solve the visible issue while breaking a neighbor lane.

**Need more:** Tiny replay suite for each worn fix.

**Fix shape:** Before wearing a fix, replay: original failure, neighbor lane, authority check, evidence check, load check, and next-anchor check.

**Pass condition:** The fix is worn only after it proves it does not dirty the core route.

---

### Boss 06 — Walk-Around Is Too Subjective

**Weak link:** Walk-around currently risks meaning “feel if it is cleaner.”

**Why it matters:** Feeling is useful as a signal, but not as verdict.

**Need more:** Walk-around checklist.

**Fix shape:** Walk around six rooms: active anchor, active guide, proof lane, parking lane, bridge lane, user-facing command lane.

**Pass condition:** Each room is marked cleaner, same, heavier, broken, or unknown.

---

### Boss 07 — Recursive Levels Are Not Numbered

**Weak link:** “Levels up” is not yet formal.

**Why it matters:** The system may grow without knowing what new responsibilities come with each level.

**Need more:** Level model.

**Fix shape:** Define levels:

* L0 Seed: one route, manual proof.
* L1 Wrapped Seed: active guides and checks.
* L2 Self-Checking Seed: ledgers and trigger checks.
* L3 Self-Balancing House: boss cycle, scoring, housekeeping.
* L4 Controlled Bridges: external tools/workers with contracts.
* L5 Promoted System: larger whole, audits, rollback, dashboards.

**Pass condition:** Promotion between levels requires evidence, not ambition.

---

### Boss 08 — Too Many Metaphors Can Become Multiple Bosses

**Weak link:** Seed, milk, house, boss, leash, suit, bridge, rack, pallet, murder-suspect, mission lock, clean milk, and patch balance all work, but they can compete.

**Why it matters:** A weak assistant may follow the metaphor instead of the route.

**Need more:** Metaphor-to-mechanism dictionary.

**Fix shape:** Every metaphor gets a mechanical equivalent. Example: “milk filter” = intake classifier; “boss” = highest-priority proven weak link; “wear fix” = install after regression proof.

**Pass condition:** Any contractor can implement the mechanism without needing the metaphor.

---

### Boss 09 — No Single System State Machine

**Weak link:** The plan names many states but lacks one master state machine.

**Why it matters:** Recursion requires clean state transitions.

**Need more:** Canonical states and allowed transitions.

**Fix shape:** `CANDIDATE -> INTAKE -> TESTING -> PARKED|QUARANTINED|REJECTED|INSTALLED -> WORN -> MONITORED -> PROMOTED|ROLLED_BACK`.

**Pass condition:** No item can jump from candidate to installed without passing through evidence.

---

### Boss 10 — Proof History Can Become Weight

**Weak link:** Proof history is correctly marked as not active doctrine, but it can still become too heavy to use.

**Why it matters:** A proof room that cannot be searched becomes a graveyard.

**Need more:** Proof index and retention classes.

**Fix shape:** Classify proof into: receipt, regression proof, audit proof, historical note, rejected evidence, backup, incident evidence.

**Pass condition:** The system can answer “what proves this?” without reading the whole proof pile.

---

### Boss 11 — Parking Lane Needs Review Clock

**Weak link:** Parked ideas are preserved, but parked can become forgotten.

**Why it matters:** Good ideas may never return, while stale ideas may keep attracting attention.

**Need more:** Parking review cadence and expiration/decay.

**Fix shape:** Each parked item gets reason, revisit trigger, owner, expiry or archive date, and promotion requirement.

**Pass condition:** No parked item has unclear reason or no revisit trigger.

---

### Boss 12 — Promotion Needs Demotion

**Weak link:** Controlled promotion is strong, but demotion/retirement is less explicit.

**Why it matters:** Self-balancing systems must remove or downgrade things that stop helping.

**Need more:** Demotion path.

**Fix shape:** Add states: installed-watch, deprecated, retired, replaced, rollback-candidate.

**Pass condition:** Every promotion includes a rollback or demotion condition.

---

### Boss 13 — Bridges Need Interface Contracts

**Weak link:** Controlled bridges say permissioned contact, not open access. Good. But every bridge needs an exact contract.

**Why it matters:** Bridges are where authority leaks happen.

**Need more:** Bridge contract template.

**Fix shape:** For every bridge: source, destination, allowed inputs, allowed outputs, transform, verifier, failure capture, approval requirement, logs, rollback.

**Pass condition:** No bridge can pass raw authority, only verified output.

---

### Boss 14 — Human Approval Can Become Vague

**Weak link:** Human approval exists, but approval phrase, scope, and lifetime can blur.

**Why it matters:** The system might treat one approval as broad permission.

**Need more:** Approval object model.

**Fix shape:** Approval must name action, files/lane, time/scope, destructive/non-destructive status, rollback, and whether it is one-time or standing.

**Pass condition:** Approval cannot be reused outside its named scope.

---

### Boss 15 — Done/Pass Split Needs Tooling

**Weak link:** “Done is not PASS” is excellent, but needs automatic verdict scope labels.

**Why it matters:** Completion can be mistaken for correctness.

**Need more:** Output footer standard.

**Fix shape:** Every closeout ends with: movement done, proof checked, pass scope, unknowns, next state.

**Pass condition:** No final answer says done without saying whether it is PASS, PARTIAL, YIELD, INVALID, or BLOCKED.

---

### Boss 16 — Evidence Before Repair Needs Snapshot Requirement

**Weak link:** Evidence capture is present, but repair can still begin before a reproducible snapshot exists.

**Why it matters:** Without pre-repair evidence, the system cannot learn from failure.

**Need more:** Failure snapshot object.

**Fix shape:** Capture input, output, state, log, verdict, repair target, and disproof condition before repair.

**Pass condition:** Repair does not start until the failure event is classified.

---

### Boss 17 — Names Are Strong, But Renames Need Migration

**Weak link:** Names are compression keys, but name changes can break references.

**Why it matters:** A better name can create stale docs, broken commands, or duplicate identities.

**Need more:** Rename protocol.

**Fix shape:** Old name, new name, reason, affected files, aliases, migration date, redirect, retirement date.

**Pass condition:** A rename does not create two active identities.

---

### Boss 18 — Mission Lock Needs Interrupt Protocol

**Weak link:** Mission lock protects focus, but interruptions are inevitable.

**Why it matters:** Recursive systems must intake interruptions without losing the active route.

**Need more:** Interruption stack with severity.

**Fix shape:** Classify interruption as safety-critical, blocker, clean milk, parking, duplicate, dirty, or unrelated.

**Pass condition:** Only safety-critical or active-blocking interruptions can preempt the mission.

---

### Boss 19 — Patch Intake Needs Batch Size Limit

**Weak link:** Many good fixes can arrive together.

**Why it matters:** Too many simultaneous patches make cause/effect unprovable.

**Need more:** Patch batch size rule.

**Fix shape:** Allow one behavior patch, one anchor update, one proof receipt per cycle unless explicitly doing a scoped batch.

**Pass condition:** Each patch’s effect can be isolated.

---

### Boss 20 — Level-Up Criteria Need Hard Gates

**Weak link:** Growth through promotion is clear visually, but promotion must be hard to fake.

**Why it matters:** Systems often fail by scaling before they can recover.

**Need more:** Level-up gate.

**Fix shape:** To level up: prove stable route, failure handling, rollback, evidence sufficiency, load budget, bridge safety, and no stale anchors.

**Pass condition:** No level-up without a signed promotion receipt.

---

### Boss 21 — Self-Balancing Needs Load Budget

**Weak link:** The plan watches bloat but lacks a numeric load budget.

**Why it matters:** A system can be correct but too heavy to use.

**Need more:** Load budget per layer.

**Fix shape:** Track active rules, active commands, active files, active tools, open bosses, parked items, proof volume, and response complexity.

**Pass condition:** If load crosses threshold, housekeeping becomes candidate boss.

---

### Boss 22 — Too Much Active Guide Material

**Weak link:** The active guides are long.

**Why it matters:** Long guides are harder for assistants and humans to execute consistently.

**Need more:** Kernel/expanded split.

**Fix shape:** Keep a tiny always-on kernel and move expanded doctrine to referenced modules.

**Pass condition:** The assistant can run the core from a short kernel and pull modules by trigger.

---

### Boss 23 — Command Surface Can Expand Too Much

**Weak link:** Many commands are useful, but command count can create confusion.

**Why it matters:** Users should not need to remember a giant command menu.

**Need more:** Command grouping and aliases.

**Fix shape:** Group commands under Intake, Proof, Balance, Repair, State, Promotion, Bridge, Cleanup.

**Pass condition:** A user can ask naturally and the assistant maps to the right command/check.

---

### Boss 24 — Unknown Command Handling Needs Suggestions

**Weak link:** Unknown command handling says do not run, but should suggest likely corrections safely.

**Why it matters:** Typos should not stall the system or trigger wrong behavior.

**Need more:** Did-you-mean matcher with confirmation.

**Fix shape:** For typo commands: show top 1–3 likely commands, explain difference, require exact confirmation.

**Pass condition:** No typo silently runs.

---

### Boss 25 — Recursive Child Systems Need Parent Contract

**Weak link:** Promoted systems become part of a larger whole, but parent/child responsibilities need precision.

**Why it matters:** Recursion fails when children inherit too much authority or parents micromanage.

**Need more:** Parent-child seed contract.

**Fix shape:** Child exposes inputs, outputs, evidence, failure, recovery, authority limit, and promotion state to parent.

**Pass condition:** Parent can verify child without taking over child internals.

---

### Boss 26 — Clean Seed Needs Health Dashboard

**Weak link:** Current system is text-ledger heavy.

**Why it matters:** Self-balancing improves when health is visible at a glance.

**Need more:** Health metrics.

**Fix shape:** Dashboard fields: anchor freshness, proof freshness, open bosses, load score, parked count, quarantine count, last failure, last rollback, next action.

**Pass condition:** The next action is obvious from the dashboard.

---

### Boss 27 — No Drift Trend Tracking

**Weak link:** Drift is checked per event, but trends are not tracked.

**Why it matters:** Recurring small drift reveals structural weakness.

**Need more:** Drift ledger.

**Fix shape:** Track drift type, trigger, affected lane, recurrence count, fix, regression status, and last seen date.

**Pass condition:** Third recurrence automatically becomes boss candidate.

---

### Boss 28 — Weak Failure Taxonomy

**Weak link:** Failures are captured, but failure categories should be standardized.

**Why it matters:** Good recursion depends on learning from comparable failures.

**Need more:** Failure codes.

**Fix shape:** Example codes: ANCHOR_STALE, AUTH_LEAK, PROOF_MISSING, PATCH_BLOAT, BRIDGE_OVERREACH, COMMAND_AMBIGUOUS, LOAD_RED, PASS_UNSCOPED.

**Pass condition:** Every failure is coded before repair.

---

### Boss 29 — Repair Target Can Be Too Broad

**Weak link:** Evidence before repair is strong, but repair target must stay narrow.

**Why it matters:** Broad repair hides cause/effect.

**Need more:** Repair target bounding.

**Fix shape:** Repair target must name exact seam, affected lane, not-touched list, and smallest expected behavior change.

**Pass condition:** If target cannot be bounded, verdict is YIELD or MOCK first.

---

### Boss 30 — No Automatic Stop-Loss

**Weak link:** The system has stop language, but needs clear stop-loss thresholds.

**Why it matters:** Self-improvement loops can become analysis loops.

**Need more:** Stop-loss gates.

**Fix shape:** Stop when load red, proof missing, anchor stale, user approval required, next action obvious, or repeated loop produces no new evidence.

**Pass condition:** The assistant stops before it starts polishing forever.

---

### Boss 31 — Bloat Detection Needs Positive Definition

**Weak link:** Bloat is named, but “necessary complexity” versus “bloat” can blur.

**Why it matters:** Some complexity is needed for safety.

**Need more:** Complexity justification rule.

**Fix shape:** Complexity is allowed only when it reduces risk, recurrence, ambiguity, or manual load more than it adds operational load.

**Pass condition:** Every complex addition has a tradeoff note.

---

### Boss 32 — Stiffness Needs Measurement

**Weak link:** Stiffness is listed as a risk driver, but not yet detected mechanically.

**Why it matters:** A system can become safe but hard to adapt.

**Need more:** Stiffness test.

**Fix shape:** Ask: can this rule handle a new but similar case without rewrite? Does it require too many approvals? Does it block harmless variation?

**Pass condition:** Rules protect the core without freezing the edges.

---

### Boss 33 — Obscurity Needs Reader Test

**Weak link:** The language is powerful but dense.

**Why it matters:** A contractor or future assistant may not understand it fast enough.

**Need more:** Cold-reader test.

**Fix shape:** Give a new reader one page and ask them to identify active file, next move, blocked move, proof path, and pass judge.

**Pass condition:** Cold reader gets at least 4/5 correct.

### Boss 34 — Misfit Detection Needs Placement Check

**Weak link:** Good ideas may be placed in the wrong lane.

**Why it matters:** A good rule in the wrong home becomes dirty.

**Need more:** Placement matrix.

**Fix shape:** Each item must choose one: active guide, check, command, receipt, proof history, public note, test lane, shed, parking, quarantine.

**Pass condition:** No item is saved without lane and role.

### Boss 35 — Public vs Local Boundary Still Needs Sync Proof

**Weak link:** The public/local distinction is better, but public repo and local truth can drift.

### Boss 36 — Receipts Need Stable IDs

## Main Critique in One Sentence

The current plan has the right soul, language, and safety instincts; now it needs a smaller executable kernel, measurable scorecards, explicit state transitions, regression replay, anchor sync, and a boss ledger so the recursion is not just described but mechanically enforced.
