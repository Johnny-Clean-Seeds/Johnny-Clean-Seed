## 4. Deepened Weak Lanes

### 4.1 Symbolic Output Forensics

Second-pass correction: symbolic recurrence needs an **anti-HARKing forensic ledger**, not just a symbol ledger.

Useful fields:
- Symbol observed.
- Exact output text or artifact.
- Timestamp / order.
- Source system or model.
- Prompt/input immediately before output.
- Human cue present? yes/no/unknown.
- Prior refeed present? yes/no/unknown.
- Platform memory/thread/context carryover possible? yes/no/unknown.
- Independent recurrence? yes/no/unknown.
- Control condition available? yes/no/unknown.
- Base-rate/null expectation.
- Alternative explanations.
- Decision: anomaly, motif, cued recurrence, contaminated recurrence, parked.

New source anchors:
- Benjamini-Hochberg FDR: many symbol searches raise false-positive risk.
- Kerr HARKing: do not present post-hoc symbol interpretations as if predicted first.
- Blind analysis: hide labels/outcomes before evaluating recurrence when stakes are high.
- National Academies forensic science report: forensic pattern disciplines need standards, validation, and bias control.
- Karlin-Altschul/BLAST E-values: sequence similarity needs significance against a random model.
- Matrix profile / motif discovery: repeated subsequences can be useful, but window size and distance choices matter.

INFERENCE: For Crown Thread-style judgment, the smallest safe ledger is not "symbol recurred." It is **symbol + chain of custody + cue/refeed map + null/control + interpretation status**.

### 4.2 Emotion / Feeling As Signal

Second-pass correction: prediction-error language is useful, but active inference and interoception make the rule sharper.

Useful distinction:
- Feeling as **sensor**: "my map and current input do not match."
- Feeling as **attention allocator**: "inspect this region."
- Feeling as **value signal**: "this matters to goals/needs."
- Feeling as **verdict**: blocked; not proof.
- Feeling as **command authority**: blocked.

Better anchors:
- Seth and Friston: active interoceptive inference links emotion to embodied prediction and regulation.
- Barrett constructed emotion: emotion categories involve prediction, categorization, and prior experience.
- Scherer appraisal theory: emotion involves appraisal components, action tendencies, physiology, expression, and feeling.

INFERENCE: Clean Seed should avoid saying "fog proves drift." A cleaner statement is: **fog is a prediction-error alarm that authorizes inspection, not conclusion**.

### 4.3 Hidden-State / Ghost-Boss Detection

Second-pass correction: "causal states" are powerful but too heavy for immediate use. Start with simpler hidden-state triage:

1. Observable symptom string:
   - repeated action, repeated wording, repeated blocked move, repeated urge, repeated authority confusion.
2. Candidate hidden parent:
   - stale anchor, source-ore command leak, missing receipt, unclear authority, unresolved conflict, user/codex role confusion.
3. Alternative parents:
   - fatigue, ambiguity, new user command, bad file state, tool limitation.
4. Intervention:
   - inspect anchor, inspect path/hash, ask if unclear, park source ore, write bookmark.
5. Result:
   - symptom reduced, unchanged, worsened, unknown.

Better anchors:
- Computational mechanics defines causal states as histories with the same predictive futures.
- HMMs model hidden states generating observations.
- Probabilistic suffix trees and context-tree weighting provide variable-history sequence prediction without inventing a hidden boss.

INFERENCE: The ghost-boss card should not claim hidden state. It should ask: "What hidden parent would make these repeated symptoms predictable?"

### 4.4 Proof Receipts / Provenance

Second-pass correction: the first report over-focused on hashes and Merkle trees. Proof receipts need **identity, lineage, custody, and replayability**.

Useful ladder:
1. Plain receipt: path, action, before/after, timestamp.
2. Hash receipt: add content hashes.
3. Hash chain: each receipt includes previous receipt hash.
4. Manifest bundle: BagIt-style file manifest and checksums.
5. Provenance model: W3C PROV-style entity/activity/agent.
6. Attested workflow: in-toto/SLSA-style materials/products/steps.
7. Transparency log: append-only Merkle log with inclusion/consistency proofs.

INFERENCE: Clean Seed should test hash chains before Merkle trees. A hash chain is the smaller proof object.

### 4.5 Authority Boundaries

Second-pass correction: bitmasks are good, but not enough. Authority boundaries need role, attribute, capability, and enforcement.

Useful model:
- RBAC: who/role.
- ABAC: who + what + action + context.
- Capability: exact unforgeable permission token for a named operation.
- Biba integrity: high-integrity active doctrine should not read down from low-integrity source ore without review.
- PEP/PDP split: decision and enforcement are separate.
- Policy-as-code: declaration can be inspected, but runtime is not needed now.

INFERENCE: Clean Seed's authority card should be human-readable first:
`SUBJECT -> ACTION -> OBJECT -> CONDITIONS -> ALLOW/BLOCK -> RECEIPT NEEDED`.

### 4.6 Lifecycle State Representation

Useful state ideas:

### 4.7 Anti-Bloat / MDL

Second-pass correction: MDL is still strong, but it must be kept crude and practical.

Do not ask:

Ask:
- "What repeated mess does this rule remove?"
- "What extra carry does this rule add?"
- "What evidence would this compression erase?"
- "Can a smaller card do the job?"
- "Can we park instead of install?"

New anchors:
- MDL / MML / AIC / BIC all penalize model complexity in different ways.
- Algorithmic sufficient statistic sharpens the ideal: the model captures regularity and leaves noise as noise.

INFERENCE: The Clean Seed version should not claim formal MDL. It should be called a **Compression Burden Test** or **MDL-style Anti-Bloat Gate**.

## 7. Attack On MDL Anti-Bloat Gate

### Is MDL still the best first object?

INFERENCE: Yes, but only as a tiny practical gate. It remains the best first object because the current danger is not lack of math; it is attractive concepts wanting to become structure. MDL-style thinking asks the right first question: "Does this reduce the real mess enough to justify its added carry?"

### What would beat it?

1. **Symbol Forensics Ledger** would beat MDL if the next real problem is Crown Thread / symbolic recurrence / prompt contamination.
2. **Hamming Drift Check** would beat MDL if the next real problem is known expected-state vs actual-state mismatch.
3. **Typestate File-Lane Card** would beat MDL if the next real problem is illegal moves between source/proof/active lanes.
4. **Hash Chain Receipt Test** would beat MDL if the next real problem is receipt lineage and tamper evidence.
5. **Emotion-as-Sensor Bit** would beat MDL if the next real problem is fog/off-feeling being mishandled as either command or noise.

### What failure case would prove MDL useful?

It proves useful if a proposed new rule/tool:
- seems attractive,
- would add wording or workflow burden,
- and the MDL gate either parks it or shrinks it while preserving the useful effect.

Concrete test:
- Take one Work Shed source-ore idea.
- Fill the gate in under five minutes.
- If the gate identifies no repeated mess removed, the idea parks.
- If it identifies repeated mess removed but a smaller object can solve it, shrink it.

### What would prove it is just more bloat?

The MDL gate is bloat if:
- it is used on every tiny decision,
- it takes longer than the issue itself,
- it adds jargon,
- it becomes another mandatory ritual without changing decisions,
- it cannot name a real repeated mess removed,
- it blocks useful action by over-reviewing.

### Should it be paired?

Yes, but with one line only.

Best pairing:
- **Rate-Distortion Summary Loss line**
  - Field: "What nuance/evidence would this compression lose?"
  - Reason: MDL without loss accounting can compress away truth.

Optional situational pairings:
- Pair with **Hamming Drift Check** only when expected/actual state exists.
- Pair with **Symbol Chain Ledger** only when symbolic recurrence is involved.
- Pair with **Authority Boundary Card** only when action permissions are involved.

Do not bundle all pairs into one object. That would violate the anti-bloat purpose.
