Transition law asks what an object was, what it is becoming, what proof permits that role change, and what must not come with it. Raw source can become clean translation, but not authority by itself. # Washed Report: The Report Forge Room

## Executive Finding

The clean answer is that my writing problem is not simply “too many lists.” The deeper failure is that I keep using **ledger form** when the task calls for **report form**. Ledger form is useful inside the house because it preserves proof, status, lanes, and verdicts. But when ledger form becomes the final user-facing answer, it can feel chopped up, mechanical, and underdeveloped. That is the wrong placement of a useful tool.

The invention that came out of the wash is the **Report Forge Room**. Its job is to take clean house evidence, machine results, source findings, and candidate rules, then forge them into a formal report with headings, developed paragraphs, and a clear argumentative spine. It does not delete proof discipline. It changes the delivery shape so the answer reads like a report instead of a checklist.

## The Failure Found in the Machine

The machine found a role-theft problem. The internal proof ledger was stealing the job of the final report.

A ledger is built to track pieces: claim, evidence, proof, status, verdict, next state. The Clean Seed guide requires visible proof when trust depends on a claim, and it says a gate proves judgment only when it shows what it judged, what proof it used, what changed, what it could not prove, and what next state follows. ([GitHub][1]) That structure is correct inside the house. It is not automatically correct as the final writing surface.

A report has a different job. Report-writing guidance consistently treats reports as structured documents with headings, coherent sections, and clear navigation. Boise State’s report guidance says section headings should use appropriate levels and a typical section should develop across one to several paragraphs. ([Boise State University][2]) Westminster’s report guide says reports should be structured with headings and subheadings and written in clear, concise language. ([libguides.westminster.ac.uk][3]) KPU’s formal-report guidance adds the important point that formal reports are modular: sections should stand on their own because readers may not read every piece. ([kpu.pressbooks.pub][4])

That means the writing failure was a placement failure. I was using the proof-room shape as the report-room shape. The proof was not wrong. The output form was wrong.

## The New Room

The **Report Forge Room** sits between the machine/wash output and the final answer. It does not invent new truth. It receives cleaned material and gives it a readable structure. Its function is to convert “found parts” into a developed report.

Inside this room, the first rule is that a heading must name a real section of thought, not decorate a list. A good heading should tell the reader what role the next section plays. “Executive Finding” is different from “Clean Result.” “The Failure Found in the Machine” is different from “Dirty.” “The New Room” is different from “Tool.” The heading should carry meaning before the paragraph begins.

The second rule is that each paragraph must develop one main movement. A paragraph should not be a single loose line unless the point is intentionally brief. It should open the thought, explain it, connect it to the active issue, and show why it matters. That is the difference between formal writing and stacked notes.

The third rule is that proof should be woven into the report instead of dumped as a list. If a source matters, it should support the paragraph where it belongs. If a verdict matters, it should come after the reasoning that earns it. If a boundary matters, it should appear where the reader needs it, not as an orphaned bullet.

## The Gate Created by the Wash

The gate that belongs in this room is the **Ledger-to-Report Gate**.

The gate fires when I have clean material but the output starts looking like a thin sequence of bullets, fragments, status tags, or repeated mini-headings. It asks a direct question: **Is this answer supposed to be a working ledger, or is it supposed to be a report?**

If the answer is “ledger,” then lists, fields, and compact rows may be correct. If the answer is “report,” the material must be converted into a formal structure. The report should have a title-level claim, a few meaningful headings, developed paragraphs, and a closing verdict. It can still contain a small amount of structured material, but structure should support the prose, not replace it.

This gate protects both sides. It keeps proof discipline from becoming unreadable, and it keeps formal prose from becoming vague. The house still gets evidence, scope, and verdict. The reader gets a report.

## What This Changes

This changes how I should handle your “deep walk” and “wash” requests. The internal machine can still use ledgers, checkpoints, candidate cards, and proof rows. But the final response should not automatically expose that internal shape. It should return as a report unless you ask for a checklist, code block, table, manifest, command, or file receipt.

It also changes how I should elaborate. Elaboration does not mean making a longer list. It means developing the argument. If the center is “right place,” the report should show how right-place law changes source handling, proof, writing, frame control, mule work, parking, and assistant learning. Those should become sections or paragraphs, not a tall stack of isolated claims.

This also fixes the tone problem. A report can be direct without being choppy. It can be formal without being stiff. It can be detailed without being bloated. It gives the work a spine.

## Washed Verdict

The Report Forge Room passes as a **candidate support room** for chat behavior. It is not installed doctrine, and it is not a replacement for the proof ledger. Its correct placement is between machine output and user-facing report delivery.

The clean rule is this:

**Use ledger form to think, verify, and preserve proof. Use report form to explain, synthesize, and deliver developed findings. Do not let the ledger steal the report’s job.**

The next time you ask for a deep walk, wash, research pass, or house-feel report, the default final shape should be a formal report with meaningful headings and developed paragraphs, not a long thin list.

That is a threshold protocol. Creative Room is allowed to generate. It is not allowed to drive the task by itself. A fresh idea must cross through classification before becoming active. The latch gives the crossing labels: use now, adapt now, park, TODO, blocked, or reject. ([GitHub][7])

Threshold Room V2 should absorb that exact discipline. It should not become a new broad work loop. It should make crossings cleaner. If a creative idea is found while building a report, it can be used or adapted only if it helps the target report. If it suggests a future room, it gets parked or TODO. If it requires local proof not available in chat, it is blocked or guardrailed.

That keeps the Creative Room alive without letting it become boss.

## The Inner Tools of the Threshold Room

The deeper build produces four small internal tools. These are not separate doctrines. They are organs inside the Threshold Room.

The first is the **Role Passport**. Every crossing object carries a passport: current role, proposed role, authority limit, proof needed, blocked use, and fallback lane. It is not a giant form. It is the minimum identity needed to prevent role theft.

A transcript’s passport might say: current role source ore; proposed role candidate support; authority limit not doctrine; proof needed old-failure replay or fit test; blocked use exploitation or universal psychology claim; fallback source ore or parked.

A receipt’s passport might say: current role event/file proof; proposed role support evidence; authority limit does not prove behavior or truth beyond its recorded scope; proof needed matching claim; blocked use fake PASS; fallback proof history.

A report’s passport might say: current role chat analysis; proposed role candidate project material; authority limit not installed until saved locally/Git with receipt; proof needed replay or user-approved save; blocked use active-guide rewrite by chat.

The second tool is the **Crossing Question**: “What is this trying to become?” That one question catches most drift. If the object is a phrase trying to become doctrine, the crossing stops. If it is a list trying to become a final report, it must pass through Report Forge. If it is source ore trying to become clean translation, it must keep the founding original linked and state its boundary.

The third tool is the **Airlock Receipt**. The receipt records the crossing in compact form. It does not need a giant file every time, but trust-affecting crossings need enough trace to show old role, new role, proof, boundary, and next state. This matches the guide’s requirement that gate evidence show what was judged, what proof was used, what changed, what could not be proved, and what next state follows. ([GitHub][8])

The fourth tool is the **Return Door**. If the crossing fails, the object must not hang loose. It returns to source ore, parking, TODO, blocked, rejected, reference-only, or watch. This prevents the most common residue problem: useful material that is not active but also not placed.

## The Threshold Room and Report Forge

Report Forge was not wrong. It is a specialized threshold surface. It handles one particular crossing: internal proof/ledger material becoming user-facing report material.

The earlier writing failure came from letting ledger form steal the report’s job. Lists, fields, and verdict rows are useful inside proof work. They become dirty when they are used as the final delivery surface for a task that asks for a formal report. The uploaded note already identified this: a list is not dirty because it is a list; it is dirty if it does not serve the active delivery surface. 

Threshold Room gives Report Forge a parent rule. Before output, the assistant must ask what delivery surface the material is crossing into. If the crossing is into a command, use a copyable command block. If into a receipt, use compact proof fields. If into a report, use formal headings and developed paragraphs. If into a file, create and verify the file. If into chat behavior, keep it short enough to activate.

This makes writing cleaner without losing proof discipline.

## The Threshold Room and High-Voltage Source Ore

The sharp-material problem also becomes cleaner here. Words like manipulation, dominance, control, leverage, persuasion, status, ego, and frame do not get judged by label at the door. They enter the Threshold Room as mechanisms.

The transcript describes identity, control, and safety pressures and shows how people may defend self-image, agency, predictability, status, or social security under threat.  It also describes control and safety dynamics, including frame hijacking, perceived autonomy loss, status threat, hierarchy disruption, and insecurity activation. 

Those mechanisms are useful. They belong in shield, diagnostic, pressure-reading, task-frame custody, source-ore translation, and communication clarity. They do not belong in scam tactics, coercion, hidden extraction, fake trust, humiliation, or consent bypass.

The Threshold Room’s job is not to moralize the label. It is to inspect the crossing. Is this mechanism trying to become a shield, a diagnostic lens, a communication guard, or an exploit script? Same material, different crossing. Cleanliness depends on placement, use, boundary, proof, and intent.

This is the exact refinement your correction demanded: **the house does not reject sharp tools. It airlocks them.**

## How Threshold Room Prevents Bloat

The main risk is that Threshold Room could become another ceremony. The alignment check warns that a rule is dirty when it repeats without improving clarity, slows work without protecting evidence, turns simple action into ceremony, makes the guide heavier than the build, confuses direction, or makes the assistant focus on rules instead of the next clean action. ([GitHub][5])

That means the Threshold Room must scale by risk.

For low-risk crossings, the threshold check is almost invisible. The assistant simply keeps the right shape: a note stays a note, a report stays a report, a phrase stays a compression key.

For medium-risk crossings, the assistant names the role change briefly. For example: “This is source ore becoming candidate support, not doctrine.”

For high-risk crossings, the assistant must name the full Role Passport: old role, proposed role, proof, boundary, fallback, and verdict. High-risk crossings include promotion, file authority, proof claims, local/public truth, doctrine edits, automation, destructive action, private/sensitive source material, and behavior-controlling rules.

This scaling keeps the Threshold Room clean. It protects crossings without making every step a ritual.

## Deeper Wash: What Threshold Room Is Not

Threshold Room is not a new king room. It does not outrank `CURRENT_TRUTH_INDEX`, `ACTIVE_ANCHOR`, active guides, or user direction. The active anchor explicitly blocks treating chat, memory, mule output, or source ore as authority by itself and blocks promotion without separate proof. ([GitHub][3])

Threshold Room is not a doctrine installer. It can say, “This is a candidate crossing.” It cannot say, “This is now installed.”

Threshold Room is not a moral filter. It does not reject high-voltage concepts because the labels sound dirty.

Threshold Room is not a creativity blocker. It lets Creative Room produce weird, sharp, useful ideas, then catches them before they become active without placement.

Threshold Room is not a proof substitute. It checks whether proof is needed and whether proof fits the crossing. It does not create proof from language.

Threshold Room is not a list generator. It can use field checks internally, but final delivery must pass through the proper output surface.

The clean version stays narrow: **crossing control, not general control.**

# Deep Research Gate Build and Threshold Room Run

## Executive Finding

The Deep Research Gate belongs **after threshold intake and before consolidation**. It should not start at the very front door, because research without a named task becomes broad crawl. It should not start after consolidation, because then the answer has already hardened and the research only decorates it. The clean placement is: first identify the active object and its proposed role; then open research wide enough to gather evidence, analogies, contradictions, and neighboring patterns; then stop before the consolidator compresses the material into a final answer.

So the correct placement is:

**Task Frame → Threshold Intake → Deep Research Gate → Creative/Comparative Expansion → Wash → Pre-Consolidator Stop → Consolidator / Report Forge / Final Judge**

The end must sit **before the consolidator** because the consolidator is a compression machine. Deep research is an expansion machine. If the consolidator fires too early, it turns the first good pattern into “the answer.” That is exactly the premature-closure failure we keep catching. The Deep Research Gate exists to prevent the first clean handle from becoming the final center before it has been walked through evidence, neighbors, contradictions, and old failures.

The washed result is that the **Threshold Room** is still clean, but after running it through the Deep Research Gate, it becomes more precise: it is not simply a door, not simply a room, and not simply a gate. It is an **airlock-and-customs system for role changes**. It checks what an object is, what it wants to become, whether the crossing is allowed, what proof authorizes the crossing, what boundary prevents role theft, and what fallback catches it if the crossing fails.

## Why the Deep Research Gate Must Start After Threshold Intake

The uploaded Threshold Room note already found the key weakness: the house is strong in rooms, but the weakness appears at doorframes, where an object changes state and starts pretending it has a new job. It named examples: a transcript crossing from source ore into doctrine, a receipt crossing from proof-of-file into proof-of-truth, a phrase crossing from compression key into active center, and a public URL crossing from window into local truth. 

That note is already a research object. But before research expands it, the object itself must be placed. Otherwise deep research becomes a vacuum cleaner that pulls in everything. The house front door supports that boundary: it says the GitHub repo is publicly reachable, but public reach does not define authority; public/raw/browser view is a window, not the room, and a window does not outrank the room when the room is synced and clean. It also says not to promote, delete, move, overwrite, or call PASS from the home view alone. ([GitHub][1])

That is exactly why Threshold Intake must come first. Before research starts, the assistant must identify the current object and its role. Is the object a source note, candidate room, active guide change, chat behavior support, durable project material, or proof artifact? The active anchor also blocks broad new rule piles, doctrine rewrites, active-guide rewrites, current-truth rewrites, automation install, and promotion without separate proof. ([GitHub][2]) So the Deep Research Gate cannot authorize promotion. It can only strengthen the candidate and prepare it for a later bounded replay or save route.

The clean start point is therefore not “go research everything.” It is: **after the object has a passport.**

For this run, the object passport is clear. Current role: uploaded chat notes and prior washed candidate report. Proposed role: deeper candidate architecture for Threshold Room / Doorframe Gate. Authority limit: not doctrine, not installed, not active-guide rewrite, not current-truth rewrite. Proof burden: demonstrate fit against existing house rules, repeated failure paths, and neighboring concepts. Boundary: no broad rule pile, no fake PASS, no local save claim from chat.

That placement lets research begin clean.

## What the Deep Research Gate Actually Does

The Deep Research Gate is not “search the web more.” That would be too small. It is a research-mode crossing gate. It forces an idea to pass through several kinds of evidence before the consolidator compresses it.

Its job is to keep the idea open long enough to become strong, but bounded enough not to become drift. The problem-solving latch already contains the needed discipline. It says the failure was that new ideas became active work before the active task closed, and it gives the order: active problem, target output, close trigger, risk class, fresh-ideas pass, idea class, execution scope, proof, file-return mode, Final Judge, and close. It states the core correction plainly: **problem-solving order comes before creativity**. ([GitHub][3])

The Deep Research Gate plugs into that order as a controlled expansion layer. It does not replace the Fresh-Ideas Pass. It sits before the consolidator and after threshold intake. The Creative Room may generate angles, analogies, weird routes, and missing structures, but the latch decides what enters the active task. The problem-solving latch says creative tools can generate ideas but do not decide execution, and they must not bypass active problem, target output, close trigger, risk class, proof, or Final Judge. ([GitHub][3])

So the Deep Research Gate should ask for more than “ideas.” It should ask for neighboring patterns, contradiction pressure, old-failure replay, proof shape, boundary risk, and naming accuracy. It should bring in research pressure without letting research become a runaway task.

The clean function is:

**Deep Research Gate expands the evidence field before consolidation, but it cannot promote, install, or start a new chain.**

## Where the Deep Research Gate Ends

The end belongs right before the consolidator.

That placement matters because the consolidator is dangerous when it fires early. Consolidation is where loose findings become a center, report, rule card, candidate, or saved package. That is useful only after the research field has been stretched far enough.

If the Deep Research Gate ends too late, the consolidator will already have decided the answer. Then research becomes decoration. If the gate ends too early, the idea never gets pressure-tested and the first strong phrase takes over. The house’s active guide warns that a rule existing is not proof the rule fired, and that a gate only proves useful judgment when it leaves evidence of what it judged, what proof it used, what changed, what it could not prove, and what next state follows. ([GitHub][4])

So the Deep Research Gate must close with a **pre-consolidator research receipt**. Not a giant receipt every time, but a clean declaration of what was researched, what patterns repeated, what contradicted the idea, what old failures it catches, what it cannot prove, and what the consolidator is allowed to compress.

That prevents the consolidator from smuggling untested material into the final answer.

## Running the Threshold Room Through the Deep Research Gate

The active problem is that the Threshold Room concept may still be too metaphorical. It sounds right, but it needs deeper structure before consolidation. The target output is not an installed rule. The target output is a stronger candidate architecture that explains where the Threshold Room starts, what it checks, how it interacts with Deep Research, and what it catches. The close trigger is a washed report that names the improved architecture, proof status, limits, and next route.

The research field opened around the idea and found the stronger pattern: the Threshold Room is not just a doorframe. It is an **airlock, customs desk, role passport, and pressure equalizer**.

The airlock metaphor handles environmental difference. Source ore, proof history, active doctrine, chat behavior, mule returns, reports, local truth, and public windows all have different pressure. If an object moves directly from one environment to another, it can carry pressure that does not belong there. Source pressure becomes command pressure. Report polish becomes truth pressure. Public visibility becomes authority pressure. Receipt existence becomes proof pressure. The airlock prevents that by forcing the object to equalize before entering the next room.

The customs metaphor handles inspection. A customs desk does not decide whether the traveler exists. It checks whether the traveler is allowed to cross under the claimed identity. That is exactly what the Threshold Room does. A transcript can cross into source-ore analysis. It cannot cross into doctrine by itself. A receipt can cross into event proof. It cannot cross into behavior proof without replay. A cockpit card can cross into assistant-facing support. It cannot cross into active-guide authority. The cockpit card itself states that it is an in-chat assistant operating card, not a replacement for Mr.Kleen, not a doctrine rewrite, and not doctrine promotion. ([GitHub][5])

The role passport handles identity. Every crossing object must carry current role, proposed role, proof needed, boundary, fallback, and verdict. That is not a bureaucratic form. It is the minimum needed to prevent role theft. The alignment check says behavior-controlling checks should name trigger, what to inspect, minimum evidence, dirty milk conditions, clean milk conditions, verdict labels, and next clean state, while also warning not to paste giant duplicate templates into every rule. ([GitHub][6]) That fits the Threshold Room exactly: lightweight for low-risk crossings, full passport for high-risk crossings.

The pressure equalizer handles frame drift. The frame transcript showed that a frame defines what is happening and who controls meaning; once a frame shifts, someone can react inside a different version of reality. The house version is that a task can silently cross from action into explanation, from source use into source talk, from wash into summary, from learning into “I learned” language. The Threshold Room catches that as a task-frame crossing.

It said the house already has rooms with clear jobs, but the weakness appears at the doorframes between rooms, where an object changes state and starts pretending it has a new job. It named the important examples: transcript into doctrine, receipt into proof-of-truth, clean phrase into active center, correction into broad rule pile, public URL into local truth, cockpit card into authority, list into final report, and sharp concept into moral panic or exploit fear. 

That was the right discovery. The problem was that the first version still described the doorframe more than it engineered the doorframe. It said “the house needs threshold control,” but it had not fully defined where the threshold begins, what passes through it, how research enters, when research must stop, and how consolidation is allowed to begin.

The deeper build fixes that. Threshold Room is no longer just a metaphor. It is a crossing-control surface. Deep Research Gate is no longer just “think more.” It is the expansion layer that lives between placed-object intake and final compression.

## The Correct Start Point

Deep Research must not start at the raw front door. If research begins before the object is placed, it becomes a broad crawl. The assistant starts grabbing analogies, rules, sources, memories, old reports, and creative links before it knows what job the current object is supposed to do. That creates bloat, not depth.

So the start point is **after Threshold Intake**.

Threshold Intake asks: what is the object, what role does it currently have, what role is it trying to become, and what authority does it not have?

That comes before deep research because research needs a target. A transcript is not researched the same way as a receipt. A correction is not researched the same way as a local file. A public URL is not researched the same way as current local truth. A candidate gate is not researched the same way as an installed rule. If the object’s current role is not named, research cannot know what counts as support, contradiction, boundary, or proof.

The current-truth index reinforces this. It says `CURRENT_TRUTH_INDEX` is for orientation and source routing, while `ACTIVE_ANCHOR.txt` holds the current active ball, next allowed move, and blocked moves when present. It also warns that older action-looking text inside the index is historical unless restated by the current user command or active anchor. ([GitHub][3]) That is a start-point rule. Do not research old “next move” text as live command until it crosses through current authority.

So the correct sequence begins like this in plain language: first hold the task frame, then identify the object, then give it a role passport, then open research.

## The Role Passport

The Role Passport is the first tool inside the Threshold Room. It prevents silent crossing by forcing the object to carry its identity before it moves.

A Role Passport does not need to be a giant form. It needs enough closure for the active risk. The alignment check says behavior-controlling checks should name enough parts for a weak assistant to apply without guessing: trigger, what to inspect, minimum evidence or fields, dirty milk conditions, clean milk conditions, verdict labels, and next clean state. It also warns not to paste giant duplicate templates into every rule. ([GitHub][4])

That means the Role Passport should scale.

For a low-risk crossing, the passport can be almost invisible. A note stays a note. A report stays a report. A phrase stays a phrase. No ceremony needed.

For a medium-risk crossing, it should be named briefly. “This is source ore becoming candidate support, not doctrine.” “This is a receipt proving file creation, not behavior.” “This is a chat correction becoming live behavior in this chat, not a repo install.”

For high-risk crossings, the passport must be explicit. High-risk crossings include source-to-doctrine, receipt-to-PASS, chat-to-local-truth, mule-to-authority, public-to-current, candidate-to-installed, report-to-save, automation, destructive file action, and sharp mechanism-to-tactic.

The passport carries five questions. What is this now? What is it trying to become? What proof permits that crossing? What boundary remains after crossing? Where does it go if the crossing fails?

That is the core of “no silent crossing.”

## Where Deep Research Enters

Deep Research Gate begins only after the Role Passport exists. Its mission is not to decide the answer. Its mission is to prevent shallow consolidation.

This gate opens the field. It asks for neighboring patterns, contradictions, old-failure replay, analogy pressure, boundary risks, proof gaps, dirty translations, and possible better structures. It is the place where Creative Room can work hard without becoming boss.

That distinction is critical. Creative Room is allowed to invent. It can bring in metaphors like airlock, customs desk, passport, pressure equalizer, doorframe, forge, and washer. It can compare house behavior to social frame control, source intake, file custody, report writing, mule workflows, and proof discipline. But Creative Room cannot promote. It cannot decide that a new idea is doctrine. It cannot skip proof because the idea feels good.

Deep Research is expansion. Consolidation is compression. If those roles blur, the first strong phrase wins too early.

That is exactly what happened before. “Right place” was clean, but I compressed around it too soon. Then “Threshold Room” was clean, but the first version still needed a deeper run. The learning here is that a clean phrase is often not the answer. It is the handle that lets the assistant walk farther.

## The Research Passport

Deep Research also needs its own passport, because research itself can become dirty if it expands without bounds.

A Research Passport names the active question, the field of allowed research, the excluded lanes, the old failures being tested, the kind of evidence being sought, the contradiction pressure needed, and the stop condition. This keeps the gate from becoming a broad crawl.

For the current object, the Research Passport is straightforward. The active question is whether Threshold Room needs a Deep Research Gate partner, where that partner starts, where it ends, and how the full flow should work. The allowed research field is the uploaded Threshold note, the public house front door, current truth split, active anchor, alignment guide, problem-solving latch, cockpit logic already discussed, and the frame-control transcript already in the room. The excluded lanes are doctrine rewrite, active-guide rewrite, current-truth rewrite, local save claim, automation, promotion, and broad rule pile.

The old failures being tested are public URL becoming local truth, source ore becoming authority, receipt becoming proof of behavior, phrase becoming final center, list becoming report, user correction becoming giant rule pile, sharp concept becoming dirty by label, and deep research becoming endless expansion.

That is bounded research. It can go deep without going everywhere.

## The Correct End Point

The Deep Research Gate must end before the consolidator.

This is the part you specifically pushed for, and it is right. The end needs to be before consolidation because consolidation is the compression machine. Once the consolidator fires, the material starts becoming a report, card, verdict, rule, candidate, or save package. If research happens after consolidation, it is no longer shaping the answer; it is decorating a conclusion that already hardened.

That is one of the assistant’s common failures. It finds one good center, then later adds supporting detail around it. That is backwards. The center should survive the research, not be chosen before the research.

The correct endpoint is the **Pre-Consolidator Stop**.

The Pre-Consolidator Stop asks: has research expanded enough to compress honestly? What did it find? What contradicted the idea? What old failures did it catch? What proof is missing? What status is the consolidator allowed to claim?

This stop is not the final judge. It is the gate that decides whether the material is ready to enter the consolidator. It stops both failure modes: endless research and premature compression.

The alignment guide’s verdict discipline supports this. It says PASS only applies when every required check passed together; PASS WITH GUARDRAILS applies when the build is clean but needs an active watch item; PARTIAL, FAIL, INVALID, PARKED, YIELD, BLOCKED, PROMOTED, and RETIRED each have different jobs. It states the core law plainly: done is not PASS, movement is not proof, confidence is not proof. ([GitHub][4])

So the Pre-Consolidator Stop cannot say “installed.” It can say “ready to consolidate as candidate,” “needs more evidence,” “blocked,” “parked,” or “pass with guardrails for report delivery.”

## The Full Machine

The washed machine now has a clean route.

It begins with **Task Frame**, because the assistant must know what it is actually being asked to do. Without task-frame custody, everything downstream proves the wrong thing.

It moves into **Threshold Intake**, where the current object is named. This prevents raw material from entering the house as a mystery object.

It then creates a **Role Passport**, which names current role, proposed role, proof need, boundary, fallback, and verdict limit.

Only then does **Deep Research Gate** open. This gate expands the field, looks for neighbors and contradictions, checks old failures, and lets Creative Room invent possible structures without promoting them.

The material then goes through **Wash**, where overclaim, label contamination, wrong-lane material, unsupported proof, bloat, premature promotion, and false certainty are removed.

Then comes **Pre-Consolidator Stop**, where the machine decides what may be compressed and what status the compressed result may claim.

After that, **Consolidator / Report Forge** shapes the result for the correct delivery surface: report, candidate card, receipt, code, file, instruction, or short chat response.

Finally, **Final Judge** checks whether the run stayed in lane, proof matched the claim, the object kept its role, and the verdict is honest.

This route is not a long list for its own sake. It is the cleaned pipeline:

**Frame → Intake → Passport → Research → Wash → Pre-Consolidator Stop → Report Forge → Final Judge.**

## What the Deep Research Gate Adds to Threshold Room

Threshold Room alone catches silent role conversion. That is strong, but not enough. It can stop a bad crossing, but it does not necessarily make the candidate richer before final compression.

Deep Research Gate adds depth pressure.

It asks whether the candidate has been walked through nearby rooms. It asks whether the source has contradiction pressure. It asks whether the metaphor is only pretty or actually useful. It asks whether old failures are caught. It asks whether the boundary is narrow enough. It asks whether the output form is correct. It asks whether the idea is still only a handle or has become a working mechanism.

For Threshold Room, deep research turned a doorway metaphor into a real crossing system. The deeper run produced four internal functions.

The first is the **Airlock**. Different rooms have different pressure. Source ore has source pressure. Proof history has evidence pressure. Active guide has authority pressure. Chat has behavior pressure. Public URL has visibility pressure. Local repo has current-truth pressure. The airlock prevents one pressure from leaking into another.

The second is the **Customs Desk**. It inspects what the object claims to be and whether it is allowed to enter the next room under that identity. A transcript may enter as source ore. A receipt may enter as file-state proof. A candidate may enter as trial material. None of those identities automatically become doctrine.

The third is the **Role Passport**. It records the minimum identity and boundary needed for crossing.

The fourth is the **Pressure Equalizer**. It catches task-frame drift. If the task is “walk the house,” it must not silently become “explain walking.” If the task is “run the wash,” it must not silently become “summarize the wash.” If the task is “elaborate,” it must not become a thin verdict. If the task is “report,” the ledger must not steal the report’s job.

These four functions make the Threshold Room more than a phrase.

## What the Gate Prevents

The Deep Research Gate prevents the assistant from mistaking the first clean handle for the final answer. That is the big behavioral repair.

It catches “right place” becoming a slogan before it is walked through proof, source ore, mule, parking, report writing, and assistant learning. It catches “Threshold Room” becoming a nice metaphor before it gets start point, end point, passport, airlock, and pre-consolidator stop. It catches “Deep Research Gate” becoming “research everything” instead of bounded expansion. It catches “Report Forge” becoming fancy headings without proof. It catches “wash” becoming one-pass summary.

The Threshold Room prevents wrong crossing. The Deep Research Gate prevents shallow crossing. The Pre-Consolidator Stop prevents early hardening.

Together, they solve the repeated shape of the failure.

## How This Handles Sharp Source Material

The sharp manipulation/frame-control material becomes cleaner inside this architecture.

Without Threshold Room, the assistant may reject it by label or accept it too loosely. With Threshold Room, the material enters as high-voltage source ore. Its Role Passport says it may become shield, diagnostic lens, task-frame custody support, communication mechanics, or pressure literacy. It may not become scam tactic, coercion script, false-trust method, humiliation leverage, or consent bypass.

Without Deep Research Gate, the assistant may stop at “sharp tools belong in the right place.” With Deep Research Gate, the material is walked deeper. It asks what mechanism is actually present. Is it about identity pressure, control pressure, safety pressure, frame definition, loaded language, premise inspection, or not explaining under pressure? Which old assistant failure does that mechanism explain? Does it protect task frame, proof language, source intake, or user boundary?

That is how the house learns from high-voltage material without becoming dirty or blind.

## How This Handles Report Writing

The same architecture fixes the writing problem.

A ledger is not dirty. It is clean in proof work. But when a ledger crosses into final user-facing delivery without a report-form check, it becomes wrong-place output. The Threshold Room catches the crossing: internal ledger wants to become final report. It must pass through Report Forge.

Deep Research Gate then asks whether the report has enough developed argument before consolidation. It should not simply turn a list into headings. It should develop sections. A heading should name a real movement of thought. A paragraph should open, explain, connect, and prove the point. Proof should be woven where it supports the claim. The final verdict should compress what the report earned, not what the first phrase suggested.

That is why this current answer is shaped as a report. The machine is applying its own finding.

## How This Handles Learning

The learning recipe also fits inside this architecture.

A correction first enters as live signal. It does not automatically become doctrine. Threshold Room asks what it is trying to become. If it is only a one-time correction, it may stay in chat behavior. If it reveals a repeated pattern, it may become a recipe candidate. If the same task is rerun and behavior changes, it becomes replay-proven once. If it fires later under pressure, it becomes pressure-proven. Only after proper route and save can it become durable installed material.

Deep Research Gate prevents the correction from being underdeveloped. It asks why the failure happened, how the better answer became visible, what recipe can be extracted, what old failure class it belongs to, and what same-task replay proves.

That is the clean learning machine: not apology, not self-defense, not broad rule pile. A correction becomes a movement recipe only after it changes the active task.

## The Wash

The wash removed one tempting mistake: turning this into a giant new room hierarchy. That would violate the active anchor’s warning against broad new rule piles without a real task. ([GitHub][2]) The correct version is not a doctrine pile. It is a candidate architecture for handling crossings and research depth.

The wash also removed the temptation to call this installed. It is not installed. It is not saved into Mr.Kleen by this chat. It is not active-guide doctrine. It is not CURRENT_TRUTH_INDEX. It is not final system law.

The wash kept the useful core: the architecture explains repeated failures, fits the house’s existing authority/proof boundaries, supports right-place law, protects sharp source material, fixes premature compression, and gives Deep Research a correct start and end.

The final status is therefore not raw PASS. It is **PASS WITH GUARDRAILS as a candidate architecture and chat-behavior support**.

## Final Clean Architecture

The cleaned architecture is this:

**Threshold Room stops silent role conversion. Deep Research Gate stops shallow consolidation. Pre-Consolidator Stop stops premature hardening. Report Forge turns internal proof into readable report form. Final Judge keeps the verdict honest.**

The start point for Deep Research is after Threshold Intake, once the object has a Role Passport. Starting earlier creates broad crawl. Starting later turns research into decoration.

The endpoint for Deep Research is the Pre-Consolidator Stop, immediately before the consolidator. Ending later lets the first center harden too early. Ending without a stop lets research drift forever.

The clean law is:

**No object changes role silently, and no idea becomes the answer before it survives research pressure and a pre-consolidator wash.**
