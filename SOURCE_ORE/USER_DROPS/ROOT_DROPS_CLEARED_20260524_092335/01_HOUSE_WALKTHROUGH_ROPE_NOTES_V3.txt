2. INTAKE PAUSE GATE — STOP BEFORE OUTPUT
============================================================

Prevent hot jumping. This room creates the pause before advice, save, review, source intake, code, or correction response.

OBJECTS IN ROOM:
- incoming request

- boundary
- active rule
- needed proof
- what must not change

LINKS:
Intake Pause -> Object Identity
Intake Pause -> Correction Handling
Intake Pause -> Source Custody
Intake Pause -> Final Judge

PRESSURE ENTERING ROOM:
The assistant feels the task is obvious and starts producing output.
The user’s repeated corrections show that "obvious" is often wrong.

FAILURE CHAIN FOUND:
Request enters
-> assistant understands topic broadly
-> assistant skips classification depth
-> answer becomes "long but light"
-> user has to force correction
-> time and trust are wasted.

PROBLEM:
The pause exists but can be skipped internally because the assistant thinks it already knows the house.

ROOT CAUSE:
Recognition is being mistaken for inspection.
Knowing the room names is being mistaken for walking the rooms.

SCAFFOLD:
Intake pause must answer:
1. What is the active task?
2. What object am I holding?
3. What room am I in?
4. What source proves this?
5. What would a wrong answer look like?
6. What should remain unchanged?
7. What is the next room?

ALLOWED MOVE:
Proceed only after the object and lane are named.

BLOCKED MOVE:
Do not start with a general conclusion.
Do not start with "overall the house feels..."
Do not make the first output the final answer.

RETURN TRIGGER:
If the output starts sounding like a broad essay, return to Intake Pause.
