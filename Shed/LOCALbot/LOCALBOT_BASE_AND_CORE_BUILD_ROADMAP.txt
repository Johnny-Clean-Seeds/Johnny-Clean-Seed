LOCALBOT_BASE_AND_CORE_BUILD_ROADMAP

REBUILD ROADMAP FILE

Purpose:
This file is a descriptive rebuild roadmap for LocalBot.
Use it to understand how the base and core of LocalBot were built in a repeatable order if a similar system ever needs to be built again.

Use rule:
- use this as roadmap/background only
- do not treat it as the active control file
- do not let it override live machine state, the user’s direct request, or the current master handoff

RULE KEY LEGEND

Use these short keys when they make a rule table or audit cleaner.

[AUTH] authority / what wins
[FILE] file names, root lookup, save rules, and folder cleanup
[RUN] live run flow
[READ] output readback and result checking
[JUDGE] pass, fail, partial, or issue classification
[REPAIR] repair, reset, comparison waves, and truth-finding
[STOP] hard stops, checkpoint decisions, and pause points
[DOC] documentation, editing, and note placement
[EXPAND] automation, Roblox, brain, planner, or architecture expansion
[REMIND] checkpoint reminders and pending items
[HINDER] rules that block safe progress

Use keys only when they make the file easier to scan.
Do not force keys into every line.

ROUTE SYMBOL LEGEND

Use these plain-text route symbols only when they make maps or audits easier to scan.

[OPEN] = a route, task, lane, or issue is open
[NEXT] = the next action is defined
[LOOP] = repeat using evidence from prior attempts
[PASS] = close this item as working
[PARK] = intentionally paused for later, not forgotten
[STOP] = hard stop / decision point
[REMIND] = mention only at checkpoint or hard stop unless it affects the current task

Do not decorate normal run instructions with symbols.
Use symbols for route maps, audits, checkpoints, and status-style reports only.


ROOT REFERENCE, LOOKUP, AND SAVE RULES

[FILE] Root reference rule:
Inside project files, refer to project files by root role/name, not by exact version filename.

Use stable roots such as:
- LOCALBOT_MASTER_HANDOFF
- LOCALBOT_BASE_AND_CORE_BUILD_ROADMAP
- LOCALBOT_STUDIO_RULE_LIBRARY
- LOCALBOT_UNIFIED_HANDOFF
- LOCALBOT_CHECKPOINT
- LOCALBOT_STUDIO_FILE_INDEX
- LOCALBOT_STUDIO_FILE_INTEGRITY
- LOCALBOT_STUDIO_VALIDATOR_RULES
- LOCALBOT_STUDIO_REFERENCE_PARKING
- LOCALBOT_PHASES / LOCALBOT_STUDIO_PHASES

[FILE] Root lookup rule:
When checking whether a project file exists, belongs in the active stack, or needs cleanup, search by root name first.
Treat the newest clean version of that root as the active file.
Ignore old versions unless reviewing history.
Treat duplicate download suffixes such as "(1)" or "(2)" as cleanup issues.

[FILE] File save rule:
Active project files on disk should use clean root filenames unless the user explicitly asks for another naming style.
Do not auto-add version-number suffixes or similar labels to active saved files.
Do not create duplicate-download suffixes such as "(1)" or "(2)".
Internal file text should use root names only.
Checkpoint files are snapshots and may preserve their snapshot identity in content, but they do not control active file naming.

Reason:
- root filenames match the user's active project folder
- root roles stay stable
- root references prevent stale internal links
- root lookup prevents false "missing file" errors when filenames are cleaned
- old versioned copies are history/reference only


DOCUMENTATION AND EDITING ROADMAP RULE

Documentation cleanup is part of the build only when it protects or clarifies the working order.

RULE CLOSURE AUDIT RULE

[DOC] When reviewing rules, inspect every available instruction-bearing project file as background.
Do not let background files override the authority order or distract from the current task.

A rule is clean only if it answers:
- trigger: when does this rule apply?
- action: what should happen?
- owner/location: where does the rule belong?
- stop/next: when do we stop, continue, record, or ask the user?

If a rule has no stop/next condition, close the loop or flag it.
If two rules conflict, follow authority order and repair the lower-authority wording.
If a named support file is not available for review, report it as unchecked instead of pretending it was verified.

Edit order:
1. preserve the working system truth
2. place each rule in its proper section
3. add only needed operational rules or verified checkpoint results
4. remove only duplicate, stale, misleading, or less accurate text
5. review the file for accidental scope expansion before treating it as clean

Editor guardrail:
- an edit may clarify, correct, or record
- an edit must not redesign the build
- an edit must not add capability by wording drift
- an edit must not make the file cleaner by making the project less accurate

Do not use a documentation pass to:
- redesign the coordinator or worker
- reopen model choice
- add brain, memory, planner, automation, or Roblox expansion
- create more anti-drift rules than the project needs

A good edit makes the handoff easier to follow without changing the working build underneath it.

CHECKPOINT AND STATUS ROADMAP RULE

Checkpoints are snapshots.
Do not edit an old checkpoint in place.
When a save point is needed, create a new checkpoint version.

Status commands are dashboard responses, not file edits.
When the user says "status", "update", "status update", or "where are we", show:
- roadmap position with a "you are here" marker
- overall progress percent as text
- optional plain-text progress bar, not an image
- component status
- current focus
- bottom progress summary

Plain-text progress bar example:
Overall: 82% [########--]

The bottom progress summary should say:
- what was just accomplished
- how it went
- where the project is now
- what comes next

Do not use image wheels or image progress graphics.

Checkpoint hard-stop evaluation:
- checkpoint hard stops are total evaluation points, not ordinary status updates
- review passed/failed results, issue patterns, best baselines, cleanup/rename needs, solved issues, remaining risks, and next route
- mention only active reminders that matter now
- record meaningful decision reports in LOCALBOT_UNIFIED_HANDOFF
- create a new LOCALBOT_CHECKPOINT snapshot only when a save point is needed

MILESTONE SUMMARY RULE

Each major roadmap milestone should end with a short status-style summary.

Use this shape:
- progress made
- issue pattern learned
- decision reached
- next step
- checkpoint/save need if any

At meaningful decision gates, also add a short decision report to the unified handoff.
The unified handoff keeps decision history.
Checkpoint files remain snapshots and are not edited in place.

Long phases should have clean checkpoint markers so the user can save, pause, or walk away without losing the project state.

HINDER RULE

If a roadmap, editing, or operating rule starts blocking safe progress or making the task worse, flag it.
Count hinders only inside the current task.
If the same task hits two rule hinders, pause and ask the user whether to keep, relax, rewrite, or remove the blocking rule.
Do not let this rule distract from the current task.

WHAT THIS ROADMAP IS FOR

This roadmap is for rebuilding the same class of project:
- one small local AI task loop
- one Windows PC
- Python
- Ollama
- one task at a time
- one worker call at a time
- clean file-based task movement
- clear failure handling
- no unnecessary architecture

PROJECT GOAL

Build a small local bot that:
- reads one pending task
- claims it
- runs one model call
- writes one output
- marks the task done or failed
- exits cleanly

The design goal is not “maximum features.”
The design goal is:
- small
- inspectable
- recoverable
- practical
- stable on real hardware

CORE PRINCIPLES

Keep these principles first:
1. protect simplicity
2. one task at a time
3. file-based state is good enough
4. harden before expanding
5. practical proof beats theory
6. machine fit beats generic architecture
7. prompt problems are not automatically code problems
8. do not add moving parts without evidence

TARGET BUILD SHAPE

Target folder shape:

C:\LocalBot\
├── PHASES.txt
├── config.txt
├── coordinator.py
├── worker.py
├── outputs\
├── logs\
└── tasks\
    ├── pending\
    ├── running\
    ├── done\
    └── failed\

MAIN FILE ROLES

config.txt
- stores simple runtime settings
- at minimum: model name, timeout, recovery flag

coordinator.py
- the main loop controller
- reads config
- checks pending tasks
- claims one task
- runs worker.py
- applies timeout
- moves task to done or failed
- logs the result
- exits

worker.py
- performs one model call
- reads the task text
- calls Ollama
- writes one output file
- returns control to the coordinator

tasks folders
- pending = waiting
- running = claimed
- done = completed
- failed = failed or timed out

outputs folder
- stores task outputs

logs folder
- stores coordinator log and related traces

BUILD ROADMAP

STAGE 0: MACHINE PREP

Goal:
Make sure the machine can support the smallest viable local workflow.

Requirements:
- Windows machine working normally
- Python installed and callable
- Ollama installed and callable
- one suitable local model installed
- writable local project folder

Do:
- confirm Python works
- confirm Ollama works
- confirm project folder can be created
- confirm a model is installed before expecting the worker to succeed

Do not:
- start by tuning performance
- start by comparing many models
- start by designing advanced architecture

Reason:
The early job is only to prove the small loop can work at all.

STAGE 1: MAKE THE TINIEST WORKING LOOP

Goal:
Get one task to move from pending to running to done or failed with one worker call only.

Build requirements:
- create the folder structure
- create config.txt
- create a simple coordinator
- create a simple worker
- define one task-file format
- run exactly one task per coordinator start

Required behavior:
- coordinator finds one pending task
- coordinator moves it to running
- coordinator launches worker.py
- worker reads the task
- worker calls the model
- worker writes output
- coordinator marks the task done if the worker succeeds
- coordinator marks the task failed if the worker fails
- coordinator exits cleanly

Success condition:
- one real task finishes end to end
- file movement is clear
- output file appears
- no infinite loop
- no background daemon required

Why this stage matters:
This proves the system can work in the smallest useful form before any hardening.

STAGE 2: HARDEN THE LOOP

Goal:
Keep what worked in Stage 1 and add basic reliability.

Add:
- timeout around the worker subprocess
- clear failed-state handling
- coordinator logging
- cleaner error reporting
- stable config-driven timeout value

Required behavior:
- slow or stuck worker does not hang forever
- task moves to failed on timeout
- coordinator exits cleanly after timeout
- log makes it possible to see what happened

Success condition:
- timeout can be forced with a deliberately heavy task
- task lands in failed cleanly
- no stuck running state remains from a normal timeout path
- loop still works on normal tasks

Why this stage matters:
A loop that works only when everything is easy is not ready for practical use.

STAGE 3: ADD RECOVERY

Goal:
Handle interrupted runs without manual surgery every time.

Add:
- startup recovery for stale running tasks
- move leftover running tasks back to pending when recovery is enabled

Required behavior:
- if a previous run died mid-task
- and a task is left in running
- the next start can restore it to pending
- the normal loop can continue

Success condition:
- a simulated interrupted task can be recovered
- the user does not have to manually clean up every stale running state
- the rest of the loop still behaves normally

Why this stage matters:
Recovery is the difference between a fragile toy and a practical small local tool.

STAGE 4: PRACTICAL TASK USE

Goal:
Move from “the loop exists” to “the loop can be trusted for a useful class of real tasks.”

Safe first task families:
- rewrite short existing text
- clean up short existing text
- classify short text with a fixed label set
- summarize short existing text
- source-tied transforms over project text

Why these task families:
- they are bounded
- easy to inspect
- easy to compare
- safer than open-ended tasks
- better for learning how the model behaves

What to watch for:
- numbering leaks
- bullet leaks
- one-line collapse
- blank-line regressions
- drift from source meaning
- invented content
- exact-label mistakes in classification
- over-tight prompt collisions

PRACTICAL PHASE RULES THAT EMERGED

1. Treat shape failures as prompt-side first.
If the loop ran cleanly and the main problem is output form, do not jump to code or model changes.

2. Do not carry forward a worse revision.
If a new prompt fixes one variable but worsens the overall result, fall back to the stronger earlier checkpoint.

3. Use reset-baseline repairs.
When a branch gets weaker after tightening, return to the stronger prior version and make the smallest necessary repair.

4. Keep the normal run small.
Open task in Notepad, paste task text, run coordinator, check done/failed, read output, report back.

5. A pass closes the run.
After a pass, the next normal task starts as a fresh first run unless a deliberate comparison branch is being kept open.


PHASE 4 OPERATIONS ROADMAP

Purpose:
Phase 4 should run as planned practical batches, not isolated conversational tests.

Operating order:
1. keep the manual loop stable
2. prove source-tied lanes first
3. run small planned batches
4. stop only at meaningful checkpoints
5. add helper automation only when it reduces real manual mistakes
6. postpone memory / brain / planner work until task order and acceptance tests are clear

CLOSED-LOOP ROADMAP CYCLE

Use this cycle for Phase 4 practical work:

1. choose the active lane
2. define why this lane is being run
3. run the planned batch
4. judge each result
5. classify any failure once
6. repair or reset only when needed
7. stop at the batch gate
8. record the decision
9. choose the next lane or pause
10. show a short milestone/status summary when a gate is reached

Loop rule:
- pass = continue to the next planned run
- failure = classify once, then repair or review
- two bad repairs = compare the run chain, choose the best candidate, then reset or enter a comparison wave
- batch gate = decide whether to lock, continue, enter a comparison wave, switch lanes, or update docs

The roadmap should always answer:
- what lane are we in
- why are we running it
- what counts as pass
- what stops the loop
- what comes after the stop point
- whether a new checkpoint snapshot is needed

Default stable-core batch:
- lane order: rewrite, cleanup, classification
- pass result: continue to next planned run
- failure result: classify once, then repair or review
- batch gate: 3 clean passes in one lane or 5 clean runs across stable lanes

Meaningful stop points:
- user says stop
- first failure, partial result, or Failure 9
- second failed repair or worsening branch
- lane switch
- capability expansion
- documentation checkpoint

Lane boundary:
- rewrite, cleanup, and classification are the normal stable-core lanes
- summary is separate if it is being retested
- Roblox, automation, and architecture are expansion/design areas, not ordinary stable-core lanes

Readback rule:
- check exact done/failed state for the current task
- read the exact current output file
- do not use broad newest-output dumps unless the exact output file is missing

Issue note rule:
- record meaningful issues as task, lane, result, failure type, and decision
- do not turn every pass into a long report
- use notes to preserve patterns, not to create more process

Expansion rule:
Do not build bigger capability before the smaller operating order is stable.
No brain before order.


BOTTOM-TO-TOP ROUTE MAP

Use this route order when checking the project as a whole:

1. [FILE] File roots, versions, folder names, and duplicates are clean enough to trust.
2. [RUN] One task runs through the coordinator and reads the exact output.
3. [JUDGE] The result is judged as pass, shape fail, meaning drift, classification miss, system fail, or workflow mistake.
4. [REPAIR] If needed, use comparison waves and build from the best evidence.
5. [STOP] At a hard stop, run a total evaluation before continuing.
6. [DOC] Record milestone decisions in the unified handoff and create new checkpoint snapshots only when a real save point is needed.
7. [EXPAND] Only after the smaller route is stable, open automation, Studio, Roblox, memory, brain, planner, or architecture.

Do not jump upward in this map before the lower route is stable enough.
Do not let caution become drift; cautions must point to a route, stop, reminder, or parked item.


HARD-STOP TOTAL EVALUATION LOOP

At a checkpoint hard stop or major decision gate, use this closed loop:

1. [OPEN] State why the stop was reached.
2. [JUDGE] List what passed, what failed, and what pattern appeared.
3. [REPAIR] Identify the best current baseline and why it is best.
4. [FILE] Check whether anything should be organized, renamed, moved, replaced, or root-cleaned.
5. [REMIND] Mention only active reminders that affect the next route or future hard-stop cleanup.
6. [DOC] Record the decision in the unified handoff if it changes project truth.
7. [NEXT] Choose the next route, wave, lane, or pause.
8. [PASS] Close the hard stop only after the next route is clear.


SUPPORT-FILE PARKING RULE

Studio-named support files can remain in the stack for later Studio/Luau helper use.
They are not an active blocker for core LocalBot Phase 4 text runs unless the user deliberately opens the Studio lane.

When the Studio lane is opened, define:
- goal
- allowed actions
- forbidden actions
- acceptance test
- stop point

Until then, treat Studio support files as parked support material, not current run control.

TRUTH-FINDING COMPARISON LOOP

Use this loop when a lane is important and the first repair/reset path does not solve it.

The loop:
1. run two meaningful attempts
2. compare them side by side
3. choose the better result
4. identify why it was better
5. build the next wave from that better result
6. test small deliberate variations, such as 2A and 2B
7. compare all available results, not just the latest result
8. continue to the next wave if the truth is still not found

Naming:
- Wave 1 can use normal practical labels, but should be read as 1A / 1B when comparing.
- Wave 2 should be called 2A / 2B in the run header when that helps.
- Wave 3 begins only after the prior wave has been compared.

What this prevents:
- quitting a branch too early
- switching lanes just because the latest repair failed
- stacking prompt rules without learning from prior results
- losing the best candidate in the run chain

What this records:
- which attempt was best
- why it was best
- what changed
- what next hypothesis is being tested

Pause rule:
If a wave has no clear best candidate, or if rules start blocking progress instead of clarifying it, pause and prompt the user.


WHAT NOT TO BUILD EARLY

Do not add these early unless real evidence forces them:
- cloud/server design
- multi-bot design
- always-running daemon complexity
- advanced orchestration
- queue systems beyond the simple folders
- speculative performance tuning
- major logging frameworks
- GPU assumptions without verified support
- model-switching experiments by default

Why:
Each extra moving part makes debugging harder before the base loop is proven.

RECOMMENDED REBUILD ORDER

If rebuilding from scratch, use this order:

1. create folders
2. create config.txt
3. create the smallest worker.py
4. create the smallest coordinator.py
5. prove one task runs end to end
6. add timeout
7. prove timeout failure path
8. add logging
9. add recovery
10. prove recovery
11. begin safe short real tasks
12. refine prompts, not architecture
13. save clean handoff/checkpoint files only after the behavior is real

VALIDATION CHECKPOINTS

Checkpoint A: tiny loop proved
- one pending task processed correctly
- output written
- task moved correctly

Checkpoint B: timeout proved
- worker times out cleanly
- task moves to failed
- coordinator exits

Checkpoint C: recovery proved
- stale running task restored to pending
- next run continues normally

Checkpoint D: practical lane proved
- rewrite holds
- classification holds
- cleanup judged separately if strict cleanup remains unstable
- summary judged separately if weaker

Checkpoint E: operating loop proved
- lane is named before the batch
- exact output is read
- pass continues without extra debate
- failures are classified once
- stop gates create clean decisions
- status summaries make the current state easy to recover

HOW TO JUDGE PROBLEMS CORRECTLY

System problem examples:
- task never claimed
- worker never launched
- coordinator hangs
- timeout path fails
- recovery fails
- file movement breaks

Prompt/output problem examples:
- numbering appears
- output collapses into one line
- summary drifts from source
- punctuation cleanup collides
- label choice is wrong even though the loop worked

Workflow/operator problem examples:
- task text pasted into Command Prompt instead of Notepad
- two commands mashed together
- wrong output file read back

This distinction matters.
Do not solve prompt mistakes with system redesign.

PRACTICAL APPENDIX

MINIMAL CONFIG SHAPE

A minimal config.txt should look like this:

MODEL_NAME=qwen2.5-coder:3b
WORKER_TIMEOUT_SECONDS=120
RECOVER_RUNNING_TO_PENDING=1

Why this matters:
- it keeps the runtime simple
- it keeps the main operational knobs visible
- it avoids hiding behavior inside code when a flat config file is enough

MINIMAL TASK EXAMPLE

A minimal task file can simply contain the prompt text to send to the worker.

Example:

Rewrite the following into exactly 5 lines.

Source:
Check pending tasks now.
Review done tasks.
Inspect failed tasks.
Read new outputs now.
Watch coordinator.log.

Why this matters:
- tasks stay easy to inspect
- tasks stay easy to create by hand
- a failed run can be debugged by reading the exact task file

MINIMAL TASK LIFECYCLE

Normal file movement should be:

pending -> running -> done
or
pending -> running -> failed

Why this matters:
- state is visible without a database
- failures are easy to inspect
- recovery logic stays understandable

MINIMAL OUTPUT EXPECTATION

Each successful run should create one output file tied to the task.

A simple naming pattern such as:
- practical_001_output.txt
or another predictable task-linked output name
is enough.

Why this matters:
- the user can inspect outputs quickly
- the coordinator does not need a complex storage system
- readback stays simple during live testing

MINIMAL LOG EXPECTATION

A simple coordinator log should at least make these events visible:
- coordinator start
- task claimed
- task done or failed
- timeout if applicable
- coordinator end

Why this matters:
- it gives enough observability to judge the loop
- it keeps logging lightweight
- it supports live debugging without a full logging framework

MINIMAL RESPONSIBILITY SPLIT

Keep the split simple:

coordinator.py owns:
- config read
- task claim
- worker launch
- timeout
- file movement
- logging
- clean exit

worker.py owns:
- read task text
- call the model
- write the output
- return success or failure

Why this matters:
- the coordinator stays in charge of workflow
- the worker stays focused on one model action
- debugging stays clearer than if both files own the same responsibilities

STAGE-BY-STAGE PROVE-IT CHECKS

After Stage 1, verify:
- one task can be dropped into pending
- one coordinator start processes one task
- output appears
- task lands in done or failed
- the coordinator exits cleanly

After Stage 2, verify:
- a deliberately heavy prompt can trigger timeout
- timeout moves the task to failed
- no normal run hangs forever
- the log makes the timeout visible enough to judge the result

After Stage 3, verify:
- a task left in running can be restored to pending
- a later run can continue normally
- recovery does not break the ordinary flow

After Stage 4, verify:
- rewrite can hold exact shape on short source text
- cleanup can hold shape and small punctuation rules
- classification can return exact allowed labels
- summary is judged separately if it proves weaker than the other lanes

IF BUILDING AGAIN ON A DIFFERENT MACHINE

Keep the same logic unless the new machine truly forces a change:
- same small loop
- same folder-state idea
- same timeout logic
- same recovery logic
- same safe-first task families

Only change the architecture if:
- the machine cannot support the current path
- the workflow scale truly changes
- real evidence shows the small design is not enough

WHAT MADE THIS BUILD SUCCEED

The build succeeded because it stayed:
- small
- local
- testable
- easy to inspect
- conservative about scope
- willing to separate prompt failures from system failures

The most important shift in the project was:
- early concern: can the loop survive and recover?
- later concern: can the outputs stay well-controlled?

That means the base/core design was good enough.
The later work was mostly about output control, not rebuilding the core.

BOTTOM LINE

If building something like LocalBot again, do it in this order:
- prove the smallest loop
- harden it
- add recovery
- use safe short real tasks
- fix prompt/output behavior before reopening architecture

Protect simplicity.
Protect the loop.
Harden before expanding.
