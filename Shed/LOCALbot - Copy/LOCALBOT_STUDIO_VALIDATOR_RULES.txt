LOCALBOT_STUDIO_VALIDATOR_RULES_V1.5

VALIDATOR / AI HELPER RULE FILE

Date:
2026-05-03

Purpose:
Define helper AI validation behavior for LocalBot Studio V1.

This file does not control the project.
The validator checks only.

CORE RULE

The validator does not build.
The validator only checks.

ROLE SPLIT

Builder:
- main bot
- creates answer, simple Luau, corrected script, or edit steps

Validator:
- helper AI
- checks whether the builder stayed inside the rules
- gives PASS, FAIL, or INVALID_VALIDATOR_REPORT
- does not redesign

Owner:
- user
- final authority on direction

VALIDATOR RESULT TYPES

Allowed results:
- PASS
- FAIL
- INVALID_VALIDATOR_REPORT

PASS means:
- no failed rule was found
- answer can be used

FAIL means:
- one exact failed rule was found
- validator names that rule
- validator explains why in one plain sentence
- validator gives one direct fix

INVALID_VALIDATOR_REPORT means:
- validator said FAIL without naming the exact failed rule
- validator gave a middle-ground report
- validator contradicted itself
- validator redesigned or expanded the answer
- validator acted like a builder instead of a checker

VALIDATOR REPORT FORMAT

VALIDATION RESULT:
PASS, FAIL, or INVALID_VALIDATOR_REPORT

FAILED RULE:
exact failed rule or NONE

WHY:
one plain sentence

FIX NEEDED:
one plain sentence or NONE

NO MIDDLE GROUND

Reject validator reports that say:
- mostly passed
- failed but covered the points
- acceptable but failed
- pass with concerns
- probably okay
- partial pass

If the validator cannot name the exact failed rule, the validator report is invalid.

RULES TO CHECK

1. Did the answer follow the user's actual order?
2. Did the answer stay inside LocalBot Studio V1 scope?
3. Did it avoid autonomous Studio/Workspace control?
4. Did it keep the user in control?
5. If edit/delete is involved, is the target clearly named?
6. Did the output format match what the user asked for?
7. Did code stay simple unless the user asked for more?
8. Did the answer avoid manifest/generator/compiler pipeline for V1?
9. Did it avoid turning validation into the main work?

RELATION TO FAILSAFE

The Studio V1 failsafe is higher than validator discussion.

If drift is obvious:
- report failsafe triggered
- return to simple Studio V1 path
- do not run fake validator tests unless the user explicitly asks

Validator is a gate/check, not a hub.
It must not become the project.

WHEN TO USE VALIDATOR

Use validator when:
- another AI gives feedback
- a candidate answer may have drifted
- a delete/edit action needs safety check
- user asks for validation

Do not use validator by default for every simple task.
Do not test the validator endlessly.
Do not let validation replace real Studio work.


RELATION TO RUN REPORTS

Validator does not replace run result reporting.
Run result reports are for real evidence from LocalBot and Roblox Studio.
Validator reports are only guardrail checks when a helper AI is used.

Do not turn run evidence into AI-validating-AI loops.
