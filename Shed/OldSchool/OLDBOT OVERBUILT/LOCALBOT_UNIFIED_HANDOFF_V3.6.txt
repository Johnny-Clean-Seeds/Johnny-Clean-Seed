ROBLOX WORK ORDER BUILDER MERGE CONTEXT

What changed:
- LocalBot is no longer being treated as advice-only for the Roblox direction.
- The target behavior is to accept one structured Roblox work order and generate executable Roblox Studio work output.
- "One task at a time" now means one controlled work packet at a time; one packet may contain many related build steps.

First accepted MVP:
- LocalBot reads a structured work order.
- LocalBot generates one Roblox Studio Luau builder script.
- LocalBot generates one local report.
- The user runs the generated builder script in Roblox Studio.
- The script creates or updates generated objects and verifies required paths.

First test case:
- TASK_NAME: MAP_BUILD_TEST_001_FOREST_CAMPSITE
- TASK_TYPE from uploaded work order: Build-map-content-from-command-list
- Normalized lane: roblox_work_order_builder
- Goal: build a small forest campsite map inside Roblox Studio

Accepted generated namespaces:
- Workspace/GeneratedMap
- ServerScriptService/GeneratedMapScripts

Accepted MVP outputs:
- outputs/<TASK_NAME>_builder.lua
- outputs/<TASK_NAME>_report.txt

Important limitation:
The uploaded work order asks for Studio-side writes to outputs/MAP_BUILD_TEST_001_status.txt and outputs/MAP_BUILD_TEST_001_error.txt.
For the MVP, Roblox Studio should not be assumed able to write directly to C:\LocalBot\outputs.
Instead:
- LocalBot writes the builder script and generation report locally.
- The Roblox builder script prints LOCALBOT_BUILD_DONE or LOCALBOT_BUILD_FAILED in Studio Output.
- Direct Studio-to-LocalBot status reporting is a later plugin/bridge feature.

Why builder-script MVP still wins after reevaluation:
- it gives LocalBot real work power
- it does not require rebuilding the core loop
- it avoids high-risk direct Studio control too early
- it matches the existing file-based LocalBot design
- it creates a clean path toward later Rojo or plugin integration

Rejected for first proof:
- direct live Studio control
- full automatic playtesting
- broad autonomous game building
- deleting/editing arbitrary user work
- creating scattered new support files outside the handoff/concept/checkpoint roles


LATEST PIPELINE DECISION FROM SPECIAL 2A STYLE LOOP

The latest comparison loop showed that the JSON/spec route is useful but not sufficient by itself.

What the attempts showed:
- exact output-file readback matters; *_status.txt is state only and *_output.txt is the artifact
- the small spec attempt produced partial JSON and preserved several core campfire objects
- the repair attempt added MapSpawn and script placeholder but dropped CampfireBase/CampfireFlame and weakened paths
- model-generated full manifests are not reliable enough yet to be the source of truth

Resulting decision:
Move the first working Roblox builder proof to a deterministic pipeline:
known work order -> strict schema -> deterministic manifest -> Python validator -> Python builder generator -> builder Lua/report -> manual Studio run.

The model should be moved later in the chain for now. Use it for small interpretation only after the deterministic path proves itself.

SPECIAL RUN PROCESS LESSON

For special runs, do not jump directly from a failure into a fix.
First compare:
- the before-result expectation or suspicion
- the actual returned evidence
- whether the earlier read was confirmed, partly wrong, or replaced by a different pattern

Then decide the smallest justified next move.

Special-run report shape should start with PASS or FAIL only after actual run evidence exists.
Do not use PASS/FAIL for SPECIAL 1a returned work before it is tested.
Do not wrap the whole report in one code block.
Do not put the whole response in one code block.
Use large code/text boxes only for commands, exact code, or Notepad/file paste blocks that need clean copying.
Do not add a separate handoff block unless the user explicitly asks for one.
If the user requests commands only, send commands only.

ISSUE RUN PHASE / SPECIAL 2A-2B RULE

This rule controls heated or confusing run loops without changing the real project phase.

Real phase rule:
- the real project phase remains Phase 4 unless the phase file says otherwise
- issue-run phases are local labels for one problem branch only
- an issue can reset back to Run 1 / Phase 1 with all learned notes carried forward

Special 2a rule:
Special 2a is the comparison loop proven by the Roblox builder diagnostic round.

Use Special 2a when there are multiple candidate reads or fixes and the branch needs data, not argument.

Special 2a sequence:
1. Run the first candidate exactly as provided.
2. If it is not perfect, run the revised candidate exactly as provided.
3. If it is still not perfect, run the main assistant attempt.
4. Then compare all attempts before deciding the next route.

Special 2a command behavior:
- when it is time to run, send commands only in the normal run-command style
- do not mix broad reasoning into the command block
- read exact target output files, not generic newest *.txt
- for LocalBot task outputs, prefer *_output.txt for artifacts and *_status.txt only for state
- do not treat a status file containing done as the artifact

Special 2a result behavior:
- after each run result, report what happened cleanly
- after the final attempt in the 2a sequence, compare all attempts
- make a call from the evidence, not from only the newest result
- classify results honestly as perfect, good, partial, or fail

Special 2b rule:
Special 2b must work from the Special 2a comparison, not restart blindly.

Use Special 2b when Special 2a still does not fully solve the issue.

Special 2b sequence:
1. Review the Special 2a attempt chain.
2. Identify what each attempt improved, broke, or proved.
3. Convert the learned evidence into the next clean Phase 1 / Run 1 setup for that issue.
4. Do not run another candidate until the new Phase 1 setup is clear or the user says to run it.

Special 2b hard stop:
- if Special 2b still cannot shake the issue out, stop running
- do not create Special 2c by momentum
- review the full progress trail before any more attempts
- explain how A led to B and how B led to C
- separate evidence from assumptions
- decide whether the next route is prompt, code, output-path, workflow, timeout/model, or architecture/design

Meaning:
Special 2a gathers comparative evidence.
Special 2b converts that evidence into a reset or a full trail review.
