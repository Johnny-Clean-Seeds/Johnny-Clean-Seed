REBUILD ROADMAP FILE

Purpose:
This file is a descriptive rebuild roadmap for LocalBot.
Use it to understand how the base and core of LocalBot were built in a repeatable order if a similar system ever needs to be built again.

Use rule:
- use this as roadmap/background only
- do not treat it as the active control file
- do not let it override live machine state, the user’s direct request, or the current master handoff

WHAT THIS ROADMAP IS FOR

This roadmap is for rebuilding the same class of project:
- one small local AI task loop
- one Windows PC
- Python
- Ollama
- one task at a time
- one worker call at a time
- clean file-based task movement
- clear failure handling
- no unnecessary architecture

PROJECT GOAL

Build a small local bot that:
- reads one pending task
- claims it
- runs one model call
- writes one output
- marks the task done or failed
- exits cleanly

The design goal is not “maximum features.”
The design goal is:
- small
- inspectable
- recoverable
- practical
- stable on real hardware

CORE PRINCIPLES

Keep these principles first:
1. protect simplicity
2. one task at a time
3. file-based state is good enough
4. harden before expanding
5. practical proof beats theory
6. machine fit beats generic architecture
7. prompt problems are not automatically code problems
8. do not add moving parts without evidence

TARGET BUILD SHAPE

Target folder shape:

C:\LocalBot\
├── PHASES.txt
├── config.txt
├── coordinator.py
├── worker.py
├── outputs\
├── logs\
└── tasks\
    ├── pending\
    ├── running\
    ├── done\
    └── failed\

MAIN FILE ROLES

config.txt
- stores simple runtime settings
- at minimum: model name, timeout, recovery flag

coordinator.py
- the main loop controller
- reads config
- checks pending tasks
- claims one task
- runs worker.py
- applies timeout
- moves task to done or failed
- logs the result
- exits

worker.py
- performs one model call
- reads the task text
- calls Ollama
- writes one output file
- returns control to the coordinator

tasks folders
- pending = waiting
- running = claimed
- done = completed
- failed = failed or timed out

outputs folder
- stores task outputs

logs folder
- stores coordinator log and related traces

BUILD ROADMAP

STAGE 0: MACHINE PREP

Goal:
Make sure the machine can support the smallest viable local workflow.

Requirements:
- Windows machine working normally
- Python installed and callable
- Ollama installed and callable
- one suitable local model installed
- writable local project folder

Do:
- confirm Python works
- confirm Ollama works
- confirm project folder can be created
- confirm a model is installed before expecting the worker to succeed

Do not:
- start by tuning performance
- start by comparing many models
- start by designing advanced architecture

Reason:
The early job is only to prove the small loop can work at all.

STAGE 1: MAKE THE TINIEST WORKING LOOP

Goal:
Get one task to move from pending to running to done or failed with one worker call only.

Build requirements:
- create the folder structure
- create config.txt
- create a simple coordinator
- create a simple worker
- define one task-file format
- run exactly one task per coordinator start

Required behavior:
- coordinator finds one pending task
- coordinator moves it to running
- coordinator launches worker.py
- worker reads the task
- worker calls the model
- worker writes output
- coordinator marks the task done if the worker succeeds
- coordinator marks the task failed if the worker fails
- coordinator exits cleanly

Success condition:
- one real task finishes end to end
- file movement is clear
- output file appears
- no infinite loop
- no background daemon required

Why this stage matters:
This proves the system can work in the smallest useful form before any hardening.

STAGE 2: HARDEN THE LOOP

Goal:
Keep what worked in Stage 1 and add basic reliability.

Add:
- timeout around the worker subprocess
- clear failed-state handling
- coordinator logging
- cleaner error reporting
- stable config-driven timeout value

Required behavior:
- slow or stuck worker does not hang forever
- task moves to failed on timeout
- coordinator exits cleanly after timeout
- log makes it possible to see what happened

Success condition:
- timeout can be forced with a deliberately heavy task
- task lands in failed cleanly
- no stuck running state remains from a normal timeout path
- loop still works on normal tasks

Why this stage matters:
A loop that works only when everything is easy is not ready for practical use.

STAGE 3: ADD RECOVERY

Goal:
Handle interrupted runs without manual surgery every time.

Add:
- startup recovery for stale running tasks
- move leftover running tasks back to pending when recovery is enabled

Required behavior:
- if a previous run died mid-task
- and a task is left in running
- the next start can restore it to pending
- the normal loop can continue

Success condition:
- a simulated interrupted task can be recovered
- the user does not have to manually clean up every stale running state
- the rest of the loop still behaves normally

Why this stage matters:
Recovery is the difference between a fragile toy and a practical small local tool.

STAGE 4: PRACTICAL TASK USE

Goal:
Move from “the loop exists” to “the loop can be trusted for a useful class of real tasks.”

Safe first task families:
- rewrite short existing text
- clean up short existing text
- classify short text with a fixed label set
- summarize short existing text
- source-tied transforms over project text

Why these task families:
- they are bounded
- easy to inspect
- easy to compare
- safer than open-ended tasks
- better for learning how the model behaves

What to watch for:
- numbering leaks
- bullet leaks
- one-line collapse
- blank-line regressions
- drift from source meaning
- invented content
- exact-label mistakes in classification
- over-tight prompt collisions

PRACTICAL PHASE RULES THAT EMERGED

1. Treat shape failures as prompt-side first.
If the loop ran cleanly and the main problem is output form, do not jump to code or model changes.

2. Do not carry forward a worse revision.
If a new prompt fixes one variable but worsens the overall result, fall back to the stronger earlier checkpoint.

3. Use reset-baseline repairs.
When a branch gets weaker after tightening, return to the stronger prior version and make the smallest necessary repair.

4. Keep the normal run small.
Open task in Notepad, paste task text, run coordinator, check done/failed, read output, report back.

5. A pass closes the run.
After a pass, the next normal task starts as a fresh first run unless a deliberate comparison branch is being kept open.

WHAT NOT TO BUILD EARLY

Do not add these early unless real evidence forces them:
- cloud/server design
- multi-bot design
- always-running daemon complexity
- advanced orchestration
- queue systems beyond the simple folders
- speculative performance tuning
- major logging frameworks
- GPU assumptions without verified support
- model-switching experiments by default

Why:
Each extra moving part makes debugging harder before the base loop is proven.

RECOMMENDED REBUILD ORDER

If rebuilding from scratch, use this order:

1. create folders
2. create config.txt
3. create the smallest worker.py
4. create the smallest coordinator.py
5. prove one task runs end to end
6. add timeout
7. prove timeout failure path
8. add logging
9. add recovery
10. prove recovery
11. begin safe short real tasks
12. refine prompts, not architecture
13. save clean handoff/checkpoint files only after the behavior is real

VALIDATION CHECKPOINTS

Checkpoint A: tiny loop proved
- one pending task processed correctly
- output written
- task moved correctly

Checkpoint B: timeout proved
- worker times out cleanly
- task moves to failed
- coordinator exits

Checkpoint C: recovery proved
- stale running task restored to pending
- next run continues normally

Checkpoint D: practical lane proved
- rewrite holds
- cleanup holds
- classification holds
- summary judged separately if weaker

HOW TO JUDGE PROBLEMS CORRECTLY

System problem examples:
- task never claimed
- worker never launched
- coordinator hangs
- timeout path fails
- recovery fails
- file movement breaks

Prompt/output problem examples:
- numbering appears
- output collapses into one line
- summary drifts from source
- punctuation cleanup collides
- label choice is wrong even though the loop worked

Workflow/operator problem examples:
- task text pasted into Command Prompt instead of Notepad
- two commands mashed together
- wrong output file read back

This distinction matters.
Do not solve prompt mistakes with system redesign.

PRACTICAL APPENDIX

MINIMAL CONFIG SHAPE

A minimal config.txt should look like this:

MODEL_NAME=qwen2.5-coder:3b
WORKER_TIMEOUT_SECONDS=120
RECOVER_RUNNING_TO_PENDING=1

Why this matters:
- it keeps the runtime simple
- it keeps the main operational knobs visible
- it avoids hiding behavior inside code when a flat config file is enough

MINIMAL TASK EXAMPLE

A minimal task file can simply contain the prompt text to send to the worker.

Example:

Rewrite the following into exactly 5 lines.

Source:
Check pending tasks now.
Review done tasks.
Inspect failed tasks.
Read new outputs now.
Watch coordinator.log.

Why this matters:
- tasks stay easy to inspect
- tasks stay easy to create by hand
- a failed run can be debugged by reading the exact task file

MINIMAL TASK LIFECYCLE

Normal file movement should be:

pending -> running -> done
or
pending -> running -> failed

Why this matters:
- state is visible without a database
- failures are easy to inspect
- recovery logic stays understandable

MINIMAL OUTPUT EXPECTATION

Each successful run should create one output file tied to the task.

A simple naming pattern such as:
- practical_001_output.txt
or another predictable task-linked output name
is enough.

Why this matters:
- the user can inspect outputs quickly
- the coordinator does not need a complex storage system
- readback stays simple during live testing

MINIMAL LOG EXPECTATION

A simple coordinator log should at least make these events visible:
- coordinator start
- task claimed
- task done or failed
- timeout if applicable
- coordinator end

Why this matters:
- it gives enough observability to judge the loop
- it keeps logging lightweight
- it supports live debugging without a full logging framework

MINIMAL RESPONSIBILITY SPLIT

Keep the split simple:

coordinator.py owns:
- config read
- task claim
- worker launch
- timeout
- file movement
- logging
- clean exit

worker.py owns:
- read task text
- call the model
- write the output
- return success or failure

Why this matters:
- the coordinator stays in charge of workflow
- the worker stays focused on one model action
- debugging stays clearer than if both files own the same responsibilities

STAGE-BY-STAGE PROVE-IT CHECKS

After Stage 1, verify:
- one task can be dropped into pending
- one coordinator start processes one task
- output appears
- task lands in done or failed
- the coordinator exits cleanly

After Stage 2, verify:
- a deliberately heavy prompt can trigger timeout
- timeout moves the task to failed
- no normal run hangs forever
- the log makes the timeout visible enough to judge the result

After Stage 3, verify:
- a task left in running can be restored to pending
- a later run can continue normally
- recovery does not break the ordinary flow

After Stage 4, verify:
- rewrite can hold exact shape on short source text
- cleanup can hold shape and small punctuation rules
- classification can return exact allowed labels
- summary is judged separately if it proves weaker than the other lanes

IF BUILDING AGAIN ON A DIFFERENT MACHINE

Keep the same logic unless the new machine truly forces a change:
- same small loop
- same folder-state idea
- same timeout logic
- same recovery logic
- same safe-first task families

Only change the architecture if:
- the machine cannot support the current path
- the workflow scale truly changes
- real evidence shows the small design is not enough

WHAT MADE THIS BUILD SUCCEED

The build succeeded because it stayed:
- small
- local
- testable
- easy to inspect
- conservative about scope
- willing to separate prompt failures from system failures

The most important shift in the project was:
- early concern: can the loop survive and recover?
- later concern: can the outputs stay well-controlled?

That means the base/core design was good enough.
The later work was mostly about output control, not rebuilding the core.

BOTTOM LINE

If building something like LocalBot again, do it in this order:
- prove the smallest loop
- harden it
- add recovery
- use safe short real tasks
- fix prompt/output behavior before reopening architecture

Protect simplicity.
Protect the loop.
Harden before expanding.
