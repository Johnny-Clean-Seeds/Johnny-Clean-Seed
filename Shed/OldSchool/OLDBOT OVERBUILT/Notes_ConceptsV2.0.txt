QUANTIZATION

Quantization reduces the precision of model weights so a model uses less RAM and usually runs faster, with some quality trade-off.

Common precision steps:
- FP32: full precision, highest memory use
- FP16 / BF16: about half the size of FP32
- INT8 / INT4 / INT2: lower precision, much smaller, more quality loss

GGUF

GGUF is the model file format used by Ollama through llama.cpp.
It stores model weights and metadata, including quantized variants.

Common GGUF quantization levels:
- Q8_0: near-full quality, largest quantized size
- Q6_K: very good quality
- Q5_K_M: strong quality / size balance
- Q4_K_M: common practical balance
- Q3_K_M: smaller, more noticeable degradation
- Q2_K: very aggressive compression

K-QUANT

k-quant is a GGUF / llama.cpp quantization family designed to preserve more quality than older uniform low-bit methods.

Core idea:
- weights are handled in blocks instead of one global scale
- most values use low precision
- important scaling values and outliers keep higher precision

This improves quality at the same approximate bit width compared with simpler methods.

K-QUANT VARIANTS

Common suffixes:
- _M: medium balance
- _S: smaller, more aggressive compression
- _L: larger, less aggressive compression

BLOCK QUANTIZATION

Block quantization divides weights into small groups instead of applying one global scale to a whole tensor.

Each block gets:
- its own scaling factor
- low-bit values within that block

This preserves local dynamic range better than uniform quantization.

MIXED PRECISION

Mixed precision in k-quant means not every value inside a block uses the same precision.

Typical pattern:
- most weights use low-bit storage
- scales, offsets, or outlier values keep higher precision

This reduces error without giving up most of the compression benefit.

BLOCK SIZE IMPACT

Block size is the number of weights that share a scale.

Trade-off:
- smaller blocks: better quality, more overhead
- larger blocks: lower overhead, more quality loss

Very small blocks improve fidelity but cost more metadata.
Very large blocks behave more like coarse uniform quantization.

UNIFORM QUANTIZATION

Uniform quantization uses one scale, or one simple min / max mapping, across a larger tensor.

Why it is weaker for LLMs:
- large weight ranges and outliers are handled poorly
- precision is wasted on some regions and insufficient in others

That is why newer block-wise methods are generally preferred for LLM inference.

K-QUANT VS UNIFORM

Uniform quantization:
- simpler
- cheaper to compute
- more quality loss

k-quant:
- block-wise
- mixed precision aware
- better meaning and structure retention at similar size

K-QUANT VS GPTQ

k-quant:
- built around GGUF / llama.cpp workflows
- strong fit for CPU-first inference stacks
- widely used through Ollama

GPTQ:
- a post-training quantization method often associated with GPU-oriented workflows
- can perform well at low bit widths
- usually depends on different tooling than standard GGUF / Ollama flows

QUANTIZATION METHOD FAMILIES

Main categories:
- Post-Training Quantization (PTQ): quantization applied after training
- Quantization-Aware Training (QAT): quantization effects simulated during training

Common PTQ approaches:
- uniform / RTN
- calibration-based methods such as GPTQ or AWQ
- block-wise mixed precision methods such as k-quant

MODEL OPTIMIZATION CONTROLS

General knobs that affect local LLM behavior:
- model size
- quantization level
- context length
- prompt / task design
- CPU vs GPU execution path

OFFLOADING

CPU offloading:
- work stays on CPU and system RAM when GPU VRAM is too limited

GPU layer offloading:
- some model layers are placed in GPU VRAM while the rest remain on CPU / RAM

Possible trade-off:
- can help on suitable hardware
- can hurt when the GPU is too weak or VRAM is too small

TIMEOUT HANDLING

A simple timeout around the worker subprocess is often the safest way to prevent hangs in a small local task loop.

Practical shape:
- coordinator launches worker.py with subprocess.run(..., timeout=...)
- timeout expiration raises TimeoutExpired after Python kills the child and waits for it
- task can then be marked failed and the coordinator can exit cleanly

Why this approach is attractive:
- no extra dependencies
- no custom threading model
- fewer Windows-specific edge cases than signal-based designs
- easy to adjust from config without redesigning the worker path

Alternatives that may exist but add complexity:
- soft timeout warnings inside the worker
- per-task timeout rules
- process-tree cleanup with psutil
- async or threaded wrapper designs

Concept rule:
Prefer the simplest timeout path that has already proved reliable on the real machine.
Only add deeper process control if real runs show orphan processes, blocked cleanup, or other concrete gaps.

TIMEOUT TEST DESIGN

For a local LLM worker, the clean timeout test shape is a heavy or deliberately slow prompt.
This is more realistic than shell-style sleep ideas when the worker is a model call rather than a command runner.

Good timeout checks:
- task lands in failed state cleanly
- log message is clear
- no stale running state remains
- no stray child process remains after the timeout
- coordinator exits cleanly

OLLAMA CLI AND ENCODING

On Windows, subprocess capture can be tripped up by console codepage or decoding issues.
A byte-safe capture path with explicit UTF-8 decoding and replacement for bad bytes is a practical protective pattern.

Concept points:
- PATH problems can exist on Windows even when the CLI is installed
- restart or full-path invocation may be needed after some installs
- encoding problems are separate from model quality problems
- a byte-safe worker path can avoid codepage-related crashes

GPU HANDLING ON THE CURRENT CLASS OF MACHINE

For an older Windows desktop with weak or unsupported GPUs, CPU-only execution may be the safest path.

Practical concept points:
- unsupported or too-old NVIDIA GPUs may not qualify for official acceleration paths
- integrated graphics may not provide a stable or worthwhile LocalBot path even if experimental routes exist elsewhere
- if GPU support is unverified, CPU-only is often the cleaner and safer default
- VRAM only matters if a real GPU execution path is actually being used

Machine-fit rule:
If GPU acceleration is not verified, do not assume it is helping.
Keep the working CPU path unless a real test proves otherwise.

CPU OPTIMIZATION

For a small local task loop on an older 4-core CPU, default behavior is often good enough until measurements show otherwise.

Useful general ideas:
- backend choice matters
- ISA support matters
- thread count can matter
- context length matters
- prompt/task design can matter more than low-level tuning on short runs

Safe tuning rule:
Do not add thread flags, env vars, or backend-specific knobs just because they exist.
Only tune when a real bottleneck has been observed and measured.

AVX2

AVX2 is a 256-bit SIMD extension available on many x86 CPUs, including Skylake-era chips.
It helps vectorizable workloads such as matrix math and other tensor-heavy inner loops.

Practical takeaway:
If a CPU supports AVX2, modern inference backends may benefit without any custom project code.

FMA

FMA allows fused multiply-add operations such as a * b + c with one fused operation and one rounding step.
It is useful for dense floating-point math and often appears alongside AVX-era optimization discussions.

Practical takeaway:
FMA support can help numerical kernels, but it is still only one part of the full backend and hardware path.

AVX2 VS AVX512

High-level contrast:
- AVX2 uses 256-bit vectors
- AVX-512 uses 512-bit vectors and adds wider masking and more specialized feature subsets

Important caution:
AVX-512 is hardware-dependent and often comes with workload-specific trade-offs.
It should not be assumed to help on hardware that does not actually support it.

Machine-fit rule:
If the CPU does not have AVX-512, AVX-512 ideas are concept material only.
They are not an actionable optimization path.

VNNI

VNNI is an AVX-512-era instruction family aimed at efficient low-precision dot-product style neural inference.

Concept takeaway:
- useful in some quantized inference paths on supported CPUs
- irrelevant on machines that do not have AVX-512 / VNNI hardware

AVX-512 FOR NON-NEURAL WORKLOADS

AVX-512 can also matter outside neural workloads, for example in some scientific, analytics, crypto, media, or compression kernels.

Concept takeaway:
- the presence of a powerful ISA does not make it relevant to every project
- if the project does not expose a path that uses it, it remains background material rather than a live action item

ARM SVE

SVE is Arm's scalable vector ISA.
Unlike fixed-width AVX-512, SVE is vector-length agnostic and can be implemented at different widths.

Concept takeaway:
- SVE is important background for cross-architecture inference
- it does not affect a Windows x86 LocalBot machine directly

RISC-V RVV

RVV is the RISC-V vector extension.
Like SVE, it is designed around scalable vector length rather than one fixed width.

Concept takeaway:
- RVV is useful background for future cross-platform or research discussions
- it is not an active optimization path for a Windows x86 Ollama setup

PRUNING

Pruning removes low-importance parameters or structures from a trained model to reduce size or inference cost.

Main forms:
- unstructured pruning: remove individual weights
- structured pruning: remove larger units such as neurons, heads, or channels

UNSTRUCTURED VS STRUCTURED PRUNING

Unstructured pruning:
- creates sparse weight matrices
- often needs sparse-kernel runtime support for speed gains

Structured pruning:
- removes full structures
- easier to deploy as a dense model
- can cause larger quality loss per removal step

LOTTERY TICKET HYPOTHESIS

The Lottery Ticket Hypothesis proposes that large dense networks contain smaller subnetworks that can match the larger model's performance when trained from the right initialization.

ITERATIVE MAGNITUDE PRUNING

Iterative magnitude pruning is a common pruning method:
1. start with a trained model
2. remove the smallest-magnitude weights
3. fine-tune
4. repeat until the desired sparsity is reached

The goal is to increase sparsity gradually while preserving as much performance as possible.

AUTOMATION FIT

Small helper automation can support LocalBot by reducing manual repetition during repeated coordinator runs.
For this project shape, simple automation is preferable to larger orchestration.

Useful goals:
- reduce typing mistakes
- keep repeated runs consistent
- keep console output visible
- avoid changing the core LocalBot loop

BATCH FILE AUTOMATION

A minimal batch launcher is a natural fit on a Windows machine when the goal is only to start the coordinator cleanly.

Typical responsibilities:
- change into the project folder
- launch the coordinator
- keep the window open long enough to read the result

Why this can help:
- native Windows workflow
- no extra dependencies
- lower friction for repeated manual runs
- does not require changing coordinator.py or worker.py

Concept rule:
A helper batch file is support tooling only.
It should not replace or complicate the core loop unless a real need appears.

PYTHON AUTOMATION ALTERNATIVE

A tiny Python launcher can serve the same purpose as a batch file while staying closer to the rest of the project stack.

Possible advantages:
- easier future extension
- consistent subprocess handling style
- better fit if run wrappers later need small logic

Possible disadvantages:
- more overhead than a batch file for very simple launching
- less native to the default Windows console workflow
- unnecessary if the batch file already solves the real problem

Preference rule:
Choose the smallest launcher that solves the repeated-run problem cleanly.

BATCH VS PYTHON LAUNCHER

Batch launcher:
- smallest Windows-native option
- almost zero maintenance
- best for simple repeated manual runs

Python launcher:
- easier to extend later
- more flexible if wrapper logic grows
- still lightweight, but less minimal than batch

Project-fit takeaway:
For a tiny local Windows task loop, batch is often the first practical helper.
A Python wrapper becomes more attractive only if repeated-run tooling needs logic beyond simple launching.

COORDINATOR LOGIC OVERVIEW

At a high level, the coordinator is responsible for one small task loop cycle.

Typical flow:
- read config values
- inspect pending tasks
- claim one task
- run the worker subprocess with a timeout
- read the result state
- move the task to done or failed
- append log output
- exit cleanly

This single-task claim-and-run structure is well suited to a small local project because it is easy to inspect, easy to recover, and easier to harden than a larger always-running loop.

BATCH SCRIPTING COMPLEXITY

- harder debugging
- more Windows-specific fragility

Complexity rule:

PROMPT ENGINEERING FOR STRICT REWRITE TASKS

When a small local model is already staying near the source text but keeps slipping on formatting or normalization rules, prompt design may matter more than more cleanup retries.

Useful prompt-design ideas:
- keep the task tied to the given source text
- state the output shape clearly
- separate meaning-preservation rules from formatting rules
- reduce ambiguity about order, line count, and allowed interpretation
- avoid adding so many simultaneous rules that the model drops one halfway through

Prompt-design rule:
Prefer the smallest prompt change that addresses the observed failure pattern.

ZERO-SHOT PROMPTING

Zero-shot prompting means giving the model the task and rules directly without examples.

Strengths:
- shortest prompt
- lowest token cost
- easiest to reuse across many small tests

Weaknesses:
- weaker pattern lock on small models
- more likely to miss one of several formatting rules
- can be less reliable when exact shape matters

Use case:
A good baseline when testing whether a task family is naturally stable without added scaffolding.

CHAIN-OF-THOUGHT STYLE INSTRUCTIONS

A step-by-step instruction style can sometimes improve rule compliance by making the transformation sequence more explicit.

Typical pattern:
- read the source lines
- apply one normalization step
- apply another normalization step
- preserve order
- output only the final result

Possible benefits:
- stronger rule salience
- better compliance on multi-step cleanup tasks

Possible downside:
- the model may still miss a final output rule even after following the internal sequence
- extra instruction text can add complexity without guaranteeing better results

Use rule:
Use step-by-step instructions only when they stay short and directly tied to the observed problem.

RULE PRIORITIZATION

When several rules matter at once, grouping or prioritizing them can help.

Examples of useful priority groupings:
- output shape first
- order and line count second
- meaning preservation third
- punctuation and style normalization last

Why this can help:
- makes the task easier to parse
- reduces hidden conflicts between cleanup and preservation rules

Caution:
Over-prioritizing one rule can cause the model to weaken another, so the grouping must stay balanced.

FEW-SHOT PROMPTING

Few-shot prompting gives the model one or more input/output examples before the real input.

Strengths:
- stronger style lock
- stronger formatting mimicry
- often better than pure instructions when exact output shape matters

Weaknesses:
- more token cost
- examples can accidentally overconstrain wording
- poor examples can teach the wrong pattern

Use rule:
Use few-shot prompting when the desired shape is simple, repeatable, and easy to demonstrate with short examples.

ADVANCED FEW-SHOT PROMPTING

Advanced few-shot prompting uses more than one example or varies the examples to better anchor style and structure.

Potential advantages:
- stronger pattern anchoring
- better reliability for style normalization tasks
- less chance that the model treats one example as a one-off case

Potential risks:
- prompt grows faster
- example quality matters even more
- more room for subtle mismatch between examples and the real task

Project-fit rule:
Use advanced few-shot only when simple prompts or single-example prompts are not stable enough.

ROLE PROMPTING

Role prompting assigns the model a narrow job identity such as editor, classifier, or rewriter.

Possible benefits:
- reduces creative drift
- makes the task frame feel narrower
- can improve style consistency on constrained outputs

Possible downside:
- the role can become decorative text if it is not clearly tied to the actual task
- a role prompt does not replace precise shape rules

Use rule:
A role prompt is best used as a small stabilizer, not as the main control mechanism.

SELF-CONSISTENCY STYLE PROMPTING

Self-consistency style prompting asks the model to internally consider multiple candidates and output the best one.

Concept benefit:
- can reduce output variance
- can improve consistency on rule-heavy tasks
- may help when one-pass outputs are unstable

Concept cost:
- more prompt complexity
- more token use
- no guarantee that the final chosen answer will obey every visible rule

Use rule:
Treat self-consistency as a higher-cost prompt technique, not the default for tiny practical tasks.

ROLE PLUS FEW-SHOT COMBINATION

Combining a narrow role prompt with short examples can provide stronger shape control than role or examples alone.

Why this can work:
- the role narrows behavior
- the examples anchor the exact output style
- the task remains tied to a visible pattern

Risk:
- the prompt can become too large or too busy for a small model if too many examples or rules are added

Combination rule:
Use combinations only when the simpler prompt forms are still unstable and the task value justifies the extra prompt weight.

REWRITE VS CLEANUP PROMPT SHIFTS

When strict cleanup prompts become brittle, it may be better to shift back to rewrite wording instead of piling on more micro-rules.

Why rewrite can help:
- feels more natural to the model
- gives room to normalize text while still staying close to the source
- can reduce brittle rule collisions seen in cleanup-only phrasing

Why cleanup can still matter:
- useful when the task is genuinely about normalization
- good when the source and target differ only in formatting or small wording cleanup

Shift rule:
If cleanup prompts keep dropping lines or colliding with preservation rules, test a rewrite framing instead of endlessly tightening the same cleanup wording.

SHAPE RULE REUSE

Reusable shape rules help keep later tests consistent.

Examples of reusable shape constraints:
- exact line count
- same order as source
- plain text only
- no numbering or wrapper text
- each line starts directly with a word
- end each line with a period when sentence output is required

Use rule:
Keep reusable shape rules centralized and reapply them consistently rather than rewriting them from scratch each time.

TERM AND VERB LOCKING

For strict source-tied rewrite tasks, it can help to lock a small set of source terms and action verbs when those elements must survive unchanged.

Typical use:
- preserve required task nouns exactly
- preserve the original action verb for each source line
- allow cleanup of casing or punctuation around those locked terms

Why this can help:
- reduces synonym drift
- protects task identity words that matter operationally
- keeps rewrites close to the source even when other cleanup rules are active

Risk:
- too many locked elements can make the prompt brittle
- heavy locking can collide with cleanup or natural sentence flow if overused

Use rule:
Lock only the terms and verbs that are actually load-bearing for the task.

PUNCTUATION NORMALIZATION LOCKS

When the model is close on meaning and line structure but keeps leaking stray punctuation, a small punctuation rule block can be useful.

Typical targets:
- remove one known stray punctuation mark
- require a single final period when sentence-style output is needed
- forbid all other punctuation except explicitly allowed characters

Why this can help:
- turns vague cleanup requests into measurable output constraints
- reduces repeated punctuation regressions on short rewrite tasks
- can stabilize outputs without changing the broader task family

Risk:
- broad punctuation bans can collide with legitimate source tokens such as file names
- over-tightening punctuation rules can improve one variable while breaking shape or meaning elsewhere

Use rule:
Keep punctuation locks narrow, explicit, and compatible with required source terms.

HARD-LOCK REWRITE PROMPTS

A hard-lock rewrite prompt is a stricter rewrite form that explicitly fixes line count, order, required terms, required verbs, allowed punctuation, and output-only behavior.

Potential benefit:
- stronger control when lighter rewrite prompts are close but still unstable
- clearer separation between source preservation rules and cleanup rules
- useful as a deliberate comparison point against simpler prompt forms

Potential risk:
- prompt weight grows quickly
- hidden conflicts become more likely
- one problem may improve while another regresses

Project-fit rule:
Treat hard-lock rewrite prompts as a comparison tool, not the default baseline.
Use them when a simple rewrite prompt is nearly correct and the remaining failures are specific enough to target cleanly.


PROMPT FAILURE INTERPRETATION

When a prompt test fails, classify the miss by what actually broke.

Common categories:
- shape miss: formatting or line structure broke
- preservation miss: meaning changed or lines were dropped
- normalization miss: the model kept source noise that should have been cleaned
- drift: the model wandered beyond the source task
- system failure: the task loop itself broke

Interpretation rule:
Use the smallest accurate failure label first.
Do not escalate a prompt-design problem into a code or model problem unless the evidence clearly shows that.

PROMPT COMPARISON RULE

When testing multiple prompt styles for the same small task family:
- keep the source block the same
- change one prompt dimension at a time when practical
- compare the result by shape, meaning preservation, and consistency
- avoid mixing too many experimental variables at once

This makes it easier to tell whether zero-shot, few-shot, role prompting, or step-by-step instructions actually changed the outcome.

LOCALBOT-FIT PROMPT TAKEAWAY

For a tiny local Windows task loop using a small model:
- prompt design can matter a lot
- simpler prompt styles should be tried first
- examples can help when strict output shape matters
- role prompting can help reduce drift
- step-by-step instructions can help, but should stay short
- more prompt complexity is not automatically better


GENERATION CONTROLS FOR STRICT OUTPUT TASKS

When a prompt family is already close to correct but still leaks formatting variation, generation settings can matter alongside prompt design.
These are model-sampling controls, not prompt techniques.

Concept rule:
Treat sampling controls as optional tuning knobs.
Do not change them casually in live project control without a real test reason and a clean backward-compatible path.

TEMPERATURE

Temperature controls randomness in token selection.

High-level effect:
- lower temperature: more deterministic, less variation
- higher temperature: more variation, more creativity, more formatting drift risk

Practical takeaway for strict shape tasks:
- lower values tend to help with exact formatting, sentence case, punctuation consistency, and reduced numbering leaks
- very high values increase variance and make strict cleanup or rewrite tasks less stable

Safe concept read:
Temperature can improve consistency, but it is not a substitute for good prompt design.

TOP-P SAMPLING

Top-p, or nucleus sampling, limits next-token sampling to the smallest set of likely tokens whose combined probability exceeds a chosen threshold.

Why it can matter:
- adapts to model confidence at each token
- can preserve coherence while still reducing wide random sampling
- often works well with low temperature on constrained rewrite tasks

Practical takeaway:
For strict output tasks, top-p can be a useful companion to lower temperature when deterministic but not totally rigid behavior is desired.

TOP-K SAMPLING

Top-k sampling limits next-token choice to the k most likely tokens.

Concept trade-off:
- simpler fixed-size filtering
- easier to reason about in narrow settings
- less adaptive than top-p when probability distributions change from token to token

Practical takeaway:
Top-k is an optional sampling control, but top-p often maps more naturally to constrained text-generation tasks.

GREEDY DECODING

Greedy decoding always selects the highest-probability next token.

Potential benefit:
- strongest determinism
- useful for maximum repeatability when exact formatting matters

Potential downside:
- can become too rigid
- may reduce naturalness or amplify repetitive phrasing

Use rule:
Greedy-style behavior is useful as a comparison baseline for strict formatting tasks, but it is not automatically best for every rewrite lane.

SAMPLING CONTROL COMBINATIONS

Useful combination patterns:
- low temperature + moderate top-p for constrained consistency
- greedy or near-greedy settings for strict repeatability checks
- higher randomness only when the task genuinely benefits from freer generation

Interpretation rule:
When comparing sampling settings, change as few variables as possible so the outcome can still be judged cleanly.

SAMPLING VS PROMPT DESIGN

Prompt design and sampling controls solve different problems.

Prompt design affects:
- what the model is being asked to do
- how clearly order, shape, and meaning are specified
- whether the task feels natural or brittle

Sampling controls affect:
- how much variation the model allows while producing tokens
- how deterministic the output is
- how likely formatting noise or drift is under the same prompt

Project-fit rule:
Improve the prompt first when the task is unclear.
Consider sampling controls only when the task is already clear and the remaining problem is output variance.

WORKER SUPPORT CONSIDERATIONS

Sampling controls only matter if the worker path exposes them.

Concept implications:
- the worker may need to pass generation parameters to the model call
- config-driven support is cleaner than hardcoded values
- backward compatibility matters if support is added later

Safe rule:
Treat sampling support as an implementation option, not as assumed current behavior.

SAMPLING TEST DESIGN

A clean sampling comparison should:
- keep the same source text
- keep the same prompt text
- change only the sampling setting being tested
- compare shape, meaning preservation, and consistency across runs

Why this matters:
- avoids confusing prompt changes with generation-setting changes
- makes the result easier to interpret in handoffs

LOCALBOT-FIT SAMPLING TAKEAWAY

For a tiny local rewrite or cleanup workflow:
- low-randomness settings are often the most relevant conceptually
- temperature and top-p can matter for strict output consistency
- sampling changes are secondary to a strong prompt and a stable loop
- any real sampling change should be introduced minimally and tested cleanly


REPETITION PENALTY

Repetition penalty is a generation control that discourages the model from reusing the same tokens or phrasing too easily.

High-level effect:
- values above the neutral setting reduce repeated wording and stuck patterns
- overly strong values can distort natural phrasing or push the model away from the source wording

Why it can matter:
- can help reduce repeated formatting artifacts
- may help when a small model keeps echoing punctuation or casing patterns
- is usually a secondary control, not the first fix

Safe concept read:
Use repetition penalty as a light tuning option only after the prompt is already clear and stable enough to compare fairly.

FREQUENCY PENALTY

Frequency penalty is a generation control that reduces the likelihood of tokens being reused based on how often they have already appeared in the generated output.

High-level effect:
- default 0.0 = no effect
- small positive values can reduce repeated words, repeated punctuation habits, and repeated formatting artifacts
- stronger values can push the output toward unnecessary variation

Why it can matter:
- can help reduce repeated formatting patterns in short strict-output tasks
- may help when a model keeps reusing noisy casing, punctuation, or repeated wording
- is best treated as a light stability control rather than a primary fix

Safe concept read:
For small rewrite or cleanup tasks, low values are the most relevant comparison range.
Frequency penalty should be evaluated only after the prompt itself is already clear enough to judge fairly.

PRESENCE PENALTY

Presence penalty is a generation control that reduces the likelihood of tokens being reused once they have appeared at least once in the generated output.

High-level effect:
- default 0.0 = no effect
- small positive values discourage reuse of already-seen tokens
- stronger values can force excessive novelty and weaken parallel structure or source fidelity

Why it can matter:
- can help reduce repeated wording or repeated formatting habits
- may help when a small model keeps echoing the same token patterns across short outputs
- can become counterproductive when the task needs repeated source terms or parallel line structure

Frequency penalty versus presence penalty:
- frequency penalty scales with how often a token has already appeared
- presence penalty only checks whether the token has appeared at least once

Safe concept read:
Presence penalty is a secondary tuning option for strict-output work.
Use small values only, and compare it carefully against source-term preservation and output consistency.


PROMPT OVER-TIGHTENING

When a small model is already near the source but not fully stable, adding too many constraints at once can make results worse instead of better.

Typical over-tightening signs:
- meaning drift increases after adding more preservation rules
- line shape collapses after adding more cleanup rules
- punctuation still leaks even though the prompt becomes longer
- one problem improves while another problem gets noticeably worse

Why this happens:
- too many simultaneous constraints create hidden conflicts
- the model may satisfy a new rule by weakening an older one
- long rule stacks can reduce reliability instead of improving it

Use rule:
When a revision gets worse, step back and identify which variable actually improved and which one regressed before writing the next version.

PROMPT REGRESSION INTERPRETATION

Not every new revision is progress.

Useful comparison points:
- shape preservation
- line count and order
- meaning preservation
- wording stability
- punctuation cleanup
- extra formatting noise

Interpretation rule:
Judge a revision against the strongest earlier version, not just against the immediately previous failure.
A newer prompt can still be worse overall even if it fixes one narrow issue.

MINIMAL REFINEMENT RULE

When a prompt family is still viable but unstable, prefer minimal refinement over adding a whole new layer of control.

Good minimal refinements:
- clarify one ambiguous rule
- lock one drifting term
- reinforce one shape constraint
- remove one conflicting instruction

Bad refinements:
- adding many new constraints at once
- mixing multiple new techniques in one step
- changing prompt style, examples, and rules all at once

Project-fit takeaway:
For small LocalBot rewrite tests, narrow revisions are easier to judge and less likely to create hidden prompt conflicts.

BOTTOM LINE

Keep concepts rich.
Keep live control elsewhere.
Protect the working loop.
Only turn a concept into an action when the machine or a real test proves it matters.


TIMEOUT TEST INTERPRETATION

When evaluating a timeout recheck, separate core behavior from observability.

Core timeout behavior is confirmed when:
- the task is claimed
- the task fails cleanly after the timeout window
- the coordinator exits without hanging
- no stale running state remains
- config is restored correctly if it was changed for the test

Observability may still be incomplete when:
- the log only shows a generic failed state
- no explicit timeout message appears
- there is no before / after process baseline to judge whether a resident process is normal or orphaned

Interpretation rule:
A timeout path can be operationally good enough even when logging and process-baseline proof are weaker than ideal.

PROCESS CLEANUP STRATEGIES

A small local coordinator can often rely on the default timeout cleanup path before adding deeper process-control logic.

Simple current-path concept:
- the coordinator times out the worker subprocess
- the task is marked failed
- the coordinator exits cleanly
- running-state cleanup is verified separately

When deeper cleanup might be justified:
- repeated evidence of orphan child processes
- blocked cleanup after timeout
- persistent stale running state
- real machine instability after timeout events

Complexity-adding alternatives:
- process-tree cleanup helpers such as psutil
- Windows process-group or job-object handling
- custom polling loops after timeout
- more elaborate async wrappers

Cleanup rule:
Do not add stronger cleanup machinery until real runs show the simple path is insufficient.

TIMEOUT LOGGING OBSERVABILITY

Timeout handling and timeout observability are related but not identical.

A system may handle timeouts correctly while still logging them too generically.

Weak observability pattern:
- the log shows only a generic failed line
- there is no explicit timeout marker
- duration or exception type is not visible

Possible small improvement path:
- add a timeout-specific log line
- include the task name
- include the timeout duration
- keep the existing failure-path behavior unchanged

Observability rule:
Treat clearer timeout logging as a small quality improvement, not as proof that timeout protection is broken.

STRUCTURED LOGGING OPTIONS

For a tiny local Windows project, simple text logging may be enough until debugging needs prove otherwise.

Possible logging styles:
- plain text append logs
- stdlib logging module with levels and formatters
- structured key=value style
- JSON-style logs
- third-party logging helpers

Trade-off pattern:
- simpler logging: lower risk, easier to inspect manually
- more structured logging: better filtering and automation, more setup and maintenance

LocalBot-fit rule:
Do not add a logging framework just because it is available.
Prefer the smallest logging approach that still supports real debugging needs.

SHAPE TEST RULES

Use these rules for practical output-shape tests when strict formatting matters.

OUTPUT CONTROL
- Output exactly the content requested by the task and nothing else.
- No introductions, conclusions, explanations, or wrapper text.
- Strictly follow any specified line count.
- Plain text only.

FORMATTING RULES
- Use normal sentence case unless the task explicitly asks otherwise.
- End every sentence with exactly one period when sentence-style output is required.
- No numbering, bullets, dashes, symbols, list markers, or stray prefixes.
- No markdown, backticks, code blocks, or formatting markup.
- No extra blank lines unless the task explicitly requires them.

CONTENT RULES
- Preserve original meaning as closely as the task requires.
- Keep lines short, clear, and consistent in style and tone.
- Remove unnecessary punctuation and filler when cleanup is part of the task.
- Keep the output tied to the provided source text.

SHAPE FAILURE INTERPRETATION

Not every bad output is the same kind of failure.

Useful distinction:
- shape failure: content is mostly right, but format is wrong
- drift: output moves away from the source or task
- invention: output adds unsupported content
- system failure: loop, execution, timeout, or file handling breaks

Interpretation rule:
Do not treat a prompt-shape miss as a code or model failure unless evidence shows the problem is deeper than output form.

NOTES MAINTENANCE RULE

When new notes are added, clean them before folding them into concept notes.

Remove:
- live next-step commands
- active phase instructions
- stale model references
- exact run verdicts that belong in handoffs
- unsupported certainty claims
- control-layer wording that belongs in the master or unified handoff

Keep:
- reusable technical explanation
- evaluation criteria
- machine-fit reasoning
- trade-offs and limits
- small conceptual rules that help future judgment

ROBLOX WORK ORDER BUILDER CONCEPTS

These notes explain the Roblox Work Order Builder idea as reusable concept material.
They are not live run instructions.
Control truth belongs in the master handoff.
Current status belongs in the checkpoint.
Historical context belongs in the unified handoff.

CORE CONCEPT

LocalBot can become more useful by moving from advice-only text output to work-order output.

Advice-only shape:
- user describes a Roblox issue
- LocalBot gives advice
- user manually copies and decides everything

Work-order builder shape:
- user gives one structured Roblox work order
- LocalBot parses the work order
- LocalBot generates executable Roblox Studio builder output
- user runs or applies that output inside Roblox Studio
- result is verified before the next work order

This preserves the small LocalBot core while making the output more useful.

ONE TASK VS ONE WORK PACKET

"One task at a time" should not mean "one tiny object at a time."

For Roblox work:
- one LocalBot task can be one work packet
- one work packet can contain many related build steps
- the packet should stay focused on one system, feature, map section, or script set

Good packet:
- build one campsite test map
- create required folders
- create required parts/models
- create one support script
- verify required objects

Bad packet:
- build a whole full game from vague instructions
- change unrelated systems at the same time
- control Studio, publish the game, and debug every script in one pass

BUILDER-SCRIPT MVP

The safest first bridge between LocalBot and Roblox Studio is a generated Luau builder script.

Flow:
1. LocalBot reads a structured work order.
2. LocalBot generates one builder script.
3. LocalBot generates one local report.
4. User runs the builder script inside Roblox Studio.
5. The script creates generated objects and verifies required paths.
6. User reports the result back.

Why this is the best first bridge:
- does real work
- avoids direct Studio control too early
- fits the existing file-output LocalBot design
- keeps human review in the loop
- gives a clear proof before adding Rojo or plugins

MVP BOUNDARY

The first version should not claim to:
- directly control Roblox Studio
- automatically playtest the game
- write status files from Studio back to C:\LocalBot\outputs
- publish the place
- repair every runtime bug automatically
- build full games from vague prompts

The first version should only prove:
structured work order -> generated Roblox builder script -> user runs script -> generated content appears and verifies.

WORK ORDER FORMAT CONCEPT

A structured work order should contain predictable sections.

Useful sections:
- TASK_NAME
- PROJECT or PROJECT_NAME
- BOT_ROLE
- TASK_TYPE
- GOAL
- RULES
- REQUIRED_STRUCTURE
- COMMAND_LIST or BUILD_STEPS
- VERIFY
- SUCCESS_OUTPUT
- ERROR_OUTPUT

The exact names can be normalized later, but the first parser should stay strict enough to avoid ambiguity.

TASK TYPE NORMALIZATION

Uploaded work orders may use plain labels such as:
- Build-map-content-from-command-list

The internal lane can normalize them to:
- roblox_work_order_builder

A later version can support more lane labels:
- roblox_map_builder
- roblox_script_placer
- roblox_ui_builder
- roblox_remote_event_builder
- roblox_module_builder
- roblox_datastore_review
- roblox_bugfix_packet

BUILD MANIFEST CONCEPT

Before generating Luau, the work order should be mentally or programmatically normalized into a manifest.

The manifest is the clean middle layer between human instructions and builder code.

Conceptual fields:
- task_name
- generated_workspace_root
- generated_script_root
- folders
- models
- parts
- spawn locations
- lights
- scripts
- lighting settings
- verification paths
- warnings

The manifest does not have to be saved as JSON in the first version, but the code should behave as if it is building from one.

WHY A MANIFEST HELPS

A manifest helps because it separates:
- understanding the work order
- generating Roblox code
- checking safety
- reporting what was planned

This makes errors easier to diagnose.
If the parser is wrong, the manifest looks wrong.
If the generator is wrong, the manifest can still be correct.
If the Studio result is wrong, the verification list helps locate the miss.

ROBLOX BUILDER SCRIPT SHAPE

A generated builder script should prefer helper functions over repeated one-off code.

Useful helper functions:
- getOrCreateFolder(parent, name)
- clearGeneratedFolder(folder)
- getOrCreatePart(parent, name)
- getOrCreateModel(parent, name)
- applyPartProperties(part, spec)
- createOrReplaceScript(parent, name, source)
- verifyRequiredObjects()

The script should print clear status messages:
- LOCALBOT_BUILD_DONE: <TASK_NAME>
- LOCALBOT_BUILD_FAILED: <TASK_NAME>
- Missing: <PATH>

SAFETY NAMESPACE CONCEPT

Generated Roblox objects should stay inside approved generated namespaces.

Initial safe namespaces:
- Workspace/GeneratedMap
- ServerScriptService/GeneratedMapScripts

Reason:
These are clearly generated areas.
They reduce the risk of touching unrelated user work.

Safety rule:
A builder script may clear or replace generated content inside approved generated namespaces only.
It should not delete arbitrary user-created Workspace objects, scripts, UI, ReplicatedStorage assets, or services.

COLOR NORMALIZATION

Work orders often use human color names.
Builder output needs exact Roblox values.

Examples:
- dark green -> Color3.fromRGB(35, 100, 35)
- dark gray -> Color3.fromRGB(70, 70, 70)
- orange -> Color3.fromRGB(255, 140, 0)
- brown -> Color3.fromRGB(120, 72, 32)
- blue -> Color3.fromRGB(40, 90, 180)
- dark blue -> Color3.fromRGB(25, 45, 120)
- green -> Color3.fromRGB(40, 140, 40)
- gray -> Color3.fromRGB(120, 120, 120)
- light green -> Color3.fromRGB(120, 220, 120)
- soft gray -> Color3.fromRGB(120, 120, 120)
- dark blue-gray -> Color3.fromRGB(45, 55, 75)

This table should be treated as a starting convention, not universal truth.

SHAPE NORMALIZATION

Work orders may say:
- Cylinder if supported, otherwise Block
- Ball if supported, otherwise Block

Roblox Part shapes can be generated deterministically by the builder script when supported.
The generator should avoid vague fallback wording in final builder code.

SCRIPT PLACEMENT CONCEPT

Generated scripts should go into predictable generated script folders.

Initial script namespace:
- ServerScriptService/GeneratedMapScripts

This keeps generated scripts separate from user-written scripts.
Later lanes may add:
- ReplicatedStorage/GeneratedModules
- StarterPlayer/StarterPlayerScripts/GeneratedClientScripts
- StarterGui/GeneratedUI

However, these should only be added when the work order type requires them.

REPORT CONCEPT

LocalBot should write a local report beside the builder script.

Useful report fields:
- task name
- lane detected
- generated builder script path
- number of planned folders
- number of planned models
- number of planned parts
- number of planned scripts
- approved generated namespaces used
- warnings
- manual next step

The report is not the Studio test result.
The MVP report only confirms what LocalBot generated.

STUDIO RESULT REPORTING

Roblox Studio cannot be assumed to write directly to C:\LocalBot\outputs during the MVP.

For the MVP:
- Studio reports through the Output window
- user reads back done/failed status
- LocalBot's local report remains generation-side only

Later bridge options:
- Rojo file sync
- local plugin bridge
- local HTTP bridge if safe and explicitly designed
- Studio plugin that can send status back to LocalBot

Do not treat bridge options as active until built and proved.

ROJO AS A LATER OPTION

Rojo is conceptually useful because it maps a local filesystem project into Roblox Studio.

Possible future value:
- LocalBot writes project files
- Rojo syncs them into Studio
- script files and model files stay versionable

Why not first:
- adds setup complexity
- requires project mapping choices
- still needs user/plugin workflow
- builder script is simpler for proving the first work-order idea

STUDIO PLUGIN BRIDGE AS A LATER OPTION

A Studio plugin bridge is the closest to the user's long-term vision.

Possible future value:
- plugin receives work orders or generated manifests
- plugin creates/updates objects directly in Studio
- plugin can verify results and report status back

Why not first:
- more moving parts
- harder debugging
- plugin permissions and UI design matter
- local communication must be designed safely

BEST DEVELOPMENT ORDER

Recommended order:
1. Generate one builder script from one strict work order.
2. Run that script manually in Studio.
3. Verify the map appears.
4. Improve script safety and reporting.
5. Add more work-order types only after the first one is reliable.
6. Add Rojo writer only if file-sync workflow becomes valuable.
7. Add Studio plugin bridge only after generated manifests/scripts are trustworthy.

DO MORE WITH LITTLE TAKEAWAY

The most efficient design is not a giant autonomous agent.

The high-leverage design is:
- small LocalBot core
- strict work-order input
- normalized build manifest
- generated builder script
- safe generated namespaces
- clear report
- human-reviewed Studio execution

This gives LocalBot real work ability while preserving the project's original strength: small, local, inspectable, and recoverable.
