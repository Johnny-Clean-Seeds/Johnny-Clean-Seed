# Problem-Solving Order Latch — Mule Work, Idea Routing, and File-Return Control

Status: WORKING SUPPORT RULE / REVIEWED DRAFT  
Date: 2026-05-23  
Scope: Clean Seed / Mr.Kleen / Desktop\\123 / mule handoff discipline  
Boundary: This is not doctrine by itself. It is a support rule candidate until saved through the proper house lane. It does not rewrite ACTIVE_GUIDES, CURRENT_TRUTH_INDEX, or doctrine. It does not authorize Git, delete, public change, broad folder restructuring, or authority promotion.

---

## 1. Why This Exists

The failure was not that the mule worked. The failure was that the problem-solving order was wrong before the mule was used.

The original task was small:

- clean the local mule packet;
- remove duplicated/stale `_0` packet material;
- add the Next Spin Sequencer update;
- clean porch/root loose scripts;
- repair stale teleporter target facts;
- clean the `$NewAddendum` packet hygiene issue;
- capture the Rule-Touch Spiral result files.

That work became messy because new ideas were treated as new active work before the active task was closed.

The system needs a latch that forces the assistant to answer these questions before work expands:

1. What is the active problem?
2. What exact output proves the task is done?
3. What closes the task chain?
4. Which new ideas are active, parked, TODO, rejected, or blocked?
5. Which actions execute now, and which must not?
6. When files exist, does the system return them instead of making new work?

The key correction:

**Problem-solving order comes before creativity.**

Creative tools, Rule-Touch Spiral, Proven-Better Adoption, and Next Spin Sequencer are useful only after the active problem, target output, and close trigger are named.

---

## 2. Core Rule

Every mule task, local cleanup, gate review, or problem-solving run must pass through this order before execution.

```text
ACTIVE PROBLEM
→ TARGET OUTPUT
→ CLOSE TRIGGER
→ RISK CLASS
→ FRESH-IDEAS PASS
→ IDEA CLASS
→ EXECUTION SCOPE
→ PROOF
→ FILE-RETURN MODE
→ FINAL JUDGE
→ CLOSE THIS TASK CHAIN
```

No step may skip ahead because the idea sounds useful.

A useful idea is not automatically active work.

## 3. Problem-Solving Order Latch

## 3A. Creative Room Fresh-Ideas Pass

Status: CONTROLLED FRESH-IDEAS LAYER  
Purpose: bring in fresh thinking without turning the task into a runaway loop.  
Boundary: this is not a spin loop, not a new work chain, not proof, and not authority. It is a short idea-finding pass that must feed back into the active problem-solving order.

The Creative Room may be entered only after these are already named:

1. active problem;
2. target output;
3. close trigger;
4. risk class.

Fresh ideas are allowed, but they must be held and routed. They do not get to drive the task by themselves.

Clean order:

```text
THINK
→ NAME ACTIVE PROBLEM
→ NAME TARGET OUTPUT
→ NAME CLOSE TRIGGER
→ FRESH-IDEAS PASS
→ CLASSIFY IDEAS
→ EXECUTE ONLY ACTIVE/SUPPORT ITEMS
→ PROOF
→ FILE-RETURN MODE
```

### What the Fresh-Ideas Pass Does

The pass asks for new angles around the current task without expanding the job.

It should look for:

- simpler route;
- cleaner order;
- missing blocker;
- better proof shape;
- better file-return shape;
- safer boundary;
- better home for the object;
- better wording;
- better gate neighbor;
- hidden stale assumption;
- old useful idea that now fits;
- repeated failure pattern;
- more efficient method.

### What the Fresh-Ideas Pass Must Not Do

It must not:

- start a new task;
- run another loop because it found an idea;
- promote a gate;
- rewrite doctrine;
- mutate ACTIVE_GUIDES;
- mutate CURRENT_TRUTH_INDEX;
- move into Git/public work;
- expand cleanup scope;
- delay file return after files already exist.

### Fresh-Idea Routing

Every idea found must be routed immediately.

```text
USE NOW
ADAPT NOW
PARK
TODO
BLOCKED
REJECT
```

#### USE NOW

The idea is required to hit the named target output.

#### ADAPT NOW

The idea improves the current task without changing the close trigger.

#### PARK

The idea is useful but belongs to a later boss, room, or review.

#### TODO

The idea deserves a future bounded task.

#### BLOCKED

The idea needs missing facts, unclear paths, or authority not available in this task.

#### REJECT

The idea is stale, duplicate, unsafe, distracting, or wrong-lane.

### Fresh-Ideas Safety Rule

The Creative Room can suggest; the latch decides.

Allowed wording:

```text
Find fresh ideas around the named task.
Use/adapt only what helps the current target output.
Park or TODO everything else.
Do not start a new task chain.
```

Blocked wording:

- “continue if safe”;
- “run another pass”;
- “keep working through the next bounded action”;
- “expand into all related cleanup”;
- “test every idea now.”

### When To Use It

Use Fresh-Ideas Pass when:

- the task is stuck;
- the task keeps repeating a failure;
- the route feels stale;
- the user asks for creative-room thinking;
- a better order may exist;
- a rule/gate needs broader but non-vague coverage.

Do not use it after file-return mode begins. Once the target files exist, return the files.

### Fresh-Ideas Template

```text
FRESH-IDEAS PASS

Active problem:
[one sentence]

Target output:
[exact target]

Close trigger:
[exact close]

Fresh ideas found:
1.
2.
3.

Routing:
- USE NOW:
- ADAPT NOW:
- PARK:
- TODO:
- BLOCKED:
- REJECT:

Decision:
[execute current task / adapt current task / park ideas and continue original task / blocked]
```

### Why This Is Safe

This gives the system fresh ideas without recreating the runaway failure.

The Creative Room brings options.  
The latch classifies them.  
The active task stays bounded.  
Proof verifies.  
File-return mode closes.

### Step 1 — Name the Active Problem

The active problem must be written as one clear sentence.

Good:

```text
The local mule packet has stale duplicated `_0` material and must be repaired before mule reads it again.
```

Bad:

```text
Make the house better.
```

The active problem must not be a vibe, theme, or broad desire. It must name the concrete weakness being repaired.

Required fields:

- active problem;
- parent boss;
- child task if any;
- blocked issue if any;
- what is not the problem.

Example:

```text
Active problem:
Desktop\\123 has loose root `.ps1` files that should return home to a receipt-backed cleanout lane.

Parent boss:
Porch/root custody and return-home discipline.

Not the problem:
Do not restructure folders, repair Git, or rewrite doctrine.
```

---

### Step 2 — Name the Target Output

The target output is the thing that must exist before the task can close.

Examples:

```text
Target output:
- repaired packet files;
- superseded stale `_0` files;
- updated read-first pointer;
- receipt with SHA256;
- validation showing active `_0` files = 0.
```

For file work, the target output must include:

- files expected;
- receipts expected;
- ledger if needed;
- hash/proof expected;
- validation expected.

If the target output is not named, the mule does not know when the task is complete.

---

### Step 3 — Name the Close Trigger

The close trigger prevents runaway work.

Examples:

```text
Close trigger:
When the seven scripts are moved, hashes match, receipt exists, and Desktop\\123 root `.ps1` count is 0, return the report.
```

```text
Close trigger:
When the Rule-Touch Spiral review files and receipt exist, return the file set. Do not run a second spiral pass.
```

A close trigger must say what ends the chain.

Blocked close trigger language:

- “continue if safe”;
- “keep going until nothing remains”;
- “run the next bounded action”;
- “do one more pass” unless that pass is part of the named target output.

---

### Step 4 — Classify Risk Before Approval

Approval is not based on anxiety. Approval is based on risk class.

#### EXECUTE

The mule may execute without asking when the action is:

- local-only;
- bounded;
- non-destructive;
- clear target path;
- no delete;
- no Git;
- no public change;
- no doctrine change;
- no ACTIVE_GUIDES change;
- no CURRENT_TRUTH_INDEX change;
- no broad folder restructuring;
- has backup/hash/receipt behavior.

Example:

```text
Move seven named loose root `.ps1` files into a dated porch cleanout lane, hash before move, write ledger and receipt.
```

This is EXECUTE, not ASK.

#### SKIP / PARK

The mule should skip or park an item when it is:

- useful but not needed for the target output;
- lower importance than the active boss;
- unclear but not blocking safe work;
- unproven;
- too broad for the current task;
- a future idea;
- a possible gate, but not needed to close the task.

Example:

```text
Second Rule-Touch Spiral pass after first result exists.
```

This is TODO/PARK unless explicitly part of the target output.

#### ASK / HOLD

The mule should not execute automatically when an action involves:

- delete/permanent removal;
- Git commit/push;
- public change;
- doctrine rewrite;
- ACTIVE_GUIDES rewrite;
- CURRENT_TRUTH_INDEX rewrite;
- authority promotion;
- broad folder restructuring;
- destructive overwrite;
- credentials, auth, tokens, secrets, cache/security risk;
- unclear live target path that would require guessing;
- scope expansion beyond the handoff.

Important correction:

If a risky item appears, the mule should not freeze the whole run unless the whole run depends on that item. It should skip/park the risky item, continue the safe parts, and report the blocker at the end.

### Step 5 — Classify New Ideas Before Acting

Every new idea gets a lane before it can affect the active task.

Allowed idea classes:

```text
ACTIVE
SUPPORT
PARK
TODO
BLOCKED
REJECT
```

#### ACTIVE

The idea is required to finish the current target output.

Example:

```text
Packet hygiene check is ACTIVE if the task is to repair a packet placeholder.
```

#### SUPPORT

The idea improves the active task without expanding the target output.

Example:

```text
Hash-before-move supports a porch cleanup.
```

#### PARK

The idea is useful but not needed now.

Example:

```text
Rule-Touch Spiral as a possible future gate after first scaffold proof.
```

#### TODO

The idea deserves a later bounded task.

Example:

```text
Use Rule-Touch Spiral one more time on a future next-spin review.
```

#### BLOCKED

The idea cannot be safely acted on because a required fact is missing.

Example:

```text
Repair live Mr.Kleen pointer when the live repo path is missing on mule machine.
```

#### REJECT

The idea is stale, duplicate, wrong-lane, or harmful.

Example:

```text
Treat stale `_0` packet files as current authority.
```

### Step 6 — Apply Boss-Importance Routing

Ideas are routed by boss importance.

Order:

```text
1. BLOCKER
2. Parent boss
3. Active-route child
4. Neighbor support
5. Parked source ore
6. TODO
7. Rejected stale
```

Rules:

- A blocker may interrupt if the active work cannot continue without it.
- A parent boss may redirect if the current task is working on the wrong problem.
- An active-route child may join if it is necessary to close the target output.
- Neighbor support may be used only if it does not expand the task.
- Parked source ore stays available but inactive.
- TODOs do not execute during the current chain.
- Rejected stale items are named with reason and left behind.

---

### Step 7 — Separate Problem-Solving From Creativity

Creative tools must not steal the wheel.

Rule-Touch Spiral, Creative Room, Wildcard, algorithms, bitstrings, and ratio-based rotation can generate ideas, but they do not decide execution.

They must feed into:

```text
Creative Synthesis
→ Proven-Better Adoption
→ Parking / TODO
→ Next Spin Sequencer
→ Hindsight
→ Omega / Final Judge
```

They do not bypass:

- active problem;
- target output;
- close trigger;
- risk class;
- proof;
- Final Judge.

Use creativity only when:

- the active task is idea generation;
- the current repair needs a creative route;
- the target output includes creative review;
- a blocker requires alternate-route thinking.

Do not use creativity to keep working after the files already exist.

---

### Step 8 — Apply Proven-Better Adoption Correctly

Proven-Better Adoption decides whether to use an idea. It does not automatically execute the idea.

Decision table:

| State | Decision |
|---|---|
| Unproven | Test, park, or TODO |
| Proven but not clean fit | Adapt, merge, or park |
| Proven + more efficient + clean fit + boundary-safe | Adopt/adapt through proof lane |
| Proven but conflicts with higher boss | Hold and escalate to parent boss |
| Old rule only wins by age | Not enough |
| New idea only wins by novelty | Not enough |

Adoption requires:

- proof;
- clean fit;
- boundary safety;
- better efficiency or better correctness;
- receipt/record when applied.

If an idea is proven useful but not part of the active target output, it becomes TODO/PARK, not immediate work.

---

### Step 9 — Execute Only the Active Task

Execution language must be narrow.

Good:

```text
Repair the current packet placeholder, write backup, hash before/after, receipt, validate `$NewAddendum` count 0, then return report.
```

Bad:

```text
Repair this and continue to the next safe bounded action.
```

Execution must name:

- allowed paths;
- allowed file types;
- allowed edits;
- blocked actions;
- receipt path type;
- validation checks;
- close trigger.

Execution must not include open-ended chains unless the target output itself is a chain.

---

### Step 10 — Validate Proof

Proof must match the task.

For local file work:

- file exists;
- path is correct;
- hash recorded;
- before/after hash recorded if edited;
- backup exists if edited;
- ledger exists if moved;
- receipt exists;
- validation counts pass;
- blocked items are named;
- boundaries held.

For mule reports:

- files read;
- files changed;
- files created;
- receipts;
- hashes;
- validation;
- blocked moves;
- final verdict.

Verdict language:

```text
PASS
PASS WITH GUARDRAILS
PARTIAL
FAIL
BLOCKED
INVALID
NEEDS PROOF
```

Do not call DONE a PASS.

---

### Step 11 — Enter File-Return Mode When Target Files Exist

This was the largest missed switch.

Once the mule creates the target file set and receipt, the system enters file-return mode.

Allowed in file-return mode:

- list the completed files;
- list receipt path;
- list SHA256;
- ask user to upload/send files if needed;
- summarize what the files mean;
- close the chain.

Blocked in file-return mode:

- new mule handoff;
- new loop;
- new creative pass;
- “one more test”;
- “continue if safe”;
- extra cleanup;
- second spiral pass;
- next bounded action;
- expanding the goal.

File-return mode ends only when the user gives a new command.

---

### Step 12 — Close This Task Chain

Closing the task chain does not mean the project stops.

It means:

```text
This exact work order is closed.
Extra ideas are parked/TODO.
The user gets the files/status.
A new chain requires a new explicit command.
```

This prevents useful ideas from hijacking file retrieval.

Use this template for future mule tasks.

```text
MULE WORK ORDER — BOUNDED TASK

Active problem:
[one sentence]

Parent boss:
[name]

Target output:
[list exact files/receipts/results expected]

Close trigger:
[when these outputs + validation exist, return file list/report]

Allowed actions:
- [local-only action]
- [bounded edit/move/read]
- [backup/hash/receipt]

Blocked actions:
- no Git
- no delete
- no doctrine
- no ACTIVE_GUIDES
- no CURRENT_TRUTH_INDEX
- no public change
- no broad folder restructuring
- no destructive overwrite
- no new task chain

New idea rule:
Extra ideas must be classified ACTIVE / SUPPORT / PARK / TODO / BLOCKED / REJECT.
Only ACTIVE and SUPPORT may affect this run.
PARK and TODO must not execute now.

Risk rule:
Execute bounded local non-destructive actions.
Skip/park risky or unclear items and continue safe work.
Report blockers at the end.

Proof:
Write receipt, hashes, backups/ledger if needed, validation checks, and final verdict.

File-return mode:
When target output exists, close this work chain and return:
1. files created/changed;
2. receipt path;
3. receipt SHA256;
4. validation;
5. parked TODOs;
6. blocked items.
```

## 6. What Went Wrong in the Run

### Failure 1 — Fake Approval Checkpoint

The seven `.ps1` files were safe to move because they were:

- local-only;
- loose root files;
- named;
- hashable;
- movable;
- no folders;
- no delete;
- no Git;
- no authority change.

The correct class was EXECUTE.

The wrong behavior was asking for approval.

Root cause:

The assistant treated “file move” as automatically risky instead of classifying the actual move.

Fix:

Risk class before approval.

---

### Failure 2 — Overcorrection Into Open-Ended Work

The user corrected the fake approval checkpoint and said the mule should work without babysitting.

Correct interpretation:

```text
Finish the assigned bounded task without pointless approval.
```

Wrong interpretation:

```text
Continue into the next safe bounded task if one exists.
```

Fix:

Execution continuity must always be tied to the target output and close trigger.

---

### Failure 3 — Missing File-Return Mode

After the Rule-Touch Spiral pass, the mule had already produced:

- review file;
- parking/TODO output file;
- TODO file;
- receipt;
- SHA256.

That was the close trigger.

The correct action was:

```text
Close this task chain and get the files.
```

The wrong action was:

```text
Run another spiral pass.
```

Fix:

When target files exist, file-return mode activates automatically.

---

### Failure 4 — Good Ideas Became Active Work Too Early

Proven-Better Adoption, Boss-Importance Routing, Multi-Idea Parking, and Rule-Touch Spiral were useful.

But useful does not mean active.

The correct route:

- use only what supports the current target;
- park the rest;
- write TODO with return trigger;
- close the active task.

Fix:

Idea Interrupt Filter.

## 8. Edge Cases

### Edge Case 1 — Safe Work Plus One Risky Item

Do not halt all work.

Action:

- skip/park risky item;
- continue safe items;
- report risky item at end.

### Edge Case 2 — Missing Live Target Path

Do not guess.

Action:

- mark FALSE or UNKNOWN;
- do not repoint to parked/source-only copy;
- continue unrelated safe work.

### Edge Case 3 — New Brilliant Idea Appears Mid-Run

Do not execute unless it is required for the target output.

Action:

- classify it;
- park/TODO unless ACTIVE or SUPPORT;
- continue active work.

### Edge Case 4 — Target Files Exist But New TODO Exists

Close task chain.

Action:

- return target files;
- list TODO;
- wait for user command.

### Edge Case 5 — User Says “Keep Working”

Interpret as:

```text
Finish the current bounded task without pointless approval.
```

Do not interpret as:

```text
Invent and execute the next task chain.
```

### Edge Case 6 — User Says “Send File” or “He Stopped”

Switch to damage control/file-return mode.

Do not send another handoff.

Action:

- identify latest completed file set;
- get/read/send files;
- summarize/close.

## 9. Final Operating Logic

```text
If problem is not named, name it.
If target output is not named, name it.
If close trigger is not named, name it.
If new idea appears, classify it before action.
If action is safe/local/bounded, execute it with receipt.
If action is risky/unclear, skip/park it and continue safe work.
If target files exist, enter file-return mode.
If in file-return mode, do not create new work.
If proof passes, report PASS/PASS WITH GUARDRAILS.
If proof is missing, report PARTIAL/BLOCKED/NEEDS PROOF.
```

Verdict: **PASS WITH GUARDRAILS / BROAD SUPPORT RULE CANDIDATE**

- It directly addresses the overcorrection into open-ended work.
- It directly addresses the missed file-return mode.
- It separates new ideas from active work.
- It adds a bounded Creative Room Fresh-Ideas Pass without reopening the task chain.
- It preserves bounded local execution.
- It protects Git/public/doctrine/authority boundaries.
- It includes risk classification, proof, receipts, parking/TODO, and close behavior.
- It gives exact operational templates.

Guardrails:

- It should not become another giant checklist used on every tiny action.
