START HERE — CLEAN SEED MODEL BUILD

What This Is

This is a startup guide for building a clean seed.

A clean seed is a small controlled first version of a bot, helper, workflow, or project
assistant.
It is not the final machine.
It is the first living piece that proves the build can work without drifting, bloating, hiding
errors, or letting the assistant become the boss.

This README does one job:
It gets you from a loose idea to the first clean starting point.

What This Does

This README helps you:

use the three guide files
put the guide files somewhere easy to find
give the guide files to the assistant
define the build direction or current build state before project-specific names lock in
let the assistant recommend clean names and file placement
create or confirm one clean project home
create one first seed card
fill the seed card one field at a time
prove the route before calling anything PASS

How The Model Works

A clean seed starts as one small named build with:

one home
one route
one proof path
one failure path
one recovery path
one authority boundary
one parking lane
one promotion rule
one close condition

The seed grows only after the current stage is proven, wrapped, and allowed by the promotion
rule.

First, the direction is named.
Then the home and first files are placed.
Then one tiny route is proven.
Then failure and recovery are made visible.
Then useful future ideas can be reopened only when the current stage can safely hold them.

Done is not PASS.
PASS needs proof.
Growth waits for wrap.

To Get Started

Follow the steps below in order.

First, make sure the three guide files have the clean filenames.
Second, put the guide files in the holding folder.
Third, give all three guide files to the assistant.
Fourth, paste the startup request that matches your situation.
Fifth, let the assistant ask for or confirm the starting situation before project-specific files
are created.

Once the assistant has the three guide files, it should guide the next move.
Your job is to choose the direction, approve names, run or paste only the exact steps given, and
send back the result that proves what happened.

Do Not Start By

Do not start with a giant bot.
Do not start with many agents.
Do not lock in project names before the build direction is known.
Do not scale before the seed is named, proven, and wrapped.
Do not treat motion as proof.
Do not call anything clean if the route, evidence, failure, recovery, or next state is unclear.

Do not make the user choose how to build the core.

Plain Meaning Of The Seed Card Fields

Build direction:
The real build idea or current build state.
This is not a giant architecture plan.
It is the plain direction needed before names lock in.

Name:
What this seed is called right now.
Use a small honest name.
Do not name it after a future giant system.

Mission:
What this seed is being built to do right now.
Keep future features out of the mission.

Home:
Where the seed lives on the computer.
Use one clean project folder.

Input / Output Route:
What goes in, what happens, what comes out, and where the result lands.

Done / Pass Rule / Pass Judge:
Done means the route finished.
PASS means the result met the required check.
The pass judge says who or what decides PASS.

Failure Path:
What happens when something fails.
Failure must be visible.
Failure evidence should not be erased before repair.

Recovery Path:
What happens if work is interrupted, stuck, stale, or incomplete.
The seed should not stay trapped forever.

Authority Boundary:
What the assistant can touch.
What the assistant must not touch.
What needs user approval.
The user owns direction.

Parking Lane:
Where useful future ideas go when they are not active yet.
Future ideas are not trash, but they are not allowed to control the current seed.

Promotion Rule:
What must be proven before the seed can grow, add a new lane, add a helper, connect a bridge, or
move to the next stage.

Close Condition:
What proves the current setup stage is finished.
A stage is not finished just because files exist.
It closes when the required parts are named, checked, and accepted.

Minimum File Roles Later

These files may appear later, but only when the stage needs them.

SEED_BUILD_CARD.txt
Holds the seed direction, name, mission, home, route, done/pass rule, pass judge, failure path,
recovery path, authority boundary, parking lane, promotion rule, and close condition.

SEED_EVIDENCE_LOG.txt
Records real proof checks, outputs, failures, repairs, and verdicts.

PARKED_IDEAS.txt
Holds useful future ideas that are not active yet.

SEED_CLOSEOUT.txt
Records what passed, what failed, what was repaired, what remains parked, what the seed can be
trusted to do, and what it should not be trusted to do yet.

Do not create extra files just because they sound useful.
Create the next file only when the active stage needs it.


If Something Goes Wrong During Startup

If the assistant starts naming files before asking what the build is:
Tell it to run Build Direction Intake first.

If the assistant gives too many options:
Tell it to give one recommended clean option unless a real choice is needed.

If a folder name is wrong:
Stop and fix the folder name before continuing.

If a file has a numbered suffix:
Rename the kept active copy to the clean filename.

If two copies seem different:
Do not guess.
Open both.
Keep the correct newest copy.
Rename only the kept copy to the clean filename.

If the assistant tries to skip setup:
Tell it to return to the setup stage and fill the seed card one field at a time.

If the assistant tries to build big too early:
Tell it to park the idea and finish the current clean seed stage first.
