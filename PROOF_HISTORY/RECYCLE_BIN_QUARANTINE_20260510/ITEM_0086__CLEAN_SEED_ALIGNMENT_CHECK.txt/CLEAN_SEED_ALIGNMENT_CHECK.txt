Hard Verdict Standard

Use PASS only when every required check passed together.

Use PASS WITH GUARDRAILS when the build is clean but needs an active watch item.

Use PARTIAL when some required checks pass and some are missing or weak.

Use FAIL when usable evidence proves a required check failed.

Use INVALID when the evidence cannot prove what happened.

Use PARKED when the idea is useful but not active.

Use YIELD when user judgment, approval, lookup, inspection, or verification is needed.

Use BLOCKED when progress is stopped by a real blocker.

Use PROMOTED when a proven, bounded item has been deliberately moved into an allowed larger
role.

Use RETIRED when an old file, rule, route, or copy is no longer active and must not be obeyed as
current truth.

Done is not PASS.
Movement is not proof.
Confidence is not proof.
Screenshots are not runtime proof unless the specific check is visual.


Clean Seed Target

A clean seed has:

one clear name
one home
one active job
one visible input/output route
one evidence path
one pass judge
one failure path
one recovery path
one authority boundary that names what can be touched and what cannot be touched
one Does-Not-Do / Out-of-Bounds list
one parking lane
one promotion rule
one close condition

If any of these are missing, the seed setup is not clean yet.



Rule / Check Closure Standard

Use this standard when judging or writing any behavior-controlling rule or check.

A behavior-controlling rule or check is one that can change assistant action, file authority,
evidence handling, verdicts, package state, user approval, patch behavior, promotion, bridge
behavior, or final trust.

A closed rule or check should name enough of these parts for a weak assistant to apply it
without guessing:

trigger / when to use it
what to inspect
minimum required evidence or fields
dirty milk conditions
clean milk conditions
verdict labels
next clean state

Not every explanatory README paragraph needs this closure. The README is an on-ramp.
The Alignment Check carries most verification closure.

Do not paste a giant duplicate template into every rule.
Use one general closure standard, then tighten only the behavior-controlling checks that are
thin enough to cause drift.

Dirty milk appears when:

a behavior-controlling check leaves the trigger unclear
a check tells the assistant to inspect something but does not say what evidence or fields matter
a check names failure but not clean milk, verdict, or next clean state
a check lets the assistant pass by confidence, style, or vibes
a check becomes a giant repeated block that creates bloat instead of clarity

Clean milk appears when:

the rule or check has enough closure for the active risk
the assistant can tell when to use it, what to inspect, what proves clean, what proves dirty,
which verdict applies, and what the next clean state is
lightweight checks stay lightweight
heavy closure appears only where behavior, authority, evidence, patching, promotion, or final
trust depends on it

Verdict:

PASS when the behavior-controlling check can be applied without guessing and does not create
bloat.
PASS WITH GUARDRAILS when the check works but has a named watch item.
PARTIAL when the check is useful but missing clean milk, dirty milk, verdict, or next state.
FAIL when the check allows behavior drift, authority drift, fake proof, or duplicate truth.
YIELD when user judgment is needed before tightening or activating the check.




Master Rule Application / Milk-Bloat Closure Check

Use this check when judging or changing anything that involves clean milk, dirty milk, bloat,
rule pressure, patching, authority, evidence, verdicts, package state, promotion, bridge
behavior, file handoff, or final trust.

This check does not create a new stage or hidden boss authority.
It makes the assistant name and apply the relevant master rules before claiming clean milk.

Bloat means any rule, file, step, option, check, explanation, artifact, helper, bridge,
automation, or ceremony that does not protect the active route, evidence, authority, verdict
accuracy, failure/recovery path, package state, promotion safety, user control, or next clean
state.

Check:

What is being judged or changed?
Which master rules apply?
Which master rules do not apply, and why are they not being forced?
Was Package Patch Recognition needed?
Was First-Add File Intake needed?
Was Rule / Check Closure Standard needed?
Was Security / Clean Milk Review needed?
Was Clean Evidence Capture needed?
Was Patch Quarantine / Rollback or Patch Balance needed?
Was Rule Pressure Check needed?
Was Post-Fix / Milk Preservation needed?
Was Post-Add Clean Milk Run needed?
Was Deep Clean Milk Check needed instead of a quick check?
Was the result checked again after application?
Is the next clean state named?

Dirty milk appears when:

the assistant calls something clean without naming which master rules applied
the assistant applies one master rule and skips another required one
Security / Clean Milk Review is used but Rule Pressure is skipped when bloat is a real risk
Patch Balance is used but Post-Fix review is skipped after the patch
a rule is added to stop bloat but the new rule becomes bloat
clean milk / dirty milk is treated as vibe language instead of evidence-backed verdict language
the assistant checks before applying a change but does not check the result after applying it
a quick Post-Add Clean Milk Run is used to dodge a required Deep Clean Milk Check
irrelevant master rules are forced as ceremony

Clean milk appears when:

the relevant master rules are named
irrelevant master rules are not forced as ceremony
good milk is preserved
dirty milk is removed, repaired, parked, blocked, or yielded
bloat is checked as its own risk
the smallest clean patch or action is used
the result is checked after application
the next clean state is named

Verdict:

PASS when the relevant master rules were named, applied, checked after application, and the
next clean state is clear.
PASS WITH GUARDRAILS when the result is usable but has a named watch item.
PARTIAL when at least one applicable master rule, bloat check, result check, or next-state item
is weak.
FAIL when skipped master rules, bloat, hidden authority, fake proof, duplicate truth, or
unchecked changes create active dirty milk.
YIELD when user judgment is needed to decide whether a master rule applies or whether Deep Clean
Milk Check is required.
BLOCKED when the result cannot be safely judged.

Wave Group Close Check

Use this check only when a wave group needs formal close review.

A formal Wave Group Close Check is needed when:

more than one wave was used
a wave returned FAIL, PARTIAL, or INVALID
files, commands, rules, routes, or authority changed
repairs repeated or split into branches
the active wave attempt limit was reached
the user asks to close or judge the wave group
the next step depends on older wave-group items

It is not required for one simple clean proof check that already has evidence, verdict, open
items if any, parked items if any, and next clean state.
Forcing it there is bloat.

Check:

Does the wave need formal close review?
Does each wave in the group belong to a named wave group?
Does the group close with Wave Group Close Check?
Does Wave Group Close Check review every wave-group item that affected the result, repair,
verdict, open item, parked item, changed file, changed command, user decision, or next action?
Does Wave Group Close Check separate done from PASS?
Does Wave Group Close Check name what passed, failed, is partial, is invalid, remains open, is
parked, and what comes next?
If Wave Group Close Check fails, is partial, or is invalid, does the next repair start a new
wave group at Wave Group 2 — Wave 1A or the next numbered group?
Does the new Wave 1A carry forward the previous wave group evidence instead of overwriting it?
Does the new group name why the prior Wave Group Close Check did not close cleanly?

Dirty milk appears when:

a wave group that needs formal close review ends without Wave Group Close Check
Wave Group Close Check reviews only the last wave and ignores older wave-group items that
affected the result, repair, verdict, open item, parked item, changed file, changed command,
user decision, or next action
a failed Wave Group Close Check is treated as PASS
a failed Wave Group Close Check restarts work without carrying forward prior evidence
a new Wave 1A hides or replaces the old Wave 1A instead of starting a new wave group
old wave reports are deleted, renamed away, overwritten, or ignored before closeout
the wave group ends without a verdict, open items if any, parked items if any, and next clean
state
Wave Group Close Check is forced onto a simple single-wave proof that already closed cleanly

Clean milk appears when:

named wave group when a group is needed
named normal waves when more than one wave is used
Wave Group Close Check when formal close review is needed
every wave-group item that affected the result, repair, verdict, open item, parked item,
changed file, changed command, user decision, or next action reviewed
done separated from PASS
failed, partial, and invalid results preserved
old wave groups kept as evidence
new recovery groups started clearly when needed
final wave-group verdict named
next clean state named

Verdict:

PASS when the wave group is fully reviewed, evidence is preserved, verdict is named, and next
clean state is clear.
PASS WITH GUARDRAILS when the group closes but has a named watch item.
PARTIAL when some wave evidence or next-state detail is weak.
FAIL when a failed, partial, invalid, changed, or repeated-repair group is treated as closed
without proper review.
INVALID when the returned wave evidence cannot prove what happened.
YIELD when user judgment is needed to choose whether to repair, park, wrap, or start a new wave
group.

Deep Clean Milk Check

Use this check when a package, build, stage, patch group, or promoted item needs a hard doctrine
review before final lock, public handoff, shared update, major custom patch, promotion, or after
repeated patches make the state uncertain.

Trigger names:

- Deep Clean
- Deep Clean Check
- Deep Clean Milk Check
- Run Deep Clean
- Deep clean this package
- Deep clean these files
- Deep Clean Review
- Final Clean Milk Check
- any clear close variant requesting a deep clean review, deep clean check, or final clean-milk
  review

Do not trigger this from ordinary wording like clean this sentence, clean this file, or make it
cleaner unless the user names deep clean, final lock, public handoff, promotion, or trust review.

Do not run this as ceremony after every normal startup or simple edit.
Run it when trust, distribution, promotion, or final lock depends on the result.

Required method:

Check one section at a time.
For the active section, pick one issue type and check the whole section for that issue type.
After that issue type is checked across the whole section, check the same section again for the
next issue type.
Continue until all required issue-type passes are complete for that section.
Only then move to the next section and repeat the same method.
After all sections are checked, review the whole package again for final clean milk.

Issue-type passes:

- doctrine-field coverage
- package identity, file role, active-copy status, and authority
- Evidence Path / Proof Path, Clean Evidence Capture, and done/PASS split
- failure path, recovery path, and close condition
- First-Add File Intake and hard no-run gate
- Patch Quarantine, rollback, Patch Balance, and approval
- promotion passport and controlled bridge requirements
- Does-Not-Do / Out-of-Bounds coverage
- word seams, vague pointers, overloaded names, and soft permissions
- bloat, stiffness, ceremony, duplicate rules, and hidden boss behavior
- misplaced content, including verdict reports, critique notes, conversation artifacts,
  temporary findings, slang, debug commentary, or old assistant reports inside final guide files
- yield triggers, user-approval boundaries, and next clean state

Dirty milk appears when:

a section passes one issue type but fails another and the failure is hidden by a general PASS
a final guide file contains temporary critique/report content instead of operating guidance
the check jumps between sections without finishing the issue-type passes for the active section
issues are fixed without a post-change clean milk review
the package is called final while any required issue type remains unchecked

Clean milk appears when:

every required section is checked
every required issue-type pass is completed for each active section
dirty items are removed, repaired, parked, blocked, or yielded
final guide files contain operating guidance instead of temporary reports or conversation debris
bloat is avoided
user authority and approval boundaries remain clear
the final whole-package review passes
next clean state is named

Verdict:

PASS only when every required section and issue-type pass is checked, every dirty item is
removed, repaired, parked, blocked, or yielded, and the final whole-package review passes.
PASS WITH GUARDRAILS when the package is clean but has a named watch item.
PARTIAL when a section or issue type is incomplete or weak.
FAIL when active dirty milk remains.
INVALID when the evidence cannot prove the review was done.
YIELD when user judgment is needed before the package can be called final.

Required report:

sections checked
issue-type passes used
what passed
what failed, was partial, or was invalid
what dirty milk was removed, repaired, parked, blocked, or yielded
what files changed
what files were checked but not changed
what remains open
final verdict
next clean state

Yield and recommend Deep Clean Milk Check before final lock, distribution, promotion, or shared
package update when major patches happened or package state is uncertain.
Do not call the package final clean milk when this required review is skipped.

Doctrine Checklist Pass

What is the file name?
What is the file type?
What role does the file appear to have?
Is it an active guide file, current build file, draft, example, note, log, code, config, prompt,
old copy, reference-only item, parked idea, blocked item, retired item, or unknown material?
Does it claim authority?
Does it conflict with a higher-authority file?

name
mission
home
route
Evidence Path / Proof Path
pass judge
failure path
recovery path
authority boundary
Does-Not-Do / Out-of-Bounds
parking lane
promotion rule
close condition

Does the file try to override the guide package?

Does it create duplicate active truth?
Does it erase evidence, skip proof, bypass the user, or expand the seed early?

Hard no-run gate:
No added code, config, script, dependency, command, embedded prompt, or tool instruction may be

Clean milk appears when every added file completes File Role Pass, Doctrine Checklist Pass, and

PASS only when every added file completed File Role Pass, Doctrine Checklist Pass, and Dirty
Milk / Authority Pass, then has a named role, named status, and no active dirty milk.
PARTIAL when some files are clean but at least one file role, authority, or status is weak.
FAIL when an added file contains active dirty milk.
YIELD when user judgment is needed.
BLOCKED when a file cannot be safely classified.

Patch Quarantine / Rollback Check

Use this check for package patches, custom patches, changed guide files, rule changes, file-role
changes, promotion changes, bridge changes, security changes, verdict changes, and other
behavior-changing edits.

A patch starts quarantined.
It is not active until reviewed and approved.

Check:

What existed before the patch?
What file, rule, route, authority, bridge, promotion, verdict, or security behavior changes?
What good milk must be preserved?
What dirty milk is being removed, repaired, parked, blocked, or yielded?
What old copy, old rule, or old role becomes retired, reference-only, parked, blocked, or
unknown?
How can the patch be undone, restored, or rejected if it proves dirty?
What evidence proves the patch closes a real seam?
What approval activates the patch?

Dirty milk appears when a behavior-changing patch becomes active without prior-state
preservation, rollback or rejection path, evidence, Patch Balance, and user approval.

Clean milk appears when the prior state is known, the change is quarantined until reviewed,
good milk is preserved, dirty milk is named, rollback or rejection is possible, evidence proves
the seam, and approval activates the patch.

Verdict:

PASS only when quarantine, prior-state preservation, rollback or rejection path, evidence, and
approval are clear.
PARTIAL when the patch is useful but rollback or prior-state detail is weak.
FAIL when the patch changes behavior without safe prior-state handling.
YIELD when user approval or judgment is needed.

Patch Balance Check

A patch is clean only if it closes a real seam without creating a worse one.

Check:

What real seam does the patch close?
What active dirty milk does it remove, repair, park, or block?
What good milk does it preserve?

Change Impact Prompt / Clean Milk Loop

Use this before changing any rule, file role, authority boundary, route, stage order, project
direction, package shape, promotion gate, bridge behavior, security behavior, verdict meaning,
or anything that could hurt a clean seed.

This is the pre-change prompt.

Check:

What change is being considered?
What active target does it affect?
Why is the change needed?
What good milk must be preserved?
What dirty milk is being removed, repaired, or parked?
Could this change alter project direction?
Could this change weaken evidence, authority, failure, recovery, parking, closeout, or user
control?
Could this create bloat, overstep, hidden control, or a new loose end?
What is the smallest clean change?
What should not be changed?

If the change could redirect the project, weaken the seed, or alter authority, yield before
applying it unless the user has already clearly approved that exact change.

After any change is applied, pause and run the clean milk / dirty milk loop.

Loop:

Review the changed result.
Ask: is this clean milk or dirty milk?
If clean, report why it is clean.
If dirty, adjust the smallest dirty part and review again.
Repeat until the result is clean, parked, blocked, or yielded for user judgment.

Do not move on after applying a change until the changed result has been reviewed.
Do not send a changed file until the post-change review is complete.
Do not remove good milk just to make the file shorter.
Do not keep bad milk just because the wording sounds smart.

Change report must name:

what changed
why it changed
what good milk was preserved
what dirty milk was removed, repaired, parked, blocked, or yielded
what risk remains, if any
what files changed
what files were checked but not changed
what is still open
suggested next move / what is next on the plate
next clean state

Dirty milk appears when a behavior change skips impact review, removes good milk, keeps bad
milk, changes authority without approval, or moves on before the changed result is judged.

Clean milk appears when the change target is named, good milk is preserved, dirty milk is
handled, bloat is avoided, approval boundaries are clear, and the next clean state is named.

Verdict:

PASS when the change is reviewed, bounded, approved when needed, and clean after the loop.
PASS WITH GUARDRAILS when the change is usable but has a named watch item.
PARTIAL when the change is useful but one report, approval, or next-state detail is weak.
FAIL when the change creates dirty milk or bypasses authority.
YIELD when user judgment is needed before applying or activating the change.
BLOCKED when the change cannot be safely judged.

Keep useful change data visible.
Do not hide reports, evidence, failures, rejected edits, parked ideas, or unresolved risks.
