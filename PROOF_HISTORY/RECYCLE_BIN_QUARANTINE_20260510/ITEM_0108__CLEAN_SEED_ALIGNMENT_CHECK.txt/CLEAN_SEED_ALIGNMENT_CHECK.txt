Purpose

This file checks whether a clean seed build is staying small, clear, proven, bounded, and
easy to follow.

This file is not the start guide.
This file is not the assistant walk-through guide.
This file is not runtime proof.
This file is a guardrail and alignment check.
It does not add a separate test phase.
It checks proof only to prevent dirty milk.

Use this file with:

README_START_HERE.txt
CLEAN_SEED_WRAP_GUIDE.txt

The README starts the user.
The guide controls the assistant walk-through.
This alignment check catches drift and runs the Security / Clean Milk Review.


Hard Verdict Standard

Use PASS only when every required check passed together.

Use PASS WITH GUARDRAILS when the build is clean but needs an active watch item.

Use PARTIAL when some required checks pass and some are missing or weak.

Use FAIL when usable evidence proves a required check failed.

Use INVALID when the evidence cannot prove what happened.

Use PARKED when the idea is useful but not active.

Use YIELD when user judgment, approval, lookup, inspection, or verification is needed.

Use BLOCKED when progress is stopped by a real blocker.

Use PROMOTED when a proven, bounded item has been deliberately moved into an allowed larger
role.

Use RETIRED when an old file, rule, route, or copy is no longer active and must not be obeyed as
current truth.

Done is not PASS.
Movement is not proof.
Confidence is not proof.
Screenshots are not runtime proof unless the specific check is visual.


Clean Seed Target

A clean seed has:

one clear name
one home
one active job
one visible input/output route
one evidence path
one pass judge
one failure path
one recovery path
one authority boundary that names what can be touched and what cannot be touched
one Does-Not-Do / Out-of-Bounds list
one parking lane
one promotion rule
one close condition

If any of these are missing, the seed setup is not clean yet.



Rule / Check Closure Standard

Use this standard when judging or writing any behavior-controlling rule or check.

A behavior-controlling rule or check is one that can change assistant action, file authority,
evidence handling, verdicts, package state, user approval, patch behavior, promotion, bridge
behavior, or final trust.

A closed rule or check should name enough of these parts for a weak assistant to apply it
without guessing:

trigger / when to use it
what to inspect
minimum required evidence or fields
dirty milk conditions
clean milk conditions
verdict labels
next clean state

Not every explanatory README paragraph needs this closure. The README is an on-ramp.
The Alignment Check carries most verification closure.

Do not paste a giant duplicate template into every rule.
Use one general closure standard, then tighten only the behavior-controlling checks that are
thin enough to cause drift.

Dirty milk appears when:

a behavior-controlling check leaves the trigger unclear
a check tells the assistant to inspect something but does not say what evidence or fields matter
a check names failure but not clean milk, verdict, or next clean state
a check lets the assistant pass by confidence, style, or vibes
a check becomes a giant repeated block that creates bloat instead of clarity

Clean milk appears when:

the rule or check has enough closure for the active risk
the assistant can tell when to use it, what to inspect, what proves clean, what proves dirty,
which verdict applies, and what the next clean state is
lightweight checks stay lightweight
heavy closure appears only where behavior, authority, evidence, patching, promotion, or final
trust depends on it

Verdict:

PASS when the behavior-controlling check can be applied without guessing and does not create
bloat.
PASS WITH GUARDRAILS when the check works but has a named watch item.
PARTIAL when the check is useful but missing clean milk, dirty milk, verdict, or next state.
FAIL when the check allows behavior drift, authority drift, fake proof, or duplicate truth.
YIELD when user judgment is needed before tightening or activating the check.




Master Rule Application / Milk-Bloat Closure Check

Use this check when judging or changing anything that involves clean milk, dirty milk, bloat,
rule pressure, patching, authority, evidence, verdicts, package state, promotion, bridge
behavior, file handoff, or final trust.

This check does not create a new stage or hidden boss authority.
It makes the assistant name and apply the relevant master rules before claiming clean milk.

Bloat means any rule, file, step, option, check, explanation, artifact, helper, bridge,
automation, or ceremony that does not protect the active route, evidence, authority, verdict
accuracy, failure/recovery path, package state, promotion safety, user control, or next clean
state.

Check:

What is being judged or changed?
Which master rules apply?
Which master rules do not apply, and why are they not being forced?
Was Package Patch Recognition needed?
Was First-Add File Intake needed?
Was Rule / Check Closure Standard needed?
Was Rule Closure Cycle needed because a new or changed rule/check/master rule could create affected-scope work or follow-through?
Was Security / Clean Milk Review needed?
Was Clean Evidence Capture needed?
Was Patch Quarantine / Rollback or Patch Balance needed?
Was Rule Pressure Check needed?
Was Post-Fix / Milk Preservation needed?
Was Adjustment Stack Yield needed because repeated chat corrections or assistant tuning could
become unclear?
Was Prompt And Command Boundary needed because long prompts, command packets, reports, or
verdict text could be cut off, mixed, or hard to verify?
Was Clarification Checker needed because action steps, command packets, save instructions, or
return-output instructions could confuse the OS, shell, app, target, or evidence?
Was Setup Shortcut needed because repeated prompt, terminal, folder, PowerShell, Command Prompt,
or command-center use would be safer through a bounded shortcut?
Was Save Location And Backup CYA needed because files, rules, prompts, packages, reports,
evidence, shortcuts, or build artifacts were saved, replaced, exported, overwritten, or handed
off?
Was Slow Review / Interruption Warning needed because heavy clean-seed review could look slow,
or because an interruption could mix the active fix stack?
Was Interruption Recovery / Last Actual Fix Check needed because work was paused, corrected, interrupted, restarted, or resumed after a partial response?
Was Post-Add Clean Milk Run needed?
Was Affected-Scope Clean Milk Sweep needed?
Was Rule Closure Cycle needed after the addition was applied, so the changed package was checked again?
Was Clean Finding Placement needed?
Was New File / Directory Certification needed?
Was Deep Clean Milk Check needed instead of a quick check?
Was Verdict Scope / Stage Proof / Wave Boundary Check needed?
Was the result checked again after application?
Is the next clean state named?

Dirty milk appears when:

the assistant calls something clean without naming which master rules applied
the assistant applies one master rule and skips another required one
a new rule or master rule is added without closing trigger, scope, evidence, verdicts, affected scope, follow-through, and next state
Security / Clean Milk Review is used but Rule Pressure is skipped when bloat is a real risk
Patch Balance is used but Post-Fix review is skipped after the patch
a rule is added to stop bloat but the new rule becomes bloat
clean milk / dirty milk is treated as vibe language instead of evidence-backed verdict language
the assistant checks before applying a change but does not check the result after applying it
the assistant continues after a long prompt, command packet, report, or verdict text looks cut
off, mixed, or hard to verify
the assistant keeps using long manual command packets when a bounded shortcut is the cleaner
route for repeated prompt, terminal, folder, PowerShell, Command Prompt, or command-center work
a quick Post-Add Clean Milk Run is used to dodge a required Deep Clean Milk Check
files are saved, overwritten, exported, or handed off without save-location and backup CYA when
that risk is active
irrelevant master rules are forced as ceremony

Clean milk appears when:

the relevant master rules are named
irrelevant master rules are not forced as ceremony
good milk is preserved
dirty milk is removed, repaired, parked, blocked, or yielded
bloat is checked as its own risk
the smallest clean patch or action is used
the result is checked after application
long prompts, command packets, reports, and verdict text are bounded or split when needed
shortcuts are created only when repeated use proves they reduce confusion without hiding proof
the next clean state is named

Verdict:

PASS when the relevant master rules were named, applied, checked after application, and the
next clean state is clear.
PASS WITH GUARDRAILS when the result is usable but has a named watch item.
PARTIAL when at least one applicable master rule, bloat check, result check, or next-state item
is weak.
FAIL when skipped master rules, bloat, hidden authority, fake proof, duplicate truth, or
unchecked changes create active dirty milk.
YIELD when user judgment is needed to decide whether a master rule applies or whether Deep Clean
Milk Check is required.
BLOCKED when the result cannot be safely judged.

Build Direction And Naming Check

Use this check during startup and whenever project-specific names are being chosen.

Check:

Has the user defined the build direction, provided current build state, or asked for help
shaping the first build idea?
If only the guide files are loaded and no build content exists, did the assistant ask what the
user is starting from before doing build work?
Did the assistant use the same clean start screen with no more than 3 options when a menu was
needed?
Did the assistant recommend one clean option instead of dumping a pile?
Do the seed name, project folder name, existing project home path if present, and first build
files fit the build direction or current build state?
Are the names plain, role-based, and tied to the active seed instead of future scale?
Did the three guide filenames stay fixed?
Did the assistant avoid forcing a starter name onto a real build without approval?
Did the assistant avoid creating project-specific files before the starting situation and names
were accepted?

Clean milk appears when:

the build direction or current build state is known before names lock in
the user provides the real build idea or current build state
the assistant chooses the smallest clean seed structure needed to prove it
current build context is recognized before any duplicate blank seed is created
the assistant asks for direction when direction is missing
the assistant helps name the seed around the build direction or current build state without old
prototype names, mocking labels, insulting labels, or future-scale names
the core guide filenames stay unchanged
old starter names are used only when the user deliberately accepts them as real names
build-specific names serve active file roles

Dirty milk appears when:

project-specific names lock in before the build direction or current build state is known
the assistant makes the user choose internal seed type, brain part, architecture lane, route
lane, proof system, workflow category, worker type, or core build method
current build context is ignored and the assistant starts a duplicate blank seed
the assistant assumes a starter name is the real build without approval
old prototype names leak into the new build
core guide files are renamed to match the project
file names claim future architecture too early
option menus become a pile
naming creates unclear file roles or multiple active truths

Verdict:

PASS only when the build direction or current build state is known, names fit that situation,
and core guide filenames remain fixed.

PARTIAL when the direction is known but one or more names are weak, vague, or too future-facing.

FAIL when names are locked, files are created, or project work begins from an assumed direction
the user did not give or accept.

YIELD when the starting situation is missing and the next step would create project-specific
names, folders, or files.


Build Alignment Check

Check every active stage against these questions:

Build Direction:
Is the build direction or current build state known before project-specific names lock in?
If current build files, folders, code, notes, logs, or project state were provided, did the
assistant recognize the current build instead of starting a duplicate blank seed?
Did the assistant avoid asking the user to choose internal seed type, brain part, architecture
lane, route lane, proof system, workflow category, worker type, or core build method?
If the starting situation was missing, did the assistant ask what the user is starting from
before doing build work?
Did the assistant give only the clean start-screen options, with one recommended option, when
the user needed help naming the starting situation?

Name:
Is the seed name plain, honest, and tied to what is being built now?
Does the name avoid claiming future scale too early?

Mission:
Is the mission short and active?
Does it say what is being built now?
Does it leave future features parked?

Home:
Does the seed have one project folder?
Are tasks, outputs, evidence, support files, and closeout files placed clearly?

Route:
Is the input/output route visible?
Can the user say what enters, what happens, what exits, and where evidence is stored?

Evidence Path / Proof Path:
Is the proof path named as its own field?
Can the user tell where evidence is captured, what evidence is required, and how it will be
judged?

Done / Pass:
Is done separated from PASS?
Does PASS require evidence?
Is the judge clear?

Failure:
Can failure happen visibly?
Is failure evidence protected before repair?

Recovery:
Can interrupted, stale, or incomplete work be recovered or safely failed?
Is recovery evidenced?

Authority:
Does the user own direction and approval?
Does the authority boundary name what can be touched, what cannot be touched, and what needs
approval?
Does the assistant stay inside the active mission and approved boundary?

Does-Not-Do / Out-of-Bounds:
Does the seed name what it explicitly does not do?
Does every helper, bridge, patch, route, or promoted item have enough out-of-bounds wording to
prevent authority drift?

Parking:
Are useful future ideas preserved without controlling the active build?
Does every parked idea have a reopen condition?

Promotion:
Is the promotion rule named before the seed grows, adds a new lane, adds a helper, connects a
bridge, or moves to the next stage?

Close:
Does the current stage have a clear close condition?
Is the stage wrapped before scaling?

The build should be as easy as possible without hiding proof.

Safe means inside the active mission, inside the approved boundary, non-destructive,
inspectable, and not hiding or erasing evidence.

Check:

Is the proposed shortcut inside the active mission and approved boundary?
Does it preserve evidence, user approval, and inspectability?
Would a safe command, file packet, or tool action reduce needless manual work?
Does the assistant explain risky commands in plain terms before use?
Does the assistant give one clean default when the next action is obvious?
Are options limited to 2 or 3 when a real choice exists?

Dirty milk appears when ease:

skips evidence
hides errors
overwrites useful state
bypasses user approval
expands the seed too early
treats convenience as proof
uses options to avoid judgment
makes the user carry work the assistant can safely simplify

Clean milk appears when:

the easy path stays inside the active mission and approved boundary
the route remains inspectable
evidence remains visible
errors are not hidden
the user gets one recommended clean default unless a real choice exists
risky actions are explained before use

Verdict:

PASS when the easier route preserves proof, authority, safety, and the next clean state.
PASS WITH GUARDRAILS when the easier route is usable but has a named watch item.
PARTIAL when the route is easier but one proof, approval, or clarity detail is weak.

YIELD when user judgment or approval is needed before using the easier route.

Mechanical Clarity Check

Use this check before giving the user instructions that require opening, pasting, saving,
running, sending, or judging anything.

Check whether the assistant told the user exactly:

what to open
where to go
what to paste or run
what to save
where to save it
what result to send back
what evidence will be judged

Dirty milk appears when the user has to guess the app, folder, file, command target, paste
target, save location, return report, or proof result.

Clean milk appears when the instruction names the exact target, action, expected result,
return evidence, and next clean state without making the user infer the missing step.

Verdict:

PASS when the user can follow the instruction without guessing and can return judgeable
evidence.
PARTIAL when the instruction is mostly clear but one target, result, or evidence detail is weak.
FAIL when the instruction creates a guess point that could change the file, route, proof, or
verdict.
YIELD when the target is unclear and user judgment is needed before proceeding.

Prompt And Command Boundary Check

Use this check when a prompt, command, report, verdict, or file-content packet is long enough
that the user could accidentally cut it off, mix old and new verdicts, paste only part of a
command, or lose the start/end boundary.

Check:

Is there one clear target for the packet?
Is there one clear expected return?
Are old verdicts separated from new verdicts?
Is file content separated from commands and reports?
Is the start and end of long content clear enough?
Does the command print short judgeable PASS lines or a named report?
Does the returned evidence show the packet completed?
Does the returned text look cut off, mixed, stuck in continuation mode, or missing the final
completion lines?
Should the assistant recommend closing and reopening a fresh prompt or PowerShell window before
continuing?

Dirty milk appears when:

old verdict text and new verdict text blur together
the user has to guess where a prompt starts or ends
a long packet is cut off and the assistant continues anyway
PowerShell or Command Prompt output is incomplete but treated as PASS
file content, commands, and reports are mixed in a way that creates paste-boundary risk

Clean milk appears when:

long packets are split or clearly bounded
each packet has one target and one expected return
completion evidence is short and judgeable
cut-off, mixed, or incomplete output triggers a yield instead of a guess
the user can close and reopen a fresh prompt, terminal, folder, or command center without losing
the active route

Verdict:

PASS when the packet boundary is clear and the return evidence is judgeable.
PASS WITH GUARDRAILS when usable but the packet length or paste boundary needs a named watch
item.
PARTIAL when one boundary, target, or return detail is weak.
FAIL when cut-off or mixed content is treated as clean.
YIELD when the user should close/reopen a fresh prompt or rerun a smaller packet before
continuing.

Command Return Evidence Check

Use this check whenever a command, script packet, file-write packet, terminal step, or shortcut
asks the user to run something and return evidence.

Check:

Did the assistant tell the user exactly what output to send back?
Did the assistant ask for only the short printed output text, such as PASS, FAIL, error, or
named report lines, by default?
Did the assistant avoid identifying output only by color, such as white text or colored text?
Did the assistant use color only as secondary clarification when useful, after the printed output
target was named?
Did the assistant avoid asking for the whole command block unless error context is needed?
Is the return evidence short enough to judge without old/new text bleed?
If output is cut off, mixed, stuck in continuation mode, or missing the expected line, did the
assistant yield instead of guessing?
Did the assistant recommend closing and reopening a fresh prompt or PowerShell window when that
is the safest reset?

Dirty milk appears when:

the user has to guess what part of the output to return
the assistant identifies output only by color, even though colors vary by OS, terminal, and theme
the assistant asks for excessive output by default
missing PASS or error lines are treated as completion
cut-off terminal state is treated as clean
old command text and new verdict text blur together

Clean milk appears when:

return evidence is named before or near the action step
printed output text is named before color cues
the expected output is short and judgeable
errors are returned with enough context to repair safely
cut-off or mixed output triggers a reset or smaller packet
completion is judged from visible evidence

Verdict:

PASS when the return evidence is clearly named and judgeable.
PASS WITH GUARDRAILS when the return works but packet length or output clutter needs a watch item.
PARTIAL when one return detail is weak.
FAIL when missing, mixed, or cut-off output is treated as clean completion.
YIELD when the user should return a smaller output, close/reopen a fresh prompt or PowerShell
window, or rerun a smaller packet.



Interruption Recovery / Last Actual Fix Check

Use this check when work resumes after the user interrupts, pauses, corrects, restarts, or returns
during or after a long review, patch, save, file handoff, command packet, rule update, verdict
review, Deep Clean, or project sync.

Check:

What was the last actual verified fix or saved checkpoint?
What work was interrupted?
What was the active target before interruption?
Did the user interruption pause the work, correct a mistake, add a new rule, raise a safety
concern, change the target, or ask to continue?
Are accepted fixes named?
Are rejected fixes blocked from reappearing?
Are parked concepts still parked?
Are blocked or yielded items still marked?
Are unsaved thoughts, partial drafts, or unfinished plans excluded from the recovery point?
Did any rule-implied task remain after the interruption?
Did any saved change need post-save clean milk review?
Did any project file become stale after a core-rule update?
Is the next clean action named from the last actual verified fix instead of from memory?

Dirty milk appears when:

the assistant resumes from a half-remembered thought
the assistant treats an unsaved draft as an actual fix
the assistant forgets an accepted correction
the assistant applies a rejected idea
the assistant loses a parked concept
the assistant skips a rule-implied task created before the interruption
the assistant continues from conversation momentum instead of the last verified checkpoint
the assistant restarts too far back and discards valid saved fixes
the assistant moves forward without naming the recovery point

Clean milk appears when:

the last actual verified fix or saved checkpoint is named
accepted, rejected, parked, blocked, yielded, and open items are separated
new user input is classified before work resumes
post-save clean milk review is completed when needed
rule-implied follow-through is completed, yielded, parked, blocked, or marked not affected
the next clean action starts from the verified recovery point

Verdict:

PASS when the recovery point is verified, the active stack is rebuilt, and the next clean action is
clear.
PASS WITH GUARDRAILS when recovery can continue but has a named watch item.
PARTIAL when one recovery detail is weak but not dangerous.
FAIL when the assistant continues from a thought, loses an accepted fix, applies a rejected fix,
or skips a required follow-through task.
YIELD when user judgment is needed to choose the recovery point.
BLOCKED when no safe recovery point can be identified.


Verdict Scope / Stage Proof / Wave Boundary Check

Use this check whenever command output, file saves, checked inputs, test cases, stage closeout,
package review, or waves could blur together.

Check:

Is the verdict scope named?
Is this a file-save PASS, command-output PASS, checked-input verdict, test-case verdict, stage
verdict, package verdict, or wave verdict?
Did the assistant avoid treating a file-save PASS as content PASS?
Did the assistant avoid treating a command PASS as stage PASS?
If a dirty input received FAIL, did the assistant explain that the checker route may still PASS
because it caught the dirty input correctly?
If a cut-off input received INVALID, did the assistant explain that the test case may still PASS
because it proved the checker did not guess?
Is Stage PASS withheld until required evidence and closeout satisfy the stage close condition?
Is Clean Evidence Capture used instead of broad data collection language?
Is captured evidence necessary, bounded, and tied to the active target?
Are stage proof tests kept separate from waves unless a named wave repair process is active?
If repeated repair attempts are happening against the same target, did the assistant start or
continue a named wave group and preserve prior evidence?
Is the next clean state named?

Dirty milk appears when:

file-save PASS is treated as proof that content is clean
command-output PASS is treated as proof that a stage passed
a dirty input receiving FAIL is described as the dirty input passing
a test-case route proof is confused with the checked input verdict
stage PASS is claimed before evidence and closeout
data collection language expands clean evidence into hoarding
normal stage proof tests are relabeled as waves after the fact
wave rules are skipped when repeated repair attempts are actually happening
verdict scope is unclear

Clean milk appears when:

file-save, command-output, checked-input, test-case, stage, package, and wave verdicts are separated
when needed
the assistant names exactly what passed and what did not pass
dirty, partial, failed, or invalid checked inputs can still prove the checker route when the
expected verdict is correct
Clean Evidence Capture stays bounded and necessary
stage proof tests are not waves unless a named wave process is active
wave groups are used only when repeated repair attempts require them
next clean state is clear

Verdict:

PASS when verdict scope is clear, evidence is bounded, wave boundaries are correct, and next clean
state is named.
PASS WITH GUARDRAILS when usable but a scope label needs a watch item.
PARTIAL when one scope, evidence, or wave-boundary detail is weak.
FAIL when file-save PASS, command PASS, checked-input verdict, test-case proof, stage verdict, or
wave verdict are mixed.
INVALID when evidence cannot prove which scope passed or failed.
YIELD when user judgment is needed before continuing.


Clarification Checker Check

Use this check before action steps, command packets, file-write packets, save instructions,
shortcut instructions, handoff instructions, or return-output instructions when confusion about
the operating system, shell, app, folder, filename, paste target, save target, output text, or next
action could cause wrong work.

Check:

Is the active operating system known?
Is the active shell, terminal, command app, editor, or file tool known when the instruction depends
on it?
Does the instruction name the exact action, target, and expected result?
Does the instruction name what output, file, or report the user should send back?
Does the instruction avoid relying only on color words such as white text or colored text?
If color is mentioned, is it secondary to the real target, such as printed PASS or error lines?
If the environment is unknown, does the assistant use neutral wording or clarify before giving an
OS-specific command?
Would the instruction still make sense if terminal colors, theme, or OS changed?

Dirty milk appears when:

the user must guess the OS, shell, app, folder, file, paste target, save target, or return evidence
white text, colored text, prompt, terminal, output, this, that, or it is used without enough target
clarity
color is the main identifier for returned command evidence
the assistant gives system-specific instructions while the active system is unknown and the
system difference matters
the assistant gives alternatives by default instead of one easiest clear route

Clean milk appears when:

the active environment is named when known
neutral wording is used when the environment is unknown
color is optional help, not the main identifier
printed output text and PASS/error lines are named clearly
exact action, target, expected result, return evidence, and next clean state are clear

Verdict:

PASS when the instruction is clear for the active environment and no mechanical guess point remains.
PASS WITH GUARDRAILS when usable but a named environment or return-output watch item remains.
PARTIAL when one target, environment, or return-output detail is weak.
FAIL when ambiguity could cause wrong run, wrong paste, wrong save, wrong file, or wrong evidence.
YIELD when environment or target must be clarified before continuing.


Slow Review / Interruption Warning Check

Use this check when heavy clean-seed rules may make the assistant slower than a normal answer,
or when user interruption could cause accepted fixes, rejected fixes, file targets, backup state,
verdicts, or next clean state to become mixed.

Heavy review can happen during Deep Clean, package patching, file updates, CYA save and backup
work, verdict reviews, rule changes, promotion reviews, wave group close checks, or repeated
adjustment-stack cleanup.

Check:

Was the user warned that strong clean-seed rules can slow the assistant during heavy review?
Did the assistant give short progress updates when useful and possible?
Did progress updates avoid replacing proof or promising background work?
If the user interrupted from concern that the assistant was slow, did the assistant name the
active review state and continue from the last clean checkpoint?
If a long review, patch, save, or file handoff was interrupted, did the assistant assume the fix
stack may be at risk until accepted changes, rejected changes, parked concepts, saved files, and
next clean state were named again?
If the user interrupted with a real correction, safety concern, approval boundary, or target
change, did the assistant pause, preserve state, and run the relevant adjustment, CYA, boundary,
or change-impact rule before continuing?
Are accepted fixes, rejected fixes, parked ideas, save locations, backup state, active files, and
next clean state still clear after the interruption?

Dirty milk appears when:

the user is not warned that heavy reviews can be slower
the assistant appears stalled and gives no orientation when a short update is possible
a speed-concern interruption causes the assistant to restart from memory
accepted fixes, rejected fixes, parked ideas, save targets, backups, or verdicts get mixed
the assistant continues after an interruption without naming the last clean checkpoint and active
fix stack
interruption becomes an excuse to skip proof, backup, CYA, or final review
the assistant promises future background work instead of completing or reporting current work

Clean milk appears when:

the user understands slow review is expected during heavy clean-milk work
short progress updates are used when useful
the active target and last clean checkpoint stay clear
interruption type is handled correctly
accepted fixes, rejected fixes, parked ideas, save locations, backup state, verdicts, and next
clean state remain clear

Verdict:

PASS when heavy-review slowdown is explained, progress orientation is adequate, and no active
state is lost or mixed.
PASS WITH GUARDRAILS when the work can continue but needs a named watch item.
PARTIAL when the user was partly oriented but one checkpoint, fix-stack, save, or next-state
detail is weak.
FAIL when slowdown or interruption causes lost fixes, mixed state, skipped proof, skipped CYA,
or false clean verdict.
YIELD when user judgment is needed before continuing.
BLOCKED when the active review state cannot be safely reconstructed.


Save Location And Backup CYA Check

Use this check whenever the assistant or user is saving, updating, replacing, exporting, handing
off, backing up, moving, renaming, or overwriting rules, guide files, build files, seed cards,
prompts, reports, verdicts, evidence, configs, scripts, shortcuts, or project artifacts.

CYA means Cover Your Ass: preserve the prior state, save to the right place, keep proof visible,
and leave a recovery path before progress depends on memory or guesswork.

Check:

What active target is being saved or changed?
What is the active project home or package home?
What exact destination folder will receive the file?
What exact filename will be used?
Is the file new, changed, unchanged, backup, exported, retired, parked, blocked, or unknown?
Does an old copy already exist?
Is a backup required before overwrite, replacement, handoff, promotion, stage close, wave close,
Deep Clean lock, package export, or after 3 meaningful saved changes?
If the assistant can save directly with a safe available tool or bounded command packet, did it
do that or provide that packet instead of leaving placement to guesswork?
After saving, did the assistant check the saved result again for clean milk and dirty milk?

Change Impact Prompt / Clean Milk Loop

Use this before changing any rule, file role, authority boundary, route, stage order, project
direction, package shape, promotion gate, bridge behavior, security behavior, verdict meaning,
or anything that could hurt a clean seed.

This is the pre-change prompt.

Check:

What change is being considered?
What active target does it affect?
Why is the change needed?
What good milk must be preserved?
What dirty milk is being removed, repaired, or parked?
Could this change alter project direction?
Could this change weaken evidence, authority, failure, recovery, parking, closeout, or user
control?
Could this create bloat, overstep, hidden control, or a new loose end?
What is the smallest clean change?
What should not be changed?

If the change could redirect the project, weaken the seed, or alter authority, yield before
applying it unless the user has already clearly approved that exact change.

After any change is applied, pause and run the clean milk / dirty milk loop.

Loop:

Review the changed result.
Ask: is this clean milk or dirty milk?
If clean, report why it is clean.
If dirty, adjust the smallest dirty part and review again.
Repeat until the result is clean, parked, blocked, or yielded for user judgment.

Do not move on after applying a change until the changed result has been reviewed.
Do not send a changed file until the post-change review is complete.
Do not remove good milk just to make the file shorter.
Do not keep bad milk just because the wording sounds smart.

Change report must name:

what changed
why it changed
what good milk was preserved
what dirty milk was removed, repaired, parked, blocked, or yielded
what risk remains, if any
what files changed
what files were checked but not changed
what is still open
suggested next move / what is next on the plate
next clean state

Dirty milk appears when a behavior change skips impact review, removes good milk, keeps bad
milk, changes authority without approval, or moves on before the changed result is judged.

Clean milk appears when the change target is named, good milk is preserved, dirty milk is
handled, bloat is avoided, approval boundaries are clear, and the next clean state is named.

Verdict:

PASS when the change is reviewed, bounded, approved when needed, and clean after the loop.
PASS WITH GUARDRAILS when the change is usable but has a named watch item.
PARTIAL when the change is useful but one report, approval, or next-state detail is weak.
FAIL when the change creates dirty milk or bypasses authority.
YIELD when user judgment is needed before applying or activating the change.
BLOCKED when the change cannot be safely judged.

Keep useful change data visible.
Do not hide reports, evidence, failures, rejected edits, parked ideas, or unresolved risks.
