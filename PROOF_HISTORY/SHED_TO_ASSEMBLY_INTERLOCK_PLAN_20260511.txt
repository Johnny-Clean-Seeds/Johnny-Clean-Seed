SECTION 1 â€” REAL-LIFE MODEL

House:
The current living system.
Represents CURRENT_TRUTH_INDEX.txt and the three active guide files.
Proof-history material does not enter the house by identity.

Shed:
Useful non-active storage.
Represents PROOF_HISTORY, drafts, receipts, old ideas, source-ore, candidate rules, review reports, and saved concepts.
The shed is valuable, but it is not authority by itself.

Workbench:
The current active target.
Only the current reviewed pile belongs here.
If everything is on the workbench, nothing is truly active.

Bins / Shelves:
Labeled placement lanes.
Examples:
- evidence
- parked idea
- rest stop
- blocked / quarantine
- source-ore
- proof-history rule
- technician report
- promotion candidate
- delete candidate

Lockbox:
Proof custody.
Stores receipts, hashes, paths, backup notes, boundary proof, and save reports.

Trash Can:
Delete gate only.
Nothing goes there because it is old, replaced, ugly, duplicate-looking, or annoying.
Deletion requires exact path, proof, user approval, and post-delete verification.

Caution Tape:
Freeze before movement.
Use when something weird, risky, unclear, duplicated, replaced, damaged, or trust-affecting appears.
Freeze blocks blame, fixing, cleanup, deletion, overwrite, move, rename, promotion, and old-file cleanout until evidence is named.

Label Maker:
Words are keys.
Labels route meaning, authority, evidence, verdict, and next state.
A label never creates authority by itself.

Toolbox / Technician:
Local worker with broader file visibility and bounded trust.
The technician may inspect, review, recommend, and give provisional judgments inside approved scope.
The technician does not decide final PASS, promote active truth, override active guides, or override CURRENT_TRUTH_INDEX.

Assembly Line:
The movement path for material.
It controls how useful stored material becomes usable action without drift.

SECTION 2 â€” OPERATING CYCLE

Current rough cycle:
1. Intake
2. Freeze
3. Inventory
4. Sort
5. Judge
6. Place
7. Save
8. Seal
9. Handoff
10. Review
11. Promote / Park / Block
12. Continue

Cycle meaning:

Intake:
New material enters.
Examples: user idea, file, worker report, old draft, proof-history candidate, screenshot, receipt, unknown file, cleanup concern.

Freeze:
Stop unsafe movement.
No deletion, movement, overwrite, rename, active-guide edit, current-truth edit, folder creation, script run, zip extraction, shortcut execution, cleanup, or promotion.

Inventory:
Record what it is.
Minimum fields:
- name
- path if local
- source
- role
- status
- evidence available
- evidence missing
- why it matters

Sort:
Make piles before judging.
Pile-making is not judging.
Judging is not fixing.
Fixing is not deleting.

Judge:
Assign provisional status from evidence.
Possible statuses:
- active truth candidate
- proof-history reference
- parked
- rest stop
- blocked
- unknown needs review
- duplicate / no-new-value candidate
- delete candidate
- promotion candidate
- technician review candidate

Place:
Put the item into the correct lane.
Placement does not mean promotion.

Save:
Create proof-history file, receipt, backup, hash, or decision record when useful.

Seal:
Close with boundary proof:
- what changed
- what did not change
- path
- size
- hash
- verdict
- remaining parked items
- next clean action

Handoff:
Send bounded work to technician when local file visibility or structured review is needed.

Review:
Check technician output for:
- path proof
- hash proof
- boundary proof
- overreach
- parked items
- yielded items
- recommendation quality
- authority correctness

Promote / Park / Block:
Every item gets a next state.
Nothing floats.

Continue:
Only move on after current ball is closed, saved, parked, blocked, yielded, or carried forward.

SECTION 3 â€” TRADEOFF / EVOLUTION CHECK

Use a small Tradeoff / Evolution Check before handoff or before changing:
- structure
- authority
- storage
- technician role
- folder count
- naming
- deletion route
- proof burden
- active-guide doctrine
- current-truth routing

Core law:
Fit beats universal best.

Questions:
1. What does this improve?
2. What does this cost?
3. What new failure mode appears?
4. What situation is it best for?
5. What situation makes it bad?
6. Does it feed the active decision?
7. Should it continue, park, block, or yield?

This check must stay small.
It is not a new master boss.
It is not endless analysis.
It exists to improve the next judgment call.

SECTION 6 â€” NAME WRAP PROVENANCE / WORD KEYS

Core law:
A better label must not cut the wire to the original key.

Words are paths.
Words are keys.
Names route meaning, authority, evidence, verdict, and next state.

A new name is an alias first, not a replacement.

Before wrapping or renaming a concept, record:
1. Original word / name
2. Proposed new name
3. Why the new name is needed
4. What the new name improves
5. What the new name might hide
6. What meaning stays unchanged
7. What meaning changes
8. Source path / evidence path
9. Authority boundary
10. Collision risk
11. Old name visibility
12. Review gate before replacement

Dirty state:
A clean-sounding wrapper severs lineage, hides the source meaning, or creates false authority.

Clean state:
The new name improves routing while keeping the original path visible until replacement is reviewed and approved.
