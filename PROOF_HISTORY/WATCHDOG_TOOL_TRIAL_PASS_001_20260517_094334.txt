TOOL NAME:
Watchdog

TOOL CLASS:
Control-pattern candidate.

PROBLEM IT TARGETS:
The assistant or local process appears to be running, but no real progress is being returned, proven, or bounded.
This includes:
- fake background-work posture
- silent stall
- endless "running" language
- no state-change return
- misleading PASS-style completion after missing state
- local process that hangs or fails without clear status

WATCHDOG JOB:
Detect when a run is no longer making visible, bounded, provable progress.

FIT TEST:
A Watchdog is useful in Clean Seed work if it can do all of these:
1. Notice when progress is no longer visible.
2. Force a state return instead of pretending continuation.
3. Classify the state as clean, blocked, defect found, save candidate, or decision needed.
4. Avoid chat bloat.
5. Avoid false background-work claims.
6. Preserve boundaries.
7. Point to the next clean move.

TRIAL AGAINST RECENT FAILURE:
Recent failure:
Assistant said it was running/continuing as if it could work silently after the response ended.

Watchdog response that should have happened:
- Stop the fake-running posture.
- Return state immediately.
- Say whether real work was completed.
- If more work is needed, provide a script or bounded next action.
- Do not imply background execution.

PASS CONDITION:
Watchdog passes if it turns "I am still working silently" into a returned state inside the current turn.

FAIL CONDITION:
Watchdog fails if it creates more ceremony, more labels, or more chat noise without changing behavior.

OBSERVED RESULT:
Watchdog directly fixes the fake-running / silent-stall class of defect.
It provides a clear stop-return rule:
No invisible continuation. Return state or provide local executable route.
