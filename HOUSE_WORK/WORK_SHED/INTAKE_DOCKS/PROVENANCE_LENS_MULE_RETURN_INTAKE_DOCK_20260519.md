# Provenance Lens Mule Return Intake Dock

Date: 2026-05-19.
State: Work Shed intake dock.
Authority: intake/sorting only.

## Job

This dock receives the mule’s Deep Slam + Gold Search report for the Provenance / Chain-of-Custody Thread Lens.

It prevents the report from becoming a loose pile, premature doctrine, active-guide material, promotion proof, or model truth.

The mule report is expected to contain critique, outside research, source citations, gold candidates, rejection candidates, and next-move recommendations. This dock sorts that material before anything is adopted.

## Hard Boundary

This dock does not promote the Provenance Lens.

This dock does not rewrite active guides.

This dock does not install doctrine.

This dock does not create runtime automation.

This dock does not expand bitstring work into a full system.

This dock does not make outside research authoritative.

This dock does not disturb the mule’s active handoff file.

This dock does not touch the Provenance Lens files while the mule is working.

## Why This Exists

The mule is being asked to read, judge, slam hard, deep-search for gold, and cover all aspects of the need.

That kind of output can be powerful, but it can also create a flood.

Without an intake dock, the report could cause:

- gold mixed with bad ideas,
- critique mixed with proof,
- outside standards mistaken for doctrine,
- source ore mistaken for active rule,
- promotion pressure,
- duplicate tools,
- neighbor conflicts,
- too many next moves,
- stale or unsupported claims,
- loss of useful rejected ideas,
- wrong-lane saves.

This dock gives the report a landing shape.

## Intake Rule

When the mule report returns, do not immediately install anything.

First freeze the report.

Then sort it.

Then decide the next clean move.

## Required Intake Phases

### Phase 1 - Freeze

Capture the mule report as received.

Do not edit the report while judging it.

Preserve source citations.

Preserve claims, critiques, and recommendations.

Mark the report as mule output, not house truth.

### Phase 2 - Source Check

For each cited outside item, classify:

- primary standard / official documentation,
- academic or research source,
- government / institutional guidance,
- mature engineering source,
- weak but interesting source ore,
- unsupported claim,
- irrelevant source,
- source not checked.

Do not let weak sources become authority.

### Phase 3 - Aspect Coverage Check

Judge whether the mule covered:

1. source/origin,
2. actor/custodian,
3. authority boundary,
4. artifact/output,
5. transformation/change,
6. timestamp/version/hash,
7. proof/receipt,
8. handoff/transfer,
9. integrity/tamper risk,
10. rollback/recovery,
11. retrieval/indexing,
12. lifecycle state,
13. lane placement,
14. promotion pressure,
15. proof-history vs live-state confusion,
16. source ore vs doctrine confusion,
17. support note vs active rule confusion,
18. duplicate/overlapping tools,
19. naming/key precision,
20. ceremony/bloat risk,
21. missing failure path,
22. auditability,
23. trust scoring,
24. clean-lane placement,
25. Soft Suit / Hard Suit lifecycle,
26. human workflow burden,
27. mule/Codex handoff use,
28. future automation risk,
29. adopt candidates,
30. reject candidates.

If a major aspect is missing, mark it as a gap.

### Phase 4 - Ring Sort

Sort every useful item into one of these lanes:

- ADOPT NOW CANDIDATE,
- ADAPT CANDIDATE,
- SOURCE ORE ONLY,
- REJECT,
- PARK FOR LATER,
- NEEDS PROOF,
- NEEDS SEPARATE TEST,
- PROMOTION-REVIEW CANDIDATE,
- WORD/KEY IMPROVEMENT,
- NEIGHBOR CONFLICT,
- DO NOT TOUCH.

### Phase 5 - Authority Check

For each item, ask:

- Is this a source fact or a mule inference?
- Does it fit Mr.Kleen’s core?
- Does it preserve clean lanes?
- Does it respect source/support/proof/doctrine/runtime boundaries?
- Does it reduce confusion or create ceremony?
- Does it duplicate an existing tool?
- Does it require a separate test?
- Does it require a separate promotion review?

### Phase 6 - Boss Ranking

Do not process the whole mule report at once.

Pick the top boss only.

Ranking order:

1. highest risk of authority contamination,
2. highest source/proof confusion,
3. highest promotion-pressure risk,
4. biggest missing proof,
5. strongest useful gold that fits cleanly,
6. easiest small repair with high leverage.

Park the rest.

### Phase 7 - Save Decision

After sorting, choose one clean save path:

- save mule report as source ore,
- save an intake summary,
- revise the lens as Soft Suit only,
- create a separate test card,
- park all findings,
- reject the lens,
- prepare a separate promotion-review candidate,
- do nothing and wait.

No installation without proof.

No promotion inside intake.

## Output Shape After Mule Returns

The intake result should include:

- mule report received,
- report source/citation quality,
- aspect coverage result,
- strongest criticism,
- strongest gold,
- rejected ideas,
- source-ore items,
- adoption candidates,
- adaptation candidates,
- missing proof,
- neighbor conflicts,
- top boss,
- parked list,
- one clean next move.

## Pass Condition

This dock passes when the mule report can be received without contaminating doctrine, active guides, proof standards, runtime lanes, or promotion state.

## Failure Condition

This dock fails if the mule report is pasted straight into authority, if outside research becomes doctrine without routing, if many ideas are installed at once, or if critique causes uncontrolled rewriting.

## Current Next Move

Wait for the mule report.

When it returns, freeze it and sort through this dock before making any project change.
