## What was missing before

The prior Intake Gate plan was strong on sorting, chunking, piles, decision tables, ledger fit, map routing, proof, and Final Judge.

It was weaker on helper readiness.

A future helper needs more than sorted chunks. It needs orientation, context carry, capability limits, a handoff, readiness review, feedback, balance, escalation, and a repeatable route.

## Cross-exam against house rules

No king gate:
Intake still does not decide final destiny alone.

No broad crawler:
Selected object only.

Misc Drawer fit:
Intake is the read/classify/route processor. Misc Drawer remains temporary mixed-material custody.

Code Gate fit:
Intake routes code to Code Gate. Intake does not execute code.

Flight Recorder fit:
Each meaningful helper action should leave event, proof, ledger, reverse summary, and handoff.

Key/Ledger/Map Driver fit:
Keys identify, ledgers remember, maps route, proof closes.

Deep work checkpoint fit:
Large work uses named batches and clean stop states.

## Main risks

Too many gates:
Mitigation: the gate stack is a map, not always-on ceremony. Use proportional depth.

Dirty labels:
Mitigation: use clean support terms and avoid loaded words at the front.

Pseudo-automation:
Mitigation: no automation; no broad crawler; manual test first.

Gate overreach:
Mitigation: Final Judge and role fidelity checks.

Bloat:
Mitigation: backpressure, carry key not furniture, one next action.

## Required proof before helper scripting

One manual intake test must prove:

- every chunk has a fate;
- every pile has a route;
- every route has proof or missing-proof label;
- every helper-facing next action has context/capability/stop/handoff;
- code is routed but not run;
- no broad crawl;
- no save;
- one next action.
