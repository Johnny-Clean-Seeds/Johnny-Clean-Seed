2. FRONT-GATE RULE

Rule name: POWERSHELL_FRONT_GATE_NO_DEFAULT_BYPASS_RULE

Rule:

Do not use `-ExecutionPolicy Bypass` by default for generated helpers, repair scripts, save scripts, test scripts, or Code Gate launches.

Meaning:

PowerShell execution policy is the front script-start gate. It decides whether PowerShell will run `.ps1` scripts and whether signing or restrictions apply.

`-ExecutionPolicy Bypass` tells the launched PowerShell process not to enforce that script-start policy for that process. It skips warnings/prompts from that policy.

Why this matters:

Even if execution policy is not a complete security system, it is still a protective layer against accidental script execution. During helper-trust testing, every layer matters. A helper-proving lane must not begin by skipping a gate.

First try normal launch.

If normal launch fails, read the exact error.

`Communicator` — sends only tiny rope-state pings when allowed or when user interrupts.
`Judge` — checks B-close: object, scope, sources, boundaries, limitations, next move.

This is important: the assistant should hold all four roles internally, but not let any one role take over. Worker cannot go silent forever. Communicator cannot flood chat. Scribe cannot turn checkpoints into a second giant file. Judge cannot claim done before the file closes.

The harness starts with a **Work Seal** before deep work:

`File/Object:` exact output name.
`Output Lane:` chat answer, downloadable file, repo save candidate, or local-only note.
`User Delivery Rule:` file only / brief chat allowed / checkpoints allowed.
`Scope:` what this file must do.
`Forbidden Drift:` what it must not become.
`Chunk Map:` named sections.
`Source Map:` what kinds of sources will be used and why.
`Checkpoint Mode:` embedded in file, sidecar, or tiny chat pings.
`Close Condition:` what proves the file is ready.

That Work Seal is the first rope knot. Without it, a big task begins as fog.

Then the file gets **A-checkpoints**. These are not summaries. They are custody proof. Each checkpoint answers:

`Object still held?`
`Chunk completed?`
`Sources used?`
`What was kept?`
`What was parked?`
`What was dropped?`
`Drift blocked?`
`Next chunk?`

If chat silence is requested, these A-checkpoints go inside the file or sidecar. If you interrupt with “hold the rope,” I surface the current A-checkpoint in one line and continue.

The live carry rule becomes:

`CURRENT OBJECT + CURRENT CHUNK + CURRENT SOURCE SET + CURRENT BLOCKER + NEXT CHECKPOINT`

Everything else gets parked. This is the practical version of “carry the key, not the furniture.” The cockpit already supports that concept directly. 

The harness needs a **Rope Alarm Table**:

`hold the rope` = emit current object / chunk / next checkpoint / blocked drift.
`stay focused` = reduce scope to active chunk; park side branches.
`smoke break` = drop live clutter, re-lock object, resume from last checkpoint.
`send when ready` = no content dump; finish file; B-close only.
`make chunks/checkpoints` = create or expose chunk map immediately.
`use links` = source ledger required; no unsupported broad claims.

This is the real control language. It turns your phrases into process controls.

The harness also needs **B-close** at the end of the file:

`What this file is.`
`What it proves.`
`What it does not prove.`
`Sources used.`
`Parked material.`
`Risks or weak spots.`
`Next clean move.`
`Boundary held.`

This prevents a big file from becoming a blob. It makes the file close as a handled object.

The fuller design should include a **sidecar option**. Sometimes the main report should stay clean. In that case, checkpoints go into:

`<FILE_NAME>__ROPE_LEDGER.md`

That sidecar holds Work Seal, A-checkpoints, source map, parked ideas, and B-close. The main report remains readable. The sidecar proves custody.

This solves the “no chat but still need rope” conflict. The user gets quiet chat, but the work still has a visible trail.

`LONG_FILE_ROPE_LEDGER_TEMPLATE_V1`

Fields:

`A1_CHECKPOINT`

The rough spot log matters because the rough spot during the big file was not a script error. It was a **custody signal**:

`USER_ROPE_INTERVENTION / LONG INVISIBLE WORK / POSSIBLE DRIFT OR SILENCE / REANCHOR REQUIRED`

That should be logged once, not repeated ten times.

The rule should be:

If the user says “hold the rope” more than once during the same long file build, the assistant must create or update a rope ledger. If the user says it three times, the assistant must stop expanding and return to the Work Seal before continuing.

That is the missing safety catch.
