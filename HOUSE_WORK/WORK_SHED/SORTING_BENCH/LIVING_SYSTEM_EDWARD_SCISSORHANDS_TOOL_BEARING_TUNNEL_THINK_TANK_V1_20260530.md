The strongest carry lines are:

- URL / LEDGER / MAP / KEY = SCENT / STATE / RELATION / LANGUAGE.

## Raw Edward pass source

Edward pass result: the small chat source is not small because the system is simple. It is small because the system is now trying to behave like a **controlled living interface** instead of a file dump.

The hard cut is this:

`URL / LEDGER / MAP / KEY` is not just four items.

It means:

`SCENT / STATE / RELATION / LANGUAGE`

URL gives scent: where to enter and what is on the other side.
Ledger gives state: what is current, blocked, proven, stale, or next.
Map gives relation: what connects, what owns, what returns, what proves.
Key gives language: what labels mean, what moves are legal, and what must stop.

That is the front-door nervous system.

Current house proof: the repo README points to `START_HERE_CURRENT_HOUSE_LEDGER.md`, the start file points to the V1.4 front-door ledger and the URL/LEDGER/MAP/KEY chat source, and the chat-source file explicitly says the live shape is `URL -> LEDGER -> MAP -> KEY -> only needed tunnel`. ([GitHub][1])

The Edward cut says the old mistake would be “all rooms, all tools, all ideas, all inventions, all suits loaded at once.” That is wrong. That makes a monster pile. The stronger version is:

`ALL ROOMS AVAILABLE / ONE ROOM ACTIVE`

`ALL TOOLS AVAILABLE / ONE TOOLBELT WORN`

`ALL IDEAS AVAILABLE / ONE CURRENT FRAME`

`ALL PROOF PRESERVED / ONE PROOF POINTER CARRIED`

`ALL SUITS AVAILABLE / ONE SUIT WORN`

That matches information architecture: navigation should make information findable through labels, structure, and choices, not by dumping every possible path at the user. ([Nielsen Norman Group][2]) It also matches information scent: the visible entry point must give enough cue-value that the assistant can decide where to go without wandering. ([Nielsen Norman Group][3]) Progressive disclosure confirms the same principle from another angle: hide advanced or rarely used detail until it is needed, so the system is easier to use and less error-prone. ([Nielsen Norman Group][4])

So the house principle becomes:

`MAXIMUM REACH / MINIMUM CARRY`

The map can reach everything. Chat should not carry everything.

The “all rooms all tools” meaning should be cybernetic, not chaotic. Ashby’s Law says control needs enough response-variety to match the system’s variety; in house terms, that means the system needs many possible toolbelts, not one crude tool. ([BusinessBalls][5]) But that does not mean all tools fire at once. It means the front door can select the right response variety at the right time. That is the difference between a toolbox and a junk pile.

The hub toolbelt registry is already the right home for that. It separates hub kits instead of making one giant universal toolbag. The current house registry names the toolbelt/workbench/toolbag/backpack/locker distinction, which is exactly the right grammar: what is worn now, what is available nearby, what is task-specific, what is carried forward, and what stays behind. ([GitHub][6])

Edward chop-up across rooms:

Front Door is not a welcome mat. It is a scent gate. Its job is to prevent wrong entry.

Ledger is not a diary. It is a state instrument. Its job is to say current/proven/blocked/next.

Map is not a spiderweb. It is a relation grammar. Its job is to say why two things connect and what that connection authorizes.

Key is not a glossary. It is a permission grammar. Its job is to define legal movement.

Toolbelt is not decoration. It is method selection. Its job is to stop the wrong method from firing in the wrong room.

Proof room is not paperwork. It is trust custody. Its job is to stop fake pass.

Crime scene is not panic. It is root-cause sensing. Its job is to stop surface repair loops.

Chat Drop is not authority. It is live-load convenience. Its job is to reduce chat burden.

Creative Room is not random imagination. It is recombination after boundaries are known.

Grandmaster is not a king file. It is node/feeler/root/return judgment. Its job is to prevent unowned links and false authority.

The wildest but cleanest idea from this pass is:

`TOOL-BEARING TUNNELS ARE BEHAVIORAL LINKS`

A normal link says: go there.

A tool-bearing tunnel says: go there, wearing this toolbelt, for this reason, with this proof need, carrying this result, leaving this weight behind, and returning through this path.

That is closer to a living path than a hyperlink.

The old internet/hypertext idea supports this, but the house needs a stronger version. Vannevar Bush’s memex imagined a private file/library with associative trails that could be followed quickly and flexibly. ([The Atlantic][7]) The house version should not be just associative trails. It should be **qualified associative trails**: trails with proof, boundary, toolbelt, load state, and return trigger.

So the upgraded tunnel format should eventually be:

`SourceNode`

`RelationType`

`TargetNode`

`Direction`

`RequiredToolbelt`

`LoadShelf`

`ProofNeed`

`RouteCost`

`FailureRisk`

`CarryOutLine`

`LeaveBehind`

`ReturnPath`

`StopCondition`

That adds two important Edward cuts: `RouteCost` and `FailureRisk`.

Information foraging says people navigate by estimating value/cost from cues. ([Nielsen Norman Group][8]) House translation: a tunnel should tell the assistant whether it is cheap or expensive to open. A proof-history tunnel may be high-value but high-cost; a chat-load manifest tunnel is low-cost and should be first.

The second strong invention is:

`SCISSOR MODE / SEW MODE`

Scissor Mode cuts. Sew Mode reconnects.

Scissor Mode asks:

What is duplicate?
What is stale?
What is proof-only?
What is a surface symptom?
What is a dirty patch?
What is furniture, not key?
What is a wrong-room object?
What must be severed from live carry?

Sew Mode asks:

What owns it?
What proves it?
What relation type connects it?
What toolbelt travels it?
What key explains it?
What return path closes it?
What smoke break resets it?
What should become living rule?

Edward without Sew Mode is destruction. Sew Mode without Edward is bloat. The clean pattern is: cut hard, sew clean.

Morphological analysis fits this because it handles multi-dimensional, non-quantified complexity by laying out dimensions and testing combinations instead of reducing away “trivial” parts too early. ([Wikipedia][9]) House translation: when the problem is weird, do not just choose one method. Build a small morphology box:

Dimension 1: room.
Dimension 2: toolbelt.
Dimension 3: load state.
Dimension 4: proof need.
Dimension 5: relation type.
Dimension 6: risk.
Dimension 7: return path.

Then reject impossible combinations. Example: `Chat Drop + source authority` is invalid. `Proof receipt + live carry by default` is usually invalid. `Error family + no scratch-and-sniff` is invalid. `Tunnel + no toolbelt` is invalid.

TRIZ also fits because the house is full of contradictions. TRIZ is designed around resolving contradictions rather than accepting a tradeoff. ([TRIZ40][10]) The core contradiction here is:

Need broad reach, but less chat load.
Need many tools, but fewer active methods.
Need proof, but less proof furniture.
Need creativity, but more boundaries.
Need automation-like consistency, but no watcher/automation.
Need current source, but no stale file pile.

The solution is not compromise. It is separation by layer:

Broad reach lives in the map.
Low chat load lives in the manifest.
Many tools live in toolbelt registry.
One active method lives in the current hub.
Proof lives in proof history.
One proof pointer lives in chat.
Creativity lives in Creative Room.
Boundary lives in Key and Final Judge.

That is good systems design.

Sensemaking says data and frame adjust each other: the data fits the frame, and the frame changes to fit the data. ([garyklein][11]) House translation: the ledger is the current frame, but crime-scene evidence can force the frame to revise. That is exactly what happened in the save-route harness failures. The data said the problem was not one line; the frame changed to `UNSTABLE LOWER-LAYER SAVE ROUTE HARNESS`.

OODA gives the timing discipline: observe, orient, decide, act, then repeat under feedback. ([The Decision Lab][12]) House translation:

Observe: read URL/ledger/proof/status.
Orient: select hub/toolbelt/key.
Decide: choose tunnel or stop.
Act: run the smallest legal move.
Feedback: receipt/readback/crime scene.

Smoke break belongs between loops because without it, orientation smears. That was the clunky failure you caught.

Double-loop learning adds the deeper rule: if the same type of failure repeats, do not only change the action; change the governing assumption. ([EBSCO][13]) House translation: after V1.2, V1.3, V1.5, V1 harness failures, the fix was not another line edit. The governing assumption changed: save scripts need a lower-layer harness.

That is exactly what the saved Push-to-Shove rule means. The current house rule says repeated surface failures are cause-and-effect evidence, warns against surface patches, and requires freeze, evidence preservation, scratch-and-sniff, lower-layer cause naming, root mechanism repair, need-to-know helper updates, and proof before resuming. ([GitHub][14]) The save-route harness rule then names the repeated family and the mechanics the harness must own: string safety, automatic-variable guard, Git wrapper standard, probe/direct split, dirty-state recovery, staged-set verification, manifest/receipt order, and final proof. ([GitHub][15])

So the system now has a deeper rhythm:

`SCENT -> FRAME -> TOOL -> TUNNEL -> PROOF -> SMOKE BREAK -> LEARN`

The Zettelkasten/atomic-notes angle gives another useful cut: linked knowledge works because notes are atomic and connected, not because everything is stuffed into one note. ([Zettelkasten][16]) House translation: do not make one giant ledger of everything. Make atomic nodes, typed tunnels, and a front-door manifest. The key is not the notes alone; it is the typed relation.

ADR thinking gives the decision-record layer. ADRs capture decisions with context, rationale, tradeoffs, and consequences; a collection becomes a decision log. ([Architectural Decision Records][17]) House translation: whenever the house chooses a structural rule—URL/LEDGER/MAP/KEY, no tunnel without toolbelt, smoke break between waves, no surface patch—it should eventually have a decision record or ledger entry. Not every thought gets promoted, but every structural decision needs a reason and consequence.

Wardley mapping adds value/evolution. Components are mapped by user need and by evolution from uncertain/custom to commodity/stable. ([Wikipedia][18]) House translation: some pieces are still genesis/custom, like Creative Room “Edward mode.” Some are becoming productized, like the save-route harness. Some should become commodity, like `Invoke-Git -GitArgs`, staged-set guards, and crime-scene readback. The more a lower-layer mechanic repeats, the more it should become boring, standard, and reusable.

The Viable System Model angle says a surviving/adaptive system needs multiple interconnected regulatory layers; it is not one central file controlling everything. ([BusinessBalls][19]) House translation: Front Door, Grandmaster, Proof Room, Chat Drop, Creative Room, and Code Gate are different regulatory organs. The mistake is letting one organ pretend to be the whole body.

Now, the full Edward recombination:

`URL` is the nose.

It smells the correct entry. If the scent is weak, the assistant wanders.

`LEDGER` is the heartbeat.

It says current state. If the heartbeat is stale, every move is suspect.

`MAP` is the nervous system.

It routes signals through rooms. If the map overconnects, the system twitches everywhere.

`KEY` is the language center.

It defines meanings. If the key is fuzzy, words become fog.

`TOOLBELT` is the hand.

It decides what can be done now. If all tools are held at once, the hand drops everything.

`PROOF` is the skeleton.

It gives structure. If proof is missing, the body slumps.

`CRIME SCENE` is pain.

It signals damage. If pain is numbed with a surface patch, the injury grows.

`SMOKE BREAK` is breath.

It resets the cycle. Without breath, the system starts thrashing.

`CREATIVE ROOM` is mutation.

It generates options. Without boundary, mutation becomes tumor.

`FINAL JUDGE` is immune response.

It decides fit, block, park, save, or stop.

That metaphor is actually useful because it makes the system living without making it mystical. Every part has a job.

The biggest hidden risk now is **front-door overgrowth**. Because the front door is important, the temptation will be to make it explain everything. Do not. The front door must stay a doorway. It should smell right, point right, and stop wrong movement. The deeper explanation belongs behind the door.

Second hidden risk: **key inflation**. If the key gains too many labels, it becomes another unreadable doctrine pile. The key needs a small stable core and an overflow/experimental area.

Third hidden risk: **toolbelt bloat**. If each hub has too many tools, the assistant again carries too much. Toolbelt should be capped; workbench can hold more; locker holds the rest.

Fourth hidden risk: **proof drag**. Proof must exist, but proof should not ride live unless proof is the task. That is the whole difference between proof custody and proof furniture.

Fifth hidden risk: **creative bypass**. “Edward mode” could bypass proof if not bounded. It should always end by assigning outputs to ADOPT / ADAPT / PARK / BLOCK / PROOF / LEAVE-BE.

A cleaner internal command set could be:

`SCISSOR` — cut stale/duplicate/surface/dirty carry.

`SEW` — reconnect with typed tunnel/proof/key.

`SMELL` — evaluate scent and source freshness.

`BONE` — identify structural proof/receipt/commit.

`NERVE` — map signal path from front door to target.

`HAND` — choose toolbelt.

`BREATH` — smoke break reset.

`SCAR` — inspect repeated failure family.

`SEED` — mark living rule candidate.

`LOCK` — save only after proof.

That is “all rooms/tools” compressed into verbs.

The current house should not save all this immediately as doctrine. The best fit is a **Think Tank / Source Fit / Creative Room report**. It should be candidate support, not authority. The ledger is already done; this is meaning expansion and future tool grammar.
