## One-line root

The revise method failed because no pre-answer execution harness forced the house method to mount before synthesis.

The visible symptom was "I did not revise." The deeper disease was that stored rules, methods, proof, and house history entered the answer as content, but did not take control of the answer process.

## What happened

The user asked for a whole-project sorting method for the status/history/growth substrate and gave strong method triggers:

- build a plan;
- use all tools/resources;
- follow the house;
- bring back the list;
- tell the process;
- hidden pop quiz;
- find the root, not symptoms.

Expected process:

Raw list -> pause/review -> critique -> revise -> improve -> final list -> explain what changed.

Actual process:

A polished synthesis was produced directly.

Gap:

- no visible raw list;
- no critique pass;
- no revision pass;
- no "what changed";
- no proof that the final was better than the first version.

## Why this matters

This was not a content-only failure. It was an activation failure.

A rule, method, or house tool can exist and still fail if it does not fire at point of work. That is the same family as Rule Storage Mistaken For Rule Firing. The revise method was known, but not worn.

This creates clean-looking dirty milk: the answer may contain useful ingredients, but the operating path is not proven.

For a business-facing model, this is dangerous because polished output can hide missing process. Fake confidence is worse than rough draft, because it makes the system look organized while the growth substrate stays dirty.

3. Compression in the wrong place.

4. Outside-method magnet.

5. No pre-final self-check.

9. Untraced synthesis.

## MTER method

MTER is not a new king. It is a small pre-answer harness for complex work.

### 1. MOUNT

Before answering, mount the operating stack.

Questions:

- What did the user command?
- Is this chat-only, local save, file work, mule work, report, or artifact?
- Which house method applies?
- What parent boss does this sit under?
- What is blocked?
- What proof must be visible?

For the revise failure, the mounted stack should have been:

- chat-only;
- complex sorting-method design;
- Parent Boss: Rule Activation / Work-Order Control;
- Neighbor Boss: Recursive Carryover / Point-of-Work Link Failure;
- business risk: dirty growth substrate;
- required method: revise loop;
- blocked: local save, dashboard, doctrine, active guide rewrite, CURRENT_TRUTH_INDEX rewrite, mule pass.

### 2. TRACE

Split the task into layers:

- content layer: design a sorting method for the status/history/growth substrate;
- process layer: use the revise method;
- proof layer: cite/stack house history and show this is a repeated pattern;
- business layer: protect trust, handoff clarity, product story, and inspectable growth;
- boundary layer: no forced tests, no dashboard, no doctrine rewrite, no local save unless selected.

### 3. EXPOSE

Expose the failure before fixing it.

Surface symptom:
Revise method did not fire.

Root:
No execution harness mounted before synthesis.

Pattern:
Stored method entered answer content but did not become answer control.

Danger:
Polished output hid missing process.

### 4. REVISE

Run the actual revise loop.

- Raw pile: collect all ideas, methods, risks, proof, and triggers.
- Group: sort by parent boss, child symptom, proof state, business risk, boundary, and trigger.
- Cut/merge: remove duplicates, merge overlapping methods, reject bad options.
- Upgrade: strengthen the root, replace weak names, add rejection ledger, add pass condition.
- Final: return the improved list.

### 5. DELTA

State what changed.

For this case:

- The root changed from "missed revise method" to "missing execution harness."
- The method changed from EXPOSE-only to MOUNT-TRACE-EXPOSE-REVISE.
- The fix changed from "remember to revise" to "mount the house method before synthesis."
- The output standard changed from "good answer" to "inspectable construction."

### 6. BOUNDARY

Hold the boundary.

- No local save unless selected.
- No forced test.
- No doctrine rewrite.
- No active guide rewrite.
- No CURRENT_TRUTH_INDEX rewrite.
- No dashboard.
- No runtime.
- No automation.
- No knowledge graph.
- No full lesson index.
- No mule pass by inertia.
- No turning MTER into king.
