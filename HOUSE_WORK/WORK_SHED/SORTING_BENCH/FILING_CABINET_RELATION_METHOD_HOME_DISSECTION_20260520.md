## Why this exists

Two new ideas were exposed during live work:

1. Filing Cabinet / Reusable Workflow Skeletons.
2. Relation Method / Across-the-Board Relation Lens.

The first visible case was repeated deep-scan PowerShell generation, but the user corrected that this is broader than deep scans.

The parent issue is not only code repetition. The parent issue is repeated workflow generation without retrieving proven skeletons and without mapping how a method touches neighboring lanes.

## Idea 1 — Filing Cabinet / Reusable Workflow Skeletons

Meaning:
Repeated local workflows should not be regenerated from scratch when a proven skeleton exists.

Use prior skeletons for:

- deep scans;
- intake scripts;
- save/lock scripts;
- proof receipts;
- mule handoffs;
- kickoff cards;
- file-placement validators;
- artifact self-checks;
- readback scripts;
- test runners;
- status checks;
- hash/checksum blocks;
- restore/cleanup guards.

Default movement:

1. Find closest prior skeleton/tool card.
2. Tweak only task-specific inputs.
3. Run proof.
4. If the tweak is broadly useful, improve the skeleton instead of creating another cousin script.

## Idea 2 — Relation Method / Across-the-Board Relation Lens

Meaning:
Do not inspect a problem only in its home lane when the issue touches many lanes.

Map relations across:

- source;
- trigger;
- lane owner;
- proof;
- TODO;
- artifact;
- handoff;
- PowerShell;
- mule;
- memory;
- status/anchor;
- active guide boundary;
- next-start effect.

This is near pattern recognition, but not identical.

Pattern recognition asks:
Have I seen this shape before?

Relation Method asks:
What other lanes does this touch, and does the real issue live across the board?

## Why this matters

The filing cabinet point was first seen through repeated deep-scan code generation, but relation mapping shows it also touches:

- compute/context waste;
- copy/paste risk;
- PowerShell proof consistency;
- false-success hazards;
- Gear Rack;
- artifact checks;
- mule handoff packets;
- receipts;
- future start surfaces.

Artifact self-check:
artifact creation -> carryover latch -> TODO -> memory boundary -> next-start improvement.

Broad status search:
retrieval weight -> start-surface fog -> targeted section search.
