### Failure 3 — It proposed too many files too early


Only after those pass fresh-reader testing should the next file be chosen.

### Failure 4 — It blurred doc types

The packet knew the docs had different jobs, but it did not enforce the split.

A house path fails when one file tries to be:

```text
entry map
explanation
reference
proof index
runbook
decision record
manifest
```

That creates burden.

### Failure 5 — It used “scent” too loosely

A title is not enough.

A sign must tell a specific reader whether the path answers a specific question.

Every path sign needs:

```text
target reader
target question
what this file answers
what this file does not answer
next pointer
proof pointer
```

### Failure 6 — It left custody too vague

“Source/custody” is not enough.

A stronger house object needs a fixed mini-shape so each future file does not improvise custody in a new format.

### Failure 7 — It underbuilt the ADR threshold

Decision records are useful only for important forks.

A decision record for every tiny move becomes another pile.

## 5. Missing Gates To Add

### 5.1 Doc-Type Split Gate

Every proposed file must declare one doc type.

Allowed first types:

```text
NAVIGATION / CURRENT STATE
REFERENCE
EXPLANATION
HOW-TO / PLAYBOOK
INDEX / CATALOG
RECEIPT / PROOF
DECISION RECORD
MANIFEST / SIGN
```

Rule:

```text
One file may contain light supporting context,
but it must not try to do every job.
```

Mapping:

```text
ASSISTANT_ENTRY_PATH_V1.md
  -> NAVIGATION / CURRENT STATE

CONTROLLED_VOCABULARY_V1.md
  -> REFERENCE

HOUSE_WAYFINDING_PROOF_MEMORY_SYSTEM_V1.md
  -> EXPLANATION

HELPER_REGISTRY_REPAIR_PLAN_V1.md
  -> HOW-TO / PLAYBOOK

PROOF_INDEX_V1.md
  -> INDEX / CATALOG

FOLDER_MANIFEST_CANDIDATES_V1.md
  -> INDEX / CATALOG / PLANNING

CLEAN_WRAP_GATE_V1.md
  -> GATE / REVIEW TOOL
```

Problem fixed:

```text
Stops one mega-file from becoming new dead chat.
```

### 5.2 Target-Reader Scent Gate

Every path/sign/manifest must say who it is for and what question it answers.

Mini-shape:

```text
Target reader:
Target question:
This file answers:
This file does not answer:
Best next file:
Proof pointer:
Stale trigger:
```

Examples:

```text
Target reader: fresh assistant
Target question: Where do I start right now?
This file answers: live lane, parked branches, proof status, next move.
This file does not answer: full doctrine or all historical decisions.
Best next file: CONTROLLED_VOCABULARY_V1.md
Proof pointer: latest proof/status card.
Stale trigger: next commit changing current lane.
```

Problem fixed:

```text
A good title is not enough.
The file must smell right for a specific question.
```

### 5.3 Fixed Custody Mini-Shape

Every important object uses the same custody shape.

```text
Object:
Entity/source:
Activity/action:
Actor/tool:
Inputs:
Outputs:
Receipt/proof:
Hash/commit:
Trust state:
Current status:
What must not be assumed:
Drift trigger:
```

House translation of provenance:

```text
object + action + actor/tool + inputs + outputs + receipt + trust state
```

Problem fixed:

```text
No vague “source/custody” field that every file invents differently.
```

### 5.4 ADR Threshold Gate

Do not make decision records for every tiny move.

Create ADR-style records only when at least one is true:

```text
the choice changes house structure;
the choice affects proof/safety/quality;
the choice is hard to reverse;
the choice supersedes an old route;
the choice keeps getting relitigated;
the choice creates a new durable path, not just a one-off action;
the choice affects local-only vs Git-durable boundaries.
```

ADR mini-shape:

```text
Decision:
Status:
Date:
Context:
Options considered:
Chosen route:
Rejected routes:
Why:
Consequences:
Proof needed:
Supersedes:
Superseded by:
```

Problem fixed:

```text
Decision records stop old forks from reopening,
but too many records become another burden.
```

### 5.5 Freshness / Drift Gate

Every path file needs a drift condition.

```text
Last verified:
Proof pointer:
Stale when:
Review trigger:
Owner/lane:
Superseded by:
```

Examples:

```text
ASSISTANT_ENTRY_PATH becomes stale when:
- current lane changes;
- new proof is pushed;
- a parked branch is resumed;
- helper registry counts change;
- Git HEAD/origin changes after a save route.

CONTROLLED_VOCABULARY becomes stale when:
- new status word is introduced;
- a word is used inconsistently;
- Code Gate/warning states change.
```

Problem fixed:

```text
A good map can rot.
A stale map is worse than no map when readers trust it.
```

### 5.6 Burden Budget Gate

Every new path file must prove it reduces more burden than it adds.

Burden budget questions:

```text
What repeated explanation does this remove?
What broad scan does this avoid?
What old chat does this replace?
What giant helper/code does this prevent?
What future question can it answer in under one minute?
What new maintenance burden does it create?
Who must update it, and when?
```

Verdicts:

```text
REDUCES BURDEN
REDUCES BURDEN WITH MAINTENANCE COST
UNCLEAR
ADDS BURDEN
```

Problem fixed:

```text
Prevents “helpful” maps from becoming new furniture.
```

Test question:

```text
Using only CONTROLLED_VOCABULARY_V1.md and ASSISTANT_ENTRY_PATH_V1.md,
can a fresh reader answer:
```

Checklist:

```text
1. What is the main goal?
2. What lane is live?
3. What is parked?
4. What is proven?
5. What is NOT PASS?
6. What helper/status map matters next?
7. What should not be touched?
8. What is the next clean move?
9. What proof would close that move?
10. What does PASS mean here?
```

Pass standard:

```text
8/10 answers cleanly from the two files
no broad scan needed
no old chat needed
no helper code needed
no doctrine claim
```

Fail standard:

```text
reader needs old chat;
reader confuses PASS with job proof;
reader cannot find current lane;
reader cannot name the next move;
reader cannot tell parked from live;
reader needs broad scan to orient.
```

If fail:

```text
repair the two files before creating any new path file.
```
