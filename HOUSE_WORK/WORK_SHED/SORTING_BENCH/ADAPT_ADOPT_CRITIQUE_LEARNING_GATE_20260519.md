# Adapt / Adopt Critique Learning Gate

Date: 2026-05-19.
State: Work Shed gate.
Authority: sorting / learning gate.
Starting base: main @ 14c762b.

## Job

Turn critiques and house material into tested learning without creating a rule swamp.

## Gate Procedure

### 1. Intake Boundary

Name the input type:

- user instruction,
- critique,
- outside-review report,
- proof output,
- failed run,
- successful run,
- note/idea/source ore,
- handy-bag item,
- bitstring/sorting material,
- tool/check script,
- parked candidate.

State whether it has command authority, source value, proof value, or no current value.

### 2. Related Material Pull

Before judging the input alone, ask what the house already knows:

- Is there proof history for the same failure?
- Is there a prior receipt?
- Is there a bitstring/sorting review with useful dimensions?
- Is there a handy-bag lens for this issue?
- Is there a parked candidate that already solves part of it?
- Is there a prior rejection reason?
- Is there a neighbor file or lane that owns the job?

### 3. Atomic Extraction

Split the input into small pieces:

- claims,
- triggers,
- patterns,
- warnings,
- proof gaps,
- wording fixes,
- tool ideas,
- lane fixes,
- rejects.

### 4. Pattern Recognition

Ask:

- What pattern is actually useful here?
- What failure does it prevent?
- What does it make easier to see?
- What would break if copied blindly?

### 5. Sort

For each piece, choose one:

- ADOPT,
- ADAPT,
- PARK,
- REJECT,
- TEST.

### 6. Lane Map

Name the destination:

- BRAIN_LEARNING,
- Work Shed / Sorting Bench,
- Proof History,
- Source Ore,
- Handy Bag,
- Incoming File Parking,
- Active guide candidate only,
- PC-only/private hold,
- rejected-with-reason.

### 7. Fit / Neighbor Check

Check:

- core fit,
- authority boundary,
- proof need,
- current load,
- neighbor overlap,
- promotion risk,
- whether this is a parent boss or child fix.

### 8. Small Install

Install the smallest behavior-changing object possible.

Prefer:

- one card,
- one gate,
- one template,
- one hot rule,
- one test.

Do not install a giant framework from one critique.

### 9. Proof Use

Test against:

- a recent failure,
- a live next task,
- a known false-pass case,
- an existing proof gap.

### 10. Lock / Save

Only after fit and proof, save into Mr.Kleen with:

- artifact,
- receipt,
- status update,
- commit,
- push,
- clean final status.

## Boundary

This gate does not itself promote doctrine.
It does not replace proof.
It does not make critiques command authority.
It does not make newest report win.
It prevents blind mimicry.
