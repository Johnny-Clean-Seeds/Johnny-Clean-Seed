## 1. Main Goal

Reduce consumption burden by making the house carry more of its own orientation, proof, rules, maps, and next-action guidance.

The house should become easier to read without rereading the whole house.

Clean short form:

```text
Build a house that explains itself.
```

More exact form:

```text
Move the thinking burden out of live chat and into the house,
using paths, signs, maps, receipts, proof, and clean next actions.
```

This is not about making more helpers.

Helpers, registries, manifests, cards, receipts, and reports are only useful when they serve the main goal:

```text
future workers / assistants / helpers / user
can orient faster,
trust more safely,
write less giant code,
scan less,
repeat less,
and make the next clean move without chewing old chat.
```

---

## 2. What This Is Not

This is not a new door.

Blocked:

```text
No MCP.
No service layer.
No exposed protocol.
No assistant portal.
No helper in every folder.
No crawler that eats the house.
No “make everything smart.”
No new throne gate.
No doctrine promotion.
```

House wording:

```text
Open a path, not a door.
```

A path is:

```text
a readable file,
a map,
a proof pointer,
a manifest,
a registry entry,
a receipt,
a next-action card,
a warning label,
a status marker.
```

A door is:

```text
a new moving service,
a protocol,
a broad automation surface,
a helper swarm,
a system that can cause mayhem.
```

This concept chooses paths.

## 4. External Concept Wash

These are not authorities over the house. They are useful outside concepts translated into house language.

### 4.1 Wayfinding

Outside idea:

People navigate using paths, landmarks, regions, signs, sight lines, and decision points.

House translation:

```text
Do not label every object equally.
Place signs at decision points.
Use landmarks for orientation.
Keep paths structured.
Make the next turn obvious.
```

House objects:

```text
ASSISTANT_ENTRY_PATH.md
folder MANIFEST.md
proof index
helper registry
current lane card
next-action card
status markers
```

### 4.2 Information Architecture

Outside idea:

Information architecture organizes, labels, navigates, searches, and uses metadata so people can find and manage information.

House translation:

```text
The house needs:
organization,
labels,
navigation,
search hooks,
metadata,
and consistent categories.
```

House objects:

```text
controlled vocabulary
folder purpose maps
registry fields
proof/status indexes
source custody fields
```

### 4.3 Information Scent

Outside idea:

A person chooses where to go based on cues that suggest what they will find there.

House translation:

```text
Every important file/folder/path should smell like what it contains.
Weak scent causes backtracking, rereading, broad scans, and giant helper code.
```

House objects:

```text
short purpose lines
status badges
“when to use / when not to use”
related-file links
next-action lines
```

### 4.4 Cognitive Load Reduction

Outside idea:

Some load belongs to the task; extra load comes from bad structure, noise, unclear presentation, and irrelevant material.

House translation:

```text
The work can be complex.
The path should not add extra burden.
```

Reduce:

```text
duplicate logs,
stale cards,
unclear names,
overlong scripts,
ambiguous PASS,
missing proof,
old chat furniture.
```

Increase:

```text
chunking,
labels,
maps,
proof pointers,
clean next steps,
stable status words.
```

### 4.5 Records Management / File Plan

Outside idea:

A file plan says what records exist, where they live, how they are identified, and how they are indexed for retrieval.

House translation:

```text
The house needs a file plan for important project objects.
```

Not every file needs ceremony.

Important objects need:

```text
class,
home,
index path,
custody,
retention/disposition,
status,
proof pointer.
```

### 4.6 Taxonomy + Controlled Vocabulary

Outside idea:

Controlled terms reduce confusion and improve retrieval.

House translation:

```text
Same words must mean same things.
```

Candidate vocabulary:

```text
PASS

NOT PASS
BLOCKED
PARKED
STALE
DEAD
LIVE
PROOF
RECEIPT

CANDIDATE

```

Rule:

```text
Do not invent a new label when an existing one fits.
If a new label is needed, define it before using it.
```

### 4.7 Provenance / Custody

Outside idea:

Trust depends on knowing what produced an object, what was used, who/what changed it, and what proof exists.

House translation:

```text
Every important object needs custody:
source -> action -> output -> receipt -> current trust level.
```

Minimum custody fields:

```text
source path
created by / tool
date
input files
action taken
output files
proof pointer
current status
what must not be assumed
```

### 4.8 Data Catalog + Lineage

Outside idea:

A catalog helps people find assets and understand context; lineage shows where data came from and how it moved or changed.

House translation:

```text
Helper registry = tool catalog.
Proof index = outcome catalog.
Source custody = lineage.
Folder map = asset map.
```

Important distinction:

```text
Catalog tells what exists.
Lineage tells how it got here.
Proof tells whether it can be trusted.
```

### 4.9 Decision Records

Outside idea:

A decision record captures a significant decision, context, alternatives, decision, and consequences.

House translation:

```text
When the house chooses a route, capture why.
Do not force future readers to relitigate old forks.
```

Decision record fields:

```text
decision
date
status
context
options considered
chosen route
why
consequences
proof needed
superseded by
```

### 4.10 Runbooks / Playbooks / SOPs

Outside idea:

Different operating documents serve different tasks:
routine repeatable steps, tactical responses, and standard process.

House translation:

```text
Do not make one giant “how to” for everything.
Split by job.
```

House versions:

```text
SOP = ordinary house rule / stable process
runbook = exact repeatable command path
playbook = response ladder for variable situations
```

Examples:

```text
Code Gate runbook
Git save runbook
stuck process playbook
helper failure playbook
porch/drop intake SOP
```

### 4.11 Knowledge Mapping

Outside idea:

A knowledge map shows where knowledge lives, how it flows, and where gaps exist.

House translation:

```text
The house needs maps of:
what is known,
where it lives,
what depends on what,
what is missing,
what is stale.
```

This is not a broad graph for everything.

It is a useful map at pain points.

### 4.12 Digital Garden / Atomic Notes

Outside idea:

Useful knowledge grows through linked notes; smaller single-purpose notes are easier to reuse than giant mixed notes.

House translation:

```text
Use small cards for single ideas.
Link them through indexes and maps.
Do not create giant mixed cards that become another dead chat.
```

Guardrail:

```text
A garden without pruning becomes a jungle.
Every note needs status and connection.
```

### 4.13 Documentation Drift / Freshness

Outside idea:

Documents can become wrong when the world changes.

House translation:

```text
A map can rot.
A registry can rot.
A proof claim can rot.
```

Every important file needs one of:

```text
last verified date
proof pointer
drift trigger
owner/lane
stale marker
superseded-by pointer
```

## 8. Clean Object Standards

### 8.1 Clean File Standard

A project file is clean when it has:

```text
title
purpose
status
authority level
scope
source/custody
related files
proof pointer
next action
what not to assume
stale/dead/parked marker if needed
```

### 8.2 Clean Helper Standard

A helper is clean when the registry can say:

```text
name
path
lane
duty
input contract
output contract
risk boundary
Code Gate status
job proof status
last verified
local-only or durable
when to use
when not to use
failure ladder
related proof/report
next action
```

### 8.3 Clean Folder Standard

A folder is clean when a future reader can know:

```text
what belongs here
what does not belong here
key files
entry file
status/proof pointer
whether this folder is source, tool, proof, map, inbox, archive, or workspace
what should never be changed here without explicit route
```

### 8.4 Clean Decision Standard

A decision is clean when it captures:

```text
question
context
options
choice
why
consequences
proof needed
status
superseded-by pointer if changed
```

### 8.5 Clean Proof Standard

A proof claim is clean when it has:

```text
object proven
proof file path
date/time
gate result
job result
warnings classified
outputs verified
commit/hash if saved
remaining gaps
```

### 8.6 Clean Next Action Standard

A next action is clean when it states:

```text
one object
one lane
one command or judgment
expected output
stop condition
what not to do
```
