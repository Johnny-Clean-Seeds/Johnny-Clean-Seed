# Handy Idea Bag Index

Status: SUPPORT TOOL / WORK SHED
Authority: support only; not command authority
Current base before package: main @ 61dc4aa

## Purpose

The Handy Idea Bag is a helper room the assistant may open during work to find fitted patterns, tools, methods, and candidate fixes.

It is not a mandatory step before every action.

It is not a random idea pile.

It is not doctrine.

It is meant to save motion, prevent patch loops, and help the assistant choose better fitted moves when the active issue needs more than the first visible fix.

## Core Rule

The Idea Bag is not a step.

It is a shortcut when the task needs better choice.

Open the bag only when doing so is likely to reduce steps, prevent bad cycles, improve fit, or find a better buried pattern.

If opening the bag adds drag without better movement, adjust or stop using that route.

## When To Open The Bag

Open the Handy Idea Bag when one or more bag bells ring:

- repeated issue
- unclear fix
- multiple plausible fixes
- high neighbor impact
- proof-gate risk
- patch-loop smell
- user asks why something is not firing
- assistant notices a missing behavior
- task touches autonomy, source, proof, authority, lifecycle, status, anchor, or word keys
- a known pattern may exist elsewhere
- a current move could create more steps if chosen poorly

Do not open the bag for:

- simple obvious actions
- already-selected mechanical saves
- clean status checks
- direct small fixes with low risk
- tasks where checking the bag adds more fog than movement

## Bag Bell Test

Open the bag when choosing badly would cost more than checking the bag.

Do not open it by habit.

## Front Working Order

1. Name the active issue.

What problem am I actually messing with?

2. Identify the lane.

Possible lanes:
- proof
- wording
- autonomy
- lifecycle
- capture
- source ore
- guide
- status
- anchor
- tool
- template
- user workflow

3. Identify the current state.

Possible states:
- raw
- captured
- support-only
- tested
- worn
- installed
- promoted
- parked
- blocked
- rejected-with-reason

4. Check authority.

Can this idea command action, or is it source/support only?

5. Check whether the bag should open.

If no bag bell rings, do the direct clean move.

6. If the bag opens, use fit-slot retrieval.

Do not grab random ideas.

7. Compare candidates by fit.

Use:
- active issue fit
- lane fit
- neighbor impact
- proof need
- risk
- simplicity
- long-term placement value

8. Adapt, do not blindly copy.

Shrink, split, or reshape the pattern for this room.

9. Prove in the pudding.

The fix is not right because it sounds clever.

The fix is right only if the result shows cleaner movement, fewer leaks, less fog, and a stronger gate.

10. Save or park.

If proven and project-relevant, save.

If useful but not ready, park with a return trigger.

## Fit-Slot Retrieval

When the bag opens, search named slots tied to the active issue.

Do not pull random alternatives.

### Same-Failure Pattern

Have we seen this exact kind of failure before?

Use when:
- the issue repeated
- a previous repair pattern may fit
- the same leak is wearing a new mask

### Same-Lane Pattern

What has worked before in this lane?

Use when:
- proof needs a proof pattern
- wording needs a word-key pattern
- autonomy needs an autonomy pattern
- capture needs a capture pattern
- lifecycle needs a lifecycle pattern

### Neighbor-Lane Pattern

Is another lane solving the same shape of problem cleanly?

Use when:
- a neighboring room has a working companion
- the current room is lonely
- the current room needs a smaller version of a neighbor method

### Proof-Gate Pattern

Is this a case where a proof/check/receipt method should be copied?

Use when:
- PASS/FAIL is involved
- hashes/receipts/checks are involved
- false PASS risk exists
- the result must be proven before save

### Smaller-Version Pattern

Is there a big rule that needs a smaller fitted version here?

Use when:
- the parent rule is good but too large
- the room needs a child helper
- the method needs to be wearable in a smaller place

### Prior-Success Pattern

What recently worked and could be adapted?

Use when:
- a recent fix passed cleanly
- the current issue has the same shape
- the pattern can bless the current lane without stealing its job

### Blocked / Rejected-With-Reason Pattern

Did we reject or block a similar idea before, and why?

Use when:
- a tempting fix may already have a known failure
- old rejection reasons may prevent repeated mistakes
- a parked idea may now have the right companion

## Candidate Requirements

A candidate is not usable until it answers:

- What active issue does this fix?
- Which fit slot produced it?
- Why does that slot fit?
- What neighbor lanes does it touch?
- What proof would show it worked?
- What would make it fail?

If no fit-slot candidate appears, say so and proceed with the current best fix plus proof.

## Bad Wording Rejected

Rejected phrase:
open the bag and find 2-3 alternatives

Reason:
That is too vague. It sounds like grabbing random letters from a Scrabble bag.

Replacement:
Open the bag only when a bag bell rings, choose the relevant fit slots, retrieve candidates by issue shape, compare fit, adapt, and prove.

## Guardrails

- Do not use the bag by habit.
- Do not make random moves.
- Do not use arbitrary counts as a method.
- Do not install a candidate because it sounds good.
- Do not let the bag become a new boss.
- Do not use the bag to avoid direct action when the next move is already clean.
- Do not treat source/support ideas as command authority.

## Verdict

The Handy Idea Bag is ready as a support tool.

It needs companion gates:
- No Random Moves / Calculation Before Candidate
- Problem-To-Growth Repair Gate

The proof is in the pudding.
