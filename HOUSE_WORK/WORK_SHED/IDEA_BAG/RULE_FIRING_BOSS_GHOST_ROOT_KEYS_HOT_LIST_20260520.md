# Rule-Firing / Boss-Ghost Root Keys Hot List

Date: 2026-05-20
Lane: HOUSE_WORK / WORK_SHED / IDEA_BAG
Status: hot list / root keys; not doctrine

## Purpose

These are short operational keys. They are not slogans unless they trigger behavior.

Each key must help the assistant route, stop, sort, prove, or place work.

## Root keys

### Control state first

Trigger:
Before any nontrivial task.

Meaning:
Find the state that controls whether the task is allowed before acting.

Stops:
surface-task mimicry.

Allows:
clean lane/rule selection.

### Worker present, hands off

Trigger:
A mule/worker is active or may be active.

Meaning:
No house-touching work unless user explicitly pauses/cancels worker or requests read-only support.

Stops:
side fixes during worker custody.

Allows:
non-invasive support.

### Rule exists is not rule fired

Trigger:
Any time a rule is cited after a failure.

Meaning:
Stored rule is not enough. It must fire at the point of work.

Stops:
fake confidence from memory/rules files.

Allows:
pre-action rule confirmation.

### Surface task is not always controlling task

Trigger:
User asks for a task that might be blocked by state.

Meaning:
The visible request may be lower priority than custody, dirty tree, proof gap, or authority boundary.

Stops:
mimic-only obedience.

Allows:
suit-based routing.

### Boss first, ghosts under it

Trigger:
Large lists, TODO, critique, recovered chat, mule output, idea floods.

Meaning:
Find parent bosses before fighting symptoms.

Stops:
flat issue lists and ghost chasing.

Allows:
ordered idea capture.

### Complete means proven to this task's standard

Trigger:
Before calling anything done.

Meaning:
Each task type has its own completion standard.

Stops:
done equals PASS drift.

Allows:
clear proof.

### Use the tools that fit

Trigger:
When user says use all tools or when evidence is needed.

Meaning:
Use relevant tools, not everything literally.

Stops:
over-crawling and under-tooling.

Allows:
task-scoped tool use.

### Open intake, bounded authority

Trigger:
New ideas, concepts, links, mule output, old chat, screenshots.

Meaning:
Many ideas may enter, but each must get a lane before it can command.

Stops:
idea loss and authority confusion.

Allows:
big creative intake without dirty milk.

### Mule sharpens, house decides

Trigger:
Mule kickoff, mule return, outside review.

Meaning:
Mule can critique and return material. The house/user decide what becomes active.

Stops:
mule supervisor drift.

Allows:
bounded outside review.

### No side fixes under custody

Trigger:
Worker active, dirty tree, selected active task, or narrow save.

Meaning:
Do not add helpful extras while another controlling state is active.

Stops:
scope creep.

Allows:
clean stopping.

### Current task first, later issue parked

Trigger:
New issue appears mid-task.

Meaning:
If it does not block current task, park it.

Stops:
scatter.

Allows:
capture without derailment.

### Claim-scoped proof

Trigger:
Receipts, PASS claims, proof history.

Meaning:
Proof supports specific claims only.

Stops:
global PASS overreach.

Allows:
supersession and partial-proof notes.

## Use

Before work, pick only the relevant root keys for the task.

Do not recite the whole list unless the task is about the list.
