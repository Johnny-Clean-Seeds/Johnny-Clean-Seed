# Problem-To-Growth Repair Gate

Status: SUPPORT GATE
Authority: support only; not command authority
Current base before package: main @ 61dc4aa

## Purpose

When a problem appears, treat it as a chance to learn and grow, not merely as a nuisance.

Do not only patch the symptom.

Ask what the problem revealed and what needs to change so it does not recur.

## Core Rule

Problem happened -> growth chance -> diagnose why it was allowed -> install or park the smallest clean prevention.

## Required Questions

When a problem appears, ask:

- What did this problem reveal?
- What needs to change so it does not recur?
- Is this a real boss?
- Is this a ghost boss?
- Is it a repeated boss wearing a new mask?
- Is a neighbor acting out?
- Is a neighbor missing?
- Is a rule lonely and needing a friend?
- Is a parent rule missing a child helper?
- Is the trigger too vague?
- Is the proof gate weak?
- Is the issue in the wrong lane?
- Is the assistant making a random move instead of a calculated move?
- Is the real fix different from the symptom patch?

## Diagnosis Classes

### Symptom Patch

The issue is small and direct. Fix it without overbuilding.

### Real Boss

The issue reveals a true parent weakness that needs structured work.

### Ghost Boss

The issue looks important but is only a symptom or noise. Park it or collapse it under the real boss.

### Neighbor Conflict

A neighboring rule, lane, or boundary is disturbing the current fix.

### Lonely Rule

A rule exists but needs a companion rule to fire cleanly.

### Missing Child Helper

A parent rule is too large and needs a small helper/template/checker.

### Missing Trigger

The rule exists but does not fire at the right time.

### Weak Proof Gate

The system can still write PASS or move forward without enough proof.

### Wrong Lane

The idea is useful but placed in the wrong room/state/authority.

### Random-Move Leak

The assistant chose or proposed a vague move without calculation.

## Repair Options

Choose the smallest clean repair:

- direct symptom patch
- trigger rewrite
- companion rule
- child helper
- proof gate
- lane move
- neighbor check
- boss capture
- ghost-boss park
- parent-boss design
- return trigger
- calculated candidate retrieval

## Current Example

Problem:
The assistant proposed vague Idea Bag wording.

First repair:
Tighten trigger wording.

Remaining leak:
The replacement still said find 2-3 alternatives, which was vague and random.

Deeper diagnosis:
The capture line needed a companion.

Companion:
No Random Moves / Calculation Before Candidate.

Clean repair:
Fit-slot retrieval and calculation gate.

Proof:
Future bag use must retrieve by named fit slot, not arbitrary count.

## Relation To Handy Idea Bag

The Handy Idea Bag helps find better patterns.

The Calculation Gate blocks random movement.

The Problem-To-Growth Gate tells the assistant what to do when a problem appears:

learn from it, find the missing friend/child/neighbor/proof, and prevent recurrence.

## Guardrails

- Do not turn every tiny issue into a parent boss.
- Do not overbuild when a symptom patch is enough.
- Do not ignore repeated failures.
- Do not keep patching the same leak without asking why it recurs.
- Do not save a rule just because it sounds right.
- Prove by cleaner movement.

## Verdict

This support gate makes problems productive.

Problems are not just failures.

Problems are growth signals.

The proof is in the pudding.
