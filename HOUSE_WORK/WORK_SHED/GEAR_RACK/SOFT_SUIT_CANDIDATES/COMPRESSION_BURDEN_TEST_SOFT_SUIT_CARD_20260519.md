# Compression Burden Test Soft Suit Card

Status: Soft Suit candidate
Date: 2026-05-19

Source:
Bitstring abyss first and second-pass source ore.

Core finding:
MDL / compression is useful, but compression alone is dangerous.

A smaller rule is not automatically a truer rule.

A compressed rule can hide evidence, nuance, timing, lane boundaries, or living structure.

Purpose:
Before installing a new rule, tool, card, gear, or concept, test whether it actually removes repeated mess without hiding something important.

Use when:
- a new rule wants to install,
- a new helper/tool seems attractive,
- a concept feels powerful but may become bloat,
- a dense idea is being compressed,
- or the assistant wants to turn source ore into operating gear.

Do not use when:
- doing emergency capture,
- saving raw source ore,
- making a receipt,
- or when no install/change is being considered.

Test fields:

## Candidate
What wants to be added?

## Repeated mess removed
What real recurring problem does this reduce?

## Evidence / nuance lost
What could this compression hide, flatten, or damage?

## New burden added
What new carrying cost does this create?

## Smaller object possible
Can this be a line, lens, tag, or parking note instead of a full rule/tool?

## Lane
Where does it belong right now?
- source ore
- parking
- Soft Suit
- Hard Suit
- BRAIN_LEARNING
- proof/history
- active guide
- rejected-with-reason

## Decision
Choose one:
- keep as candidate
- shrink
- park
- reject
- test
- promote later only after proof

Proof of usefulness:
This card works only if it changes a real decision.

Examples of success:
- stops a bloated install,
- shrinks a rule,
- parks a shiny idea,
- exposes missing evidence,
- or protects a living idea from being flattened.

Boundary:
Soft Suit candidate only.
No doctrine install.
No active guide rewrite.
No full bitstring system.
