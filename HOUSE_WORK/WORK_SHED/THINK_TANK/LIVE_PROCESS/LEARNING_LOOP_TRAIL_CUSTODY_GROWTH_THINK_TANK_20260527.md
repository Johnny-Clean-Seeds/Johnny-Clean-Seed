## Object

Learning-through-failure operating loop for compounding failure trails.

## Think Tank finding

The issue is not a single bad command.

The issue is repeated movement before custody:
- code before route resolution;
- patch before root cause;
- save before Soft Suit adoption;
- cleanup before residue classification;
- final success treated as full closure.

## Outside pattern fit

This rule agrees with mature learning systems:

- experiential learning: experience must be transformed through reflection and testing;
- PDSA: study is required before action becomes improvement;
- after action review: activity/project knowledge must be captured so performance improves;
- high reliability practice: weak signals and near misses matter;
- double-loop learning: repeated failure may require changing governing assumptions, not only tactics.

## House translation

Experience alone is not learning.

A receipt alone is not learning.

A rule file alone is not learning.

Learning means the system changes its next move in a provable way.

## Trail custody process

1. Freeze.
2. Name the active issue.
3. Name the full trail, not only the last symptom.
4. Sort trail evidence:
   - symptom;
   - assumption;
   - failed rule trigger;
   - repair;
   - new lesson;
   - residue.
5. Check existing rules.
6. Extract clean lessons.
7. Apply to Soft Suit.
8. Test against original failure.
9. Save one object.
10. Shed weight.
11. Resume only when route is accepted.

## Failure trail from this event

Representative trail:

- giant paste scripts became the delivery risk surface;
- the assistant repeated giant paste anyway;
- file custody was assumed before .ps1 existence was proved;
- PowerShell reserved variable $Args collapsed the Git wrapper route;
- receipts existed but were not staged before commit;
- ignored lanes needed git add -f --;
- root helper scripts accumulated;
- the assistant moved toward cleanup by code before resolving the process issue;
- final repairs passed, but the trail still required custody.

## Clean correction

The assistant must not use final success as an eraser.

Final success says:
the last repair worked.

Trail custody asks:
what did the whole path teach, and did that teaching change the next move?

## Save classification

Save as living process rule / candidate.

Do not promote to doctrine.

Do not create a new boss.

Do not rewrite ACTIVE_GUIDES or CURRENT_TRUTH_INDEX.

## Soft Suit pointer after save

Failure cannot pass through unchanged.

Trigger:
repeated failure, user correction, dirty trail, root residue, or "we got there but the path was a mess."

Return:
open full rule when a compounding failure trail needs autopsy.
