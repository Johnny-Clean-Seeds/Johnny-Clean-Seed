# Redundancy / Waste Gate Candidate

Date: 20260527

## Object

Redundancy / Waste Gate candidate.

## Reason

This candidate is created from live failure evidence, not theory alone.

The repair loop showed repeated waste patterns:

- long pasted scripts caused avoidable parser/delivery failures;
- saved ideas remained in active chat as heavy explanation instead of small triggers;
- same-family fixes appeared separately without merge/shed review;
- verification sometimes happened after user burden instead of inside the save path;
- route assumptions persisted after the better rule had already been named.

## Core operating question

What is still useful, and what is now redundant weight?

## Gate placement

Use after CONSOLIDATOR when overlap appears.

Use before Final Judge closes to ask:

- What should remain live?
- What should be pointer-only?
- What should be filed and dropped from suit?
- What should merge with existing saved material?
- What should be parked with return trigger?
- What needs proof before deletion/merge?

## Soft Suit relationship

If something is saved and proofed, the suit should carry only:

- short key;
- trigger;
- file/home pointer;
- return condition.

The full saved text belongs in the house, not in live chat.

## Risk controls

Do not use this as a delete machine.

Do not treat all fog as dirty fog.

Do not collapse source history that may be needed for proof, rollback, or learning.

Do not compress active uncertainty too early.

## Candidate trial standard

A successful trial is when this gate prevents one of these:

- duplicate save;
- overlong script;
- repeated explanation;
- unnecessary carried rule text;
- stale source copy;
- bloated proof;
- forgotten parking pile;
- old route assumption surviving after correction.

No promotion without live proof.
