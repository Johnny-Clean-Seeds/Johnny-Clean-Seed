# Growth Substrate Sorting Method Proposal

Status: SUPPORT EVIDENCE / PROPOSAL ONLY
Method: MOUNT -> TRACE -> EXPOSE -> REVISE
Parent Boss: Rule Activation / Work-Order Control

## Root sorting problem

EVIDENCE:
- The source packet names the target substrate as status/history material, proof summaries, idea carryover, source ore, receipts, TODO/support surfaces, mule returns, soft-suit notes, business-risk signals, natural triggers, and method harnesses.
- The MTER source names the root failure as Missing Execution Harness and Stored Method Not Fired Under Synthesis Pressure.
- The whole-house verdict says status/history is growth substrate, not the wheel, and that gold becomes weight when unplaced.

INFERENCE:
- The substrate problem is not lack of material. It is mixed material without a consistent point-of-work classification step.
- The dangerous blend is authority language, evidence, ideas, old next moves, proof receipts, method notes, and business signals sitting close enough that a rushed future pass may treat support as command.

Root sentence:

Mr.Kleen needs a small sorting harness that classifies each substrate item by source, authority, proof claim, parent boss, natural trigger, business risk, and placement state before it can influence active work.

## Method

Use this sequence when a growth-substrate item is encountered.

### 1. Mount the item

Capture the minimum facts before interpretation:

- Source path.
- Date or event time if present.
- Origin type: status, proof, receipt, TODO, mule, source ore, soft-suit, business signal, natural trigger, method harness, or unknown.
- Current authority claim: command, active guide, anchor, support evidence, history, parked idea, test evidence, or unknown.
- Parent Boss if visible.
- Blocked moves attached to the item.
- Whether the item contains old "next move" language.

Pass condition:
- The item is not summarized into doctrine before its authority and source class are named.

### 2. Trace the layers

Trace first. Boss second. Proof decides.

For each item, separate:

- Content: what the item says.
- Process: what method, if any, it asks to fire.
- Proof: which exact claim it proves.
- Business: what entry, trust, handoff, or product-story risk it affects.
- Command: whether it can direct current action.
- Boundary: what it blocks or must not be used for.

Pass condition:
- A proof note proves only its named claim.
- A status note does not become command by sounding current.
- A method note does not become king by being useful.

### 3. Group by Boss and Ghost

Use Boss/Ghost grouping after trace.

- Boss: the real parent problem or work lane.
- Ghost: the surface symptom that can distract from the parent.
- Child task: the smallest task the item suggests, if any.
- Proof need: what would have to be shown before adoption.
- Adoption state: active, support, parked, candidate, rejected, blocked, or unknown.

Pass condition:
- Items that share a symptom but not a parent boss are not merged too early.
- Items that share a parent boss can be compared without pretending they are the same kind of proof.

### 4. Assign a placement view

Use views, not a dashboard. These are sort destinations or report lenses, not command authority.

1. Event Ledger
   - What happened, when, where proof lives, and what claim the event proves.

2. Seed Bank
   - Reusable ideas extracted from history that are not active doctrine.

3. Natural Trigger Index
   - Conditions that should fire a method only when the real event appears.

4. Business Risk View
   - Signals that hurt trust, entry, handoff, product story, or public clarity.

5. Rejection Ledger
   - Moves, structures, or claims rejected in this scope and why.

6. Method Harness Index
   - Which method fires for which task type, with pass condition and No Rule King guard.

7. Authority Boundary Map
   - What can command, what can support, what can only inform, and what is blocked.

Pass condition:
- An item can sit in more than one view only if each view names a different claim.
- No view becomes live authority by existing.

### 5. Decide the placement state

Use these states:

- ACTIVE AUTHORITY: current user command, ACTIVE_ANCHOR when present, active guides within scope.
- SUPPORT EVIDENCE: proof, receipts, mule returns, verdicts, maps, TODOs, status history.
- SOURCE ORE: raw or partially digested material.
- SEED: reusable idea not adopted yet.
- NATURAL TRIGGER: a condition that can fire later.
- CANDIDATE AFTER PROOF: useful but not adopted now.
- REJECTED FOR THIS SCOPE: named and reasoned.
- BLOCKED: prohibited by active boundary.
- UNKNOWN: not classifiable without more evidence.

Pass condition:
- "Useful" is not the same as "active."
- "Repeated" is not the same as "proven."
- "Saved" is not the same as "fired."

## Proof rule

Claim-scoped proof:

- A receipt proves the action it records.
- A mule return proves a worker output exists.
- A TODO proves a support task exists, not that the task is current command.
- A status/history entry proves a historical state or note existed.
- A method note proves the method was named, not that it fired.
- An active guide controls only within its active role and current scope.
- ACTIVE_ANCHOR controls the active ball when present.

## Natural trigger rule

Natural triggers, not forced tests.

Examples:

- MTER fires when the user asks for a complex list, plan, sort, rank, root-cause pass, process explanation, critique, or revision.
- No Rule King fires when any rule, card, selector, TODO, map, tool, mule output, receipt, status note, or method tries to become master.
- Rule-Firing Confirmation fires before nontrivial house-touching action.
- Boss/Ghost grouping fires when a flat pile, repeated symptom, or route-selection conflict appears.
- Business Risk View fires when entry, trust, handoff, product story, or public clarity is affected.

## Adoption guard

This proposal should be adopted only by a later house-side intake decision.

Adoption should start with one small pilot:

- Choose one existing substrate item.
- Fill the minimal record.
- Check whether the record prevents authority drift and improves retrieval.
- Then decide whether to repeat.

Do not sort the whole history in one pass.
