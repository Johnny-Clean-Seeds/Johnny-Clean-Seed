## Input classes

### 1. Status/history material

Role:
- Growth substrate.

Contains:
- old states;
- room records;
- next-return anchors that may now be stale;
- parked risks;
- historical summaries;
- context for why something changed.

Risk:
- Old command-looking language can impersonate current command.

Sort by:
- date;
- active/stale status;
- source path;
- authority state;
- parent boss;
- whether it contains current-looking labels.

Placement:
- Event Ledger when it records what happened.
- Authority Boundary Map when it includes command-looking language.
- Seed Bank when it contains reusable ideas.
- Rejection Ledger if it proposes a move now blocked.

### 2. Proof summaries

Role:
- Claim-scoped evidence.

Contains:
- verdicts;
- pass/partial/fail notes;
- path/hash proof;
- repair evidence;
- intake/disposition notes.

Risk:
- Proof of one claim can be overread as proof of a system.

Sort by:
- exact claim proved;
- file path;
- hash;
- patch/state ID;
- scope limit;
- remaining unknowns.

Placement:
- Event Ledger.
- Method Harness Index if it proves a method fired.
- Business Risk View if it affects trust or handoff.

### 3. Idea carryover

Role:
- Seed material, not active doctrine.

Contains:
- candidate principles;
- design ideas;
- naming improvements;
- possible future tools;
- rejected or parked insight.

Risk:
- Good language can be mistaken for adopted law.

Sort by:
- seed claim;
- parent boss;
- evidence level;
- natural trigger;
- adoption gate.

Placement:
- Seed Bank.
- Candidate After Proof.
- Rejection Ledger if actively refused.

### 4. Source ore

Role:
- Raw material for later digestion.

Contains:
- unprocessed notes;
- fragments;
- source packets;
- raw observations;
- old work material.

Risk:
- Proximity to proof or active lanes can make ore feel installed.

Sort by:
- source;
- custody;
- relevance;
- risk;
- whether a child-gold sweep is needed.

Placement:
- Source Ore.
- Seed Bank after extraction.
- Event Ledger only for custody events, not content adoption.

### 5. Receipts

Role:
- Proof that a specific action happened.

Contains:
- backups;
- before/after hashes;
- patch receipts;
- staging inventories;
- install notes.

Risk:
- A receipt can prove work happened without proving the work should guide current action.

Sort by:
- action recorded;
- files touched;
- before/after hashes;
- patch/state ID;
- verdict;
- scope.

Placement:
- Event Ledger.
- Authority Boundary Map if it changes active authority.
- Business Risk View if it proves or reduces trust risk.

### 6. TODO/support surfaces

Role:
- Support routing, not command authority.

Contains:
- planned tasks;
- support queues;
- WIP status;
- blocked moves;
- historical recommendations.

Risk:
- Headings like "next" or "current" can become stale and compete with ACTIVE_ANCHOR or current user command.

Sort by:
- support-only status;
- source state;
- kind;
- authority owner;
- proof needed;
- command status.

Placement:
- Event Ledger when a TODO is opened, updated, or closed.
- Authority Boundary Map when wording may impersonate command.
- Natural Trigger Index when a TODO has a real firing condition.

### 7. Mule returns

Role:
- Worker/sharpener output and support evidence.

Contains:
- analysis;
- proposals;
- sorted lists;
- return packages;
- proof observations.

Risk:
- Mule output can feel like supervisor output if not bounded.

Sort by:
- task packet;
- source set;
- claims made;
- boundaries held;
- adoption state;
- intake/disposition status.

Placement:
- Event Ledger for the return event.
- Seed Bank for useful ideas.
- Rejection Ledger for refused options.
- Authority Boundary Map to keep mule support-only.

### 8. Soft-suit notes

Role:
- Working loadout or operational rhythm.

Contains:
- style notes;
- approach patterns;
- soft behavior guides;
- non-doctrine working posture.

Risk:
- Soft loadout can silently become command authority.

Sort by:
- trigger;
- scope;
- proof state;
- relation to active guides;
- whether it is support-only.

Placement:
- Method Harness Index if it affects a method.
- Seed Bank if it is a candidate.
- Authority Boundary Map if it risks authority drift.

### 9. Business-risk signals

Role:
- Trust, entry, handoff, product story, and public-clarity protection.

Contains:
- dirty-substrate warnings;
- fake-confidence warnings;
- entry-friction notes;
- clarity gaps;
- customer-facing risk.

Risk:
- Business risk can be treated as tone instead of a sorting facet.

Sort by:
- affected business surface;
- severity;
- evidence;
- trigger;
- smallest countermeasure.

Placement:
- Business Risk View.
- Rejection Ledger if a move creates unacceptable risk.
- Natural Trigger Index if the risk should fire a method.

### 10. Natural triggers

Role:
- Conditions that fire a method when the real event appears.

Contains:
- MTER triggers;
- Rule-Firing Confirmation triggers;
- No Rule King triggers;
- Boss/Ghost triggers;
- business risk triggers.

Risk:
- Forced tests can replace natural evidence.

Sort by:
- event condition;
- method fired;
- pass condition;
- blocked forced use.

Placement:
- Natural Trigger Index.
- Method Harness Index.

### 11. Method harnesses

Role:
- Point-of-work execution controls.

Contains:
- MTER;
- Rule-Firing Confirmation;
- Boss/Ghost grouping;
- No Rule King;
- claim-scoped proof;
- Trace first, Boss second, Proof decides.

Risk:
- A method can be stored without firing, or become king.

Sort by:
- trigger class;
- required inputs;
- visible output;
- proof that it fired;
- No Rule King guard.

Placement:
- Method Harness Index.
- Authority Boundary Map.
