## Best Fix

Selected:
Candidate 1 - Authority Baseline Stamp.

Shape:
Before any support item is used as a reason to act, stamp:
- item name;
- source;
- role: active, support, watch, parked, blocked, proof-only, candidate;
- authority: command, evidence, method, warning, reference, none;
- allowed actions;
- blocked actions;
- proof required to change status.

## Proof To Myself

Claim:
Authority Baseline Stamp is the best fit for support-surface authority pressure.

Evidence:
- NIST configuration management emphasizes baselines and controlled changes to trusted states.
- Local files already rely on role/status labels: active, support-only, parked, proof-history only, no promotion.
- The repeated house problem is not lack of ideas; it is role bleed.

Inference:
This fix converts "No Rule King" from a slogan into a quick baseline check. It directly prevents a support item from quietly becoming command authority.

Why not the others:
- DRI/RACI helps ownership but not item status.
- ADR works for durable decisions but is too heavy for every support item.
- PDSA blocks premature standardization but still needs an authority label.
- Toyota waste lens helps bloat but could be misread as cleanup permission.

Disproof test:
If stamping creates more drag than authority clarity, use it only for high-risk or disputed support items.

Verdict:
BEST / APPLY AS GUARD CANDIDATE, not install.
