## Core Finding

The first HNS V2 packet was structurally good but still thin on one root point: it named a nervous system, but it did not yet make the live "model" explicit enough.

Deep source synthesis says the HNS Mesh should not be just a set of sensors and alarms. It should carry a small live process model:

```text
ACTIVE BALL
-> HOUSE AUTHORITY STATE
-> CURRENT OBJECT ROLE
-> EXPECTED CLEAN STATE
-> OBSERVED SIGNAL
-> DIFFERENCE / SALIENCE
-> THRESHOLD
-> DEBUGGER
-> ROUTE
-> SAFE REFLEX OR STOP
-> PROOF / MEMORY / DECAY
-> CLOSE
```

This fits:

- Conant/Ashby good-regulator theorem: a regulator needs a model of the system it regulates.
- STAMP/STPA: unsafe actions often arise from flawed process models, missing feedback, delayed feedback, or mismatched control structure.
- NASA decision/risk guidance: rigor should scale with consequence, uncertainty, and stakeholder/mission context.
- Endsley situation awareness: perception, comprehension, and projection must stay tied together.
- OpenTelemetry/SRE: instrumentation and alert quality matter only when signals are named, correlated, thresholded, and actionable.
- Alarm fatigue evidence: too many false/nonactionable alarms creates desensitization and missed response.

## Deep Design Implications

### 1. HNS Needs A Model Spine, Not Just Nerves

The mesh must keep asking:

- What object is being regulated?
- What clean state is expected?
- What signal was observed?
- What difference matters?
- What is the current authority?
- What route is allowed?
- What closes the loop?

Without that spine, the mesh can become a long list of clever sensors.

### 2. Sensors Need Discrimination

Signal detection and SRE alerting both support a precision/recall shape:

- Hit: real issue caught.
- Miss: real issue not caught.
- False alarm: alarm fires without real issue.
- Correct quiet: no alarm when no issue exists.

HNS adaptation:

- Dead Nerve Detector tracks misses.
- Alarm Storm Detector tracks false alarms and nuisance alarms.
- Close condition tracks reset time.
- Threshold quality tracks whether the sensor improves work or just adds noise.

### 3. Alarm Storm Is A Safety Problem, Not A Style Problem

Alarm fatigue literature shows high alarm volume and nonactionable alarms can make users ignore alarms. For the house, overfiring alarms can make the assistant ignore the real root or freeze into ceremony.

Clean repair:

- Use a salience gate.
- Track alarm reset.
- Park weak signals.
- Keep one active ball.
- Require an alarm to route to a debugger and close condition.

### 4. Dead Nerve Is Also A Safety Problem

STAMP and homeostasis both support the idea that missing or delayed feedback can be hazardous. The house version is: a known rule should have fired but did not.

Clean repair:

- Capture the failure class.
- Identify the expected signal.
- Decide whether the sensor was absent, numb, threshold too high, routed wrong, or blocked by fog.
- Add a tiny test before any doctrine install.

### 5. No Alarm King Is A Governance Law

The source fields repeatedly warn against single-point command:

- Requisite variety says one regulator cannot cover all disturbances.
- STAMP says feedback/control structure matters across levels.
- Global Workspace says local information should not always broadcast globally.
- Alarm fatigue says every signal cannot be treated as critical.

Clean HNS adaptation:

- No one sensor family controls the house.
- No alarm can rewrite authority.
- No source concept becomes command.
- The model spine stays smaller than the house.

### 6. Mule Workshop Needs Its Own Support Rules

Workshop rules can coexist with house rules if they are lower authority, scoped, and explicitly support custody/focus:

- They guide the mule's workbench only.
- They do not install doctrine.
- They do not alter active guides or truth index.
- They sharpen output hygiene, source containment, and focus.

### 7. Focus Must Be Treated As A Nerve Family

User request explicitly asks for ways to keep Codex on task and focused.

Deep source synthesis supports a Focus Nerve:

- Situation awareness: perception, comprehension, projection.
- Good regulator: keep the model explicit.
- NASA decision analysis: define decision need and criteria before evaluating many options.
- Alarm fatigue: too many signals destroys response quality.

Focus Nerve candidate:

```text
ACTIVE BALL -> MODEL SPINE -> CURRENT FILE ROLE -> SOURCE STATUS -> NEXT OUTPUT -> STOP CONDITIONS
```
