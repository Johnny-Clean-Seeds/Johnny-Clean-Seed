# HNS Core Model And Focus Protocol

Date: 2026-05-21
Lane: Mule return / focus support
Status: Candidate focus protocol only
Boundary: Lower than house rules. Does not command Mr.Kleen.

## Purpose

The HNS Mesh can grow too wide if the worker tries to hold every source, alarm, lane, and idea at once.

This protocol keeps Codex focused on the model and the active task while still allowing deep search.

## Core Model Spine

Use this spine before and during HNS work:

```text
1. ACTIVE BALL
2. AUTHORITY STATE
3. CURRENT OBJECT
4. OBJECT ROLE
5. EXPECTED CLEAN STATE
6. OBSERVED SIGNAL
7. MEANINGFUL DIFFERENCE
8. THRESHOLD / SALIENCE
9. DEBUGGER
10. ROUTE
11. SAFE REFLEX / STOP
12. PROOF / MEMORY / DECAY
13. CLOSE CONDITION
```

If any part is unknown, mark UNKNOWN instead of inventing.

## Focus Nerve

### Signal

The work starts widening, source volume grows, or many possible links begin competing.

### Sensor

Focus Nerve sensor checks:

- Am I still inside the approved write lane?
- What is the active ball?
- Which file or model piece am I working on now?
- Is this source authority, support, source ore, proof, or candidate?
- Is this new source adding a non-repetitive claim?
- Is there a close condition?

### Threshold

Fire WARNING when:

- More than one active ball is steering.
- Source collection continues after the claim families are saturated.
- New notes repeat an existing claim without adding a new boundary, risk, or source support.
- The worker starts making doctrine/rule/install language.
- The current output file role is unclear.

Fire BLOCK when:

- ACTIVE_GUIDES or CURRENT_TRUTH_INDEX changes are about to happen without explicit approval.
- Runtime/dashboard/automation/permanent feed build begins.
- Mule output starts commanding Mr.Kleen.
- Source ore starts steering without source-to-claim mapping.

### Debugger

Ask:

- What exact claim am I improving?
- What existing file already covers it?
- What source supports it?
- What does the source not prove?
- What is the smallest useful output?
- What should be parked?

### Route

- If claim is new -> source-to-claim map.
- If claim is repeated -> no new prose unless boundary/risk/source improves.
- If source volume is high -> search index and synthesis, not more candidate doctrine.
- If focus is slipping -> return to Model Spine.

### Safe Reflex

Write the next output file in the return lane only.

Do not chase whole-project material.

Do not install.

Do not convert source search into live rule changes.

### Close Condition

Focus Nerve goes quiet when:

- Active ball is named.
- Current output file role is named.
- Source status is classified.
- Next file/output is clear.
- Blocked moves are not crossed.

## Model-Keeping Card

Keep this in front of the work:

```text
HNS IS:
Routed sensing, debugging, response, proof, memory, and decay.

HNS IS NOT:
Authority, dashboard, runtime, automation, permanent feed, all-to-all graph, or every-task ritual.

CORE MOVE:
Signal -> Sensor -> Salience -> Threshold -> Alarm -> Debugger -> Route -> Reflex/Stop -> Proof/Memory/Decay -> Close.

MODEL RULE:
The worker must know active ball, authority, object role, expected state, observed signal, and close condition.

NO ALARM KING:
Any alarm that tries to command, install, or rule every task is dirty milk.
```

## Deep Search Control

Deep search is allowed only if it feeds one of these bins:

- Better source support.
- New non-repetitive risk.
- Stronger boundary.
- Clearer model.
- Better tiny test.
- Better workshop custody.
- Better focus mechanism.

Deep search should stop when new sources only repeat the same support without changing map, boundary, risk, or test shape.

## Attention Budget

Use three live slots:

```text
SLOT 1: ACTIVE BALL
SLOT 2: CURRENT FILE / OUTPUT ROLE
SLOT 3: CURRENT CLAIM FAMILY
```

Everything else goes to:

- source index;
- parked issue ledger;
- future candidate;
- rejection ledger;
- no-action-needed.

## Anti-Drift Questions

Ask before every new output file:

- Does this file have a role not already covered?
- Is the role in the return lane?
- Does it avoid doctrine/install language?
- Does it preserve original packet hashes by adding, not rewriting?
- Does it name EVIDENCE / INFERENCE / UNKNOWN?
- Does it help Mr.Kleen without commanding him?

## Tiny Model Test

Before using HNS on a real future event:

```text
Can I fill the Model Spine in 90 seconds?

If no:
  do not act from the alarm yet.
  collect missing model facts or mark UNKNOWN.

If yes:
  run one tiny nerve test.
  keep the result support-only until house disposition.
```
