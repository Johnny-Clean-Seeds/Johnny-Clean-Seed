# Work Shed Operating Index

Status: ACTIVE SUPPORT INDEX
Authority: support only; not command authority

## Purpose

This index tells the assistant how to use the Work Shed during future work.

The Work Shed is the assistant's controlled messy workroom.

It receives rough ideas, sorts them, turns useful items into cards, keeps hot helpers near the front, parks inactive methods, preserves old ways, and keeps retrieval signs.

The Work Shed cannot command.

The Work Shed cannot promote.

The Work Shed cannot become doctrine.

## When To Use The Work Shed

Use the Work Shed when:

- many ideas arrive at once
- the user gives a rant, pile, critique, or messy task list
- the assistant sees useful side ideas during another task
- a method looks useful but is not ready to install
- an old method may be useful later
- a new helper needs testing before adoption
- a task needs sorting before decision
- the assistant needs to preserve ideas without polluting active rooms

Do not use the Work Shed when:

- the task is simple and already selected
- the answer is only a status confirmation
- the material belongs directly in PROOF_HISTORY as proof
- the material belongs directly in SOURCE_ORE as raw source capture
- the material is already active authority
- using the Work Shed would add fog instead of clarity

## Drawer Use

### INBOX

Use for:
raw idea drops, rough notes, messy loads, unsorted helper ideas.

Rule:
Capture fast. Do not judge too early.

Exit:
Move to SORTING_BENCH when the pile needs grouping.

### SORTING_BENCH

Use for:
grouping, comparing, splitting, ranking, and deciding where items belong.

Rule:
Do not let the bench become permanent storage.

Exit:
Send clean single ideas to IDEA_CARDS.
Send current helpers to FRONT_SHELF.
Send useful inactive methods to PARKED_METHODS.
Send replaced methods to OLD_WAYS.
Send maps/tags to INDEXES.

### IDEA_CARDS

Use for:
one cleaned idea per card.

Rule:
Each card needs name, job, lane, source, status, lifecycle state, neighbor risk, proof need, and next action.

Exit:
A card can point to FRONT_SHELF, PARKED_METHODS, OLD_WAYS, SOURCE_ORE, BRAIN_LEARNING, TODO, or another room only after its exit gate is clear.

### FRONT_SHELF

Use for:
current, hot, frequently used, or frequently changed helpers.

Rule:
Front shelf means fast retrieval, not authority.

Exit:
Move to PARKED_METHODS when no longer current.
Move to OLD_WAYS when superseded.
Move to BRAIN_LEARNING only through proof if it becomes a behavior rule.

### PARKED_METHODS

Use for:
good methods that do not fit now.

Rule:
Parked does not mean rejected forever.

Exit:
Re-test when a future task fits.

### OLD_WAYS

Use for:
superseded or old methods preserved for future fit.

Rule:
Old ways are not trash.

Exit:
Return to PARKED_METHODS or FRONT_SHELF only if current task conditions make them useful again.

### INDEXES

Use for:
maps, tags, drawer lists, retrieval routes, cabinet signs, front-shelf lists, and Work Shed status.

Rule:
Indexes explain. They do not command.

## Work Shed Operating Flow

1. Capture messy material in INBOX.
2. Move/summarize it to SORTING_BENCH when sorting is needed.
3. Group by job, lane, source, or method family.
4. Compare only the relevant grouped candidates.
5. Make IDEA_CARDS for cleaned single ideas.
6. Put hot/current helpers on FRONT_SHELF.
7. Park useful inactive helpers in PARKED_METHODS.
8. Preserve superseded methods in OLD_WAYS.
9. Update INDEXES when retrieval would otherwise get foggy.
10. Only let material exit after the exit gate is satisfied.

## Exit Gate

Before anything leaves the Work Shed, define:

- name
- job
- source
- lane
- status
- lifecycle state
- authority boundary
- neighbor risk
- proof need
- next action
- target room

If any field is unclear, keep the item in the Work Shed.

## Allowed Exit Targets

BRAIN_LEARNING:
only for assistant behavior rules after proof.

SOURCE_ORE:
for raw/candidate material that should remain source-only.

HOUSE_WORK/TODO:
for trace-gated candidate work, not active boss by itself.

HOUSE_WORK/INDEXES:
for support maps and retrieval aids.

PROOF_HISTORY:
for receipts and proof, not loose ideas.

ACTIVE_ANCHOR:
only through explicit anchor refresh.

ACTIVE_GUIDES:
only through separately authorized guide work.

## Blocked Exits

Do not send Work Shed material directly into:

- doctrine
- promotion
- active authority
- runtime work
- guide rewrite
- cleanup/move/delete
- status-command work

Do not use Work Shed material as proof.

Do not treat a front-shelf helper as command authority.
