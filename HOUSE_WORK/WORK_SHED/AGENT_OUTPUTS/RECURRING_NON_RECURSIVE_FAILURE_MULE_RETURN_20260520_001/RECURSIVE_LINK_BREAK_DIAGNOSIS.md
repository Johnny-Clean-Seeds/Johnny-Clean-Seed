# RECURSIVE LINK BREAK DIAGNOSIS

## The Chain

Target recursive chain:

`failure -> diagnosis -> rule/candidate -> TODO/placement -> proof -> next-start improvement`

Observed live chain:

`failure -> artifact correction -> validation -> memory continuity -> close`

Missing links:

`rule/candidate -> TODO/placement -> next-start improvement`

## Where The Chain Breaks

INFERENCE:

The chain breaks after the immediate fix and before closure.

More precisely:

`proof of current artifact` is treated as enough, while `carryover disposition` is skipped.

The assistant did not ask:

- Did this correction reveal a reusable rule?
- Is the rule one-off or recurring?
- Does it need TODO, Sorting Bench, BRAIN_LEARNING, Work Shed, receipt, or parking?
- Is memory-only capture enough?
- What makes the next artifact task start cleaner?

## Why Existing Rules Did Not Fire

EVIDENCE:

Existing rules already say:

- corrections can become candidate rules or TODO support;
- rule storage is not rule firing;
- TODO items need trace/triage;
- root keys must alter behavior at the point of work;
- cycle flow should connect work, idea capture, TODO, proof, and learning.

INFERENCE:

They did not fire because their strongest triggers are before action or during classification. The failure appeared after a successful repair, when the assistant was psychologically and procedurally ready to close. There is no explicit "after-fix, before-close" hold point that forces carryover disposition.

## What Would Have Caught The Exact Failure

Small latch:

Before final response after any meaningful correction, failed proof, user "always do this" instruction, or reusable rule exposure:

1. Name immediate fix status.
2. Ask whether the correction is one-off or reusable.
3. If reusable, choose placement:
   - TODO,
   - Sorting Bench,
   - BRAIN_LEARNING,
   - Idea Bag,
   - proof/receipt,
   - park,
   - block,
   - memory-only by explicit user request.
4. If no repo edit is authorized, offer the smallest Mr.Kleen save candidate instead of silently relying on memory.
5. Do not call the task fully closed until carryover is placed, parked, blocked, declared one-off, or offered.

## Why The Parent Boss Is Sharper Than Rule Activation

INFERENCE:

Rule Activation / Work-Order Control asks:
Did the right rule fire before or during action?

Recursive Carryover / Point-of-Work Link Failure asks:
After action exposed a reusable lesson, did the lesson alter the next start?

The current case did not mainly fail because the assistant never understood the rule. It failed because understanding and fixing did not become house placement.

## Memory Boundary

EVIDENCE:

The active packet says the rule was captured in memory continuity only, not Mr.Kleen TODO or house lane.

INFERENCE:

Memory can help the current assistant's continuity, but it is not house capture. It has no file path, no TODO status, no proof receipt, no hash, no lane owner, no audit trail, and no guaranteed re-entry behavior.

Proposed wording:

`Memory is not house capture unless the user explicitly asks for memory-only handling.`

## Motto Wheel Support

INFERENCE FROM SUPPORTING PACKET:

The Core Spin Carryover Gate is the right conceptual support:

- Proof answers whether the current thing worked.
- Carryover answers whether the next start is cleaner.

But motto alone is not enough. The motto needs a latch with fields and proof.

Best placement:

- Keep motto language as support/root key wording.
- Save operational behavior as `After-Fix Carryover Latch`.

## Overbuild Block

INFERENCE:

The evidence does not justify a dashboard, automation layer, runtime, knowledge graph, full lessons database, or active-guide rewrite. The house already has too much passive material risk. The fix should reduce the number of unlatched transitions, not add a new global system.
