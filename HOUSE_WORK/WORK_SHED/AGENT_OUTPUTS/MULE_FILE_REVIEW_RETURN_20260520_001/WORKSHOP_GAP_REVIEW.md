## Workshop Questions Answered

### Is the workshop safe enough to use now?

INFERENCE:

Yes, for one bounded mule pass with the current packet and hard bans. It should not become a standing mule cycle.

### Is the workshop better than chat-only mule handoff?

INFERENCE:

Yes. File-first active packet plus return folder improves placement, proof, and recovery. It is safer than a raw chat dump because the output contract is file-bound.

### Is the workshop better than a full dashboard or knowledge graph?

INFERENCE:

Yes for now. The current need is bounded retrieval and critique, not global relationship modeling. A dashboard/graph would add update burden and stale-pointer risk before proving enough benefit.

### Is the access map enough?

INFERENCE:

Enough for a careful mule. Not enough to call frictionless. It should gain exact packet-specific pointers only if future mules repeat the same search burden.

### Should any tool be promoted?

INFERENCE:

No. Existing text/rule/search flow is sufficient for this pass. Tool-building is not proven better yet.

## Minimal Workshop Improvement Candidate

Candidate:

`MULE_START_CARD` or equivalent packet-top pointer block.

Contents:

- Active packet path.
- Return folder path.
- Current branch/HEAD expectation.
- Current authority files to read first.
- Exact task-scoped rule paths.
- Exact mule role / anti-agreement / better-fit rule paths.
- Recent proof receipts relevant to the packet.
- Hard bans.
- Finish condition.

Constraints:

- Non-authoritative.
- Packet-scoped.
- No standing dashboard.
- No automatic promotion.
- Updated only when a new active packet is placed or when intake explicitly chooses to refresh it.

Proof needed:

- Future mule retrieves correct rule/proof files faster.
- Future mule does not mistake the card for authority.
- Future mule returns required files without missed output rules.
