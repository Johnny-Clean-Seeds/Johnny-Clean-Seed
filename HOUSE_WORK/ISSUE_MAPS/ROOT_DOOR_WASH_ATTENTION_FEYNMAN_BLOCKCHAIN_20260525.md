## My 2 Cents

The house problem this set points at is not memory shortage or idea shortage. It is state ambiguity.

The root-door pain is the simplest proof object: a file can be "there" without being in custody, "seen" without being read, "read" without being synthesized, "synthesized" without being receipted, and "saved" without being replayable. That is exactly the failure line these three documents are circling.

My spin: the strongest use is to stop treating intake as a passive folder event. Intake should be a state machine.

Proposed support-only state ladder:

1. PRESENT: file is at the door.
2. CUSTODY: file is moved to the right source lane with hash proof.
3. READ: file has metrics and a clean-center summary.
4. DECODED: file has evidence/inference separation and source-role limits.
5. SYNTHESIZED: file has been aligned with current house issues and other documents.
6. DECIDED: the best bounded use has been chosen.
7. RECEIPTED: action, proof, limits, and dirty-tree caveats are written.
8. SAVED: scoped Git save records the support material.
9. REPLAYABLE: a future assistant can recover the source, decision, and limits without guessing.

The key repair is the transition word. Nothing should silently become read, clean, doctrine, final, installed, proof, or done.
