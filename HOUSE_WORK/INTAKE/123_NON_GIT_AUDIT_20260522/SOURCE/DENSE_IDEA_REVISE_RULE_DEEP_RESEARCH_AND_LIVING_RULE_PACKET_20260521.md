# 9. The new rule as a living behavior

Name:

**Dense Idea Intake + Comparative Revise Rule**

Purpose:

When the user drops a dense message containing many ideas, fixes, problems, rules, or concepts, the assistant must not treat it as ordinary chat. It must fire a multi-rule intake path.

Trigger words and conditions:

```text
large idea cluster
many concepts in one message
user says refine, revise, deep research, cover all bases, do not hold back
user says capture this, make it a rule, make it work
user points out a failure where a known rule did not fire
user lists candidate abilities, tools, organs, or system changes
research triggers new house ideas
mule return triggers new comparisons
```

Required route:

```text
1. MOUNT the task and boundary.
2. DISSECT the cluster.
3. CAPTURE every distinct candidate.
4. LABEL state: active, partial, candidate, needs-test, save/lock, park, reject, block.
5. APPLY adopt/adapt if a clear proved fix is present.
6. APPLY revise rule when refinement/research is requested.
7. SEARCH outside sources if current information or broader research is needed.
8. MAP sources to house uses, not authority.
9. FORM abilities from comparisons.
10. RUN second missed-idea pass.
11. BROADEN rules with all-bases/guideword pass.
12. BUILD file/packet if requested.
13. EXTRACT living rule if the process itself should persist.
14. PREPARE mule handoff if useful.
15. DO NOT install doctrine/runtime/dashboard/automation unless explicitly selected and proved.
```

Failure states:

```text
DEAD NERVE:
Dense cluster arrives but assistant answers casually.

LATE FIRE:
User has to ask why ideas were not captured.

MISFIRE:
Assistant researches but does not capture state/disposition.

OVERFIRE:
Assistant tries to install everything immediately.

FALSE FIRE:
Assistant invents rule save without proof or repo cleanliness.

CHAT-ONLY:
Assistant produces a list in chat when user asked for file.

BLOCKED CORRECTLY:
Repo is dirty/current mule return unjudged, so save waits.
```

Pass condition:

```text
All major ideas are captured, sorted, state-labeled, connected to proof needs, and delivered in the requested artifact shape.
```

# 10. Rule text draft for later BRAIN_LEARNING save

```text
# Dense Idea Intake And Comparative Revise Rule

When the user sends a dense cluster of ideas, fixes, problems, concepts, research directions, or rule/tool failures, the assistant must automatically fire the dense-idea intake route.

The assistant shall dissect the cluster, capture every distinct candidate, label each item by state, map each item to lane/parent/neighbor/proof need/save-lock condition, and decide adopt/adapt/park/reject/block.

When the user requests refinement, deep research, broader search, or "cover all bases," the assistant shall apply the revise rule: preserve the old list, research outward, add new material, integrate old + new, run a second missed-idea pass, document what was done/why/how it improved the output, and produce the requested file/packet.

When the cluster exposes a clear proved fix, the assistant shall apply the adopt/adapt rule and mark the fix for save/lock if durable/high-impact, unless authority/proof/boundary blocks it.

A dense cluster shall not be treated as ordinary chat when it clearly contains house-relevant candidate rules, tools, abilities, organs, or failure repairs.

If this route fails to fire and the user has to ask why the ideas were not captured, treat that as a rule-firing failure and repair the route.
```
