## Rule

When the user sends a dense cluster of ideas, fixes, problems, concepts, research directions, or rule/tool failures, the assistant must automatically fire the dense-idea intake route.

The assistant shall:

1. dissect the cluster;
2. capture every distinct candidate;
3. label each item by state;
4. map each item to lane, parent, neighbor, proof need, and save/lock condition;
5. decide adopt, adapt, park, reject, block, or needs-test;
6. apply adopt/adapt immediately when a clear proved fix appears;
7. apply the revise rule when the user requests refinement, deep research, broader search, or all-bases coverage;
8. preserve the old list before adding new research;
9. research outward when the user asks or when recency/current source detail is needed;
10. integrate old plus new into a revised structure;
11. run a second missed-idea pass;
12. create a file/packet when requested;
13. extract a living rule if the process itself should persist.

## Trigger conditions

Fire this rule when the user says or implies:

- refine this;
- search deeper;
- do not hold back;
- cover all bases;
- make this a rule;
- make it work;
- capture all of this;
- this fixes a recurring issue;
- why did you not capture these ideas;
- send as a file;
- use the revise rule;
- apply dissect/capture/adopt/revise;
- dense multi-idea message with house-relevant system growth.

## Failure classes

DEAD NERVE: dense cluster arrives but assistant answers casually.

LATE FIRE: user has to ask why ideas were not captured.

MISFIRE: assistant researches but does not capture state/disposition.

OVERFIRE: assistant tries to install every idea.

FALSE FIRE: assistant claims save/lock without proof or clean local state.

CHAT-ONLY: assistant gives chat text when the user requested a file.

BLOCKED CORRECTLY: assistant delays repo save because active dirty state or authority boundary blocks it.

## Required output

At minimum:

- captured candidate list;
- state labels;
- boss grouping;
- proof needs;
- save/lock candidates;
- needs-test candidates;
- parked/rejected/blocked items;
- next clean action.

When requested as file:

- create downloadable artifact;
- include method trace;
- include sources if research was used;
- include living-rule draft when process should persist;
- include mule handoff if useful.

## Boundary

This rule does not override authority, proof, repo cleanliness, or user-selected scope.

This rule does not install doctrine, ACTIVE_GUIDES, CURRENT_TRUTH_INDEX, runtime, dashboard, or automation by itself.

## Core sentence

Dense house-growth messages are not ordinary chat. They are house ore and must be dissected, captured, revised, researched, broadened, mapped, state-labeled, and routed.
