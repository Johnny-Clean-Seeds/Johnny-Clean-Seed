# Broad Command Recursive Trigger List

Status: SMALL RECURSIVE SUBSYSTEM / BROAD-COMMAND TRIGGER  
Purpose: define what happens when the user gives a broad command such as "go do this," "make the list," "gather ideas," "handle this," or similar.  
Boundary: this is not the full deep-debug gate. It is the smaller recursive method for ordinary list-building, idea-gathering, source extraction, candidate sorting, and note comparison.

## 1. Trigger

When the user says a broad command like:

- go do this
- make the list
- gather ideas
- handle this
- sort this
- pull what fits
- build the candidates
- find what we missed
- clean this up
- use the notes
- bring in what fits
- drop what is bad
- make it recursive
- walk it through
- figure out what matters

the assistant should not freestyle.

The assistant should break out the correct recursive subsystem.

## 2. Core Shape

For normal list, idea, source, note, or candidate work, use:

`anchor -> gather -> branch -> fit-test -> clean / rename / drop -> order -> output -> self-proof`

This is not a lesser method.

It is the smaller complete branch of the larger recursive system.

The job decides the branch.

The shoe-fits gate decides what comes in, what gets renamed, what gets linked, what gets parked, what gets dropped, and what must be proven first.

## 3. Step Definitions

### 3.1 Anchor

Define the actual task and current state.

Ask:

- What is the user really asking for?
- What is the current start point?
- What is the intended output?
- What is blocked?
- What would count as useful?

### 3.2 Gather

Pull relevant material before building the list.

Possible sources:

- current notes
- files
- source material
- prior outputs
- user corrections
- links
- receipts
- parked ideas
- mule returns
- support packets
- known failure patterns

Do not act like the task starts from nothing when useful material already exists.

### 3.3 Branch

Sort gathered material into clean groups.

Useful branch types:

- ideas
- rules
- tools
- names
- risks
- proof gaps
- bad fits
- parking items
- next actions
- missing links
- dependency items

### 3.4 Fit-Test

Use the shoe-fit gate.

Question:

`Does this fit the task, the house, the branch, the proof need, and the current moment?`

Verdicts:

- WEAR IT: good idea fits and should be applied.
- RENAME IT: function is good, but the name is dirty, stale, vague, or authority-heavy.
- LINK IT: useful only as support to another object.
- PARK IT: useful later, not for this branch. Needs a return trigger.
- PROVE FIRST: promising, but not ready to use.
- DROP IT: bad idea, stale, unsafe, misleading, duplicate, or wrong scale.
- BLOCK IT: dangerous move or authority-risk move.

### 3.5 Clean / Rename / Drop

Remove or repair dirty material.

Clean-up targets:

- bad framing
- stale names
- authority-heavy wording
- duplicate junk
- vague labels
- unsafe moves
- fake PASS language
- ideas that sound good but do not fit

Good function can stay if the dirty name is replaced.

### 3.6 Order

Build the dependency ladder.

Put first things first.

Do not let loud items outrank root items.

Ordering questions:

- What must be true before this can be used?
- What item poisons the others if wrong?
- What is blocked until proof exists?
- What can be safely done now?
- What belongs later?

### 3.7 Output

Produce a usable list, packet, map, or next-action plan.

Good output includes:

- mission
- source basis
- grouped list
- fit decisions
- ordered next actions
- proof/status
- blocked items
- parked items with return triggers

Bad output includes:

- loose thoughts
- unowned ideas
- rule-sounding names with no proof
- unlinked anomalies
- fake completion
- giant checklist when a smaller complete branch can carry the job

### 3.8 Self-Proof

Check whether the list actually serves the task.

State:

- what fits
- what was renamed
- what was dropped
- what was parked
- what is blocked
- what proof is still needed
- what the next clean move is

## 4. Escalation Rule

Use the larger Deep Fit Debug Mode when the task shows systemic pressure.

Escalation triggers:

- repeated failure
- unclear pressure
- authority boundary risk
- proof conflict
- stale-state confusion
- house-wide relation problem
- deep research request
- deep refinement request
- debugging request
- major pressure-map work

Escalation is not punishment.

Escalation is the next branch wearing the shoe when the smaller branch reveals deeper pressure.

## 5. Growth Rule

Do not block growth by saying a method is "too heavy" as if depth is bad.

Correct framing:

Use the smallest complete recursive branch that can carry the job.

If the branch finds deeper pressure, escalate cleanly.

If the deeper branch proves useful, bring it in.

If the idea fits, wear it.

If the idea is dirty but useful, rename it and wear the clean function.

If the idea is bad, drop it.

If the idea is not ready, park it with a return trigger.

## 6. Router Tie-In

This subsystem follows the chat-cockpit router pattern:

1. classify the task;
2. route to the right house-rule family;
3. apply lane, boundary, delivery surface, tool/resource check, proof requirement, and Final Judge;
4. do not turn the router into doctrine.

## 7. Final Judge

A broad-command list pass is clean only if it proves:

- the task was anchored;
- relevant sources were gathered;
- ideas were branched;
- fit decisions were made;
- dirty names or bad frames were cleaned;
- the list was ordered by dependency;
- the output is usable;
- proof/status is clear;
- next move is named.

Verdict language:

- PASS: list is complete for the named target.
- PASS WITH GUARDRAILS: list is usable but not authority or not fully proven.
- PARTIAL: useful list exists but proof/source review remains.
- PARKED: useful item held with return trigger.
- REJECTED: bad fit.
- BLOCKED: unsafe or unproven move.

## 8. Short Form

Broad command comes in.

Use:

`anchor -> gather -> branch -> fit-test -> clean / rename / drop -> order -> output -> self-proof`

Then decide:

- stay in small list-building branch;
- link to another subsystem;
- escalate to Deep Fit Debug Mode;
- park with return trigger;
- block unsafe move.

The system is one recursive house with branching recursive subsystems.

The branch is chosen by fit, pressure, proof need, and growth path.
